## 一、文档定位

本文是模板库系统中的模板机制统一主规范。

本文承接 `A-TMP-M-T-02_Binder模板库技术主控文档.md` 的技术总控裁定，负责冻结机制基础层。

本文只回答：

1. 工作流模板对象怎么定义。
2. 工作流模板如何从用户表达进入结构化流程表示。
3. 工作流模板如何被 Agent 使用。
4. 工作流模板执行中的硬约束、可观测性与失败降级如何成立。

本文不直接回答：

1. 模板库为什么只保留工作流模板。
2. 模板库与知识库、workspace、Chat Build 的产品边界。
3. UI 编辑器与状态机实现。

这些问题分别由 `A-TMP-M-D-01_Binder Agent模板协同主控文档.md` 与 `A-TMP-M-D-02_Binder模板库模块描述文档.md` 负责。

## 二、MVP 目标

1. 建立统一工作流模板对象模型。
2. 建立“用户表达 -> 结构解析 -> 流程编译 -> Agent 执行”的统一链路。
3. 建立统一执行约束、失败降级与状态可观测规则。

## 三、工作流模板定义与边界

### 3.1 定义

工作流模板是模板库中的唯一对象类型。

它不是自由文本提示，也不是节点图本身。

它应被理解为：

**以用户友好文档形式表达、并可被编译为结构化流程表示的过程协议资产。**

### 3.2 边界规则

工作流模板只解决：

1. 任务怎么推进。
2. 阶段如何组织。
3. 步骤之间如何衔接。
4. 哪些输入、输出、约束与校验点必须成立。

工作流模板不解决：

1. 输出结构定义。
2. 输出样式定义。
3. 风格偏好定义。
4. 行为方法定义。
5. prompt 管理。

### 3.3 最小语义锚点

工作流模板至少需要稳定表达：

1. `phase`
2. `step`
3. `input`
4. `output`
5. `constraint`

如果这些语义锚点无法被稳定识别，则该对象不构成可执行的工作流模板。

## 四、统一对象模型

### 4.1 WorkflowTemplate（最小字段）

1. `template_id`
2. `name`
3. `description`
4. `owner_scope`（`workspace | user`）
5. `version`
6. `status`（`active | archived`）
7. `goal`
8. `phases`
9. `steps`
10. `created_at`
11. `updated_at`

### 4.2 payload 结构约束

1. `goal`：模板希望完成的过程目标。
2. `phases`：阶段定义与阶段顺序。
3. `steps`：步骤数组。
4. `steps[].name`：步骤名称。
5. `steps[].input`：步骤输入约束。
6. `steps[].output`：步骤输出约束。
7. `steps[].constraint`：步骤执行约束。
8. `steps[].depends_on`：前置依赖。

### 4.3 来源约束

当前阶段工作流模板来源只允许：

1. 用户创建。

因此不再存在：

1. 系统内置模板来源。
2. 执行过程自动生成模板来源。
3. 自动入库链路。

## 五、结构解析与流程编译

### 5.1 编译层位置

在执行前，工作流模板需要从用户表现层表达映射为结构化流程表示。

这一过程属于：

1. 结构解析。
2. 流程编译。

### 5.2 编译层的必要性

必须存在编译层，原因如下：

1. 避免系统退化为对原始文档文本的 prompt 理解。
2. 让模板执行建立在稳定结构而不是松散语义之上。
3. 为结构校验、顺序控制、失败定位和状态可观测提供前提。

### 5.3 编译结果的使用边界

Agent 在执行阶段使用的不是原始文档表达，而是：

**编译后的结构化流程表示。**

原始模板表达用于：

1. 用户阅读。
2. 用户编辑。
3. 用户保存。

编译结果用于：

1. 执行。
2. 校验。
3. 状态推进。
4. 失败恢复。

### 5.4 编译失败的处理原则

若工作流模板无法被稳定解析或编译，则不得直接把原始文档表达当作执行依据。

编译失败时：

1. 模板执行链中断。
2. 主任务可回退到无模板执行。
3. 系统应暴露结构错误或编译失败状态。

## 六、模板调用机制

### 6.1 入口

1. 用户显式引用（`@` 模板 / 选择模板）。
2. AI 可建议已存在模板，但不得绕过用户确认直接启用高影响模板。

### 6.2 注入策略

1. 模板注入属于上下文注入系统的高优先级来源之一。
2. 模板在注入前先完成结构解析与流程编译。
3. 编译通过后才进入执行阶段。
4. 注入或编译失败时不阻塞主任务，回退到无模板执行。

### 6.3 执行协同

1. 工作流模板进入任务规划与执行层。
2. 模板只作为过程约束，不作为事实来源。
3. 模板执行完全内嵌在 Agent 主链中，不形成独立执行引擎。

---

## 七、统一执行约束

### 7.1 执行主控

执行主控始终是 Agent。

工作流模板本身不执行。

### 7.2 硬约束

执行阶段至少遵守以下约束：

1. 每一步必须有清晰边界。
2. 每一步必须对应明确输入与预期输出。
3. 执行默认按顺序推进。
4. 未经中断、调整或明确判断，不应盲目跳过关键步骤。
5. 编译失败时不得把原始模板文本直接当作执行依据。

### 7.3 运行时 plan 的关系

工作流模板不是运行时 plan 本身。

关系如下：

1. 模板提供可复用过程约束。
2. Agent 基于编译后的结构化流程表示构造当前任务的运行时 plan。
3. plan 是当前任务上下文中的具体执行展开。

## 八、统一执行模型与状态可观测

### 8.1 统一执行模型

工作流执行采用统一执行模型。

不存在显式模式切换，不存在供用户选择的“模式一 / 模式二 / 模式三”。

运行时只有一个执行模型：

**Agent 在工作流模板约束下持续推进任务，并根据上下文、风险、不确定性与用户行为动态调整推进状态。**

### 8.2 隐性执行状态

统一执行模型下，运行时可表现为：

1. 连续推进状态。
2. 受限推进状态。
3. 人工介入状态。

这些是运行时状态，不是用户显式选择的模式。

### 8.3 执行状态可观测性

虽然执行状态不是显式模式系统，但状态本身必须可观测。

至少应让用户能够感知：

1. 当前执行阶段。
2. 当前步骤。
3. 已完成步骤。
4. 是否处于受限推进状态。
5. 是否等待人工介入或反馈。

这里定义的是可观测性原则，不展开 UI 或状态机实现。

## 九、失败、中断与恢复

### 9.1 基本原则

当某一步骤无法完成或结果不满足约束时，Agent 应允许中断当前流程，并在当前上下文基础上进行调整或重新推进，而不是盲目继续后续步骤。

### 9.2 失败处理

1. 模板缺失：降级为普通执行，记录 `E_TEMPLATE_NOT_FOUND`。
2. 模板结构不匹配：跳过模板，记录 `E_TEMPLATE_SCHEMA_INVALID`。
3. 模板编译失败：阻断模板执行链，记录 `E_TEMPLATE_COMPILE_FAILED`。
4. 模板步骤非法：阻断当前模板执行，记录 `E_TEMPLATE_STEP_INVALID`。

### 9.3 中断与恢复

1. 中断后应保留当前上下文与已完成状态。
2. 恢复执行时，应从当前可确认边界重新推进。
3. 是否允许跳过步骤，必须建立在运行时明确判断或用户介入基础上。
