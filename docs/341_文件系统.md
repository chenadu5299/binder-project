# 341_文件系统

## 一、文档定位

本文是 Workspace 系统中文件系统能力的统一主规范，定义文件对象、工作区边界、读写主链、缓存主链与协同接口。  
本文用于把“实施方案”上升为标准定义层，并补齐跨组接口约束。

---

## 二、MVP 目标

1. 建立统一文件系统主链：打开、读取、编辑、保存、回写、同步。  
2. 建立 Workspace 边界与文件状态语义，避免状态混用。  
3. 建立前后端与跨组协同接口口径，避免重复实现与职责漂移。  

---

## 三、核心对象与边界

## 3.1 Workspace 边界

1. Workspace 是文件系统运行边界与状态边界。  
2. 文件路径、缓存、pending 状态、依赖关系均绑定 workspace。  
3. 跨 workspace 文件操作必须显式切换上下文，不允许隐式越界。  

## 3.2 文件对象

1. 文件标识：`file_path`（工作区内规范路径）。  
2. 文件类型：文本、docx、媒体、表格等。  
3. 文件状态：未加载、已加载、已修改、已保存、外部变更。  
4. 文件缓存：`file_cache`（内容快照、mtime、hash、更新时间）。  

## 3.3 变更对象

1. 编辑变更：编辑器内事务导致的内容变化。  
2. 外部变更：文件系统 mtime 变化触发的变更。  
3. 待确认变更：pending diff（与文件路径绑定，独立于 UI 视图）。  

---

## 四、文件系统主链

## 4.1 打开链

1. 解析目标文件路径与工作区归属。  
2. 优先读取缓存（命中则直接加载）。  
3. 未命中时读取磁盘并构建缓存。  
4. 返回编辑器可消费内容并记录打开状态。  

## 4.2 读取链

1. 所有读取统一通过文件命令层进入。  
2. 文本类与结构类文件走各自适配策略，但返回统一内容契约。  
3. 读取错误必须结构化返回，不做静默吞错。  

## 4.3 保存链

1. 编辑器内容序列化为目标格式。  
2. 写盘成功后同步更新 `file_cache`。  
3. 写盘失败时保留内存态并暴露错误。  

## 4.4 外部修改链

1. 文件监测发现 mtime 变化。  
2. 触发外部修改处理流程（加载外部更改或保留本地）。  
3. 若加载外部更改，相关 pending 变更按规则失效。  

---

## 五、缓存与一致性规则

1. 缓存与磁盘是双向一致性链，不是单向副本。  
2. 保存后必须执行缓存同步，保证下次打开同源。  
3. 打开时缓存命中规则与失效规则必须可追溯。  
4. 文件类型适配差异不得破坏统一缓存语义。  

---

## 六、协同接口约束（补齐标准层）

## 6.1 与 AI 组

1. AI 组通过工具/命令调用文件能力，不直接操作底层文件系统。  
2. 文件读写能力由文件系统主链提供稳定接口。  
3. AI 结果回流后的文件变更仍需走统一保存链。  

## 6.2 与编辑器组

1. 编辑器只维护内存态文档，不直接承担持久化职责。  
2. 保存动作统一委托文件系统层执行。  
3. 编辑器状态切换必须与文件状态机一致。  

## 6.3 与资源管理组

1. 文件树、监听、搜索索引由资源能力提供。  
2. 文件系统主链消费资源能力，不重复实现监听与索引。  
3. 路径规范化规则由文件系统层统一落地。  

## 6.4 与 UI/UX 组

1. UI 负责提示与交互，不定义文件状态语义。  
2. 关闭警告、冲突弹窗等由文件状态驱动。  
3. UI 文案不得与状态机语义冲突。  

---

## 七、状态机定义（MVP）

1. `idle`：未打开。  
2. `loaded`：已加载且未修改。  
3. `dirty`：已修改待保存。  
4. `saving`：保存进行中。  
5. `saved`：保存完成。  
6. `external_changed`：检测到外部修改。  
7. `conflict_resolved`：冲突处理完成。  

状态转换必须有明确触发事件，禁止隐式跃迁。

---

## 八、MVP 验收口径

1. 文件打开、读取、保存、外部变更四条主链稳定可复现。  
2. 缓存与磁盘一致性可验证。  
3. 跨组接口职责清晰，无重复实现。  
4. 状态机与 UI 行为一致，无语义冲突。
