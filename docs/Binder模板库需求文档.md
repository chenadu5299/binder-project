## 一、定义与分类分析

### 1.1 三类模板的界定与准确性分析

用户提出的三类模板定义如下，本节分析其准确性、与 Binder 场景的契合度及成熟案例参考。

| 类型 | 用户定义 | 准确性分析 | Binder 契合度 | 成熟案例 |
|------|----------|------------|---------------|----------|
| **文档模板** | PDF、docx、ppt 等格式样式模板 | 定义准确，但需扩展 | 高 | Word 模板、Notion 模板、飞书文档模板 |
| **工作流模板** | 执行一套固定工作流程（需求分析→调研→大纲→内容→校对等）；本质为固化的工具调用序列 | 定义准确 | 高 | Dify 工作流、Agent 多步骤执行（steps 注入 prompt） |
| **Skills** | 用户创建、保存、复用的行为说明，告诉 AI 在某类任务里应该怎么做事；即 Skills 的用户管理界面 | 与 Cursor Skills 概念一致 | 高 | Cursor Skills、Cursor Rules |

**结论**：三类定义均准确，且均属于模板库的集合范畴。建议采用**结构化分组**，便于管理、检索与 AI 调用。**不创造新词**，直接叫「Skills」，用户理解成本最低。

### 1.2 定义细化与边界澄清

#### 1.2.1 文档模板（Document Template）

**定义**：预定义的文档格式、样式、结构，用于快速生成符合规范的文档。

| 维度 | 说明 |
|------|------|
| **内容** | 字体、字号、颜色、段落样式、页眉页脚、表格样式、图表占位等 |
| **格式** | DOCX、PDF、PPT 等导出格式；或 Binder 内部文档结构（块类型、章节模板） |
| **用途** | 新建文档时选择模板；AI 生成内容时按模板格式输出 |
| **与 AI** | 作为「格式约束」注入提示词；或作为 create_document 的 schema 参数 |

**Binder 应用场景**：
- 用户新建报告时选择「行业报告模板」
- AI 生成文档时，用户引用模板库中的「会议纪要模板」，AI 按该模板结构生成
- 与 AI 文档格式输出协议（newText 格式、字体白名单）协同

**与知识库、记忆库的区别**：文档模板是**格式与结构**，不包含具体内容；知识库是**文档内容**的检索；记忆库是**语义记忆**。

#### 1.2.2 工作流模板（Workflow Template）

**定位**：工作流模板是 Binder 对用户复杂操作流程的**结构化记录**，供 AI 在对话中按需调用或由用户主动引用，以一致、可重复的方式完成文档生产任务。

**工作流模板不是**：
- 独立运行的自动化程序（没有独立执行入口）
- 系统预置的标准流程（全部来自用户）
- 可视化节点编辑器（用户感知层永远是自然语言）

**工作流模板是**：
- 从真实任务执行中涌现出来的可复用计划
- 对话中的上下文资产，由用户主动引用或 AI 自动调起
- 以 Plan JSON 为底层结构、以自然语言为表达层的执行计划

**核心原则**：

| 原则 | 说明 |
|------|------|
| 涌现优先 | 工作流从真实执行中提炼，不由系统预设 |
| 执行内嵌对话 | 工作流只在聊天中被调用，无独立执行入口 |
| 用户永远看自然语言 | 底层节点结构对用户不可见 |
| 加权触发涌现 | 基于节点敏感度分数，而非原始节点计数 |

**与 AI 执行**：工作流完全在 Agent 能力范围内，**无需 WEE/MAR 等独立引擎**。Agent 按 Plan 的 steps 自主执行工具调用。

#### 1.2.3 Skills

**本质定位**：Skills 的**用户管理界面**，让用户能创建、保存、复用自己的 Skills。不是一个新概念，直接叫「Skills」，用户理解成本最低。

**内容形态**：Skills 里放的东西只有一种——**自然语言写的行为说明**，告诉 AI 在某类任务里应该怎么做事。**纯自然语言**，没有结构化字段，没有 steps 数组，AI 直接消费。

**示例 - 「产品需求分析」Skill**：

```
分析用户提出的需求时，按以下方式处理：
1. 先用一句话复述需求本质，确认理解是否正确
2. 识别需求背后的真实用户痛点
3. 列出实现该需求的前置条件和潜在风险
4. 给出优先级建议，说明理由
语言风格：简洁，避免废话，每条不超过两行
```

**两个子类型，但对用户透明**：系统内部区分两种 Skill，用户创建时不需要选择，系统根据内容自动判断：

| 子类型 | 描述 | 示例 | 生效方式 |
|--------|------|------|----------|
| **持续生效型** | 描述「做某类事情的方式」 | 学术写作规范、产品文档风格、代码审查标准 | 选用后整个对话持续有效 |
| **单次执行型** | 描述「完成某个具体任务的步骤」 | 周报整理、竞品分析框架、用户访谈总结 | @ 引用后单次生效 |

**两个来源，界面清晰分层**：

```
模板库
├─ 文档模板
├─ 工作流模板
└─ Skills
   ├─ 系统 Skills（Binder 内置，只读，灰色标识）
   │  ├─ 生成 Word 文档
   │  ├─ 生成 PPT
   │  └─ 处理表格数据
   └─ 我的 Skills（用户创建，可编辑）
      ├─ 产品需求分析        ← 用户自建
      ├─ 周报整理            ← 用户自建
      └─ 从文件导入...       ← 支持导入 SKILL.md
```

系统 Skills 和用户 Skills 视觉上明显区分，用户不会误以为自己能修改系统 Skills。

**使用方式**（与 PRD 引用系统完全对齐，不增加新交互）：
- **对话开始前**：在输入框上方选择「应用 Skill」，整个对话持续生效，输入框上方显示已激活标识
- **对话中随时**：@ 引用或拖拽到输入框，单次生效

**创建**：门槛极低，就是写一段话。新建 Skill 只需：名称、场景（可选）、内容（自然语言）。没有格式要求，没有字段填写，写完保存就能用。

**前置条件与授权引导**：若 Skill 依赖外部服务（如飞书），创建时可标注；**使用时**触发授权引导（如「这个 Skill 需要连接飞书 [点击授权]」），而不是让用户去找 API key。

### 1.3 三类模板的集合范畴与边界

**三类均属于模板库**：模板库是存储、管理、检索可复用模式的集合，三类模板均为可复用模式的不同形态。

**边界与重叠**：

| 边界 | 说明 |
|------|------|
| **文档模板 vs 工作流模板** | 文档模板侧重「输出格式」；工作流模板侧重「执行过程」。可组合：工作流最后一步输出时，可指定使用某文档模板 |
| **工作流模板 vs Skills** | 见下「工作流与 Skills 的最终分工」 |

**工作流与 Skills 的最终分工**（PRD 层面边界清晰，避免混淆）：

| 维度 | 工作流模板 | Skills |
|------|------------|--------|
| **用户感知** | 「帮我从头到尾做完这件事」 | 「按我的方式帮我做这类事」 |
| **启动方式** | 对话中 `@` 引用或 AI 自动调起；无独立执行入口 | 选用后注入，不需要额外操作 |
| **执行过程** | 有明确步骤和进度，内嵌在对话流中 | 透明，用户不感知执行过程 |
| **交付物** | 明确（一份文档、一张表） | 不固定 |
| **可组合** | — | 可组合 |
| **工作流调用 Skill** | 工作流的某一步可以调用 Skill | ✅ |

**建议**：采用**统一模板库**，三类作为**子类型**（type: document | workflow | skill），共享存储、检索、引用机制，差异化仅在结构与调用方式。

## 二、工作流模板详细设计

本节基于《Binder 工作流模板 PRD v1.0》整合，定义工作流模板的完整生命周期、节点架构、涌现机制与使用方式。

### 2.1 工作流生命周期

```
用户在聊天中执行任务
        │
        ▼
AI 执行并记录工具调用链（结构化步骤序列）
        │
        ▼
任务完成 → AI 整理 Plan JSON
        │
        ▼
加权分 ≥ 阈值？
   │              │
  是              否
   │              │
AI 询问：        不触发
是否保存为工作流？
   │
用户确认 ──→ 存入工作流模板库
             （可编辑 · 可删除）
   │
用户也可随时手动创建工作流
```

### 2.2 涌现触发机制

涌现触发基于**节点加权分数**，而非原始节点计数。

**触发公式**：

```
加权分 = Σ(本次执行中每个节点的敏感度权重)
加权分 ≥ 触发阈值 → AI 询问是否保存
```

**触发阈值**：初始建议值 ≥ 8，上线后根据真实使用数据校准。

**特殊规则**：执行链中出现 `memory_save` 节点时，无论加权分多少，**直接触发**涌现询问。`memory_save` 的出现本身意味着用户判断此次执行结果值得记忆，是强复用信号。

### 2.3 通用节点架构

Binder 的核心场景是文档生产，每个工作流节点代表信息在文档生产链条中的一次变换。节点之间传递的是有语义的内容（不是触发信号），架构采用**信息流转模型**而非任务调度模型。

#### 2.3.1 节点类型全集

**A 层：获取（Fetch）**——从各类来源拉取信息，是工作流的信息入口。

| 节点 | 类型标识 | 说明 |
|------|----------|------|
| 知识库检索 | `knowledge_fetch` | 从用户知识库检索相关内容 |
| 记忆读取 | `memory_fetch` | 读取用户记忆库中的历史上下文 |
| 文档读取 | `doc_read` | 读取工作区内的指定文档或段落 |
| 外部数据 | `web_fetch` | 从外部网络来源获取数据 |

**B 层：加工（Process）**——对获取到的信息进行语义变换，是工作流的核心处理环节。

| 节点 | 类型标识 | 说明 |
|------|----------|------|
| 信息提取 | `extract` | 从原始内容中提取结构化字段或关键信息 |
| 摘要整理 | `summarize` | 对内容进行压缩整理 |
| 对比分析 | `compare` | 对多份内容进行横向对比 |
| 分类归纳 | `classify` | 将内容按维度分组归类 |
| 翻译转化 | `translate` | 语言转换或术语统一 |

**C 层：生成（Generate）**——基于加工结果进行文档内容创作，是工作流的核心输出环节。

| 节点 | 类型标识 | 说明 |
|------|----------|------|
| 文档写作 | `draft_write` | 基于输入内容生成完整文档段落 |
| 模板填充 | `template_fill` | 将内容填入指定文档模板 |
| 章节扩写 | `section_expand` | 对已有章节进行扩展和深化 |
| 表格生成 | `table_gen` | 将结构化数据生成为文档内表格 |

**D 层：输出（Output）**——将生成结果写入文档系统或持久化存储。

| 节点 | 类型标识 | 说明 |
|------|----------|------|
| 格式导出 | `format_export` | 将文档导出为指定格式（docx / pdf / md） |
| 插入文档 | `doc_insert` | 将生成内容插入当前编辑器文档 |
| 记忆写入 | `memory_save` | 将关键结果写入用户记忆库 |

#### 2.3.2 节点使用规则

- 节点可跨层组合，不强制线性执行
- 同一节点可在一次工作流中多次出现（如多次 `knowledge_fetch` 调用不同知识库）
- 执行引擎按实际调用顺序记录节点序列，不按预设层次走

### 2.4 节点敏感度权重

敏感度权重反映节点的**复用价值**：替换成本越高、越依赖用户特定配置的节点，权重越高。

| 权重 | 含义 | 判断标准 |
|------|------|----------|
| 3（高敏） | 强复用信号 | 调用了用户特定资产（知识库、模板、记忆），临时重建成本高 |
| 2（中敏） | 中等复用信号 | 操作有特定意图，但通用性相对较强 |
| 1（低敏） | 弱复用信号 | 通用操作，任何时候都可重做 |

**各节点权重**：

| 节点 | 类型标识 | 权重 | 敏感度 |
|------|----------|------|--------|
| 知识库检索 | `knowledge_fetch` | 3 | 高 |
| 记忆读取 | `memory_fetch` | 3 | 高 |
| 文档写作 | `draft_write` | 3 | 高 |
| 模板填充 | `template_fill` | 3 | 高 |
| 外部数据 | `web_fetch` | 2 | 中 |
| 文档读取 | `doc_read` | 2 | 中 |
| 对比分析 | `compare` | 2 | 中 |
| 章节扩写 | `section_expand` | 2 | 中 |
| 信息提取 | `extract` | 1 | 低 |
| 摘要整理 | `summarize` | 1 | 低 |
| 分类归纳 | `classify` | 1 | 低 |
| 翻译转化 | `translate` | 1 | 低 |
| 表格生成 | `table_gen` | 1 | 低 |
| 格式导出 | `format_export` | 1 | 低 |
| 插入文档 | `doc_insert` | 1 | 低 |
| 记忆写入 | `memory_save` | 1* | 特殊（出现即触发涌现） |

**触发示例**：

- **场景 A：竞品对比报告（触发）**：`knowledge_fetch`（3）+ `doc_read`（2）+ `compare`（2）+ `template_fill`（3）+ `format_export`（1）= 加权分 11 → 触发涌现
- **场景 B：简单摘要导出（不触发）**：`summarize`（1）+ `translate`（1）+ `format_export`（1）= 加权分 3 → 不触发
- **场景 C：含记忆写入（直接触发）**：出现 `memory_save` → 直接触发，不计分

### 2.5 使用方式

工作流模板没有独立执行入口，只在对话中被调用。

**路径 A：用户主动引用**——用户在聊天中使用 `@` 语法主动引用：

```
用户：@竞品对比报告 帮我分析一下最新的飞书文档功能
```

AI 加载对应工作流的 Plan，将步骤注入当前对话上下文，按步骤执行。

**路径 B：AI 自动调起**——用户的任务意图与某个已保存工作流高度匹配时，AI 自动识别并引用：

```
用户：帮我把最新的竞品调研结果整理成分析报告
AI：检测到你有保存的「竞品对比分析报告」工作流，正在按流程执行…
```

**执行过程**：工作流执行完全内嵌在对话流中，用户看到的是语义化进度，不是技术节点：

```
✓ 已从竞品知识库检索到 12 条相关内容
✓ 已读取当前产品文档
… 正在进行横向对比分析
  正在填充报告模板（等待中）
  导出文件（等待中）
```

### 2.6 与其他模块的关系

| 模块 | 关系 |
|------|------|
| 知识库 | 工作流通过 `knowledge_fetch` 节点调用，params 中存储 knowledge_base_id |
| 模板库 | 工作流通过 `template_fill` 节点调用，params 中存储 template_id |
| 记忆库 | 工作流通过 `memory_fetch` / `memory_save` 节点读写 |
| 对话编辑（层次三） | 工作流的执行载体，Plan 步骤注入对话上下文 |
| 工作流执行 | 内嵌对话流，Agent 按 steps 自主执行 |

## 三、模板结构化分组设计

### 3.1 分组必要性

**结论**：**建议做模版的结构化分组**。

**理由**：
1. 模板数量可能增长（文档模板、工作流、Skills 混合），无分组难以管理
2. 用户检索、AI 匹配均需按类型、场景筛选
3. 与记忆库、知识库的树状结构一致，便于统一 UI 与交互
4. 分组便于权限、可见性、版本管理

### 3.2 分组结构

```
模板库根
├─ 文档模板（type: document）
│  ├─ 报告类（report）
│  │  ├─ 行业报告模板
│  │  └─ 会议纪要模板
│  ├─ 演示类（presentation）
│  └─ 表格类（table）
│
├─ 工作流模板（type: workflow）
│  ├─ 报告生成（report_generation）
│  │  ├─ 行业报告工作流
│  │  └─ 需求分析工作流
│  ├─ 调研分析（research）
│  └─ 内容创作（content_creation）
│
└─ Skills（type: skill）
   ├─ 系统 Skills（Binder 内置，只读，灰色标识）
   │  ├─ 生成 Word 文档
   │  ├─ 生成 PPT
   │  └─ 处理表格数据
   └─ 我的 Skills（用户创建，可编辑）
      ├─ 产品需求分析
      ├─ 周报整理
      └─ 从文件导入...
```

### 3.3 分组维度

| 维度 | 说明 | 示例 |
|------|------|------|
| **类型（type）** | 文档 / 工作流 / Skills | document, workflow, skill |
| **场景（scene）** | 使用场景分类 | 报告、调研、表格、代码等 |
| **可选的标签（tags）** | 用户自定义标签 | 飞书、多维表格、周报 |
| **来源（source）** | 系统 / 用户 / 导入 | system, user, imported |

### 3.4 与 Binder 数据模型的关系

参考 Binder AI 功能方案整合版 六、核心数据模型：

```
项目
 ├ 文档
 ├ 数据集
 ├ 资产
 ├ 笔记
 └ 工作流
```

**模板库的归属**：
- **项目级**：项目可拥有自己的模板库（项目私有模板）
- **用户级**：用户可拥有跨项目模板库（用户级模板）
- **系统级**：Binder 内置模板（系统模板）

**建议**：模板库支持**多级**：系统模板（只读）+ 用户模板（可编辑）+ 项目模板（项目内可编辑）。

## 四、AI 调用机制设计

### 4.1 调用目标

**目标**：模板库生成后，AI 可**稳定调用**、**执行**模板。

**调用场景**：

| 场景 | 调用方 | 调用方式 |
|------|--------|----------|
| **用户主动选择** | 用户选择模板 → 传入 AI | 用户引用模板（@ 或拖拽）→ 作为 references 传入 AI |
| **AI 自动匹配** | AI 根据用户意图检索模板 | 用户说「按行业报告格式生成」→ AI 调用 list_templates 或 search_templates 检索 |
| **工作流执行** | Agent 按 steps 自主执行 | 用户选择工作流 → 作为 system prompt 注入 → AI 按 steps 调用工具（无需 WEE/MAR） |

### 4.2 调用协议

#### 4.2.1 模板检索接口

| 协议项 | 内容 |
|--------|------|
| **命令** | `list_templates(options?: { type?, scene?, tags?, limit? })` |
| **命令** | `search_templates(query: string, options?: { type?, limit? })` |
| **返回** | `{ items: [{ id, name, type, scene, goal?, structure?, ... }], total? }` |

#### 4.2.2 模板引用格式（与 AI 与其他功能交互规范一致）

| 引用类型 | type | 说明 |
|----------|------|------|
| 模板库引用 | `template` | 模板完整内容传入；AI 按模板执行 |

**前端 → 后端**：
```typescript
references?: Array<{
  type: 'template';
  source: string;   // 模板 ID 或名称
  content: string;  // 模板完整内容（goal、steps、structure 等）
  templateType?: 'document' | 'workflow' | 'skill';
}>;
```

#### 4.2.3 工具调用（Agent 模式）

| 工具名 | 说明 | 参数 |
|--------|------|------|
| `list_templates` | 列出可用模板 | type?, scene?, limit? |
| `search_templates` | 语义检索模板 | query, type?, limit? |
| `get_template` | 获取单个模板详情 | templateId |
| `run_workflow` | （可选）执行工作流模板 | templateId, inputs |

**调用时机**：用户意图涉及「按模板」「用工作流」「应用 Skill」时，AI 调用上述工具检索或获取模板。工作流模板通过**注入 system prompt** 执行，无需独立 run_workflow 引擎。

### 4.3 提示词注入

**模板作为上下文注入**：

当用户引用模板或 AI 检索到模板时，将模板内容作为「模板库信息」段落注入提示词：

- **文档模板 / 工作流模板**：注入 name、type、goal、steps 或 structure
- **Skills**：直接注入**自然语言内容**，无结构化字段，AI 直接消费

### 4.4 稳定性保障

| 保障措施 | 说明 |
|----------|------|
| **模板 ID 稳定性** | 模板 ID 不随重命名变更；删除后 ID 不可复用 |
| **版本兼容** | 模板结构变更时，保留版本号或兼容旧字段 |
| **降级策略** | 检索失败、模板不存在时，静默降级，不阻塞主流程 |
| **超时** | 模板检索超时 500ms，与记忆库一致 |

### 4.5 与编辑模式、构建模式的关系

| 模式 | 模板调用方式 |
|------|--------------|
| **编辑模式（Agent）** | 用户引用的模板 → 作为 references 或 system prompt 注入；工作流模板通过「带 steps 的 system prompt」由 Agent 按步骤自主执行工具调用 |
| **构建模式** | （远期）工作流内嵌对话流，无独立引擎；当前阶段由 Agent 按 steps 自主执行 |

### 4.6 工作流执行核心挑战（落地难度：中等）

工作流模板不依赖 WEE/MAR，完全在 Agent 能力范围内实现。核心挑战为：

| 挑战 | 说明 | 应对方向 |
|------|------|----------|
| **顺序执行** | AI 能否稳定地按 steps 顺序执行而不跳步 | 在 system prompt 中明确「必须按步骤 1→2→3→… 依次执行」「完成当前步骤后再进入下一步」；steps 描述中可标注建议工具 |
| **错误处理** | 中间步骤失败（如网页搜索超时、文件不存在）时的处理 | 定义重试策略、降级逻辑；失败时向用户明确提示当前步骤及失败原因；支持用户选择「跳过」「重试」「终止」 |
| **进度反馈** | 多步骤执行过程中，用户无法感知进度 | 每完成一步，在聊天窗口展示「步骤 N/N 已完成：xxx」；工具调用状态（pending/executing/completed/failed）实时可见 |

## 六、核心数据模型与协议

### 6.1 模板结构（统一基础）

```typescript
interface TemplateBase {
  id: string;
  name: string;
  type: 'document' | 'workflow' | 'skill';
  scene?: string;           // 场景分类
  tags?: string[];
  description?: string;
  source?: 'system' | 'user' | 'imported';
  createdAt?: string;
  updatedAt?: string;
}
```

### 6.2 文档模板结构

```typescript
interface DocumentTemplate extends TemplateBase {
  type: 'document';
  structure: {
    blocks?: Array<{ type: string; attrs?: Record<string, unknown> }>;
    styles?: { fontFamily?: string[]; fontSize?: number[]; colors?: string[] };
    format?: 'docx' | 'pdf' | 'ppt';  // 导出格式
  };
}
```

### 6.3 工作流模板结构（Plan JSON）

工作流模板的底层存储格式为 **Plan JSON**。用户不直接接触此结构，由系统在执行记录中自动生成。工作流用于生成「带固定 steps 的 Agent 系统提示」，无需 WEE/MAR。结构可与远期 WEE 兼容，但当前由 Agent 按 steps 自主执行。

```json
{
  "id": "wf_20260318_001",
  "name": "竞品对比分析报告",
  "description": "从知识库获取竞品资料，对比分析后套用报告模板输出",
  "created_at": "2026-03-18T10:00:00Z",
  "source": "emerged",
  "weighted_score": 11,
  "steps": [
    {
      "id": "step_1",
      "type": "knowledge_fetch",
      "label": "检索竞品知识库",
      "params": {
        "knowledge_base_id": "kb_competitor_2026",
        "query_hint": "竞品核心功能与定价"
      },
      "depends_on": null
    },
    {
      "id": "step_2",
      "type": "doc_read",
      "label": "读取当前产品文档",
      "params": {
        "doc_scope": "current_editor"
      },
      "depends_on": null
    },
    {
      "id": "step_3",
      "type": "compare",
      "label": "横向对比分析",
      "params": {
        "dimensions": ["功能", "定价", "目标用户"]
      },
      "depends_on": ["step_1", "step_2"]
    },
    {
      "id": "step_4",
      "type": "template_fill",
      "label": "填入分析报告模板",
      "params": {
        "template_id": "tpl_competitor_analysis_v2"
      },
      "depends_on": ["step_3"]
    },
    {
      "id": "step_5",
      "type": "format_export",
      "label": "导出文件",
      "params": {
        "formats": ["docx", "pdf"]
      },
      "depends_on": ["step_4"]
    }
  ]
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 工作流唯一标识 |
| `name` | string | 用户可读名称（AI 生成，用户可改） |
| `source` | enum | `emerged`（涌现）/ `manual`（用户手动创建） |
| `weighted_score` | number | 触发涌现时的加权分，用于后续阈值校准 |
| `steps[].type` | string | 节点类型标识，对应节点全集（见 2.3 节） |
| `steps[].params` | object | 节点参数，记录用户当时的具体配置 |
| `steps[].depends_on` | array | 前置节点 id，支持并行步骤 |

### 6.4 Skills 结构（自然语言内容，无结构化字段）

```typescript
interface SkillTemplate extends TemplateBase {
  type: 'skill';
  content: string;          // 自然语言写的行为说明，AI 直接消费
  scene?: string;           // 场景（如「需求讨论」），可选，便于用户分类
  requiresAuth?: Array<{    // 前置条件：依赖的外部服务，使用时触发授权引导
    service: string;        // 如 'feishu' | 'notion' | ...
    label: string;          // 如「飞书」
  }>;
}
```

**与指令集模板的区别**：无 steps 数组、无 tools 字段、无 goal 结构化字段；仅 `content` 自然语言，门槛极低。

### 6.5 与 AI 功能前置协议与标准对齐

AI 功能前置协议与标准 四、模板库 / 工作流模板 已与本文档对齐：

| 协议项 | 内容 |
|--------|------|
| **结构** | Plan JSON：`{ id, name, description, source, weighted_score, steps }` |
| **与 AI 的交互** | 对话中 `@` 引用或 AI 自动调起；无独立执行入口；AI 按 steps 注入对话上下文并执行 |
| **存储** | 格式先定，存储实现可后置 |

本文档在基础上扩展：**文档模板**、**Skills** 两类，并统一为**模板库**。工作流**不依赖 WEE/MAR**，由 Agent 按 steps 直接执行。Skills 为自然语言内容，无结构化字段。

## 七、UI 与交互

### 7.1 模板库面板

| 位置 | 说明 |
|------|------|
| **入口** | 标签栏或资源管理区添加模板库标签（图标：📋 或 📁） |
| **展示** | 树状结构，按类型、场景分组展示 |
| **与记忆库、知识库** | 三者并列，支持 @ 时统一候选列表 |

### 7.2 工作流模板侧边栏（管理入口）

工作流模板在侧边栏的作用是**管理**（查看、编辑、删除、新建），不是触发执行。

**列表视图**：

```
模板库
├── 文档模板        → 树形列表 + 编辑窗口
├── Skills         → 树形列表 + 编辑窗口
└── 工作流模板  [3] → 列表 + 详情面板（非编辑窗口）
    ├── 竞品对比分析报告
    ├── 技术规范章节生成
    └── + 新建工作流
```

**详情面板内容**（点击某个工作流后，右侧展示）：
- 工作流名称（可改）
- 来源标注（涌现 / 手动创建）
- 步骤摘要（语义化描述，非节点 ID）
- 操作：编辑步骤 · 删除

**编辑权限**：

| 元素 | 是否可改 | 说明 |
|------|----------|------|
| 工作流名称 | 是 | 自由修改 |
| 步骤顺序 | 是（编辑模式） | 普通查看不暴露，编辑模式可调 |
| 节点参数（如知识库 ID） | 是 | 可更换绑定的知识库、模板 |
| 节点类型本身 | 是（编辑模式） | 高级操作，需主动进入编辑模式 |

### 7.3 引用方式

| 方式 | 说明 |
|------|------|
| **@ 引用** | 用户输入 `@` 后，模板库作为可选项出现；支持选中某一条插入；工作流支持 `@工作流名称` 主动引用 |
| **拖拽引用** | 用户将模板拖入聊天输入框 |
| **选择后发送** | 用户在选择模板后，直接作为上下文发送 |

### 7.4 模板选择与执行

| 操作 | 行为 |
|------|------|
| **选择文档模板** | 新建文档时选择；或对话中引用，AI 按模板格式生成 |
| **选择工作流模板** | 对话中 `@工作流名称` 引用；或 AI 自动识别意图调起；执行内嵌在对话流中，无独立执行入口 |
| **选择 Skills** | 对话开始前：输入框上方选择「应用 Skill」，整个对话持续生效；对话中：@ 引用或拖拽，单次生效 |

### 7.5 Skills 创建（门槛极低）

| 能力 | 要求 |
|------|------|
| **新建 Skill** | 名称、场景（可选）、内容（自然语言）；无格式要求，无字段填写，写完保存就能用 |
| **创建界面示意** | 名称输入框、场景输入框（可选）、内容文本框；保存按钮 |
| **导入** | 支持从 SKILL.md 文件导入 |
| **前置条件标注** | 若 Skill 依赖外部服务（如飞书），创建时可标注；使用时触发授权引导（「这个 Skill 需要连接飞书 [点击授权]」） |

- **内容形态**：纯自然语言行为说明，无 steps 数组、无 tools 字段，AI 直接消费
- **两个子类型**：持续生效型 / 单次执行型（对用户透明，系统自动判断）
- **两个来源**：系统 Skills（只读、灰色标识）/ 我的 Skills（用户创建、可编辑）
- **创建门槛极低**：名称、场景、内容，写完保存就能用；支持从 SKILL.md 导入
- **前置条件与授权引导**：requiresAuth 字段，依赖飞书等外部服务时使用时触发授权
- **工作流与 Skills 边界**：工作流「从头到尾做完」；Skills「按我的方式做这类事」；工作流某步可调用 Skill
