## 一、文档定位

本文是 Chat Build 的 chat / build 接管专项文档。

它直接受以下规则约束：

1. `BR-CBT-MODEL-001`
2. `BR-CBT-STATE-001`
3. `BR-CBT-RUN-001`

本文只回答：

1. 构建前 chat 承担什么职责。
2. 构建中 chat 如何降权并避免干扰当前运行。
3. 构建后如何恢复讨论能力。
4. 指令输入与自然语言输入在不同阶段如何区分。

## 二、阶段化接管规则

### 2.1 构建前：chat 主导

在以下状态中：

1. `discussion`
2. `intent_pending`
3. `outline_drafting`
4. `outline_pending_confirm`

chat 是主交互面。

它负责：

1. 自然语言讨论
2. 引用输入
3. 构建触发
4. 大纲展示与确认

### 2.2 构建中：build 主导，chat 降权

在 `building` 中：

1. Build 控制层成为主导
2. chat 只承担展示与控制入口
3. chat 不再拥有需求改写权

chat 在此阶段只允许：

1. 展示当前状态
2. 展示总体进度
3. 展示当前步骤与过程说明
4. 暴露“停止构建”控制

### 2.3 构建后：chat 恢复

在 `completed` / `failed` / `interrupted` 之后：

1. 当前运行结束
2. chat 可恢复为讨论入口
3. 新一轮自然语言输入只作用于下一轮构建，不回写当前结束态

## 三、输入限制

### 3.1 自然语言输入

| 阶段 | 自然语言输入作用 |
|---|---|
| `discussion` | 正常参与目标收敛 |
| `intent_pending` | 可取消或回到讨论 |
| `outline_drafting` | 视为下一轮补充，不直接改写正在生成的大纲 |
| `outline_pending_confirm` | 只能触发“修改大纲”或“返回讨论”，不能绕过确认直接开跑 |
| `building` | 不进入当前运行，不得改写目标 |
| `completed/failed/interrupted` | 进入下一轮讨论上下文 |

### 3.2 指令输入

当前版本只区分两类指令：

1. 构建前指令  
   例如“开始构建”“进入大纲生成”。

2. 构建中控制指令  
   当前最小版本只保留“停止构建”。

当前版本不支持：

1. 构建中的自然语言控制指令替代 UI 中断按钮
2. 构建中的自由文本作为控制面板

## 四、UI 输入限制

当前版本推荐的最小 UI 策略是：

### 4.1 构建前

1. 输入框正常可编辑
2. 构建按钮可用
3. 引用选择可用

### 4.2 构建中

1. 自由文本输入框禁用或降为只读提示
2. 保留状态展示区
3. 保留中断按钮
4. 如仍显示输入框，则任何文本都不得进入当前运行

### 4.3 构建后

1. 输入框恢复
2. 中断按钮消失
3. 上一轮运行结果作为只读记录保留

## 五、chat / build 接管切换点

### 5.1 接管开始

接管开始点：

```text
outline_pending_confirm --(confirmed)--> building
```

一旦进入 `building`：

1. chat 不再是需求编辑通道
2. Build 控制层接管状态推进

### 5.2 接管结束

接管结束点：

```text
building --> completed | interrupted | failed
```

一旦进入结束态：

1. 当前运行结束
2. chat 恢复为讨论入口
3. 新输入只作用于新一轮

## 六、验收边界

以下情况视为未通过：

1. `building` 态自然语言输入能改写当前目标
2. `building` 态仍允许自由提交需求文本进入当前运行
3. 结束态后 chat 仍被锁死，无法恢复讨论
4. 构建前阶段 chat 失去正常讨论能力
5. 中断按钮不存在或与普通聊天输入混淆
