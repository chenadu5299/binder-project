## 一、文档目标

本文只解决一个问题：

如何让层次三对话编辑提示词成为一套完整、稳定、可执行的主控规则，不再出现目标混淆、状态误导、协议断裂、旧字段残留、工具调用跑偏。

本文必须同时覆盖：

1. 提示词层级结构
2. 上游输入契约
3. 变量命名与语义
4. 目标文档判定口径
5. 零搜索与块内搜索的分流逻辑
6. 普通引用与精确引用的分工
7. 工具调用规范
8. 工具结果后的继续对话语义
9. 与业务链和协议的对齐关系

## 二、当前提示词链中的致命问题

### 2.1 pending diff 被错误描述成“已经应用”

错误语义：

- `The edit has been applied to the document`
- `The document edit is in progress`

真实机制：

- `edit_current_editor_document` 生成的是待确认 diff
- 用户未接受前，文档逻辑状态不变
- AI 不能把 pending diff 当成已落地结果继续规划

修正规则：

- 工具执行后的继续提示只能描述为：
  - diff 已生成
  - 等待用户确认
  - 不得把该修改视为已应用

### 2.2 当前文件与目标文件混淆

错误根源：

- 提示词围绕 `Current file open in editor` 组织
- 真实业务链围绕“目标编辑文件”组织

修正规则：

- 提示词中必须明确区分：
  - 当前活动编辑器文件
  - 本轮目标编辑文件
- 所有编辑规则都以“目标编辑文件”为中心

### 2.3 零搜索与块内搜索没有统一

错误根源：

- 提示词只完整教了 `block_index + target + content`
- 没有把“选区精确坐标”和“精确引用坐标”写成一级入口

修正规则：

- 有精确坐标时直接零搜索
- 无精确坐标时才走块内搜索

### 2.4 引用层与读取文件规则互相打架

错误信号：

- “引用内容已完整提供，无需再读取”
- “如有需要可以 read_file”

修正规则：

- 已完整注入的引用内容不得再次读取
- 当前目标编辑文件不得先 `read_file` 再编辑
- 只有未注入内容的外部参考资源才允许 `read_file`

### 2.5 旧字段和旧协议残留

不得再作为推荐路径出现的内容：

- `scope`
- `target_content`
- `instruction`
- `edit_target.anchor`
- `plan_workflow`

## 三、层次三提示词的唯一目标

层次三提示词必须稳定引导模型完成以下行为：

1. 识别真实目标编辑文件
2. 判断系统是否已经提供精确定位输入
3. 有精确定位输入时直接走零搜索
4. 无精确定位输入时使用 `[文档块列表] + block_index`
5. 对多块修改逐块调用编辑工具
6. 对非当前和未打开目标文档保持同一编辑主链口径
7. 不把 pending diff 当作已应用结果
8. 不调用旧字段，不触发旧路径，不使用旧工具口径

---

## 四、层次三提示词必须支撑的完整链路

层次三提示词不是一段系统文案，而是整个对话编辑业务链的语言控制层。

它必须支撑完整链路：

1. 用户发起任务
2. 系统确定目标文档
3. 系统判断有无精确定位输入
4. 系统注入 `RequestContext`
5. 系统注入块列表或精确坐标
6. 模型输出结构化编辑请求
7. 工具执行并返回 diff
8. 系统继续与模型对话时保持正确状态语义

如果提示词不能完整覆盖这八步，它就不是主控文档。

## 五、上游输入契约

### 5.1 输入来源

层次三必须承认三类输入来源：

1. 当前目标文档显式选区
2. 复制后粘贴生成的精确内容引用标签
3. 无选区时的光标位置

`@` 标签不承担内容精确定位。

### 5.2 精确定位输入契约

当系统已掌握精确定位输入时，提示词必须按完整区间坐标理解：

- `startBlockId`
- `startOffset`
- `endBlockId`
- `endOffset`

提示词层必须把它视为：

- 零搜索输入
- 直接定位真源
- 非普通参考文本

### 5.3 普通引用输入契约

普通引用只承担参考作用，不承担精确定位作用。

普通引用可以帮助模型：

- 理解任务
- 理解上下文
- 判断是否需要同步修改其他文档

普通引用不能替代精确定位输入。

### 5.4 目标文档输入契约

提示词必须始终围绕“目标编辑文件”组织，而不是围绕“当前活动文件”组织。

目标文档可以是：

- 当前已打开文档
- 非当前但已打开文档
- 未打开但可静默加载的文档
- 未打开且需先构建 canonical HTML 的文档

## 六、变量命名与语义契约

### 6.1 变量命名原则

提示词允许沿用实现中的字段名，但语义必须统一。

### 6.2 核心字段

```text
targetFile
L
revision
editorTabId
current_file
current_content
document_revision
_sel_start_block_id
_sel_start_offset
_sel_end_block_id
_sel_end_offset
_sel_text
```

### 6.3 语义约束

- `targetFile`：本轮真正要编辑的文档
- `L`：本轮目标文档逻辑状态快照
- `revision`：目标文档真实内容状态序号
- `editorTabId`：若目标文档存在真实编辑器实例则记录，否则允许为空
- `current_file/current_content/document_revision`：当进入编辑工具时，语义等同于“本轮目标编辑文档”的文件、内容、版本
- `_sel_*`：系统提供的精确定位区间，仅在已掌握精确定位输入时出现

### 6.4 禁止的语义混用

禁止出现以下混用：

- 把 `current_file` 解释成“当前活动标签文件”而不是“当前编辑工具目标文件”
- 把 `_sel_*` 当作普通参考信息而不是定位真源
- 把 `revision` 当作 pending diff 的统一失效开关

## 七、提示词拼装顺序

层次三提示词固定按以下顺序拼装：

1. 基础系统提示词
2. 目标文档上下文提示词
3. 引用提示词
4. 工具定义与编辑规则
5. 工具执行后继续对话提示

任何实现都不得调整这个责任顺序。

原因：

- 先定义角色与边界
- 再定义目标文档和定位状态
- 再注入外部参考
- 再约束工具协议
- 最后约束工具后续语义

---

## 八、五层提示词结构

### 8.1 第一层：基础系统提示词

第一层只表达四类内容：

1. 角色
2. 工具调用边界
3. 对话编辑基本规则
4. 响应风格

必须包含的核心约束：

- 对话编辑使用 `edit_current_editor_document`
- 不得把 generated diff 当成已应用结果
- 用户未确认前，不得假定文档已经变化
- 多块编辑逐块调用
- `rewrite_document` 只用于明确全文重写

禁止出现：

- `plan_workflow`
- `scope=block`
- `target_content`
- “edit 已应用”
- “先 read_file 再编辑目标文件”

### 8.2 第二层：目标文档上下文提示词

第二层必须告诉模型：

- 本轮目标编辑文件是谁
- 当前是否已掌握精确定位输入
- 当前 `L` 的语义
- 当前是否进入全文扫描任务
- 当前块列表是什么

第二层的固定结构：

#### A. 目标文件

```text
Target file for this task: <path>
This is the document to edit in this round.
```

#### B. 当前逻辑状态

```text
Use the current logical document state provided by the system for this round.
Pending diffs are not treated as applied content.
```

#### C. 精确定位状态

有精确定位输入时：

```text
Exact edit coordinates are available for this task.
Do not search the whole document again.
Do not re-read the target file.
```

无精确定位输入时：

```text
No exact coordinates are available.
Use [文档块列表] and block_index for targeting.
```

#### D. 全文扫描约束

识别到全文扫描任务时，必须追加：

```text
This is a multi-block scan task.
Edit one block at a time.
Do not use rewrite_document unless the user explicitly requests a full-document rewrite.
```

### 8.3 第三层：引用提示词

第三层必须把精确引用和普通引用拆开写。

#### A. 当前目标文件的精确引用

当用户粘贴的引用带精确坐标且属于当前目标文件时，必须写成：

```text
The user pasted a precise reference from the target document.
This reference already provides exact edit location context.
Use it as direct targeting input.
```

#### B. 普通引用

普通引用写成：

```text
The user referenced the following external content.
It is fully provided below.
Use it as context only.
Do not treat it as exact edit coordinates.
```

#### C. 冲突语义禁令

第三层禁止同时出现：

- “内容已完整提供，无需再读”
- “你可以 read_file 读取它”

这两种语义只能保留一种。

### 8.4 第四层：工具定义与编辑规则

第四层必须明确：

- 唯一编辑主链
- 零搜索与块内搜索的分流规则
- 多块逐块调用
- 严格降级白名单
- 全文重写的唯一合法场景

必须表达的规则：

- 有精确坐标时直接定位
- 无精确坐标时使用 `[文档块列表] + block_index`
- 多块局部任务必须“一块一次调用”
- 不得把局部多块任务偷懒改成 `rewrite_document`
- 只有命中严格降级白名单时才允许降级为 `block_level`

### 8.5 第五层：工具执行后继续对话提示

第五层是最危险的一层。

它必须遵守三条硬规则：

1. 不能把 pending diff 说成已应用
2. 不能诱导模型以错误基准继续编辑
3. 不能把“待确认建议状态”和“最终文档状态”混淆

标准继续提示必须表达：

```text
The edit request produced diff suggestions for the target document.
These diffs are pending user confirmation.
Do not assume the document has changed yet.

If more changes are still required for the same task:
- continue planning against the current logical document state
- generate additional diff suggestions

If the task is complete:
- provide a concise summary of the intended changes
- do not claim the changes are already applied unless the user has confirmed them
```

## 九、层次三行为规范

### 9.1 精确定位行为

当存在以下任一输入时，模型必须按精确定位任务处理：

- 用户显式选区
- 用户粘贴的精确引用

行为约束：

- 不重新全文搜索
- 不重新读取目标文件
- 不臆造 `block_index`

### 9.2 块内搜索行为

仅在没有精确定位输入时启用。

行为约束：

- 从 `[文档块列表]` 选择 `block_index`
- `target` 必须是块内纯文本原文
- `occurrence_index` 只用于同块重复命中

### 9.3 严格降级行为

提示词必须明确：

- 只有命中固定白名单才允许降级
- 降级只能降到块级替换
- 不允许局部多块任务改成全文重写

### 9.4 全文扫描行为

提示词必须明确：

- 一个块一次调用
- 多块任务拆分为多次调用
- 内容无变化的块不生成 diff

### 9.5 非当前文档行为

提示词必须明确：

- 编辑目标由任务意图决定
- 非当前文档、未打开文档，只要任务要求同步修改，系统会先解析目标文件再编辑
- 模型不得因为“文件未打开”而改走旧协议

### 9.6 无修改行为

提示词必须明确：

- 若修改结果与原内容一致，则视为无修改
- 不生成 diff
- 不得把“无修改”包装成一次成功编辑

### 9.7 失效与确认行为

提示词必须明确：

- diff 生成后等待用户确认
- 用户未确认前，文档逻辑状态不变
- 失效是系统处理规则，不需要模型额外解释

## 十、上下游衔接规则

### 10.1 上游到提示词

上游系统必须给层次三提示词提供：

- 目标文档
- 当前逻辑状态 `L`
- 精确定位输入或块列表
- 普通引用内容
- 当前任务是否为全文扫描

缺任一项都会导致提示词主控失真。

### 10.2 提示词到工具

提示词必须把模型稳定收口到统一编辑协议：

- `block_index`
- `edit_mode`
- `target`
- `content`
- `occurrence_index`

不得再把模型引导到旧字段或旧协议。

### 10.3 工具到后续提示

工具执行后，系统必须把返回结果解释成：

- 待确认 diff
- 当前逻辑状态尚未变化
- 若仍需修改，继续以当前逻辑状态为基准生成新的 diff

### 10.4 后续提示到下一轮请求

下一轮请求重新构造：

- 新的 `L`
- 新的 baseline
- 新的 `revision`

但这些变化不自动使无交集 pending diff 失效。

## 十一、层次三与协议对齐清单

层次三提示词必须与以下业务规则完全对齐：

1. 精确内容引用只来自选区与复制粘贴引用标签
2. `@` 标签不承担内容精确定位
3. 目标文档跟任务意图走，不跟当前状态走
4. `revision` 不是统一失效开关
5. diff 是否失效只看目标内容与目标区间是否已被修改
6. 无交集 diff 持续有效
7. 部分重叠 diff 禁止共存
8. 完全包含 diff 允许共存
9. 内容无变化不生成 diff
10. `rewrite_document` 只用于明确全文改写任务

若提示词和这些规则冲突，以这些规则为准。
