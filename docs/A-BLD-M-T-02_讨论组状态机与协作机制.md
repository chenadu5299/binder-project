## 一、文档定位

本文定义讨论组（Discussion Room）的完整状态机规则。

本文是 `A-BLD-M-D-03` 状态机概述的技术细化，覆盖：状态定义、转换条件、转换发起方、不可逆约束、确认协议、协作时序。

## 二、状态定义

讨论组共三个状态：

### 2.1 IN_PROGRESS（进行中）

**含义：** 讨论组处于活跃讨论阶段。

**可执行操作：**

1. 所有参与者（真人/角色 AI）可发送消息。
2. 主控 AI 可在任意时刻输出收束摘要。
3. 发起人可触发状态转换至 WAITING_CONFIRMATION。
4. 发起人可直接触发构建（跳过 WAITING_CONFIRMATION）。
5. 新参与者可加入（通过分享链接）。

**不允许操作：**

1. 不允许确认操作（确认只在 WAITING_CONFIRMATION 阶段有效）。

### 2.2 WAITING_CONFIRMATION（待确认）

**含义：** 主控 AI 已输出本轮收束摘要，等待参与者确认或发起人触发构建。

**可执行操作：**

1. 真人参与者可点击「确认」（非强制）。
2. 发起人可触发构建。
3. 发起人可撤回至 IN_PROGRESS（重新开放讨论）。
4. 角色 AI 可继续发言（补充分析）。
5. 真人参与者可继续发言（但发送消息后当前确认状态需评估是否需重新触发收束）。

> 待定：WAITING_CONFIRMATION 阶段真人继续发言是否自动回退到 IN_PROGRESS？

**不允许操作：**

1. 不允许新参与者加入（确认阶段锁定参与者名单）。

> 待定：是否允许新参与者在 WAITING_CONFIRMATION 阶段加入？

### 2.3 FINISHED（已结束）

**含义：** 讨论已锁定，不可继续修改或追加内容。

**可执行操作：**

1. 所有参与者可查看历史记录（只读）。
2. 发起人可查看结构化总结。

**不允许操作：**

1. 任何参与者（包括发起人）不可发送新消息。
2. 不可修改角色 AI 配置。
3. 不可撤回至任何其他状态（不可逆）。
4. 不可在此状态触发构建（构建只能在 IN_PROGRESS 或 WAITING_CONFIRMATION 触发）。

## 三、状态转换规则

### 3.1 转换路径

```
IN_PROGRESS
    │
    ├──[主控AI触发收束 / 发起人手动触发]──→ WAITING_CONFIRMATION
    │                                              │
    │                          ┌──[发起人撤回]──←─┤
    │                          ↓                  │
    │                    IN_PROGRESS        [发起人触发构建]
    │                                              │
    └──[发起人直接触发构建]──────────────────────→ FINISHED
                                                   ↑
                                      [构建完成/构建触发时自动]
```

### 3.2 转换详细规则

| 从 | 到 | 触发条件 | 发起方 |
|---|---|---|---|
| IN_PROGRESS | WAITING_CONFIRMATION | 主控 AI 判断收束时机成熟，或发起人手动触发 | 主控 AI / 发起人 |
| WAITING_CONFIRMATION | IN_PROGRESS | 发起人决定继续讨论，撤回确认阶段 | 发起人 |
| IN_PROGRESS | FINISHED | 发起人触发构建（跳过确认阶段） | 发起人 |
| WAITING_CONFIRMATION | FINISHED | 发起人触发构建 | 发起人 |

**禁止的转换：**

1. 任何状态 → IN_PROGRESS / WAITING_CONFIRMATION（从 FINISHED 回退）
2. 非发起人触发状态转换至 FINISHED。

### 3.3 不可逆约束

FINISHED 是终态，一旦进入不可退出。

实现要求：

1. 后端在写入 FINISHED 状态后，必须拒绝所有发言请求。
2. 前端在 FINISHED 状态下，输入框必须禁用，不可绕过。
3. 不提供"解锁讨论"的任何 API 或 UI 入口。

**设计意图：** 实现"沟通成本显性化"——每次关键决策都应当被一次独立的讨论组承载，不允许无限追加修改。

## 四、确认协议

### 4.1 确认的含义

参与者点击「确认」= 表达"我接受当前收束结论，不阻塞构建"。

确认不代表参与者同意结论中的所有内容，只代表"不否决"。

### 4.2 确认规则

1. 只有真人参与者可以确认（角色 AI 不参与确认）。
2. 非强制：不需要所有参与者确认才能触发构建。
3. 每个参与者只能对当前收束摘要确认一次（同一版本摘要）。
4. 如果发起人撤回至 IN_PROGRESS 后重新触发 WAITING_CONFIRMATION，上一轮确认状态清零。

### 4.3 确认状态记录

系统记录每个参与者的确认状态，包含：

```
{
  participant_id: string,
  confirmed: boolean,
  confirmed_at: timestamp | null,
  confirmation_version: string  // 对应哪一版收束摘要
}
```

### 4.4 触发构建时的确认快照

发起人触发构建时，系统生成确认状态快照，写入讨论结果记录：

```
{
  triggered_by: "发起人 ID",
  triggered_at: timestamp,
  confirmation_snapshot: [
    { participant: "A", confirmed: true, at: timestamp },
    { participant: "B", confirmed: false, at: null },
    ...
  ]
}
```

此快照不可事后修改，作为决策记录保留。

## 五、协作时序

### 5.1 典型完整讨论时序

```
t0  发起人创建讨论组（IN_PROGRESS）
t1  发起人配置角色 AI（可选）
t2  发起人邀请真人参与者（可选）
t3  讨论开始，各方发言
    - 真人 A 发言
    - 角色 AI_1（产品经理视角）回应
    - 真人 B 发言
    - 角色 AI_2（技术视角）回应
    ...
t4  主控 AI 触发收束（或发起人手动触发）
    → 状态转换为 WAITING_CONFIRMATION
    → 主控 AI 输出收束摘要
t5  参与者查看收束摘要
    - 真人 A 点击「确认」
    - 真人 B 未确认（表示保留意见）
t6  发起人确认已收集足够意见
    → 点击「构建」
    → 状态转换为 FINISHED
    → 主控 AI 生成 Build Outline
    → 进入构建执行阶段
t7  构建执行中（后台运行）
t8  构建完成，Project Object 写入 Workspace
t9  通知所有参与者构建完成
```

### 5.2 发起人直接触发构建时序（跳过确认）

```
t0  发起人创建讨论组（IN_PROGRESS）
t1-t3  讨论阶段
t4  发起人认为讨论已足够，直接触发构建
    → 状态直接转换为 FINISHED（跳过 WAITING_CONFIRMATION）
    → 主控 AI 生成 Build Outline
    → 进入构建执行阶段
```

### 5.3 撤回时序

```
t4  主控 AI 触发收束（WAITING_CONFIRMATION）
t5  发起人认为讨论还不充分
    → 点击「继续讨论」
    → 状态撤回至 IN_PROGRESS
    → 上一轮确认状态清零
t6  继续讨论...
t7  再次触发收束...
```

## 六、并发与边界场景

### 6.1 多人同时发言

讨论组支持多人同时发言，消息按服务器接收时间排序展示。

> 待定：消息顺序冲突处理机制（是否需要乐观锁？）

### 6.2 角色 AI 响应延迟

角色 AI 响应期间，用户可继续发言，不需要等待 AI 响应完成。

### 6.3 主控 AI 收束期间新消息

主控 AI 正在生成收束摘要时，新消息可以继续进入消息流。

主控 AI 输出的收束摘要基于触发时刻的历史内容，不包含触发后的新消息。
