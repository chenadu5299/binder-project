## 一、AI 功能架构概述

### 1.1 三层架构设计

Binder 的 AI 功能采用**三层独立架构**，每层有明确的职责和边界，确保功能互不干扰、逻辑清晰、体验流畅。

```
┌─────────────────────────────────────────────────────────┐
│  层次一：辅助续写                                          │
│  - 悬浮卡展示 3 条续写建议                   │
│  - 快捷键触发（Cmd+J / Ctrl+J）                              │
│  - 单次构建：续写指令 + 上下文，无意图识别、无工具              │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  层次二：局部修改（Cmd+K 快捷键）                          │
│  - 弹窗内多轮对话，细化编辑需求                              │
│  - 支持无选区调出（以光标所在块为上下文）                      │
│  - 弹窗内不做 Diff；原文档只显示选区                          │
│  - 无对话历史持久化（每次覆盖上一次回复）                       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  层次三：对话编辑                                          │
│  - 完整聊天界面，Agent/Chat 双模式                           │
│  - 标签栏、记忆库、模型选择                                 │
│  - 工具调用、Diff 卡片 + 文档侧删除标记                       │
│  - 多层提示词（系统 + 上下文 + 引用 + 工具定义）               │
└─────────────────────────────────────────────────────────┘
```

### 1.2 双模式

| 模式 | 定位 | 典型场景 | 当前状态 |
|------|------|----------|----------|
| **编辑模式** | 交互式 AI 辅助编辑与微构建 | 写作、修订、增量创建 | 已实现 |
| **构建模式** | 自动化多智能体项目构建 | 生成完整报告、书籍、知识库 | 未实现；Chat 模式为前身 |

### 1.3 架构设计原则

1. **独立性原则**：每层功能独立实现，不相互依赖
2. **清晰性原则**：每层有明确的职责和边界
3. **共享性原则**：共享上下文数据和配置，但不共享实现逻辑
4. **性能原则**：根据层次选择合适的模型和响应方式

## 二、层次一：辅助续写

### 2.1 功能定位

**核心特性**：
- ✅ **悬浮卡展示**：不占用面板空间，在光标位置弹出悬浮卡
- ✅ **3 条续写建议**：AI 一次返回 3 条，用户 Tab/Shift+Tab 切换，Enter 应用
- ✅ **快捷键触发**：Cmd+J / Ctrl+J，用户主动触发
- ✅ **非侵入式**：不打断用户写作流程

### 2.2 触发机制

**触发方式**：快捷键 Cmd+J（macOS）/ Ctrl+J（Windows/Linux）

**触发条件**：
- **须光标激活**：编辑器聚焦时有效；**光标未激活时快捷键无效**
- **光标无效提示**：未聚焦时按 Cmd+J，窗口底部状态栏「保存状态处」滚动显示「指令无效」提示，2 秒后清除
- **上下文不足提示**：当上下文为空（无字符）时，窗口底部状态栏「保存状态处」滚动显示「指令无效」提示，2 秒后清除
- 光标在文档中
- **上下文条件**：只要有字符且不为空即可（不再限定最低字符数）
- 光标位置不是文档末尾（末尾不续写）
- **可选中文本**：无选中时在光标后续写；有选中时按选中逻辑处理（见下）

**触发逻辑**：
- 监听快捷键事件
- 用户按下快捷键后立即调用 AI 生成续写
- 显示悬浮卡（3 条建议）

**有选中文本时的处理逻辑**：
- **上下文**：从选中字符**之前**提取内容作为续写上下文
- **用户确认**：应用当前选中项 → 直接**替换**选中文本
- **取消**：Esc 或关闭 → 不做任何修改

### 2.3 UI 实现

**悬浮卡展示**：
- **位置**：紧贴光标下方，向右展开；空间不足时向左展开
- **布局**：`position: fixed` 或 `position: absolute`，不参与文档流，零重排
- **内容**：3 条续写建议，每条可点击或 Tab+Enter 应用

**键盘导航**：
| 按键 | 行为 |
|------|------|
| **Tab** | 循环选中下一条（高亮当前项） |
| **Shift+Tab** | 循环选中上一条 |
| **Enter** | 应用当前选中项（无选中时插入到光标后；有选中时替换选中文本） |
| **Esc** | 关闭悬浮卡，不应用（有选中时不做任何修改） |

**关闭与重构**：
- **关闭**：执行与悬浮卡无关的任何操作均直接关闭，包括：手动输入、鼠标点击空白区域、保存、缩放窗口等
- **重构**：悬浮卡已显示时再次按 Cmd+J，直接重构（重新请求并展示新建议）

**边界情况**：
- 光标靠近底部：悬浮卡向上展开
- 光标靠近左侧：悬浮卡向右展开
- 光标靠近右侧：悬浮卡向左展开

### 2.4 上下文收集

**上文（context_before）**：
- **无选中时**：光标前字符，**无最低限制**，**最高 500 字符**
- **有选中时**：从选中字符**之前**提取，同上（无最低、最高 500）
- **包含格式**：提取的上文不仅包含内容，还包含**内容格式**（字体、字号、颜色等）；AI 处理内容时需**匹配格式**

**下文（context_after）**：光标后 N 字符（可选）

**其他**：
- 当前块类型、文档格式（document_overview）
- **不接入**记忆库、知识库、模板库

**AI 意图识别**：AI 收到上文后，先判断用户意图，包括但不限于：续写内容、组词、完成句子、如上文是标题则生成副标题/目录等

### 2.5 后端实现

**Rust 后端接口**：
- 命令：`ai_autocomplete`
- 参数：`context`（上下文）、`position`（位置）、`max_length`（最大长度，1-80 字符）
- 返回：**3 条续写建议**（JSON 数组或约定格式）

**性能要求**：
- 响应时间：< 2 秒（从触发到显示续写内容）
- 不阻塞编辑器操作
- **生成长度**：1-100 字
- **模型**：优先使用快速模型；若未设置快速模型，则使用默认（推荐）模型
- 快捷键触发（无自动触发，用户主动控制）

### 2.6 边界和限制

**不共享的内容**：
- ❌ 不共享聊天历史
- ❌ 不共享对话上下文
- ❌ 不显示在聊天窗口中

**共享的内容**：
- ✅ 当前文档内容（作为上下文）
- ✅ AI 模型配置（优先使用快速模型；未设置则用默认模型）

**系统提示词**：辅助续写使用**单独一套**系统提示词（不与对话编辑共享）。因场景差异大：无工具、无引用、无多轮；需意图识别（续写/组词/完成句子/标题→副标题或目录等）；需格式匹配要求。

## 三、层次二：局部修改（Cmd+K）

### 3.1 功能定位

**核心特性**：
- ✅ **弹窗**：非聊天窗口，是浮动的弹窗
- ✅ **快捷键调出**：Cmd+K / Ctrl+K
- ✅ **支持无选区调出**：无选区时以光标所在块为上下文
- ✅ **弹窗内多轮对话**：用户可反复说明或调整编辑需求，直至满意后确认应用
- ✅ **无对话历史**：每次交互覆盖上一次回复；只回复可直接插入文档的内容
- ✅ **弹窗内不做 Diff**：原文档只显示选区，弹窗内容为对话与可插入内容，无红绿高亮

**类比**：
- 类似 VS Code 的 Quick Fix（Cmd+.）
- 类似 Cursor 的 Cmd+K 功能
- 类似 Notion 的 / 命令

### 3.2 触发机制

**触发条件**：
- **须光标激活**：编辑器聚焦时有效；**光标未激活时快捷键无效**
- **光标无效提示**：未聚焦时按 Cmd+K，状态栏滚动显示「指令无效」
- 用户可选**选中文本**，也可不选（无选区时以光标所在块为上下文）
- 按下 Cmd+K（macOS）或 Ctrl+K（Windows/Linux）

**触发逻辑**：
- 监听 Cmd+K 快捷键
- 获取选中的文本（如果有）或光标所在块
- 显示弹窗
- 用户输入指令后执行

### 3.3 弹窗交互

**弹窗内容**：
- 单行输入框 + 执行按钮
- 多轮对话：用户输入指令 → AI 返回 → 用户可继续追问调整 → 循环
- **多轮提示词**：每轮需把**上轮 AI 回复内容**作为提示词构建的一部分传入下一轮
- 应用按钮：点击直接应用修改结果

**关闭**：
- 点击关闭按钮
- 弹窗已弹出时再次 Cmd+K
- 点击应用
- 其他情况不关闭

**弹窗缩放**：
- 弹窗根据**编辑窗口显示比例**缩放（如 100%、125% 等）
- 用户**拖动窗口边缘**进行缩放时，弹窗**不随之改变比例**，可被窗口边缘覆盖

### 3.4 后端实现

**Rust 后端接口**：
- 命令：`ai_inline_assist`
- 参数：`instruction`（指令）、`text`（选中文本，无选区时可为空或块全文）、`context`（上下文）、`messages`（对话历史，格式 `[{ role, text }]`，**须包含上轮 AI 回复**以支持多轮提示词构建）
- 返回：修改后的文本

**性能要求**：
- 响应时间：< 5 秒（从提交到显示结果）
- 支持流式显示（逐步显示 AI 生成内容）

### 3.5 边界和限制

**不共享的内容**：
- ❌ 不共享聊天历史
- ❌ 不显示在聊天窗口中
- ❌ 不保存对话记录

**共享的内容**：
- ✅ 当前文档内容（作为上下文）
- ✅ 选中文本（作为输入）
- ✅ AI 模型配置
- **不接入**记忆库、知识库、模板库

**系统提示词**：局部修改使用**单独一套**系统提示词（不与对话编辑、辅助续写共享）。因场景差异：弹窗内多轮、无工具、无引用、无历史；需将上轮回复纳入提示词构建。

**与层次一的区别**：
- 层次一（辅助续写）：单次续写，悬浮卡 3 选 1
- 层次二（局部修改）：弹窗内多轮对话，执行指令

**与层次三的区别**：
- 层次二（局部修改）：单次操作，无历史；弹窗内无工具调用
- 层次三（对话编辑）：对话式，有历史；可调用工具

## 四、层次三：对话编辑

### 4.1 功能定位

**核心特性**：
- ✅ **完整聊天界面**：有消息列表、输入框、历史记录
- ✅ **Agent/Chat 双模式**：Agent 可调用工具；Chat 纯对话
- ✅ **工具调用**：可以操作文档、文件等
- ✅ **标签栏**：支持多个聊天会话
- ✅ **与记忆库、知识库、模板库交互**：AI 可检索记忆库、知识库；可引用模板库；三者均支持
- ✅ **模型选择**：可以切换不同的 AI 模型
- ✅ **Diff 卡片 + 文档侧删除标记**：聊天侧展示 Diff 卡片，文档侧只显示红色删除标记

### 4.2 聊天模式

**Agent 模式**：
- AI 可以调用工具执行操作（read_file、create_file、edit_current_editor_document 等）
- 适用于需要 AI 执行实际操作的场景（如创建文件、修改文档、搜索内容等）
- 工具调用过程对用户可见
- **edit_current_editor_document**（修改部分内容）：新生成内容**仅在聊天窗口**出现，展示 Diff 效果；用户接受/拒绝后执行精准替换
- **create_file**（AI 直接创建文件）：新创建的文件**直接填充**创建的内容；聊天窗口出现**创建记录**，显示创建状态（成功 ✓ / 失败 ✗）

**Chat 模式**：
- AI 仅进行对话，不调用工具
- 适用于纯对话场景（如咨询问题、获取建议、内容讨论等）
- 不执行任何文件系统操作

**模式切换规则**：
- 创建新对话时可以选择模式（默认 Agent 模式）
- 聊天开始后（已有消息）不能切换模式（保持对话一致性）
- 每个标签页独立设置模式

### 4.3 引用系统功能

**支持引用类型**：
- **文本引用**：复制文档内容，自动识别 blockId+offset；可编辑时注入 edit_target
- **文件引用**：拖拽文件到聊天窗口，或使用 `@文件名` 语法
- **文件夹引用**：拖拽文件夹到聊天窗口
- **图片引用**：拖拽图片到聊天窗口
- **表格引用**：选中表格并引用
- **记忆库引用**：使用 `@记忆库名称` 语法；**支持选中某一条拖入**
- **聊天记录引用**：**仅当用户主动 @聊天名称** 时，才读取该标签的对话上下文；**默认不执行任何跨标签内容读取**
- **链接引用**：自动识别用户输入或复制的 URL
- **知识库引用**：引用知识库内容；**支持选中某一条拖入**
- **模板库引用**：引用模板库内容；**支持选中某一条拖入**

**三种引用用途**：
| 用途 | 场景 | AI 行为 |
|------|------|---------|
| **可编辑** | TextReference，需修改文档 | 调用 edit_current_editor_document；blockId+offset 注入 edit_target；全文/多处置换等建议显式 `edit_mode`+`scope`（见方案文档） |
| **只读** | 内容已完整传入 | 直接使用，无需 read_file；标注「无需再读」 |
| **信息获取** | 图片、链接等 | 需获取内容再传入；占位或实时拉取 |

**引用功能**（详见引用功能完整设计文档）：
- 在聊天输入中插入引用标签；消息记录内以标签形式展示，不展开为完整内容
- 引用 content 写入 references 数组传给 AI；path 可信，prompt 统一声明「路径已解析，可直接使用」
- 前端需将 references 传递给 ai_chat_stream（当前待实现）；引用层只传内容，意图判断交给 AI

### 4.4 工具调用功能（仅 Agent 模式）

**文件操作工具**：
- `read_file`、`create_file`、`update_file`、`delete_file`
- `list_files`、`search_files`、`move_file`、`rename_file`
- `create_folder`

**编辑器操作工具**：
- `edit_current_editor_document`：编辑当前打开的文档
  - **统一协议**：以 `block_index + edit_mode + target + content + occurrence_index` 为主；精确定位输入由系统注入，不再使用旧 `scope` / `target_content` / `instruction` 作为主协议
  - **返回格式**：canonical diffs 数组，含 `diffId`、`startBlockId`、`endBlockId`、`startOffset`、`endOffset`、`originalText`、`newText`、`type`、`diff_type`
  - **定位**：有精确区间坐标时直接走零搜索；无精确坐标时走块列表和 `block_index`
  - **文档逻辑状态**：AI 修改前需传入目标文档当前逻辑状态 `L`；pending diff 不进入逻辑状态

**工具调用特性**：
- 文件操作（read_file、create_file、update_file 等）：自动执行；create_file 成功后聊天窗口显示创建记录（成功 ✓ / 失败 ✗）
- 编辑器操作（edit_current_editor_document）：需用户确认（Diff 卡片接受/拒绝）
- 实时显示工具调用状态（pending/executing/completed/failed）

**工具完整性说明**（参考《工具集扩展规范》）：当前工具集已覆盖文件读写、目录操作、编辑器编辑；与方案 ATP 的 list_documents、get_project_summary、read_block 等为增强能力，可后续扩展。**Agent 模式仍需工具调用**：read_file 读取未打开文件、create_file 创建新文件、edit_current_editor_document 编辑当前文档等，均依赖工具。

### 4.5 对话编辑 Diff 方案

**双状态模型**：
| 状态 | 说明 |
|------|------|
| **文档显示状态** | 带红色删除标记的视觉层。用户可见，但不代表最终内容。 |
| **文档逻辑状态** | 只有被明确「接受」的修改才会写入。AI 下次修改以此为基准。 |

**聊天侧 Diff 卡片**：
- 每张卡片：删除区（红色删除线）+ 新增区（绿色背景）+ 接受/拒绝按钮
- 底部全局操作栏：「全部接受」「全部拒绝」
- **全部接受**：倒序执行（最新卡片优先），避免偏移漂移
- **originalText 校验**：执行前校验区间内容未变；失败时标记已过期

**文档侧**：
- 只承载红色删除标记（#FCEBEB 背景 + 删除线），不插入新增内容，版式零影响
- 接受后：精准替换（findBlockByBlockId + blockOffsetToPMRange + tr.replaceWith）

**块 ID 稳定性**：
- blockId 100% 不跨会话继承（DOCX 不保存 data-block-id）
- 单会话内 blockId 有效；跨会话未接受的 diff 标记过期

**含未应用 diff 时的关闭保护**：
- 当存在未接受的 diff 时，用户**关闭含 diff 的文档**：弹出提示「是否关闭？」，提供**确认**（关闭）和**取消**（不关闭）选项
- 用户**关闭应用**时：同样检测是否有未接受的 diff，若有则弹出上述提示

### 4.6 多层提示词架构

**提示词层次**：

1. **第一层：基础系统提示词**
   - 角色定义、行为规范
   - **意图识别原则**：嵌入系统提示词，由 AI 在单次调用内自行判断；无显式两阶段
   - 工具调用规范（仅 Agent 模式）

2. **第二层：上下文提示词**
   - 当前打开的文档：文件路径、文档内容（或逻辑状态）
   - 当前选中的文本
   - 工作区路径
   - 编辑器状态：是否可编辑、文件类型、是否已保存、文档结构（含 block 类型与 blockId，用于「修改第十段」等自然语言定位）

3. **第三层：引用提示词**
   - 用户引用的内容列表（references 由前端传递）
   - 每个引用的类型、来源、完整内容
   - 明确说明：「以上内容已完整传入，无需再读取」；含 File 时追加「以上文件路径均已解析，可直接使用，无需 list_files 或 search_files」

4. **第四层：工具定义**（仅 Agent 模式）
   - 工具名称、参数、调用格式（JSON Schema）

**提示词构建逻辑**：
- 每轮用户消息：构建一次完整提示词
- 每次工具调用后：将工具结果加入 messages，再次调用 AI，相当于「重新构建」下一轮提示词

### 4.7 Chat 模式定位（无编辑功能）

**功能定位**：
- Chat 模式**不再兼顾编辑功能**，仅做**智能聊天**
- 后期将开发**构建功能**：用户确认构建后，执行完整工作流构建项目；中途用户**只可打断**，**不可修改编辑**
- **Chat 模式的编辑支持全部删除**（如「快捷应用到文档」等）

### 4.8 记忆库功能

**记忆库标签**：
- 在标签栏或资源管理区添加记忆库标签（图标：🧠）
- 点击切换到记忆库面板
- 显示记忆项列表和详情

**记忆库、知识库、模板库与 AI 交互**：
- 辅助续写：**不接入**
- 局部修改：**不接入**
- 对话编辑：可与记忆库、知识库、模板库交互；支持 @ 引用与选中拖入

### 4.9 后端实现

**Rust 后端接口**：
- 命令：`ai_chat_stream`
- 参数：`tab_id`、`messages`、`model_config`、`enable_tools`、`current_file`、`selected_text`、`current_editor_content`、`edit_target`、`references`（待增加；格式见引用功能完整设计文档 6.1）
- 返回：流式响应（通过事件 `ai-chat-stream` 发送）

**流式响应处理**：
- 前端和后端双重去重
- 累积文本跟踪
- 空 chunk 过滤
- 工具调用实时处理

### 4.10 性能要求

**响应时间**：
- 首次响应：< 3 秒
- 流式显示延迟：< 100ms
- 工具调用响应：< 2 秒

### 4.11 边界和限制

**共享的内容**：
- ✅ 当前文档内容（作为上下文）
- ✅ 选中文本（可以引用）
- ✅ 系统提示词
- ✅ AI 模型配置
- ✅ 记忆库数据
- ✅ 引用内容

**不共享的内容**：
- ❌ 不共享辅助续写的触发逻辑
- ❌ 不共享局部修改的弹窗

## 五、AI 与其他功能交互

### 5.1 记忆库

**交互规范**：
- **接口**：`search_memories(query, workspace_path)` → 记忆项列表
- **各层次接入**：**仅对话编辑**接入；辅助续写、局部修改不接入
- **拼接方式**：作为「记忆库信息」段落，插入上下文；明确标注来源
- **降级**：检索失败或超时静默返回空，不阻塞主流程

### 5.2 引用资源

**交互规范**：
- **可编辑**：TextReference 含 blockId 时，与 edit_target 映射
- **只读**：内容完整传入，标注「无需再读」
- **信息获取**：图片、链接等需获取内容再传入；超时策略
- **传递**：前端需将 references 传递给 ai_chat_stream（当前待实现）

### 5.3 知识库

**交互规范**：
- **接口**：`query_knowledge_base(kbId, query)` → 检索结果（预定义协议）
- **与引用**：用户 @知识库 时走引用；AI 自动检索时走上下文注入
- **状态**：未实现，协议先定

### 5.4 模板库

**三类模板**：文档模板、工作流模板、**Skills**

**交互规范**：
- **工作流模板**：Plan JSON 结构（id、name、description、source、weighted_score、steps）；从真实执行中涌现（加权分 ≥ 阈值或 memory_save 时询问保存）；对话中 `@` 引用或 AI 自动调起；无独立执行入口，AI 按 steps 注入对话上下文并执行
- **Skills**：`{ id, name, content, scene?, requiresAuth? }`；**纯自然语言 content**，无 steps/tools 结构化字段，AI 直接消费；系统 Skills（只读）/ 用户 Skills（可编辑）
- **与 AI**：用户 @ 引用或拖拽 → 作为 references 传入；对话开始前选择「应用 Skill」→ 整个对话持续生效
- **状态**：未实现，详见 Binder模板库需求文档

## 六、AI 文档格式输出

### 6.1 问题

当前 Diff 格式中 `newText` 为**纯文本**，若不做约定，AI 输出的文档大概率是**不分页的纯文本**，无字体、颜色、字号等格式。

### 6.2 需定义的协议

| 协议项 | 内容 |
|--------|------|
| **newText 格式** | 纯文本 vs 简单 HTML 子集（如 `<strong>`, `<span style="...">`）vs Markdown |
| **字体白名单** | AI 可使用的字体列表；必须指定，避免系统不存在的字体 |
| **字号范围** | 10/12/14/16/18/20/24/28/32/36 px |
| **颜色格式** | 十六进制 `#RRGGBB` |

### 6.3 方案选项

- **A. 纯文本 MVP**：Phase 1 保持纯文本，文档化限制
- **B. 简单 HTML 子集**：允许 `<strong>`, `<em>`, `<span style="...">`；前端解析为 ProseMirror 节点（推荐）
- **C. Markdown 转换**：`newText` 为 Markdown，前端解析后应用

## 八、AI 模型管理

### 8.1 模型提供商支持

**支持的提供商**：
- **OpenAI**：GPT-4、GPT-4 Turbo、GPT-3.5 Turbo 等

- **本地模型**：Ollama、vLLM、LM Studio（未来支持）

**模型选择策略**：
- 辅助续写：优先使用快速模型；未设置快速模型则使用默认（推荐）模型
- 局部修改：使用标准模型（GPT-4、Claude Sonnet 等）
- 对话编辑：用户可选择模型（默认 GPT-4 或 DeepSeek Chat）

### 8.2 模型配置

**配置选项**：
- API Key 配置（用户可配置各提供商的 API Key）
- 模型选择（用户可为不同功能选择不同模型）
- 请求超时配置（默认 60 秒）
- 温度参数配置（默认 0.7）
- 最大 Token 配置（默认 4096）

**API Key 安全存储**：
- 加密存储，不记录日志
- 支持密钥链存储（macOS Keychain、Windows Credential Manager）
- 支持环境变量配置（开发环境）

### 8.3 AI 请求管理

**请求队列**：
- 支持请求队列（管理多个 AI 请求）
- 支持请求优先级：局部修改 > 聊天 > 辅助续写
- 支持请求取消（用户可取消正在进行的请求）
- **请求重试机制**：网络错误、超时、5xx 自动重试；4xx、用户取消不重试；静默重试，不提示用户

## 十二、术语表

### 12.1 功能术语

**辅助续写（Auto-Complete）**：层次一，快捷键 Cmd+J 触发，悬浮卡展示 3 条续写建议（幽灵文字已放弃）

**局部修改（Inline Assist）**：层次二，快捷键 Cmd+K 触发，弹窗内多轮对话，执行指令

**对话编辑（Chat Window）**：层次三，独立聊天界面，Agent/Chat 双模式，工具调用，Diff 卡片

**Agent 模式**：AI 可调用工具执行操作
**Chat 模式**：AI 仅进行对话，不调用工具；**不兼顾编辑功能**，纯智能聊天；编辑支持已全部删除

**引用系统**：支持文本、文件、文件夹、图片、记忆库、链接、聊天记录、知识库、模板库等引用类型；记忆库、知识库、模板库支持选中某一条拖入

**文档逻辑状态**：只有被明确「接受」的修改才会写入；AI 下次修改以此为基准

**文档显示状态**：带红色删除标记的视觉层。用户可见，但不代表最终内容。用户沉默 ≠ 接受。Diff 卡片未被明确操作前，文档逻辑状态不变，AI 的下一次修改始终以逻辑状态为基准。

**多层提示词架构**：系统提示词 + 上下文 + 引用 + 工具定义（仅 Agent）

**双状态模型**：文档显示状态与文档逻辑状态分离，确保时序安全

### 12.2 技术术语

**blockId+offset**：块级定位，blockId 为块唯一 ID，offset 为块内字符偏移量；DOCX 不保存 blockId，100% 不跨会话继承

**API Key**：访问 AI 服务的密钥，需用户配置，加密存储

**流式响应**：AI 响应逐步返回，而不是一次性返回

**工具调用**：AI 调用外部工具执行操作（如文件操作、edit_current_editor_document）
