## 一、文档定位

本文是 Binder 知识库模块的补充文档。

本文回答：

1. “结构型文档样本”在 Binder 中应被如何定义。
2. 为什么它更适合进入知识库，而不是继续作为模板库主对象。
3. 结构型知识资产与普通知识文档、工作流模板、上下文注入的边界是什么。
4. 知识库对象模型、检索语义与运行时注入语义需要补哪些扩展位，才能合法承接这类资产。

本文不回答：

1. 工作流模板对象如何设计或执行。
2. 模板编辑器、变量替换器、槽位执行器如何实现。
3. 独立模板库产品如何重做。
4. 具体数据库 schema、排序算法或 UI 交互细节。

本文与现有文档的关系是：

1. `A-AST-M-D-03_Binder知识库模块描述文档.md` 仍定义知识库主定位。
2. `A-TMP-M-D-01_Binder Agent模板协同主控文档.md` 与 `A-TMP-M-T-01_模板机制.md` 仍主控工作流模板。
3. 本文只补充“结构型文档样本作为知识库中的结构型资产”这条能力，不新建平行系统。

## 二、问题背景

当前 Binder 中，“文档模板”若继续作为模板库对象理解，容易出现混层：

1. 一部分所谓“模板”本质上并不提供执行步骤，而是提供一份常用文档的结构样本、栏目组织或排布范本。
2. 这类对象的用户心智更接近“我之前写过一份很好用的文档样本”，而不是“请按某个工作流来执行”。
3. 若把它们继续放在模板库主链中，会冲淡模板库已经冻结的“工作流模板唯一有效类型”定位。
4. 若直接把它们当作普通知识文档，又会丢失“这是结构参考，不是事实证据”的运行时语义。

因此需要补一条明确结论：

**结构型文档样本不应继续作为模板库主对象；它更适合被定义为知识库中的结构型知识资产。**

## 三、新增定位：结构型知识资产

### 3.1 定义

本文统一使用术语：

**结构型知识资产**

它的常见用户名称可以是：

1. 结构型文档样本
2. 文档范本资产
3. 结构参考资产

结构型知识资产的本质是：

**被用户长期保存于知识库中的、可检索、可引用、可注入的文档结构样本。**

本文的系统级否决句是：

**结构型知识资产仅用于提供文档结构参考，不得参与事实知识补强、证据链构建或引用体系。**

它提供的核心价值不是事实结论，而是：

1. 栏目怎么分。
2. 段落怎么排。
3. 固定槽位怎么组织。
4. 一类文档通常由哪些结构块组成。

### 3.2 它不是什么

结构型知识资产不是：

1. 传统模板引擎对象。
2. 工作流模板。
3. 事实证据型知识文档。
4. 变量替换系统。
5. 强执行槽位绑定器。

### 3.3 为什么它属于知识库

它更适合进入知识库，而不是继续作为模板库主对象，原因是：

1. 用户管理它的心智更像“文档资产管理”，而不是“过程约束管理”。
2. 用户调用它的方式更像“参考我之前那份文档”，而不是“按某个步骤编排任务”。
3. 它的核心价值在于结构召回、样本匹配与结构参考，而不是执行过程控制。
4. 它天然具备知识资产特征：可命名、可分类、可检索、可引用、可保留来源。

### 3.4 它为什么不是普通知识文档

它虽然属于知识资产，但其运行时职责与普通知识文档不同：

1. 普通知识文档主要提供事实、内容、证据、背景。
2. 结构型知识资产主要提供结构、栏目、排布、槽位、组织方式。
3. 二者可共存于同一知识库对象模型中，但注入语义必须区分。

## 四、与其他系统的边界

### 4.1 与普通知识文档的边界

结构型知识资产与普通知识文档：

1. 同属知识库。
2. 同属用户主导的文档级知识资产。
3. 同走统一对象模型、统一 ingestion pipeline、统一版本与溯源链。

但二者运行时语义不同：

1. 普通知识文档用于事实补强、证据召回、引用核验。
2. 结构型知识资产用于结构参考、排布参考、栏目参考。
3. 结构型知识资产不得伪装成事实证据来源。

### 4.2 与工作流模板的边界

工作流模板仍然留在模板库，且模板库仍以工作流模板为唯一有效类型。

边界如下：

1. 工作流模板是过程资产，回答“任务怎么推进”。
2. 结构型知识资产是文档资产，回答“文档结构可参考什么样子”。
3. 前者影响执行步骤、阶段顺序、依赖关系。
4. 后者影响文档组织、栏目框架、结构排布。

因此：

1. 工作流模板不得并入知识库。
2. 结构型知识资产不得偷换成模板执行引擎对象。
3. 模板库不应继续把“结构型文档样本”作为主对象长期保留。

### 4.3 与 Agent 的关系

Agent 可以消费结构型知识资产，但它消费的是：

1. 结构参考
2. 栏目组织方式
3. 文档样本的布局线索

Agent 不应把结构型知识资产理解为：

1. 已成立事实
2. 强制执行计划
3. 独立流程引擎

### 4.4 与上下文注入的关系

结构型知识资产可以进入上下文，但必须作为：

**结构参考注入**

而不是：

1. 事实知识注入
2. 工作流模板注入
3. 当前文档事实替代品

它在上下文中的职责是：

1. 告诉模型“可参考哪种文档结构”。
2. 提供栏目与排布样本。
3. 为生成或改写文档提供结构性约束线索。

## 五、对知识库对象模型的补充

### 5.1 补充原则

结构型知识资产不引入新的第六对象。

它仍然通过现有五对象主链承接：

1. `KnowledgeBase`
2. `KnowledgeFolder`
3. `KnowledgeEntry`
4. `KnowledgeDocument`
5. `KnowledgeChunk`

需要补的是：

1. 资产类型标记
2. 结构语义 metadata
3. 结构检索与结构注入的行为分歧

### 5.2 强制资产类型标记

知识库实现中，必须存在稳定资产类型标识位，用于区分：

1. 普通知识资产
2. 结构型知识资产

字段名可以采用：

1. `entry_kind = standard | structure_asset`
2. 或等价的 `asset_kind = knowledge | structure_asset`

但无论字段名如何，以下约束是强制的：

1. ingestion 时必须写入该类型位，不允许以“默认普通文档”兜底。
2. query 时必须读取并参与检索分流，不允许只在展示层使用。
3. injection 时必须读取并决定注入语义，不允许与普通知识片段混用。
4. 自动检索时必须读取并参与过滤，不允许统一召回后再弱裁剪。
5. UI 必须暴露该类型，不允许把结构型资产展示成普通事实知识条目。

本文冻结的不是具体字段名，而是：

1. 系统必须能稳定识别“这是一条普通知识资产”还是“这是一条结构型知识资产”。
2. 该识别必须进入检索策略、注入语义、自动检索过滤与 UI 展示。
3. 若缺少这一稳定类型位，则不得宣称系统已经承接结构型知识资产。

### 5.3 建议新增的 metadata

结构型知识资产至少需要以下 metadata 语义位：

1. `document_form`
   - 例如：周报、研究纪要、方案说明、复盘、需求分析
2. `structure_purpose`
   - 例如：生成初稿、改写重组、栏目重排、标准化输出
3. `applicable_scenarios`
   - 说明适用任务场景
4. `section_outline_summary`
   - 结构摘要，而不是事实摘要
5. `slot_hints`
   - 固定栏目、必填段、推荐槽位等
6. `source_nature`
   - 标记它是“范本样本 / 历史样本 / 标准样例”

可选补充位包括：

1. `structure_tags`
2. `style_scope`
3. `usage_notes`
4. `sample_origin`

### 5.4 与普通 entry 的关系

结构型知识资产本质上仍是 `KnowledgeEntry` 的一种类型，不需要单独新建平行对象系统。

这意味着：

1. 它仍然有 `entry_id`、`document_id`、`active_version`、`provenance`。
2. 它仍然支持 replace、delete、move。
3. 它不进入普通知识 citation 主链。
4. 它只是在“怎么被检索、怎么被注入、怎么被理解”上与普通 entry 分化。

## 六、检索语义补充

### 6.1 两类检索目标必须区分

知识库在承接结构型知识资产后，至少存在两类检索目标：

1. 内容召回
2. 结构参考召回

二者不能继续被理解为同一检索意图。

这是强规则，不是排序建议：

1. 系统不得把 `standard` 与 `structure_asset` 统一召回后再交给模型自行理解。
2. 系统必须先按资产类型分流，再决定是否进入各自检索链。

### 6.2 普通知识检索

普通知识检索的目标是：

1. 找事实
2. 找依据
3. 找材料
4. 找可引用片段

排序更偏向：

1. 内容相关性
2. verification
3. provenance 完整度
4. active version 稳定性

### 6.3 结构型资产检索

结构型资产检索的目标是：

1. 找某种文档结构样本
2. 找相似的栏目组织方式
3. 找适合当前任务的范本文档
4. 找“按哪种结构来写”而不是“事实是什么”

排序更偏向：

1. 文档类型匹配
2. 结构用途匹配
3. 适用场景匹配
4. 结构摘要与用户意图的相似度

### 6.4 如何识别用户在找内容还是结构

至少应允许以下两类路径：

1. 用户显式选择“按结构参考检索”
2. 系统在明确结构性意图时触发结构参考检索

明确结构性意图的典型例子：

1. “按我之前那份周报结构来”
2. “参考之前的方案框架”
3. “给我一个类似上次文档的栏目组织”

### 6.5 如何避免结构型资产混入普通事实检索

以下规则为强执行规则：

1. 默认行为：普通检索默认不包含 `structure_asset`。
2. 强隔离：`structure_asset` 不得进入 citation 候选。
3. 自动检索触发条件：只有结构意图明确时，`structure_asset` 才允许进入自动检索候选。
4. 显式优先：`structure_asset` 默认只通过显式引用进入主链；未显式引用时，只有满足结构性任务条件才可作为受限补强对象进入。

进一步约束如下：

1. 若当前任务是证据、出处、对照、事实问答，`structure_asset` 必须被自动过滤。
2. 若 query intent 未显式区分“内容召回”与“结构参考召回”，实现层不得默认混查。
3. 不允许采用“统一召回 -> 再排序 -> 再由模型理解”的偷渡实现。

### 6.6 显式选择路径

知识库应允许用户在知识资产中显式选择某个结构型资产作为参考对象。

用户心智应该是：

1. 参考这份样本
2. 按这类结构来组织
3. 用它做结构参考，而不是拿它当事实依据

## 七、运行时注入语义补充

### 7.1 普通知识文档的注入语义

普通知识文档进入上下文，是为了：

1. 补事实
2. 补背景
3. 提供证据
4. 提供可定位出处

### 7.2 结构型知识资产的注入语义

结构型知识资产进入上下文，是为了：

1. 提供结构参考
2. 提供栏目参考
3. 提供排布参考
4. 提供固定槽位与组织顺序参考

它不是：

1. 已成立事实
2. 可直接替代当前文档事实的证据
3. 强执行模板

### 7.2.1 禁止性约束

以下约束为硬规则：

1. `structure_asset` 不得生成知识库 citation。
2. `structure_asset` 不得进入 evidence 链。
3. `structure_asset` 不得参与事实冲突判断。
4. `structure_asset` 不得以“已验证事实内容”身份进入上下文。

补充说明：

1. 若未来需要对结构样本做质量审核，只能形成“结构可用性 / 样本适配性”之类的治理标签。
2. 这种治理标签不得等同于普通知识文档中的事实 `verified` 语义，更不得提升其证据地位。

### 7.3 注入标签必须区分

结构型知识资产进入上下文时，必须带清晰来源标签，例如：

1. `source_role = structure_reference`
2. `reference_kind = structure_asset`

其 UI 或 prompt 标签语义应明确为：

1. 结构参考
2. 范本参考
3. 结构样本

而不是：

1. 事实来源
2. 证据片段
3. 已验证结论

### 7.4 与事实知识的优先级关系

结构型知识资产不能覆盖：

1. 当前文档事实
2. 用户显式引用的事实材料
3. 当前轮 artifact 中已经成立的任务状态

它只影响：

1. 文档如何组织
2. 输出如何分段
3. 结构如何参考

### 7.5 与 workflow template 的协同关系

若一个任务同时使用工作流模板和结构型知识资产：

1. 工作流模板负责过程约束。
2. 结构型知识资产负责文档结构参考。
3. 普通知识文档负责事实与内容补强。

三者不得被压成同一运行时语义层。

### 7.6 自动注入边界

默认规则：

1. `structure_asset` 默认不参与自动检索。
2. `structure_asset` 默认不参与自动注入。

只有满足以下例外条件时，才允许进入自动检索或自动注入候选：

1. 当前任务类型明确为生成、改写或结构重组。
2. 用户已显式引用某个结构型资产。
3. 用户消息或任务语义存在明确结构意图。

同时必须满足以下限制：

1. 即使进入候选，`structure_asset` 也只能以结构参考身份进入 augmentation，不得进入事实补强层。
2. 证据型、事实型、核验型任务中，`structure_asset` 必须被自动过滤。
3. 若结构意图不明确，系统必须默认阻断，而不是先召回再裁剪。

## 八、用户心智与管理动作

用户管理结构型知识资产时，心智不应是“我在配置模板引擎”。

正确心智应是：

1. 我把一份常用范本保存进知识库。
2. 我把一份好用的结构样本标记为结构参考资产。
3. 我在写作或编辑时调用它作为参考对象。

因此用户动作应更接近：

1. 保存进知识库
2. 标记为结构型资产
3. 增加适用场景标签
4. 在生成或编辑时显式引用
5. 替换版本、删除、移动、归类

而不是：

1. 编排模板执行图
2. 绑定变量替换逻辑
3. 配置执行器节点

## 九、MVP 范围

### 9.1 本期纳入

本期只补以下能力：

1. 结构型知识资产作为知识库中的特殊资产类型被定义清楚。
2. 其对象语义、检索语义、注入语义与边界被定义清楚。
3. 它可被标记、检索、显式选择并以结构参考方式注入。
4. 它与普通知识文档、工作流模板的边界被固定。

### 9.2 本期不纳入

本期明确不纳入：

1. 工作流模板并入知识库。
2. 复杂模板引擎。
3. 强槽位绑定执行器。
4. 变量替换模板系统。
5. skills 体系回归。
