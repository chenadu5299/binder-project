## 三、字段语义与 owner（MUST）

### 3.1 核心字段定义

1. `L`：本轮请求的逻辑态快照。
2. `baseline`：本批 diff 生成时绑定的基线内容。
3. `baselineId`：本轮请求链路唯一标识。
4. `revision`：真实内容版本号，仅真实写入后递增。
5. `logicalContent`：目标文档当前逻辑内容（不含 pending）。

### 3.2 跨子系统引用矩阵

| 字段 | owner | 生产点 | 消费点 | 刷新时机 | 禁止用法 |
|---|---|---|---|---|---|
| `L` | 状态协作层 | 请求发送前 ContextBuilder | Resolver/提示词/执行校验 | 每轮发送前 | 用展示态替代 `L` |
| `baseline` | Diff 生成层 | 本轮 diff 生成入口 | 单卡/批量执行校验 | 每轮新 diff 生成 | 当作全量失效开关 |
| `baselineId` | 请求链路层 | 请求发送前注入 | Resolver/执行/观测 | 每轮发送前 | 跨轮复用同值 |
| `revision` | 文档状态层 | 写入成功回调 | 请求构建/执行校验 | 真实写入后 | 因 revision 变化就全量失效 |
| `logicalContent` | 文档状态层 | 文档逻辑态计算 | `L` 生成/执行前比对 | 每轮请求前 | 回放 pending 生成逻辑态 |

## 四、时序协作规范（MUST）

### 4.1 发送前时序

MUST：
1. 先读取目标文档 `logicalContent`。
2. 基于 `logicalContent` 生成 `L`。
3. 生成新的 `baselineId`（每轮唯一）。
4. 携带当前 `revision` 发送请求。

### 4.2 生成 diff 时序

MUST：
1. diff 绑定当轮 `baseline/baselineId/revision`。
2. 同轮 diff 的状态语义一致，不得混用旧轮字段。
3. 生成失败条目进入暴露链，不改写其余条目状态。

### 4.3 执行前时序

MUST：
1. 执行前再次获取最新逻辑态用于 `originalText` 比对。
2. 比对失败按条目失败处理，不触发整轮回滚。
3. 成功条目继续执行并写入。

### 4.4 执行后时序

MUST：
1. 仅真实写入成功时推进 `revision`。
2. 下一轮请求必须重新生成 `L + baselineId`。
3. 禁止复用旧 `baselineId` 进行新轮编辑。

## 五、状态边界规则（MUST）

### 5.1 双状态模型

MUST：
1. 显示状态与逻辑状态必须隔离。
2. 显示状态服务 UI，逻辑状态服务后续编辑基准。
3. 任何一层不得以显示状态替代逻辑状态参与计算。

### 5.2 pending 语义

MUST：
1. pending 不进入逻辑状态。
2. 用户沉默不等于接受。
3. accepted/rejected/expired 才能推进业务状态机。

### 5.3 逻辑内容重建

MUST：
1. `getLogicalContent` 只作为重建能力。
2. 禁止把 pending diff 回放进逻辑内容。
3. 重建过程要可追溯输入版本与字段来源。

### 5.4 noop 与版本推进

MUST：
1. 内容无变化时不产出 diff。
2. 内容无变化时不推进 `revision`。
3. noop 场景必须可被观测与统计。

## 六、执行协作规则（MUST）

### 6.1 批执行一致性

MUST：
1. 批量执行遵循“先读后写 + 稳定排序”。
2. 执行前对区间关系进行冲突检查（无交集/完全包含）。
3. 条目级失败不阻断整批成功条目。

### 6.2 diff 池一致性

MUST：
1. 同一 diff 池跨轮/跨标签语义一致。
2. 不允许按 UI 分组拆出不同状态语义。
3. 状态更新必须以 diffId 为主键统一推进。

### 6.3 撤销隔离

MUST：
1. Diff 执行不进入编辑器撤销栈。
2. 编辑器撤销不反向改写 diff 池业务状态。
3. 撤销事件与 diff 状态变更必须分开记录。

## 七、异常场景处理（MUST）

1. `baselineId` 缺失：拒绝进入执行，返回结构化错误。
2. `L` 与目标文档不匹配：拒绝生成并暴露上下文错误。
3. `revision` 不一致：进入重读校验分支，不直接全量失效。
4. 部分条目校验失败：标记失败并继续其余条目。
5. 执行失败与失效：必须分离处理与展示。

建议错误码映射（来源：主控 §6.3）：
1. `baselineId` 缺失 -> `E_TARGET_NOT_READY` 或 `E_BASELINE_MISMATCH`
2. `L`/区间不可解析 -> `E_RANGE_UNRESOLVABLE`
3. `originalText` 比对失败 -> `E_ORIGINALTEXT_MISMATCH`
4. 非法重叠 -> `E_PARTIAL_OVERLAP`

## 八、测试与验收矩阵

### 8.1 功能验收

1. 连续多轮请求中 `baselineId` 每轮唯一。
2. `L` 始终来源于逻辑态而非展示态。
3. `revision` 仅在真实写入成功后递增。

### 8.2 一致性验收

1. pending 不影响下一轮逻辑编辑基准。
2. 单卡/批量执行的状态推进语义一致。
3. 不存在“revision 变化即全量 expired”的错误逻辑。

### 8.3 观测验收

1. `baselineId` 可贯穿请求、执行、暴露链追踪。
2. 条目级失败具备可定位信息。
3. 失效与失败事件可独立检索。
