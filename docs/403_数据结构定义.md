# 403_数据结构定义

## 一、文档定位

本文是工程实现层的数据结构统一主规范，定义对话编辑主链涉及的核心状态对象、区间对象、diff 对象与协作约束。  
本文承接逻辑状态规范与跨 Block diff 专项方案，补齐全系统数据结构层的统一定义。

---

## 二、MVP 目标

1. 建立统一状态对象：`L`、baseline、revision、diff 状态。  
2. 建立统一区间对象：单块/跨块共用锚点结构。  
3. 建立统一 diff 对象：生成、校验、执行、失效全链一致。  

---

## 三、核心对象总览

## 3.1 文档状态对象

1. `DocumentLogicalState`  
2. `BaselineSnapshot`  
3. `DocumentRevisionState`  

## 3.2 区间对象

1. `BlockRangeAnchor`（单块/跨块统一）  
2. `PMRange`（前端执行映射）  

## 3.3 Diff 对象

1. `CanonicalDiff`（后端输出标准）  
2. `DiffEntry`（前端存储标准）  
3. `DiffExecutionExposure`（执行观测对象）

## 四、状态结构定义

## 4.1 DocumentLogicalState

```ts
interface DocumentLogicalState {
  filePath: string
  contentHtml: string
  contentHash: string
  revision: number
  baselineId: string
  updatedAt: number
}
```

约束：

1. 只包含真实已生效内容。  
2. 不包含 pending diff 的假想结果。  

## 4.2 BaselineSnapshot

```ts
interface BaselineSnapshot {
  baselineId: string
  filePath: string
  revision: number
  contentHash: string
  capturedAt: number
}
```

约束：

1. baseline 只标识生成基线，不作为统一失效开关。  
2. baseline 随新一轮发送重建。  

## 4.3 DocumentRevisionState

```ts
interface DocumentRevisionState {
  filePath: string
  revision: number
  lastAppliedDiffId?: string
  updatedAt: number
}
```

约束：

1. 仅真实内容变化时递增。  
2. revision 变化不直接导致其他 pending diff 失效。

## 五、区间结构定义

## 5.1 BlockRangeAnchor（统一锚点）

```ts
interface BlockRangeAnchor {
  startBlockId: string
  startOffset: number
  endBlockId: string
  endOffset: number
}
```

规则：

1. 单块：`startBlockId === endBlockId`。  
2. 跨块：`startBlockId !== endBlockId`。  
3. offset 永远是各自块内纯文本字符偏移。  

## 5.2 PMRange（前端执行区间）

```ts
interface PMRange {
  from: number
  to: number
}
```

规则：

1. `BlockRangeAnchor -> PMRange` 映射失败即视为区间不可执行。  
2. 跨块仍映射为一个连续 PM 区间。

## 六、Diff 结构定义

## 6.1 CanonicalDiff（后端标准）

```ts
interface CanonicalDiff extends BlockRangeAnchor {
  diffId: string
  originalText: string
  newText: string
  type: 'replace' | 'delete' | 'insert'
  diffType: 'precise' | 'block_level' | 'document_level'
  routeSource: 'selection' | 'reference' | 'block_search'
}
```

## 6.2 DiffEntry（前端标准）

```ts
interface DiffEntry extends CanonicalDiff {
  status: 'pending' | 'accepted' | 'rejected' | 'expired'
  toolCallId?: string
  chatTabId?: string
  baselineId?: string
  documentRevision?: number
  createdAt: number
  acceptedAt?: number
  expireReason?: 'content_modified' | 'original_text_mismatch' | 'range_unresolvable' | 'partial_overlap'
}
```

## 6.3 DiffExecutionExposure（观测对象）

```ts
interface DiffExecutionExposure {
  code: string
  message: string
  filePath: string
  diffId?: string
  toolCallId?: string
  timestamp: number
}
```

约束：

1. 观测事件不替代业务状态。  
2. `expired` 与 `error exposure` 必须隔离建模。  

---

## 七、文本抽取与校验结构

## 7.1 区间文本抽取

```ts
interface RangeTextExtractionResult {
  anchor: BlockRangeAnchor
  extractedText: string
  extractionHash: string
}
```

规则：

1. 单块与跨块共用同一抽取协议。  
2. 执行前必须复算当前区间文本并与 `originalText` 校验。  

## 7.2 失效判定输入

```ts
interface DiffValidityInput {
  currentLogicalStateHash: string
  anchorResolvable: boolean
  originalTextMatched: boolean
  overlapType: 'none' | 'contain' | 'partial'
}
```

---

## 八、协作约束

1. 状态语义：以 `DocumentLogicalState` 为唯一真源。  
2. 区间语义：以 `BlockRangeAnchor` 为统一定位结构。  
3. 执行语义：以 `CanonicalDiff -> DiffEntry` 的转换链为唯一入口。  
4. 失效语义：只看内容变化、区间可解析性、非法部分重叠。  
5. 观测语义：通过 `DiffExecutionExposure` 暴露，不干预状态流转。  

---

## 九、MVP 验收口径

1. 核心对象在前后端字段语义一致。  
2. 单块与跨块区间使用同一锚点结构。  
3. diff 生成、校验、执行、失效字段链可追溯。  
4. 状态流与观测流可独立验证。
