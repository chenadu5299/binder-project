## 一、文档定位

本文定义 Binder MVP 的交互模式总览，统一“用户通过什么交互入口完成任务、各模式如何切换、边界如何约束”。  
本文是产品交互模式文档，不替代具体功能实现文档。

---

## 二、交互模式清单（MVP）

当前已落地交互模式固定为四类：
1. 文档直编模式（Editor Direct Edit）。  
2. AI 层次一模式（辅助续写，Cmd+J）。  
3. AI 层次二模式（局部修改，Cmd+K）。  
4. AI 层次三模式（对话编辑，聊天窗口 + 工具调用 + Diff 决策）。  

补充：

1. 当前与项目级构建相关的唯一生效主线是 `Chat Build`。
2. Chat Build 不是独立 Build Mode 页面前提，而是以当前 `chat` 为入口壳 / 交互表面承接的构建流程。
3. Chat Build 当前仍未形成完整工程主链，因此不作为 R1 已落地交互阻塞项；但产品交互口径必须以它为唯一主线。

## 三、模式定义与入口

## M1 文档直编模式

入口：
1. 文件树打开文档。  
2. 新建文档后进入编辑区。  

核心交互：
1. 键盘输入、格式编辑、保存与回读。  
2. 多标签切换。  

退出：
1. 切换到 AI 模式（M2/M3/M4）。  
2. 关闭编辑标签。  

## M2 AI 层次一模式（辅助续写）

入口：
1. 编辑器聚焦状态下触发 `Cmd+J`。  

核心交互：
1. 生成续写建议。  
2. 键盘选择并应用建议。  
3. 取消/重触发。  

退出：
1. 应用建议后回到 M1。  
2. 取消后回到 M1。  

## M3 AI 层次二模式（局部修改）

入口：
1. 编辑器聚焦状态下触发 `Cmd+K`。  

核心交互：
1. 弹窗内输入修改指令。  
2. 弹窗内多轮调整。  
3. 应用结果写回文档。  

退出：
1. 点击应用后回到 M1。  
2. 关闭弹窗回到 M1。  

## M4 AI 层次三模式（对话编辑）

入口：
1. 打开聊天面板，进入 Agent/Chat 交互。  

核心交互：
1. 消息输入与上下文注入。  
2. 工具调用回流。  
3. Diff 卡片决策（接受/拒绝/批量操作）。  

退出：
1. 关闭聊天面板（文档保留当前状态）。  
2. 切回 M1 继续编辑。  

---

## M5 Chat Build（当前唯一生效的构建主线）

> Chat Build 是当前唯一生效的项目级构建口径。当前工程主链仍在补齐，但系统交互定义必须以此为准。

入口：
1. 在当前 chat 交互中通过明确构建指令触发。
2. 在当前 chat 交互中通过显式“开始构建”类入口触发。

阶段：
1. chat 前置讨论阶段。
2. 构建触发与轻量确认阶段。
3. 大纲生成与大纲确认阶段。
4. 正式构建执行阶段。
5. 完成 / 失败 / 中断结束阶段。

chat 在 Chat Build 中的角色：
1. 构建前，chat 是讨论与需求收敛界面。
2. 构建中，chat 是状态 / 进度 / 可读过程展示与中断控制界面。
3. 构建中不再把新聊天输入当作需求改写通道。

当前硬边界：
1. `BR-CBT-VERIFY-001`：大纲确认是正式构建前的硬边界。
2. `BR-CBT-STATE-001`：构建一旦开始，不接受自然语义打断、改向或重定向。
3. `BR-CBT-STATE-002`：构建中只允许查看状态、进度和过程说明，或手动中断。
4. `BR-CBT-ASSET-001`：当前构建只生成新的项目资源，不修改既有内容。

未来参考说明：
1. Discussion Build / Multi-Actor Build / 真人协作不属于当前模式定义。
2. 若未来重启，应作为 future/reference 方案重新规划，不反向改写当前 Chat Build 主线。

## 四、模式切换规则

## 4.1 统一切换原则

1. 交互切换不丢编辑上下文。  
2. 模式切换不绕过“先确认再生效”规则。  
3. 任一模式失败不阻断其他模式继续。  

## 4.2 允许切换路径（MVP）

1. `M1 -> M2 -> M1`  
2. `M1 -> M3 -> M1`  
3. `M1 -> M4 -> M1`  
4. `M4` 内部可连续多轮，不要求每轮退出后再进入。  
5. `M4(chat 讨论态) -> M5(Chat Build 确认/执行态)`
6. `M5(Completed/Interrupted/Failed) -> M4(chat 讨论态)`

## 4.3 禁止切换行为

1. M2/M3 在未完成写回前强行覆盖编辑器状态。  
2. M4 未经 Diff 决策直接写入文档。  
3. 用一个模式的临时状态覆盖另一个模式的真源状态。  

---

## 五、模式间共享与隔离

## 5.1 共享项

1. 当前目标文档。  
2. 模型配置与调用基础能力。  
3. 统一错误暴露口径。  

## 5.2 隔离项

1. M2 弹窗历史不等于 M4 聊天历史。  
2. M4 的 Diff 状态机不由 M2/M3 复用。  
3. 任何模式的展示态不直接替代逻辑态。  
4. M5 构建执行态不接受自然语义改向，也不把执行中的 chat 输入回流为当前构建需求。

---

## 六、交互约束总表（MVP）

1. 文档改写必须可追踪来源（人工/AI）。  
2. 对话编辑改写必须先产出 Diff，再由用户决策。  
3. 失效处理与失败暴露分离：  
失效是业务状态流转，失败是执行异常暴露。  
4. 批量操作必须遵守稳定顺序与统一刷新时序。  
5. 快捷键模式必须要求编辑器聚焦，避免误触发。  

---

## 七、MVP 交互边界

## 7.1 纳入

1. M1/M2/M3/M4 全部可用。  
2. Agent/Chat 两种对话视图可切换。  
3. Diff 卡片单卡与批量决策链路完整。  
4. 当前唯一生效的项目级构建口径为 Chat Build，并以 chat 作为入口壳与交互表面。

## 7.2 暂不纳入

1. 移动端交互模式。
2. Chat Build 的完整工程落地主链（当前只确定产品口径与交互主线，不代表已完成实现）。
3. Discussion Build / Multi-Actor Build / 真人协作交互编排。
4. 分页/T-DOCX 交互作为首版发布前置。

---

## 八、验收口径（交互层）

1. 用户能无中断完成 `M1 -> M4 -> M1` 主链闭环。  
2. 用户能分别完成 M2 与 M3 的独立任务闭环。  
3. 模式切换时不出现上下文错乱、目标文档错位。  
4. 交互失败有统一暴露，不靠 toast 堆叠提示。
