# 203_数据流状态流

## 一、文档定位

本文定义 Binder MVP 的数据流与状态流总规范，覆盖文件编辑、AI 流式响应、对话编辑 Diff、生效与观测。  
本文是系统架构层文档，不替代子系统实现细节文档。

---

## 二、目标与边界

目标：
1. 明确核心链路的数据输入、处理、输出。  
2. 明确状态推进规则与单一真源约束。  
3. 明确失败暴露与业务状态流转的边界。  

边界：
1. 关注“流如何走、状态如何变”。  
2. 不展开具体 UI 交互细节。  
3. 不展开某个单模块内部算法细节。  

---

## 三、核心对象（流转视角）

1. DocumentSnapshot  
文档某一时刻的可编辑快照（HTML/结构化文本）。  

2. RequestContext  
一次 AI 请求绑定上下文（targetFile、L、revision、baselineId 等）。  

3. StreamEvent  
流式事件单元（文本增量、工具调用增量、完成信号、错误）。  

4. DiffEntry  
待确认修改单元（pending/accepted/rejected/expired）。  

5. ExecutionExposure  
执行观测对象（错误码、阶段、失败原因、可复现信息）。  

---

## 四、全局数据流（MVP）

## 4.1 文件编辑数据流

用户编辑 -> 编辑器状态更新 -> 保存请求 -> 后端写盘 -> 缓存同步 -> 回读确认。  

关键点：
1. 写盘与缓存更新必须同链完成。  
2. 回读结果必须可验证一致。  

## 4.2 AI 对话数据流

用户消息 -> 构建 RequestContext -> `ai_chat_stream` -> Provider 流式返回 -> 前端事件消费 -> 消息状态更新。  

关键点：
1. 流式内容按增量处理，不可重复追加。  
2. 工具调用参数按完整 JSON 边界累积。  
3. 空事件与重复 chunk 必须过滤。  

## 4.3 对话编辑数据流

工具调用 -> Resolver 产出 canonical diff -> 写入 DiffStore（pending）-> 用户决策 -> 接受后生效。  

关键点：
1. 未确认前不改逻辑状态。  
2. 接受后才推进真实内容版本。  

## 4.4 非当前文档数据流

目标文件解析 -> canonical 加载/构建 -> 上下文注入 -> 执行编辑 -> diff 或落盘结果回流。  

关键点：
1. 跟需求走，不跟当前激活状态走。  
2. 就绪链未完成不得执行。

## 五、状态流总模型

## 5.1 文档状态流

`clean -> dirty -> saving -> saved`  
并行关注：`external_modified`（外部修改检测）。  

规则：
1. 仅真实内容变化推进 revision。  
2. 保存失败不改变已确认内容状态。  

## 5.2 流式任务状态流

`queued -> streaming -> tool_running -> streaming -> completed`  
异常分支：`failed`、`partial_failed`（跳过继续）。  

规则：
1. 执行层完成与业务层完成分离判定。  
2. 局部失败可跳过，不中断整轮可继续项。  

## 5.3 Diff 状态流

`pending -> accepted`  
`pending -> rejected`  
`pending -> expired`  

规则：
1. `pending` 不进入逻辑状态。  
2. `accepted` 才进入逻辑状态。  
3. `expired` 是业务状态，不等于执行异常。  

---

## 六、单一真源约束

1. 同一轮请求只绑定一套 `L + revision + baselineId`。  
2. 生成、校验、执行必须绑定同一基线语义。  
3. 展示态（装饰、卡片）不得替代逻辑态。  
4. `revision` 变化不作为“全量失效开关”。  

---

## 七、流式链路约束（可观测）

1. SSE/流式分片必须经缓冲重组后再解析。  
2. 文本增量必须做重复检测与重叠处理。  
3. 工具调用 arguments 必须按 JSON 完整性收口。  
4. 前端消息层必须具备去重、空事件过滤、顺序保护。  

---

## 八、失败处理与状态处理隔离

1. 业务状态流转  
如 diff 过期、拒绝、已接受，属于状态机行为。  

2. 执行失败暴露  
如 JSON 解析失败、工具执行失败、定位失败，属于观测事件。  

3. 强制隔离  
失败暴露不直接替代状态流转；状态流转也不掩盖执行异常。  

---

## 九、关键时序规则

1. 批量接受：先读后写，稳定排序，一次统一刷新。  
2. 刷新顺序：文档内容 -> 版本状态 -> 装饰层 -> 视图分页/列表。  
3. 任何中途刷新不得改变同批候选集。  

---

## 十、MVP 验收口径

1. 主链数据流（编辑/对话/Diff）可端到端复现。  
2. 状态流转可解释，且与 UI 表现一致。  
3. 流式重复、错序、工具参数异常可稳定收敛。  
4. 失败可暴露，且不误伤可继续任务。
