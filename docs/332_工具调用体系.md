# 332_工具调用体系

## 一、文档定位

本文是 AI 执行系统中工具调用能力的统一主规范，收口工具定义、参数解析、执行策略、结果回流与错误处理契约。  
本文用于把“扩展规范”与“重构计划”落地为可执行运行时口径。

---

## 二、MVP 目标

1. 建立可扩展、可执行、可观测的工具调用主链。  
2. 统一工具从“定义 -> 调用 -> 执行 -> 回流”的运行时契约。  
3. 保证对话编辑主链中的工具调用稳定，不因重构造成行为漂移。  

---

## 三、统一工具主链

1. 工具定义注册  
`tool_definitions.rs` 输出工具 schema（name/description/parameters）。  

2. 调用意图生成  
模型按 schema 生成 tool_call（name + arguments）。  

3. 参数解析与校验  
调用侧解析 arguments，做必填校验、类型校验、默认值填充。  

4. 工具执行  
`tool_service.rs::execute_tool` 分发到具体工具实现。  

5. 结果标准化  
结果统一为 `success/data/error` 主结构并附带扩展字段。  

6. 事件回流  
工具结果通过流式链回传前端，驱动 UI、Diff、状态更新。  

---

## 四、运行时契约（核心补齐）

## 4.1 定义契约

1. 每个工具必须有稳定 `name`，不可与已有工具重名。  
2. `parameters` 使用 JSON Schema，字段语义与后端实际实现一致。  
3. 文档字段与代码字段必须一一对应，禁止“文档可用、运行时报错”。  

## 4.2 参数契约

1. 所有 tool_call arguments 先解析再执行，禁止边解析边执行。  
2. 参数缺失分为可重试错误与致命错误。  
3. 参数纠错（默认值、兼容字段）必须可追踪。  

## 4.3 执行契约

1. 工具执行必须幂等可控（至少在重试策略下不破坏状态）。  
2. 工具执行支持受控重试，重试策略由调用层统一管理。  
3. 涉及确认动作的工具，先走确认门禁再执行。  

## 4.4 结果契约

1. 返回主结构统一：`success`、`data`、`error`。  
2. 允许工具扩展字段，但不得破坏主结构。  
3. 对话编辑工具（如 `edit_current_editor_document`）返回结构需包含规范化 `diffs`。  

## 4.5 错误契约

1. 错误分级：可重试、不可重试、确认中断。  
2. 错误必须带可观测信息（错误码/阶段/工具名）。  
3. 错误回流不允许吞错，必须进入统一观测链。  

---

## 五、扩展与重构规则

## 5.1 新增工具标准步骤

1. 在 `tool_definitions.rs` 注册 schema。  
2. 在 `tool_service.rs` 增加执行分支。  
3. 补充参数校验、错误分级、返回结构。  
4. 补充调用链回归测试。  

## 5.2 ai_commands 重构对齐

1. 调用解析、执行、重试、确认、异常处理按模块分层。  
2. 新模块上线后，工具调用语义不得变化。  
3. 计划态重构项必须落地为“可运行契约”，才能进入主链。

## 六、与其他模块边界

1. 与 `331_prompt架构`：331 负责提示词构建与工具调用意图，332 负责工具运行时契约。  
2. 与 `333_任务规划执行`：333 负责任务层编排，332 负责每次工具调用执行可靠性。  
3. 与 `204_ai执行架构`：204 给出整体执行框架，332 给出工具执行细节规则。  
4. 与 `311/312`：对话编辑链工具结果必须符合 diff 与状态协作规范。  

---

## 七、MVP 验收口径

1. 工具定义与执行实现一致，无 schema 漂移。  
2. 工具参数错误可稳定分级并回流。  
3. 工具重试、确认、异常处理链可复现。  
4. `edit_current_editor_document` 等核心工具可稳定输出规范结果并被前端消费。
