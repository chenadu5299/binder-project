## 一、文档定位与控制权威

### 1.1 文档定位

本文是 Binder Agent 的**能力描述主控文档**。  
本文用于回答以下问题：

1. Binder Agent 到底是什么，不是什么。
2. Binder Agent 由哪些能力模块组成。
3. 每个能力模块解决什么问题、对外暴露什么能力、受什么边界约束。
4. Binder Agent 如何定义完成、验证、确认、收口。
5. 后续功能设计、产品评审、技术实现应以什么口径判断“是否符合 Agent 主控要求”。

Binder Agent 的实际开发控制文档组是：

1. `A-AG-M-D-01_Binder Agent能力描述文档.md`

3. `A-AG-X-L-01_Binder Agent落地开发计划.md`

即使 `R-AG-C-D-01_Binder-Agent指导方案（Guiding Architecture & Design Doctrine）.md` 不存在，`A-AG-M-D-01_Binder Agent能力描述文档.md`、`A-AG-M-T-04_Binder Agent技术主控文档.md`、`A-AG-X-L-01_Binder Agent落地开发计划.md` 仍然必须足以支撑 Agent 的设计评审、功能落地和架构承接。

## 二、Binder Agent 的根定义

### 2.1 本质定义

Binder Agent 不是“替用户自动完成任务的黑箱代理”。  
Binder Agent 是一个面向文档、内容、创作、办公、项目协作场景的**人-AI协作运行系统**。

它的核心职责不是追求最大自治，而是：

1. 帮助用户把模糊目标变成可推进任务。
2. 帮助用户把任务过程变成可见中间态。
3. 帮助用户把结果收口为可验证、可确认、可沉淀的阶段成果。

### 2.2 核心目标

Binder Agent 的核心目标有四项：

1. **协作推进**：让用户从“想做但没法稳定推进”进入“持续可推进”。
2. **中间态可见**：让过程不是黑箱，而是可以查看、打断、修正、接管。
3. **阶段可收口**：让任务不是伪完成，而是进入明确的阶段闭合状态。
4. **资产可沉淀**：让计划、候选、diff、验证、确认结果、经验模式进入项目资产链。

### 2.3 非目标

Binder Agent 明确不以以下方向作为主目标：

1. 不以“完全自动完成任务”作为最高目标。
2. 不以“尽量少打扰用户”作为单一目标。
3. 不以“模型一次回答尽可能长、尽可能全”作为能力衡量标准。
4. 不以“看起来像做完了”替代系统级收口。

### 2.4 与其他系统的区别

| 对象 | 典型目标 | Binder Agent 的区别 |
|---|---|---|
| Coding Agent | 跑通、修复、测试通过、PR 可合并 | Binder 面向内容协作，很多任务没有可机器判定的终态通过标准 |
| 通用任务 Agent | 尽可能自治、异步代做、最终交付结果 | Binder 不追求黑箱代做，要求中间态可见和阶段确认 |
| Chat 型 AI | 回答问题、给建议、对话陪伴 | Binder 不只是对话，而是要形成任务状态、验证与收口 |
| 办公辅助 AI | 单点增强，如写一句、改一段、生成一页 | Binder 需要贯穿左侧规划、中段执行、右侧验证与资产化 |

## 三、Agent 设计总观

### 3.1 总体命题

Binder Agent 的设计总命题是：

> 在“内容任务缺少客观终态验证”的前提下，构建一个能够持续推进、稳定收口、低成本确认、可资产沉淀的人-AI协作系统。

### 3.2 基本运行观

Binder Agent 的基本运行观不是“一次生成”，而是“一轮协作”。

一轮协作至少包含五个必要问题：

1. 当前目标到底是什么。
2. 当前作用范围到底是什么。
3. 当前系统已经做出了什么可见中间态。
4. 当前结果通过了哪些层级的验证。
5. 当前任务是否已经进入本轮可结束状态。

### 3.3 Binder Agent 的最小闭环

Binder Agent 的最小闭环不是“模型输出了一段内容”，而是：

`目标清晰 -> 范围明确 -> 产出候选 -> 完成验证 -> 用户确认/豁免 -> 状态跃迁`

只要上述链条缺任一环，就不应被系统表达为“任务已完成”。

## 五、Agent 能力模型

## 5.1 子模块 A：根定义与目标边界

**承接规则**：

1. BA-CORE-001
2. BA-CORE-002

### 5.1.1 解决的问题

这一模块解决的是“系统到底想把 Agent 做成什么”的问题。  
如果没有这一层，后续所有实现都可能不断滑向：

1. 黑箱自动代做。
2. 单纯聊天增强。
3. 只有模型能力，没有系统控制。

### 5.1.2 模块定义

Binder Agent 必须同时具备以下特征：

1. 面向任务推进，而不是只面向回答生成。
2. 面向协作过程，而不是只面向最终文本结果。
3. 面向可验证中间态，而不是只面向模型自述。
4. 面向阶段性完成，而不是伪造最终完成。

### 5.1.3 设计边界

本模块要求后续所有能力都符合以下边界：

1. 任何高影响修改都不能直接从 AI 文本流落到最终状态。
2. 任何“完成”表达都必须可回溯到状态与验证层。
3. 任何“全自动”能力都必须说明用户如何接管、系统如何托底。

### 5.1.4 设计评审时必须回答的问题

1. 这个能力是在增强协作，还是在偷换成代做？
2. 这个能力有没有把系统责任丢给模型？
3. 这个能力有没有把收口责任全部丢给用户？

## 5.2 子模块 B：三元协作模型

**承接规则**：

1. BA-MODEL-001
2. BA-MODEL-002
3. BA-MODEL-003
4. BA-MODEL-004

### 5.2.1 三元职责定义

| 角色 | 必须承担的职责 | 不应承担的职责 |
|---|---|---|
| Human | 目标定义、优先级判断、主观质量判断、最终接受 | 不应负责低层运行检查和状态机维护 |
| AI | 候选生成、局部执行、启发式检查、推进建议 | 不应冒充系统判定，不应伪造“已完成” |
| System | 状态管理、验证分层、门禁控制、可视化、回退与记录 | 不应把关键职责完全下放给 AI 文本 |

### 5.2.2 标准协作链

标准协作链如下：

`Human 定义目标 -> System 固定边界 -> AI 产生候选/执行 -> System 验证与可视化 -> Human 判断/确认 -> System 更新状态`

### 5.2.3 不允许的职责塌缩

以下模式均视为职责塌缩：

1. AI 同时生成结果并宣布结果已通过系统验证。
2. 用户既要提出目标、又要自己核对全部低层执行正确性。
3. 系统只做消息转发，不承担状态与验证责任。

### 5.2.4 低认知成本确认的基本要求

用户必须参与关键判断，但不能通过“让用户读完所有过程日志”来实现确认。  
低认知成本确认至少应满足：

1. 用户能快速看出改了什么。
2. 用户能快速看出范围在哪里。
3. 用户能快速看出风险点是什么。
4. 用户能局部接受、局部拒绝、要求进一步 refinement。

## 5.3 子模块 C：完成观与状态语义

**承接规则**：

1. BA-STATE-001
2. BA-STATE-002
3. BA-STATE-003
4. BA-STATE-004

### 5.3.1 为什么不能使用“终态完成”

Binder 的核心场景是文档、内容、创作、办公协作。  
在这类场景里，大量任务并不存在像代码测试通过那样的客观终态。

因此：

1. “流程跑完”不等于“任务完成”。
2. “模型说已经做完”不等于“系统可以宣布完成”。
3. “用户暂时满意”也不一定等于“项目进入最终完成”。

更准确的表达方式是：任务进入了某个更高确定性的阶段状态。

### 5.3.2 状态集合

| 状态 | 含义 | 进入条件 | 阶段闭合阻塞项 |
|---|---|---|---|
| `draft` | 只有目标或想法，尚未结构化 | 用户表达目标或系统接收任务 | 范围不清、约束不清 |
| `structured` | 目标、范围、约束、对象已初步结构化 | 已形成明确任务骨架 | 还没有候选或执行结果 |
| `candidate_ready` | 已形成可评估候选 | 产生了候选文本、候选 diff、候选方案 | 尚未经过验证 |
| `constraint_satisfied` | 结构和约束层基本通过 | 已通过结构/约束验证 | 用户尚未确认 |
| `review_ready` | 已进入适合用户判断的状态 | 系统已整理变更、风险、范围 | 仍缺用户接受或豁免 |
| `user_confirmed` | 用户已明确接受当前结果 | 用户完成确认动作 | 后续 refinement 尚未收口 |
| `refined` | 基于确认又做了进一步优化 | refinement 完成且未破坏确认边界 | 尚未形成阶段闭合 |
| `stage_complete` | 本轮任务已达到可结束状态 | 满足阶段闭合标准 | 无 |
| `invalidated` | 先前候选或状态已失效 | 约束变化、范围变化、验证失败、用户推翻 | 需回退到前序状态 |

### 5.3.3 阶段闭合的定义

阶段闭合不是“永远完成”，而是“当前轮次可以安全结束”。  
一个状态进入 `stage_complete`，必须同时满足：

1. 当前轮目标已明确落在某个可验证范围内。
2. 当前轮产出已完成对应层级的系统验证。
3. 当前轮所需的人类确认已完成，或满足明确豁免条件。
4. 当前轮没有悬置的关键失败项。

### 5.3.4 伪完成的典型形式

以下情形都属于伪完成：

1. 只有 AI 的总结，没有系统验证结果。
2. 只有部分步骤做完，但系统直接表达为“已完成”。
3. 用户还没确认，系统却将结果写成最终结论。
4. 作用范围仍不明确，但系统已进入结束态。

### 5.3.5 状态语义的产品意义

状态语义不仅是后端对象，更是产品表达语义。  
UI、提示词、工具调用、验证结果、确认交互都必须围绕同一组状态语义对齐，否则用户会看到相互冲突的“完成”表达。

## 5.4 子模块 D：分层验证与确认

**承接规则**：

1. BA-VERIFY-001
2. BA-VERIFY-002
3. BA-VERIFY-003
4. BA-VERIFY-004
5. BA-VERIFY-005

### 5.4.1 三层验证体系

| 层级 | 验证什么 | 不验证什么 | 主责任方 |
|---|---|---|---|
| 结构验证 | 目标对象、结构完整性、作用范围、格式完整性、引用存在性 | 不判断内容“好不好” | System |
| 约束验证 | 是否满足指令约束、风格约束、范围约束、风险约束 | 不替代主观质量判断 | System + AI |
| 质量判断 | 是否符合用户审美、是否达到内容质量预期、是否值得接受 | 不负责低层系统检查 | Human |

### 5.4.2 为什么必须分层

如果不分层，系统会在两个方向上失真：

1. 把主观质量问题伪装成客观系统通过。
2. 把所有验证责任都丢给用户。

分层验证的意义，是让不同类型的问题回到正确责任方。

### 5.4.3 用户确认不是验证外包

用户确认不是把系统没做完的工作全部甩给用户。  
在 Binder 中，用户确认应建立在前两层已经完成的基础上。

系统必须先把以下内容准备好，再要求用户确认：

1. 变更范围。
2. 候选内容。
3. 风险摘要。
4. 尚未满足的约束项。
5. 可以接受或拒绝的局部入口。

### 5.4.4 低成本确认的最低机制

低成本确认至少需要以下产品能力支撑：

1. diff 或等价变更视图。
2. 作用范围高亮。
3. 变更摘要。
4. 风险提示。
5. 局部接受/拒绝。
6. 二次 refinement 入口。

### 5.4.5 验证失败后的处理

1. 结构验证失败：不得进入 review-ready。
2. 约束验证失败：可以进入待修正状态，但不得直接进入 stage-complete。
3. 质量判断未通过：允许回到 refinement，而不是伪装系统错误。

## 5.5 子模块 E：运行模型与 runtime gate

**承接规则**：

1. BA-RUN-001
2. BA-RUN-002
3. BA-RUN-003
4. BA-RUN-004

### 5.5.1 Binder Agent 的逻辑运行模型

Binder Agent 采用如下逻辑模型：

`plan -> execute -> verify -> confirm -> transition`

这个模型不是为了模仿 coding agent，而是为了把内容协作任务拆成五个可控制环节：

1. `plan`：定义目标、范围、约束、候选路径。
2. `execute`：产出候选内容、候选修改、候选结构。
3. `verify`：执行结构与约束层检查。
4. `confirm`：让用户以低成本方式完成判断。
5. `transition`：由系统决定是否发生状态跃迁。

### 5.5.2 为什么必须有 runtime gate

prompt 不能替代 runtime gate。  
Agent 一旦进入多步骤协作，没有系统门禁，就会出现：

1. 越界执行。
2. 跳过验证。
3. 跳过确认。
4. 提前宣布完成。

### 5.5.3 gate 的职责边界

runtime gate 至少要承担以下职责：

1. 判断当前目标和范围是否足够明确。
2. 判断当前上下文是否足以执行。
3. 判断是否已经产出可见中间态。
4. 判断验证是否已经完成。
5. 判断用户确认是否已经满足。
6. 判断是否允许状态迁移。

### 5.5.4 非黑盒执行原则

以下任务默认必须走非黑盒执行链：

1. 文档修改。
2. 多点重写。
3. 跨文档联动。
4. 构建模式中的高影响生成。

非黑盒执行意味着：

1. 过程对象必须可见。
2. 修改范围必须可知。
3. 关键节点必须可打断。
4. 结果必须可回退。

## 5.6 子模块 F：左右能力场景

**承接规则**：

1. BA-SCENE-001
2. BA-SCENE-002
3. BA-SCENE-003
4. BA-SCENE-004
5. BA-SCENE-005
6. BA-SCENE-006

### 5.6.1 左中右能力版图

| 位置 | 典型场景 | 主要矛盾 | Agent 参与程度 | 是否必须 human-in-loop |
|---|---|---|---|---|
| 左侧 | 需求澄清、意图对齐、范围锚定、风格锚定、计划生成、风险暴露、资料装配 | 用户目标模糊，系统不应直接动手改 | 中等 | 是 |
| 中段 | 辅助续写、局部修改、对话编辑、多候选生成、结构重组、跨文档操作、构建中间产物 | 需要高产出，但不能黑箱越界 | 高 | 大多数是 |
| 右侧 | 结构检查、约束检查、用户确认、局部接受/拒绝、refinement、阶段闭合、项目沉淀 | 需要稳定收口，不能伪完成 | 高 | 关键节点必须是 |

### 5.6.2 旧三层 AI 能力在新体系中的位置

旧体系中的三层 AI 功能，在新体系中的承接方式如下：

| 旧能力 | 在 Agent 总体系中的归口位置 | 不发生变化的原始要求 |
|---|---|---|
| 辅助续写 | 中段中的轻量候选生成能力 | 独立触发、独立提示词、无工具、无聊天历史、不得被并入对话编辑主链 |
| 局部修改 | 中段中的局部范围执行能力 | 独立弹窗、独立提示词、无聊天历史、弹窗内不进入对话编辑 Diff 主链 |
| 对话编辑 | 横跨左中右的完整协作入口 | 仍是三层中唯一默认承接完整 Agent 协作链的主入口 |

这里的“归口位置”只表示它们在 Binder Agent 总体系中的能力归属，**不表示三层运行逻辑被统一、合并或改写**。

### 5.6.3 三层独立性总约束

Binder 原始需求已经明确：三层 AI 交互系统必须保持独立设计。  
在 Agent 新体系下，这一约束继续成立，并且被提升为 Agent 主控约束。

具体约束如下：

1. 层次一、层次二、层次三必须保持独立触发入口。
2. 三层必须保持各自独立的提示词主链，不允许因 Agent 体系而强行合并。
3. 三层必须保持各自独立的运行链，不允许默认互相接管或互相升级。
4. 三层可共享模型配置、上下文能力、术语体系等基础设施，但不得因此共享业务逻辑主链。
5. 层次三不能作为层次一、层次二的默认上位运行入口。
6. 任何新优化若会改变层次一或层次二的原始交互逻辑，必须单独立项、单独说明、单独评审。

### 5.6.4 新 Agent 优化对三层的实际影响

Agent 新体系带来的不是“三层合并”，而是“上位主控和局部能力增强”。  
三层实际影响应按下表理解：

| 层次 | 当前应承接的 Agent 优化 | 当前不应被改写的部分 |
|---|---|---|
| 层次一：辅助续写 | 可吸收统一术语、统一上下文字段命名、统一错误暴露口径 | 不改为多阶段 Agent 链，不引入工具链，不并入对话编辑状态链 |
| 层次二：局部修改 | 可吸收更清晰的 scope 语义、局部确认语义、上下文字段统一 | 不改为对话编辑，不默认引入 Diff 主链，不改写其独立弹窗交互 |
| 层次三：对话编辑 | 作为主要承接位，引入状态跃迁、分层验证、确认、artifact、gate | 仍需遵守自身专项规则，不可因 Agent 文档而覆盖 `A-DE-M-D-01 / A-DE-M-T-01 / A-DE-M-T-02 / A-DE-M-P-01 / A-DE-X-L-01` 的专项主控 |

### 5.6.5 不得误读的点

以下理解都是错误的：

1. “统一 Agent 体系” = “三层以后走同一条运行链”。
2. “辅助续写属于中段能力” = “辅助续写要接入局部修改或对话编辑的流程”。
3. “局部修改属于受控执行能力” = “局部修改现在必须接入 Diff 主链或工具链”。
4. “对话编辑是完整协作入口” = “层次一和层次二都会被它接管”。

### 5.6.6 Binder 重点投入场景

Binder 应优先投入以下场景：

1. 可明确作用范围的文档改写。
2. 有结构约束的内容生成。
3. 需要多轮 refinement 的协作编辑。
4. 需要保留过程对象和项目沉淀的构建型任务。

### 5.6.7 暂不应过度自动化的场景

以下场景不应过度自动化：

1. 审美判断高度主观的最终定稿。
2. 涉及多文档高风险重写但范围尚不稳定的任务。
3. 用户目标仍模糊、约束不足的任务。

## 5.7 子模块 G：中间态资产化与项目沉淀

**承接规则**：

1. BA-ASSET-001
2. BA-ASSET-002

### 5.7.1 中间态为什么是主产品对象

在 Binder 中，中间态不是为了调试而存在，而是为了协作而存在。  
一旦没有中间态，用户就只能：

1. 盲信 AI。
2. 逐字全文核对。
3. 在结果不满意时从头来过。

这三种结果都不适合作为协作平台的产品基因。

### 5.7.2 应被资产化的对象

以下对象原则上都应具备进入资产链的可能：

1. plan
2. scope
3. candidate
4. diff
5. verification result
6. confirmation record
7. refinement history
8. memory seed
9. template seed

### 5.7.3 资产化的最低标准

一个中间态对象要进入资产链，至少应具备：

1. 稳定标识。
2. 来源信息。
3. 作用范围。
4. 当前状态。
5. 与任务或项目的关联关系。

### 5.7.4 资产化的产品意义

资产化不是为了堆历史记录，而是为了让 Binder 逐渐具备：

1. 项目级记忆。
2. 风格与模板沉淀。
3. 多轮任务接力。
4. 可恢复、可继续的协作能力。

## 七、设计评审否决条件

以下情况任一成立，都应视为不符合 Binder Agent 主控要求：

1. 设计把 Agent 直接做成黑箱自动代做。
2. 设计没有表达状态语义，却想宣布任务完成。
3. 设计要求用户承担全部验证成本。
4. 设计没有可见中间态，却允许高影响修改直接落地。
5. 设计没有确认机制，却允许系统表达“已完成”。
6. 设计没有定义作用范围，却允许跨范围修改。
7. 设计没有资产对象，却要求系统支持跨轮接力。
