## 一、文档定位与控制权威

### 1.1 文档定位

本文是 Binder Agent 的**知识库协同控制文档**。  
本文回答以下问题：

1. Binder 知识库在 Agent 体系中到底是什么，不是什么。
2. 知识库与记忆库、模板库、当前轮 artifact 的边界如何区分。
3. 知识库为何是“用户侧主动产出资产”，而不是系统自动沉淀仓。
4. AI 如何消费知识库，用户如何主导知识库。
5. Binder Agent指导方案中哪些规则落在知识库协同层，哪些不应落在知识库。

### 1.2 控制权威

本文是知识库协同层的主控文档。  
本文不替代：

1. `A-AST-M-T-02_知识库机制.md` 的对象模型、双库结构、处理流水线与检索协议。
2. `A-AST-M-P-01_上下文注入.md` 的注入顺序、预算、裁剪与降级。
3. ``A-AG-M-D-01_Binder Agent能力描述文档.md`/`A-AG-M-T-04_Binder Agent技术主控文档.md`` 的 Agent 上位语义和技术对象主控。

但凡涉及以下问题，必须以本文为准：

1. 什么情况下知识库是合格上下文来源。
2. AI 是否可以自动把内容写入知识库。
3. 用户显式引用与 AI 自动检索之间谁优先。
4. 知识库与记忆库、模板库之间的分工。

### 1.3 本文不负责什么

本文不负责：

1. 向量索引与分块算法实现。
2. 某个具体知识库 UI 页面稿。
3. 文档解析器的格式适配细节。

这些内容由 `A-AST-M-T-02_知识库机制.md` 和实现文档承接。

## 二、知识库在 Binder Agent 中的本质

知识库不是自动生成的语义记忆。  
知识库也不是工作流模板或行为规范。

知识库的本质是：

**由用户主动导入、主动存入、主动组织、可被 AI 检索消费的文档级知识资产系统。**

它解决的不是“记住聊过什么”，而是：

1. 用户已经拥有某些文档知识，希望 AI 在需要时能检索出来。
2. 用户希望把项目文档、外部资料、研究材料转成可消费知识。
3. 用户希望这些知识保持可管理、可命名、可移动、可删除。

## 四、Binder 知识库的边界

## 4.1 与记忆库、模板库的本质区别

| 对象 | 主要来源 | 主要管理者 | 主要用途 |
|---|---|---|---|
| 记忆库 | AI 自动提取的语义记忆 | 系统主导、用户只消费/引用 | 连续语义与沉淀 |
| 知识库 | 用户导入、用户存入的文档知识 | 用户主导，AI 消费 | 文档级检索知识 |
| 模板库 | 用户创建、选择、保存的过程约束资产 | 用户主导，AI 按约束执行 | 约束流程 |
| 当前轮 artifact | 当前轮协作中生成的中间态 | 系统与 AI 协同，用户确认 | 当前轮推进与收口 |

### 4.2 不得误读的点

1. 知识库不是记忆库的“可编辑版”。
2. 知识库不是模板库的“内容填充版”。
3. 知识库不是所有引用内容的兜底容器。
4. AI 可以消费知识库，但不应默认替用户管理知识库。

### 4.3 知识库在 Agent 中的正确角色

知识库在 Agent 中承担的是：

1. **外部 / 项目资料的稳定知识来源**
2. **需要证据、事实、文档片段时的检索补强层**
3. **用户主动提供给 AI 的内容资产**

它不承担：

1. 当前轮状态托管
2. 主观偏好记忆
3. 行为方法与流程约束

补充边界：

1. 结构型资产属于知识库资产，但其运行时语义是结构参考，不是过程约束。
2. 工作流模板属于模板库资产，但其运行时语义是过程约束，不是结构参考。

## 五、知识库的写入与管理原则

## 5.1 写入原则

知识库写入必须体现用户意图。  
允许的主路径包括：

1. 用户拖拽上传文档
2. 用户点击上传文档
3. 用户对项目文档执行“存入知识库”
4. 用户确认替换已有知识条目

默认禁止：

1. AI 在未提示用户的情况下自动把当前生成结果写入知识库
2. AI 把任意引用内容自动归档为知识条目
3. 系统把所有项目文档自动全量收进知识库

## 5.2 “一键存入知识库”的规则

项目文档可以成为知识库来源，但条件是：

1. 用户明确触发“存入知识库”
2. 或者后续存在单独冻结的项目规则，定义明确的自动同步边界

在未冻结自动同步规则前，默认采用：

**用户显式存入优先**

## 5.3 用户主导的资产语义

知识库之所以偏用户侧，是因为：

1. 用户负责判断哪些文档值得长期管理。
2. 用户负责知识库名称、分类、条目命名与归属。
3. 用户负责删除、替换、重命名等管理动作。

AI 在知识库中扮演的角色更接近：

1. 检索者
2. 消费者
3. 可能的存入建议者

而不是资产所有者。

## 六、知识库的读取、检索与注入协同

## 6.1 用户显式引用优先

若用户通过 `@` 或拖拽明确引用知识库条目：

1. 它应被视为用户显式引用
2. 优先级进入 `用户引用` 层，而不是 `自动检索知识库` 层
3. 内容应作为完整、可溯源知识来源进入上下文

## 6.2 自动检索只做补强

AI 自动检索知识库时：

1. 只做补强，不替代当前文档事实
2. 不覆盖用户显式引用
3. 不覆盖当前轮状态与 artifact
4. 应明确标记来源为知识库

### 6.2.1 推荐上下文位置

在 Agent 场景下，知识库自动检索建议位于：

`用户显式引用 > 当前文档/选区 > 当前轮状态 / artifact > 记忆库 > 知识库 > 聊天历史`

这意味着知识库重要，但默认不高于：

1. 用户刚刚明确指定的材料
2. 当前文档现场事实
3. 当前轮协作现场状态

### 6.2.2 证据与引用职责

当任务需要证据、引用、出处时，知识库可以承担：

1. 候选证据来源
2. 文档内容召回来源
3. 可溯源知识片段来源

但它仍不等于：

1. 最终质量判断
2. 当前任务状态判断
3. 用户偏好判断

## 九、设计评审否决条件

以下任一情况成立，都应直接否决：

1. 让 AI 默认自动把当前结果写入知识库。
2. 让自动检索的知识库结果覆盖用户显式引用。
3. 让知识库承担当前轮状态或验证职责。
4. 未单独决策就把知识库逻辑下压到层次一或层次二。
