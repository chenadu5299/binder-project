## 一、先回答核心问题：这份文档应该怎么设计

这份文档不应该继续做成“另一份总规则合集”。  
如果它只是把已有规则重新写一遍，那么它会和 `331/333/314/204` 高度重叠，继续增加阅读负担，却不增加任何真实约束力。

这份文档应该承担的是下面四个职责：

### 1. 重新定义问题

Binder 现在的问题，不是“完全没有方案”，而是：

1. 已有方案分散在不同文档中，缺少统一的生效机制。
2. 约束主要停留在自然语言提示词层，缺少运行时硬约束。
3. 规则数量增加后，模型未必更稳定，反而可能出现注意力分散、任务理解变形、机械服从局部规则而忽略真实目标。
4. 缺少“任务是否真正完成”的机器级判定，因此任务容易在自然语言总结时提前结束。

### 2. 不直接写最终方案，先做研究与归因

这份文档的任务不是立刻拍板最终架构，而是先回答：

1. 模型本身擅长什么，不擅长什么。
2. Binder 的任务复杂度在哪些地方已经超出“单次提示词+单次生成”的舒适区。
3. 从模型到 agent 的演化，本质上是在解决什么矛盾。
4. 当前优质 agent 产品到底是靠什么稳定，而不是靠“提示词更长”稳定。
5. Binder 最应该学的不是表面功能，而是哪些技术抓手。

### 3. 把“有效约束”拆层

真正有效的约束，至少要分成五层：

1. **任务层约束**：目标、边界、完成条件、失败条件。
2. **工具层约束**：工具 schema、参数白名单、输入输出格式、可逆性。
3. **运行时约束**：权限、审批、hook、状态机、超时、重试、停止条件。
4. **记忆与上下文约束**：什么长期记，什么短期记，何时压缩，压缩后如何回注。
5. **评测约束**：如何知道 agent 真的变好，而不是只会写更像样的总结。

结论先写在前面：  
**提示词只是约束系统里最软的一层。**  
如果 Binder 继续把主要控制力压在提示词上，规则越多，越可能“看起来很严格，实际上更不稳定”。

### 4. 作为最终主控文档的前置研究底稿

这份文档的产出，不是直接替代 `331/333/314`。  
它应该为后续真正的主控文档提供研究结论，最终落成三类可执行资产：

1. 一份真正上位的 `AI行为与闭环约束主控文档`
2. 一组运行时机制：权限、校验、hook、停止条件、失败回路
3. 一套评测与回归机制：验证 agent 是否更稳定、更完整，而不是更会说

## 二、Binder 当前问题的真实形态：不是缺规则，而是规则没有形成约束闭环

从现有文档看，Binder 已经有不少正确方向：

1. `A-AG-M-T-01_ai执行架构.md` 已经定义了“请求构建、模型调用、工具执行、结果回流、失败处理”。
2. `A-AG-M-T-02_prompt架构.md` 已经定义了提示词构建链和变更治理。
3. `A-AG-M-T-03_任务规划执行.md` 已经提出“规划 -> 执行 -> 验证 -> 结束”的闭环。
4. `A-DE-M-P-01_对话编辑提示词.md` 已经把对话编辑做成了接近可执行规范的专项文档。
5. `R-AG-M-R-08_AI功能前置协议与标准.md` 已经开始把引用、记忆、知识库、工作流、输出格式协议化。

问题不在“方向错了”，而在“没有形成硬闭环”。

### 2.1 当前症状

你已经明确观察到了几个关键症状：

1. 稍复杂一点的修改，遗漏、错误很多。
2. 规则越堆越多，模型像被“降智”。
3. 任务一长，模型容易中途漂移、局部正确、整体失败。
4. 文档在往控制方向靠，但没有真正控制到任务结束。

### 2.2 根因判断

这类问题通常不是单一原因，而是四类问题叠加：

#### A. 目标控制弱于局部规则控制

模型容易记住局部格式规则、禁止项、字段名，但弱化了“真实任务目标”和“完成标准”。

结果是：

1. 局部听话，整体不对。
2. 会执行动作，但不会判断是否已经完成。
3. 会避免犯某些小错，但会犯更大的结构性错误。

#### B. 约束主要写在提示词里，没有落到系统结构里

如果“必须逐步验证”“必须不要越权”“必须失败后继续补救”“必须不能提前结束”都只是自然语言说明，那么模型不一定稳定遵守。

真正稳定的 agent，通常把这些约束下沉到：

1. 权限系统
2. 工具参数结构
3. 审批流
4. hook
5. 状态机
6. 停止条件
7. 评测回路

#### C. 没有把任务拆成“模型能稳定做对的最小动作”

复杂编辑任务对模型来说，往往同时包含：

1. 理解意图
2. 定位目标
3. 识别边界
4. 规划步骤
5. 执行编辑
6. 检查遗漏
7. 验证副作用
8. 生成用户可理解的结果说明

如果这些步骤混在一次生成里，失败率会快速上升。

#### D. 缺少“任务完成”的外部判定器

只靠模型自己说“已完成”，不构成闭环。  
只靠自然语言总结，也不构成闭环。

对 Binder 来说，至少需要外部可验证的结束条件，例如：

1. 目标文档是否正确
2. 目标区间是否全部处理
3. 是否存在未处理 diff
4. 是否有失败步骤未修复
5. 是否有验证步骤未执行

## 三、模型的能力及长处：应该顺着能力设计，而不是逆着能力堆规则

### 3.1 大模型真正稳定的长处

当前一流模型的强项，不是“绝对精确执行超复杂规则”，而是：

1. **语义理解强**：能理解模糊、高层、人类式目标。
2. **模式归纳强**：能从例子、上下文、局部结构中提炼风格和意图。
3. **语言变换强**：重写、归纳、扩写、压缩、翻译、风格调整都很强。
4. **局部规划强**：对 2-6 步的小型行动链，通常表现不错。
5. **工具借力能力强**：当工具接口清晰时，模型能显著变强。
6. **从反馈自修正能力强**：拿到测试失败、执行错误、校验结果后，往往比首次生成更稳。
7. **多源信息整合强**：能把文档、引用、外部搜索、历史对话合成一个回答或方案。

### 3.2 大模型天然不稳定的地方

模型也有一些对产品非常致命的弱点：

1. **长链任务稳定性差**：步骤一多，容易丢目标、丢约束、丢上下文。
2. **形式一致性差**：规则很多、字段很多时，容易混字段、漏字段、旧字段回流。
3. **全局完整性差**：局部看似不错，但整体漏项、漏边界、漏异常处理。
4. **停止时机不可靠**：很容易“看起来差不多了”就收尾。
5. **复杂状态管理弱**：业务状态、执行状态、逻辑状态一混，模型容易乱。
6. **对冲突指令敏感**：当系统提示、工具描述、上下文、历史消息有冲突时，模型常做折中而非严格选择。
7. **上下文污染敏感**：无关信息、历史信息、过长规范会稀释当前任务的有效注意力。

### 3.3 对 Binder 最重要的启示

Binder 的核心不是聊天，而是“面向文档和资源的复杂执行”。  
这意味着 Binder 不能只利用模型的语言能力，而必须让系统去补模型的弱项。

要点是：

1. 让模型做擅长的：理解、归纳、局部规划、候选生成、错误修正。
2. 不让模型独自承担不擅长的：状态一致性、权限裁决、完成判定、强格式约束、长链闭环控制。

## 四、Binder 的任务场景，哪些已经超出了模型舒适区

Binder 的 AI 不是单一问答，它覆盖的是“文档编辑 + 对话 + 引用 + 工作区 + 资源 + 未来构建模式”的复合场景。

### 4.1 这些场景本质上是高难任务

Binder 的典型复杂任务包括：

1. 在长文档中做多点局部修改，并保证风格、格式、结构一致。
2. 根据引用、选区、块列表、工作区文件判断真正目标对象。
3. 在多轮对话中延续任务，而不是每轮只答一句话。
4. 在工具调用后根据结果继续推进，而不是停在第一次调用。
5. 把“待确认 diff”“已接受 diff”“已失效 diff”“执行失败”区分清楚。
6. 后续延伸到构建模式时，还会遇到多步骤资料搜集、内容生成、审阅、重构、交付。

这些已经不是“提示词优化”能单独解决的问题，而是典型的 agent orchestration 问题。

### 4.2 Binder 场景对 agent 的特殊要求

Binder 和通用聊天 agent 不一样，它至少有五个强约束：

1. **对象不是纯文本，而是结构化文档**  
   需要位置、块、格式、状态一起成立。

2. **结果不是回答，而是可应用的行动产物**  
   例如 diff、文件修改、资源创建、工作流输出。

3. **很多任务必须可确认、可回退、可复核**  
   这要求 agent 不只是“能做”，还要“能停在正确的中间状态”。

4. **多轮上下文极易污染**  
   尤其对文档编辑来说，过长历史会明显影响当前轮的定位和边界判断。

5. **完成标准不是语言上的满意，而是系统状态上的闭合**  
   没有闭合，就不能算完成。

### 4.3 哪些需求已经明显超出“单模型 + 长提示词”

下面这些需求，如果还用“一个模型看完所有规则后一次性搞定”的方式，风险很高：

1. 长文档多点编辑
2. 多文件联动修改
3. 多轮追问后保持任务不漂移
4. 根据工具结果动态重规划
5. 自我检查遗漏并继续补做
6. 在不同模式间切换（聊天、编辑、构建）仍保持统一语义

结论：

**Binder 做的已经不是“给模型加点提示词”，而是在设计一个 agent 化编辑执行系统。**  
更准确地说，是在设计“面向文档生产场景的 agent runtime + constraint system”。

## 五、从模型到 agent：这个方向是怎么演化出来的

这一段不是为了写历史综述，而是为了判断 Binder 现在处在演化链条的哪一步。

### 5.1 第一阶段：单次生成

早期范式是：

1. 给用户输入
2. 拼接系统提示词
3. 模型一次性输出答案

解决的问题：

1. 自然语言理解
2. 文本生成
3. 简单改写和总结

暴露的矛盾：

1. 不能访问外部真实环境
2. 不能验证结果
3. 无法处理长链任务

### 5.2 第二阶段：RAG / 上下文增强

为了解决模型知识不足和上下文不足，引入了检索增强。

解决的问题：

1. 模型不掌握最新信息
2. 缺少项目知识、文档知识、工作区知识

暴露的矛盾：

1. 检索不等于行动
2. 有信息不等于会做事
3. 仍然无法闭环执行

### 5.3 第三阶段：工具调用

再往前一步，模型开始能够调用搜索、代码执行、文件编辑、浏览器等工具。

解决的问题：

1. 让模型接触外部环境
2. 让模型获得 ground truth
3. 让模型不只会“说”，还会“做”

这一阶段的代表性思想包括：

1. **ReAct（2022）**：把 reasoning 和 acting 交替起来，让模型边想边行动。
2. **Toolformer（2023）**：强调模型学会何时调用工具、调用什么工具、如何整合结果。

暴露的矛盾：

1. 会用工具，不代表任务会完成
2. 工具调用多了，错误也会累积
3. 若没有外部控制，模型很容易越权、乱调、早停

### 5.4 第四阶段：工作流与 agent

当任务进一步复杂，系统不再满足于“模型调一个工具”，而开始要求：

1. 分解任务
2. 动态决策下一步
3. 基于环境反馈持续迭代
4. 在必要时向人类请求确认

这就是 agent 真正形成的阶段。

Anthropic 在 2024-12-19 的《Building Effective AI Agents》中给出一个很重要的区分：

1. **Workflow**：LLM 和工具走预定义代码路径
2. **Agent**：LLM 动态决定自己的流程和工具使用

这个区分很关键。  
Binder 现在其实处在 **workflow 向 agent 过渡** 的阶段。

### 5.5 第五阶段：多 agent、记忆、运行时治理

再往后，单 agent 也开始不够用了，于是又出现：

1. 任务分工
2. 子 agent
3. 并行执行
4. 记忆分层
5. 压缩与恢复
6. 审批机制
7. hook 与自动化
8. Evals 与 tracing

这一步要解决的不是“能不能做”，而是：

1. 如何让 agent 在复杂任务中更稳
2. 如何防止上下文爆炸
3. 如何防止权限失控
4. 如何让系统能观察、回放、归因、迭代

### 5.6 Binder 正处在什么位置

Binder 现在最像：

1. 已完成基础三层能力设计
2. 已有工具调用和执行链
3. 已开始做任务闭环和状态分离
4. 但还没有形成成熟的 agent runtime

所以这项工作，确实可以叫“设计 agent”，但要更准确一点：

**Binder 正在做的，不是从 0 发明 agent，而是把已有 AI 能力产品化为一个可控的文档执行 agent 系统。**

## 六、不同类型的 agent，各自的技术方向与特色

不是所有 agent 都长一个样。  
如果类型判断错了，架构就会错。

### 6.1 编程 agent

典型代表：Claude Code、Codex 类、Cursor/Cline/Aider 一类系统。

技术特点：

1. 工具环境清晰：文件、命令、测试、git 都是强反馈环境。
2. 成功标准相对明确：能跑、能编译、测试通过、diff 正确。
3. 非常依赖工具设计和权限治理。
4. 很适合“执行 -> 失败 -> 修复 -> 再验证”的闭环。

为什么编程 agent 进展最快：

1. 外部反馈最强
2. 结果可验证
3. 工具空间稳定
4. 可做自动评测

### 6.2 任务执行 agent

典型任务：读资料、调用多个系统、发起动作、跟踪状态、交付结果。

技术特点：

1. 更依赖流程编排
2. 更依赖权限与审批
3. 结果不总是像编程那样可以自动判定
4. 更容易出现“做了很多动作但没完成目标”

### 6.3 研究 agent

典型任务：多轮搜索、筛选来源、交叉验证、汇总结论。

技术特点：

1. 强依赖搜索与浏览
2. 多源信息冲突处理重要
3. 需要引用、证据链、来源质量控制
4. 往往使用 orchestrator-worker 或多 agent 并行

### 6.4 办公/生产力 agent

典型任务：文档写作、汇总、排版、表格处理、资料整合。

技术特点：

1. 用户体验要求高
2. 格式、风格、结构一致性很重要
3. “接近正确但不够完整”是最常见失败模式
4. 很需要 checkpoint、diff、人工确认和局部重试

### 6.5 Binder 属于哪一类

Binder 不是纯编程 agent，也不是纯研究 agent。  
Binder 更接近：

1. **文档生产 agent**
2. **任务执行 agent**
3. 并且在未来会带上部分 **研究 agent** 和 **多 agent 构建 agent** 的属性

因此 Binder 最终不能只学编程 agent 的表层交互，而要学它们的底层稳定性机制，再结合文档场景做改造。

## 七、优质 agent 产品分析：重点看 Claude Code 到底强在哪里

> 这一节主要看技术抓手，不看产品宣传。

### 7.1 先下判断：Claude Code 的强，不主要来自“提示词更强”

公开材料显示，Claude Code 的强项更像是“完整的 agent runtime 设计”，而不是单纯靠一段神秘系统提示词。

从公开仓库和官方文档可见，Claude Code 至少有以下关键能力：

1. **权限系统**
2. **子 agent / agent teams**
3. **项目级记忆与规则**
4. **hooks**
5. **compact / 上下文压缩机制**
6. **技能、插件、MCP 扩展**
7. **Plan Mode**
8. **思考深度控制**
9. **工作流式命令和可复用配置**

这说明 Claude Code 实际上把“控制效果”放到了提示词之外的大量结构层中。

### 7.2 Claude Code 的关键技术抓手

#### A. 权限不是提示词约束，而是运行时制度

Claude Code 官方文档明确给出了多种权限模式，例如 `default`、`acceptEdits`、`plan`、`bypassPermissions`。  
这意味着“是否允许行动”不是靠模型自己拿捏，而是由运行时权限模式直接决定。需要编辑文件、跑命令、执行测试时，是否放行取决于系统权限制度，而不是单靠提示词。  
这是一种**运行时权限架构**，不是“请模型尽量谨慎”。

对 Binder 的启发：

1. 对文档修改、文件创建、批量改写、跨文档操作，都应有显式权限与审批语义。
2. 模型是否能做，不应只由提示词决定，还应由系统判定。

#### B. subagents 的价值不是“多角色好看”，而是隔离上下文与职责

Claude Code 官方文档里，subagent 的关键点不是人格，而是：

1. 单独上下文窗口
2. 独立工具权限
3. 聚焦任务
4. 降低主会话污染

这对 Binder 非常重要。  
Binder 后续如果进入构建模式，或者哪怕只是复杂文档任务，都可能需要：

1. 主 agent 负责总体目标和结束判断
2. 子 agent 负责检索、结构检查、格式检查、改写建议、遗漏检查

#### C. hooks 提供的是确定性控制

Claude Code hooks 官方文档有一句非常关键的话：  
hooks 提供的是 deterministic control，确保某些事情一定发生，而不是依赖 LLM 自己决定要不要做。

这是 Binder 当前最缺的部分之一。

Binder 现在很多规则仍然是：

1. 希望模型自觉检查
2. 希望模型自觉别越权
3. 希望模型自觉别提前结束

但高质量 agent 产品的思路是：

1. 用 hook、validator、gate 强制执行必要动作
2. 用系统外部逻辑拦住危险动作
3. 用结构化事件记录完整过程

#### D. compact / memory 机制说明：长任务不能只靠一个大上下文硬撑

Claude Code 公开文档里明确有：

1. `/compact`
2. `PreCompact` / `PostCompact`
3. 项目记忆 `CLAUDE.md`
4. 自动 memory 目录

这说明成熟 agent 已经接受一个现实：

**上下文不是越长越好，必须治理。**

对 Binder 的启发：

1. 长会话中必须有摘要、计划、状态、关键约束的再注入机制。
2. 历史消息不应原样无限堆积。
3. 对文档编辑来说，“当前任务状态摘要”比“完整聊天历史”更重要。

#### E. Plan Mode 说明：复杂任务需要“先分析后执行”的专门运行态

Claude Code 的 Plan Mode 明确把复杂任务分成：

1. 只读分析
2. 生成计划
3. 用户确认
4. 再进入修改

这和 Binder 当前的痛点高度相关。  
Binder 很多复杂编辑任务，最大的风险恰恰是模型在未充分分析前就开始动手。

#### F. extended thinking / effort control 的价值

Claude Code 把“思考深度”做成了显式控制项，而不是只靠用户说“认真想想”。

这说明：

1. 不同任务复杂度，需要不同推理预算。
2. 复杂任务不该和简单任务使用同一默认思考强度。
3. “多想一点”应该进入系统配置和调度，而不是留在随缘提示词里。

### 7.3 Claude Code 开源仓库能看出什么，不能看出什么

截至我这轮调研时，公开仓库 `anthropics/claude-code` 已经是大体量公开项目，仓库结构里可见 `plugins`、`.claude/commands`、`examples`、`.claude-plugin` 等目录。

可以看出的：

1. Claude Code 明确把“扩展面”和“项目级配置面”产品化了。
2. 它不是一个只会聊天的模型壳，而是一个可配置的执行系统。
3. 它把项目上下文、命令、插件、技能、子 agent、权限看成一等公民。

看不出的：

1. 公开仓库不等于全部内部 orchestrator 细节公开。
2. 顶级效果未必都来自仓库里可见的脚本层。
3. 不能假设“照搬仓库结构 = 复现效果”。

所以真正该学的不是仓库表皮，而是它体现出的产品哲学：

1. 提示词只是其中一层
2. 运行时机制比提示词更重要
3. 约束要变成系统能力，而不是文档倡议

## 八、前沿 agent 思维，哪些最适合 Binder

这一节只看和 Binder 匹配度高的。

### 8.1 前沿共识一：先用简单系统，必要时才升级复杂度

Anthropic 在《Building Effective AI Agents》中反复强调：

1. 先从最简单方案开始
2. 只有在效果确实提升时再增加 agent 化复杂度

这对 Binder 很重要。  
因为 Binder 当前的风险不是“能力不够花”，而是“复杂度已经上来了，但真正有效的层还没补齐”。

因此 Binder 不应该盲目上“大而全多 agent”，而应优先补最短板：

1. 完成判定
2. 运行时 gate
3. 上下文治理
4. 子任务拆分
5. 验证闭环

### 8.2 前沿共识二：工具接口设计比总提示词更关键

Anthropic 官方文章直接给出一个非常重要的经验：  
他们在做 SWE-bench agent 时，花在工具优化上的时间比花在总提示词上的时间还多。

这对 Binder 几乎是直接命中。

Binder 当前很多问题，很可能不是“模型太笨”，而是：

1. 工具接口对模型不够友好
2. 参数过多或语义相近
3. 输入格式不符合模型自然生成习惯
4. 缺少错误恢复友好的设计
5. 缺少让模型不容易犯错的防呆接口

因此 Binder 的后续重点，应该从“继续写更长主提示词”转向：

1. tool schema 重设计
2. 参数语义收敛
3. 输出结果结构化
4. 错误信息可修复
5. 工具结果直接服务下一步验证

### 8.3 前沿共识三：agent 要靠环境反馈，而不是自我想象

成熟 agent 强调的不是“多想”，而是：

1. 行动
2. 拿到环境返回结果
3. 依据结果修正
4. 再行动

对 Binder 来说，最该强化的是：

1. edit 工具返回必须足够结构化
2. 应用前后状态必须可比较
3. 失败必须可归因、可重试、可局部修正
4. agent 每一轮要基于真实系统状态，而不是基于自己上一轮描述

### 8.4 前沿共识四：多 agent 的主要价值是上下文管理和并行，不是人格扮演

Anthropic 2025-06-13 关于多 agent research system 的文章明确显示：

1. lead agent 负责规划与综合
2. subagents 并行执行不同检索任务
3. 明确分工、限制重复、控制 effort
4. 必须教会 orchestrator 怎样委派

对 Binder 的启发：

1. 多 agent 不是为了“几个 AI 讨论一下”
2. 多 agent 的核心价值是：
   - 降低主上下文压力
   - 让不同子任务各自聚焦
   - 并行处理复杂任务
   - 把复杂问题拆成可控问题

### 8.5 前沿共识五：评测是 agent 进化的真正基础设施

agent 系统如果没有 eval，很容易出现一种假进步：

1. 看起来更会解释
2. 看起来更有步骤感
3. 实际成功率没有提高，甚至更差

Binder 未来必须建设的，不只是文档，还包括：

1. 复杂编辑任务样本集
2. 多轮任务闭环样本集
3. 错误恢复样本集
4. 文档格式保真样本集
5. “未完成却总结完成”专项样本集

## 九、Binder 最相符的 agent 思路：不是更长提示词，而是更完整的 runtime

### 9.1 Binder 最终应该采用的不是“提示词中心主义”

如果继续以提示词为控制中心，Binder 很容易出现两个问题：

1. 规则越长，模型越不稳
2. 规则越细，模型越容易在局部服从中丢失整体目标

因此 Binder 更合理的方向是：

**Prompt-centered -> Runtime-centered**

也就是把控制核心从“写给模型看的规则”迁移到“系统真正执行的机制”。

### 9.2 一个更适合 Binder 的五层控制模型

#### 第一层：任务契约层

定义：

1. 本轮目标是什么
2. 不在范围内的是什么
3. 完成条件是什么
4. 失败条件是什么
5. 允许停在哪个中间态

这是 `A-AG-M-T-03_任务规划执行.md` 应该继续强化的层。

#### 第二层：agent 运行态层

为不同复杂度任务设计不同运行态：

1. 直接回答态
2. 只读分析态
3. 执行态
4. 审阅/验证态
5. 长任务压缩恢复态

这比“所有任务都走同一对话模式”更稳。

#### 第三层：工具与防呆层

这里是 Binder 成败关键。

应重点做：

1. 工具收敛
2. 参数简化
3. 结果结构化
4. 错误可修复
5. 强防呆接口

目标不是让模型“理解更多”，而是让模型“更难犯错”。

#### 第四层：运行时 gate 层

包括：

1. 权限审批
2. 前置校验
3. 后置验证
4. hook
5. 自动中断条件
6. 超时与最大步数
7. 部分成功与失败暴露

这是当前 Binder 最需要补的“硬控制层”。

#### 第五层：评测与观测层

没有这一层，永远不知道系统是在变好，还是只是在变复杂。

至少需要：

1. 任务轨迹记录
2. 工具调用统计
3. 失败分类
4. 漏改/误改统计
5. 完成率
6. 用户二次返工率

### 9.3 Binder 最应该优先补的不是“更多规则”，而是五件事

#### 1. 任务结束条件

要从“模型说结束”改成“系统判定可结束”。

#### 2. 复杂任务先进入分析态

复杂编辑不应直接执行，而应先形成：

1. 目标对象
2. 作用范围
3. 计划步骤
4. 风险点
5. 验证方式

#### 3. 工具防呆重构

对模型而言，工具设计本身就是约束设计。

#### 4. 上下文治理

长会话需要：

1. 任务摘要
2. 当前状态摘要
3. 关键约束摘要
4. 已完成/未完成项摘要

而不是只保留全量对话历史。

#### 5. 独立验证步骤

把“是否遗漏”“是否越界”“是否完成”从主生成步骤中拆出来，交给专门验证流程或子 agent。

## 十、对 Binder 的初步判断：我们现在做的，叫不叫 agent 设计

答案是：**叫，而且已经不是轻量级 agent 设计了。**

但需要区分三个层次：

### 10.1 如果只是在写提示词

这还不算真正的 agent 设计，只能算 prompt engineering。

### 10.2 如果已经在设计工具、状态、权限、回流、闭环

这已经进入 agent system design。

### 10.3 Binder 当前状态

Binder 已经明显跨过了单纯提示词阶段，因为你们已经在处理：

1. 工具调用
2. 文档状态
3. 任务规划
4. Diff 生效链
5. 引用协议
6. 执行与观测

所以严格说，Binder 现在做的是：

**面向文档编辑与构建场景的 agent 产品化设计。**

真正的问题不是“是不是 agent”，而是：

**这个 agent 目前还缺少成熟 runtime 和有效约束层。**

这份中间态文档后续应该收敛出下面几个正式产物，但现在不要急着一次性全写死。

5. 定义验证回路
6. 定义评测口径

推荐边界如下：

1. `A-AG-M-T-02_prompt架构.md`

   保留为系统执行架构

5. 新的上位文档

### 11.3 这份研究稿的下一步，不是继续加字，而是产出决策

下一步应该从这份研究稿中抽出可决策的问题：

1. Binder 的复杂任务是否引入“分析态”
2. 是否建立任务级结束判定器
3. 是否把验证从主生成中拆出
4. 是否引入子 agent / 专项验证 agent
5. 是否建立 compact / memory 再注入机制
6. 是否将 hook / validator / permission gate 引入运行时

## 十二、当前阶段的结论

### 12.1 最重要的判断

Binder 现在的核心问题，不是模型不够强，也不是方案完全缺失。  
核心问题是：

1. 已有方案没有形成真正的系统级约束闭环。
2. 主要控制力还停留在提示词层。
3. 复杂任务已经超出“长提示词 + 单 agent 自觉执行”的稳定边界。

### 12.2 对“规则让模型降智”的判断

这个判断大概率是对的。  
更准确地说，不是“规则本身让模型降智”，而是：

1. 规则过多但层级不清
2. 规则彼此竞争注意力
3. 规则没有下沉为结构约束
4. 规则缺少运行时执行器

于是模型会表现出一种典型现象：

1. 更谨慎
2. 更机械
3. 更会解释
4. 但不一定更会完成任务

### 12.3 Binder 的正确方向

Binder 的下一步，不应是继续单纯堆提示词。  
正确方向应当是：

1. 把控制中心从提示词迁移到 runtime
2. 把“完成任务”从模型自述改成系统判定
3. 把复杂任务拆成模型能稳定做对的小闭环
4. 把验证和纠错作为正式步骤，而不是附带要求
5. 把 agent 的成长建立在评测、观测、回放和迭代上
