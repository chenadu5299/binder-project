# 502_可观测性体系

## 一、文档定位

本文是验证与运维层的可观测性主规范，定义 AI 主链与对话编辑链的观测对象、事件模型、诊断路径与隔离边界。  
本文承接流式响应专项治理与状态边界规范，补齐全局可观测性体系。

---

## 二、MVP 目标

1. 建立统一观测模型：请求、流式、工具、状态、错误五类事件。  
2. 建立统一诊断路径：从用户反馈到根因定位可追踪。  
3. 建立状态流与观测流隔离：观测不替代业务状态判定。  

---

## 三、观测域模型

## 3.1 观测对象分层

1. 会话层：请求/响应、轮次、模型调用。  
2. 流式层：chunk、去重、完成标记、流中断。  
3. 工具层：tool_call、参数解析、执行结果、重试。  
4. 状态层：diff 状态流转、逻辑状态推进、revision 更新。  
5. 系统层：性能、超时、资源占用。  

## 3.2 核心事件结构

```ts
interface ExecutionExposure {
  eventId: string
  eventType: 'request' | 'stream' | 'tool' | 'state' | 'error'
  code: string
  level: 'info' | 'warn' | 'error'
  message: string
  context: Record<string, unknown>
  timestamp: number
}
```

---

## 四、关键观测链路

## 4.1 流式响应链

1. 请求发起。  
2. SSE chunk 接收。  
3. 重复 chunk 识别与过滤。  
4. 工具调用增量参数累积。  
5. 流结束与完成判定。  

最小观测码：

1. `E_STREAM_DUPLICATE_CHUNK`  
2. `E_STREAM_JSON_PARSE_FAILED`  
3. `E_STREAM_TOOL_ARGS_INCOMPLETE`  
4. `E_STREAM_INTERRUPTED`  

## 4.2 工具执行链

1. 工具调用触发。  
2. 参数解析与 schema 校验。  
3. 执行成功/失败与重试。  
4. 工具结果回流到会话。  

最小观测码：

1. `E_TOOL_ARGS_PARSE_FAILED`  
2. `E_TOOL_SCHEMA_INVALID`  
3. `E_TOOL_EXEC_FAILED`  
4. `E_TOOL_RETRY_EXHAUSTED`  

## 4.3 diff 状态链

1. diff 生成入池。  
2. pending -> accepted/rejected/expired 流转。  
3. 批量执行阶段性结果。  

最小观测码：

1. `E_DIFF_RANGE_UNRESOLVABLE`  
2. `E_DIFF_ORIGINALTEXT_MISMATCH`  
3. `E_DIFF_PARTIAL_OVERLAP`  
4. `E_DIFF_APPLY_FAILED`

## 五、隔离与边界规则

## 5.1 状态流与观测流隔离

1. 失效是业务状态流转，不是日志事件。  
2. 错误暴露是观测行为，不是状态判定依据。  
3. 二者允许同轮并存，但禁止互相替代。  

## 5.2 完成判定隔离

1. 执行完成：可执行项处理完成，失败项已记录并跳过。  
2. 业务完成：用户目标是否达成由结果内容判定。  
3. 观测层只提供证据，不直接下业务结论。  

---

## 六、诊断与排障流程

1. 采集：按 requestId / tabId / messageId 聚合事件。  
2. 定位：先看错误码，再看上下游链路事件。  
3. 归因：区分请求层、流式层、工具层、状态层问题。  
4. 处置：执行降级策略并记录修复行动。  

标准输出：

1. 复现条件。  
2. 影响范围。  
3. 根因说明。  
4. 回归验证结果。  

---

## 七、观测指标（MVP 最小集）

1. 流式成功率。  
2. 工具调用成功率。  
3. 平均响应耗时（首包/全包）。  
4. diff 执行成功率。  
5. 状态流转异常率。  

---

## 八、MVP 验收口径

1. 核心链路均有可追踪事件。  
2. 关键错误均可映射到统一错误码。  
3. 状态流与观测流边界可验证。  
4. 用户反馈可在日志中完成闭环诊断。
