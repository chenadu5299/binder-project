## 一、模型定义

Chat Build 的当前执行模型是：

> `单主控 + 冻结式 + 一次生成 + 只产生新资源`

这四个词必须同时成立。

## 二、单主控

当前版本只允许一个主控执行体负责：

1. 综合输入和引用。
2. 生成大纲。
3. 启动正式构建。
4. 推进构建过程。
5. 输出最终结果。

当前版本不允许：

1. 多角色 AI 讨论
2. 主控 AI 与角色 AI 分工协作
3. 真人参与者与 AI 混合会话

## 三、冻结式构建

本节的控制规则统一服从 `A-CBT-I-P-01` 中的：

1. `BR-CBT-VERIFY-001`
2. `BR-CBT-STATE-001`
3. `BR-CBT-STATE-002`
4. `BR-CBT-MODEL-001`

“冻结式”指的是：

1. 大纲确认前，需求仍可变化。
2. 大纲确认后，本轮构建目标被冻结。
3. 构建启动后，不再接受自然语义改向。

冻结式构建不是“不可中断”。

当前版本允许：

1. 手动中断

当前版本不允许：

1. 边构建边补需求
2. 边构建边换方向
3. 把构建中聊天当成新的目标输入

## 四、一次生成

当前版本采用“一次构建运行对应一次目标冻结”的模型。

它意味着：

1. 一轮 Build Outline 对应一轮正式构建。
2. 构建结束前不重新进入自由讨论。
3. 若目标变化，应重新发起新一轮 Chat Build。

## 五、输出边界

本节的输出边界统一服从：

1. `BR-CBT-ASSET-001`
2. `BR-CBT-DATA-001`

Chat Build 的产出边界是：

1. 只生成新的项目资源。
2. 不修改既有工作区内容。
3. 不通过当前对话编辑 Diff 主链生效。

这里的“项目资源”可以包括：

1. 新目录
2. 新文件
3. 相关资源文件
4. 构建过程产物索引或状态信息

但这些属于执行目标，不表示当前代码已具备完整落盘主链。

## 六、过程展示模型

当前版本的过程展示是“可读展示”，不是“高交互编辑”。

当前展示表面默认承接于 chat：

1. 构建前是讨论与需求收敛界面。
2. 构建中是状态、进度、过程说明与中断控制界面。

构建运行中应暴露的信息包括：

1. 当前阶段
2. 总体进度
3. 当前动作或步骤
4. 结果状态
5. 中断状态

当前版本不要求：

1. 构建中可编辑步骤图
2. 构建中多人批注
3. 构建中自由插入角色意见

## 七、与旧 BLD 执行模型的区别

旧 BLD 文档强调：

1. Discussion Build
2. Master AI / Role AI
3. Project Object
4. 讨论组状态机

当前 Chat Build 主线不采用这些对象作为前提条件。

当前版本只冻结：

1. 大纲确认闸口
2. 单主控执行
3. 构建中冻结需求
4. 只产出新资源
