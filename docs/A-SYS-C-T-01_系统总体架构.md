## 一、文档定位

本文定义 Binder 的系统总体架构（MVP 口径），回答“系统由哪些层组成、核心链路如何流转、跨模块约束如何统一”。  
本文是总体架构主文档，不替代子系统实现文档。

---

## 二、架构目标

1. 以 Workspace 工作台作为 Binder 的工作基础，用统一主链支撑：Workspace 文件管理 + 文档编辑 + AI 三层 + 对话编辑确认生效。  
2. 保证主链可观测、可回退、可复现。  
3. 控制复杂度：分层清晰、边界明确、模块可独立演进。  

---

## 三、系统边界（MVP）

纳入：
1. 本地工作区管理。  
2. 文档编辑主链（md/txt/html/docx）。  
3. AI 三层能力（层次一/二/三）。  
4. 对话编辑 Diff 决策主链。  

不纳入发布阻塞：
1. 移动端。
2. Excel/演示文稿完整编辑。
3. Chat Build 完整工程主链（当前文档主线已生效，工程链路仍在后续阶段推进）。
4. 分页/T-DOCX 作为首版前置条件。

---

## 四、总体分层架构

## 4.1 五层模型

1. 产品交互层（UI Layer）  
负责界面与交互入口：文件树、编辑器、聊天面板、Diff 卡片。  

2. 应用编排层（Application Layer）  
负责任务编排与状态协调：场景路由、模式切换、任务状态推进。  

3. 能力服务层（Capability Layer）  
负责核心能力实现：Workspace 服务、编辑器服务、AI 服务、引用与上下文服务。  

4. 执行与协议层（Execution Layer）  
负责工具调用、策略执行、Diff 生成、错误分级与观测。  

5. 基础设施层（Infrastructure Layer）  
负责文件系统、缓存、索引、持久化、模型 Provider 连接。  

## 4.2 分层约束

1. 上层可依赖下层，下层不可反向依赖上层。  
2. 写入文档的动作只能通过执行层统一落盘。  
3. 任何 AI 改写必须经过执行层协议，不允许 UI 侧绕过。  

---

## 五、核心子系统视图

1. Workspace 子系统  
管理工作区、文件树、资源与跨文档关系，并作为其他能力进入系统的基础工作台框架。  

2. 编辑器子系统  
管理文档打开、编辑、保存、回读与编辑状态。  

3. AI 子系统（三层）  
层次一辅助续写、层次二局部修改、层次三对话编辑。  

4. 对话编辑子系统  
负责 Intent/策略/工具调用、Diff 生成、卡片决策、生效与失效。  

5. 可观测与恢复子系统
负责错误码、关键日志、失败暴露、跳过继续。

6. Chat Build 承接层（当前唯一生效构建主线）
当前项目级构建能力以 Chat Build 为唯一生效主线，挂接在现有 chat / agent / workspace / template / task-artifact / reference 等通用基础设施之上。
它当前定义为：单主控、冻结式、只生成新资源。
当前不包含 discussion room、role AI、participant、project object 独立主链。

未来说明：

1. 旧 BLD / Discussion Build / Multi-Actor Build 只保留为参考层。
2. 若未来重启，应作为 future/reference 架构专题单独规划，不属于当前系统架构事实。

补充定位：

1. Binder 不是“先有 AI，再给 AI 补一个工作区壳”。  
2. Workspace 工作台是系统基础层，其他功能更多是在该框架内完成资源进入、任务分发、结果呈现和受控回流。  
3. 任何能力若绕开 Workspace 工作台直接建立平行主链，都应视为架构越界。  

---

## 六、主链数据流（MVP）

## 6.1 文档主链

用户操作 -> 编辑器写入 -> 保存 -> 缓存/文件系统同步 -> 回读验证。  

## 6.2 AI 主链

用户输入 -> 上下文构建 -> AI 调用/工具执行 -> 结果回流 -> 用户决策 -> 生效。  

## 6.3 对话编辑主链

对话请求 -> Resolver 定位 -> 生成 canonical diff -> Diff 卡展示 -> 接受/拒绝 -> 文档更新。  

## 6.4 Chat Build 主链（当前文档主线）

chat 自然讨论 -> 构建触发 -> 轻量确认 -> Build Outline 生成 -> 大纲确认 -> 正式构建 -> 状态/进度/可读过程展示 -> 完成/失败/中断。

补充约束：

1. 大纲确认前仍属于可讨论阶段。
2. 大纲确认后进入冻结式构建。
3. 构建中只接受手动中断，不接受自然语义改向。
4. 构建输出只允许写入新的项目资源，不修改既有内容。

---

## 七、横切架构约束

1. 单一真源  
同一业务语义只允许一个状态真源，禁止并行真源。  

2. 先确认后生效  
对话编辑改写默认先经 Diff 决策，再写入文档。  

3. 可观测优先  
失败可暴露、可分级、可复现。  

4. 边界稳定  
模式切换不改变执行协议，不引入隐式分支。  

5. 大纲确认硬边界  
对应 `BR-CBT-VERIFY-001`。Chat Build 必须先完成大纲确认，才允许进入正式构建。  

6. 构建冻结规则  
对应 `BR-CBT-STATE-001`。Chat Build 一旦进入正式构建，不接受自然语义打断、修改或重定向。  

7. 新资源写入边界  
对应 `BR-CBT-ASSET-001` 与 `BR-CBT-DATA-001`。Chat Build 当前只生成新的项目资源，不修改既有工作区内容。  

---

## 八、MVP 架构验收口径

1. 五层模型在代码与文档中有明确映射。  
2. 核心主链（Workspace/编辑/AI/对话编辑）可端到端跑通。  
3. Diff 生效链与错误暴露链可独立验证。  
4. 架构边界与场景边界一致，不出现“文档纳入但架构缺位”。
