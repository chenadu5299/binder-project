## 一、文档定位

本文定义 Binder 的系统总体架构（MVP 口径），回答“系统由哪些层组成、核心链路如何流转、跨模块约束如何统一”。  
本文是总体架构主文档，不替代子系统实现文档。

---

## 二、架构目标

1. 以 Workspace 工作台作为 Binder 的工作基础，用统一主链支撑：Workspace 文件管理 + 文档编辑 + AI 三层 + 对话编辑确认生效。  
2. 保证主链可观测、可回退、可复现。  
3. 控制复杂度：分层清晰、边界明确、模块可独立演进。  

---

## 三、系统边界（MVP）

纳入：
1. 本地工作区管理。  
2. 文档编辑主链（md/txt/html/docx）。  
3. AI 三层能力（层次一/二/三）。  
4. 对话编辑 Diff 决策主链。  

不纳入发布阻塞：
1. 移动端。
2. Excel/演示文稿完整编辑。
3. 构建模式全流程（BLD 模块，设计已完成，见 `A-BLD-C-D-01`；工程实现在 MVP 后推进）。
4. 分页/T-DOCX 作为首版前置条件。

---

## 四、总体分层架构

## 4.1 五层模型

1. 产品交互层（UI Layer）  
负责界面与交互入口：文件树、编辑器、聊天面板、Diff 卡片。  

2. 应用编排层（Application Layer）  
负责任务编排与状态协调：场景路由、模式切换、任务状态推进。  

3. 能力服务层（Capability Layer）  
负责核心能力实现：Workspace 服务、编辑器服务、AI 服务、引用与上下文服务。  

4. 执行与协议层（Execution Layer）  
负责工具调用、策略执行、Diff 生成、错误分级与观测。  

5. 基础设施层（Infrastructure Layer）  
负责文件系统、缓存、索引、持久化、模型 Provider 连接。  

## 4.2 分层约束

1. 上层可依赖下层，下层不可反向依赖上层。  
2. 写入文档的动作只能通过执行层统一落盘。  
3. 任何 AI 改写必须经过执行层协议，不允许 UI 侧绕过。  

---

## 五、核心子系统视图

1. Workspace 子系统  
管理工作区、文件树、资源与跨文档关系，并作为其他能力进入系统的基础工作台框架。  

2. 编辑器子系统  
管理文档打开、编辑、保存、回读与编辑状态。  

3. AI 子系统（三层）  
层次一辅助续写、层次二局部修改、层次三对话编辑。  

4. 对话编辑子系统  
负责 Intent/策略/工具调用、Diff 生成、卡片决策、生效与失效。  

5. 可观测与恢复子系统
负责错误码、关键日志、失败暴露、跳过继续。

6. 构建模式子系统（BLD，非 MVP）
独立于编辑模式的项目构建引擎：主控 AI 编排、角色 AI 协作、Project Object 写入 Workspace。
不接入 AI 三层执行链和 Diff 机制，通过独立构建命令与 Workspace 文件系统交互。

补充定位：

1. Binder 不是“先有 AI，再给 AI 补一个工作区壳”。  
2. Workspace 工作台是系统基础层，其他功能更多是在该框架内完成资源进入、任务分发、结果呈现和受控回流。  
3. 任何能力若绕开 Workspace 工作台直接建立平行主链，都应视为架构越界。  

---

## 六、主链数据流（MVP）

## 6.1 文档主链

用户操作 -> 编辑器写入 -> 保存 -> 缓存/文件系统同步 -> 回读验证。  

## 6.2 AI 主链

用户输入 -> 上下文构建 -> AI 调用/工具执行 -> 结果回流 -> 用户决策 -> 生效。  

## 6.3 对话编辑主链

对话请求 -> Resolver 定位 -> 生成 canonical diff -> Diff 卡展示 -> 接受/拒绝 -> 文档更新。  

---

## 七、横切架构约束

1. 单一真源  
同一业务语义只允许一个状态真源，禁止并行真源。  

2. 先确认后生效  
对话编辑改写默认先经 Diff 决策，再写入文档。  

3. 可观测优先  
失败可暴露、可分级、可复现。  

4. 边界稳定  
模式切换不改变执行协议，不引入隐式分支。  

---

## 八、MVP 架构验收口径

1. 五层模型在代码与文档中有明确映射。  
2. 核心主链（Workspace/编辑/AI/对话编辑）可端到端跑通。  
3. Diff 生效链与错误暴露链可独立验证。  
4. 架构边界与场景边界一致，不出现“文档纳入但架构缺位”。
