## 一、记忆库定位与架构

### 1.1 功能定位

记忆库是 Binder 对话编辑（层次三）的**上下文增强系统**，为 Agent/Chat 模式下的 AI 对话提供**持久化语义记忆**，支持跨轮次、跨会话的上下文连贯性。

**核心价值**：
- 使 AI 能在对话范围内保持连贯理解
- 支持用户通过引用快速注入相关记忆
- 与引用系统、知识库、模板库协同，形成完整的上下文增强体系

### 1.2 与 AI 三层架构的关系

| 层次 | 记忆库接入 | 说明 |
|------|------------|------|
| 层次一：辅助续写 | 可选接入 | 当前可接入工作区级记忆；与标签级记忆库隔离 |
| 层次二：局部修改 | 不接入 | 弹窗内无历史、无工具，不依赖记忆库 |
| 层次三：对话编辑 | **必须接入** | 每个聊天标签（Agent/Chat）拥有独立记忆库 |

**上下文优先级**（CCE 简化，见 AI 与其他功能交互规范）：
```
用户引用 > 当前文档/选区 > 工作区相关 > 标签记忆 > 项目内容记忆 > 长期记忆 > 用户级记忆 > 知识库 > 聊天历史
```

### 1.3 树状结构设计

**记忆库采用树状结构**，便于层级浏览、按需展开和引用。

**树状层级**：

```
用户级记忆库（跨项目，见第四章）
└─ 用户偏好 / 跨项目概念 / 操作模式等

工作区记忆库（项目锚定，切换项目后不复存在）
├─ 项目内容记忆（见第五章）
│  └─ 按项目类型适配：人物/剧情/章节（小说）| 术语/结论/数据（报告）| 接口/模块/依赖（技术）等
│
├─ 长期记忆库（见第三章）
│  └─ 概念记忆 / 结构记忆 / 知识记忆 / 标签删除记录等（不可修改）
│
└─ 标签级记忆库（每个 Agent/Chat 标签独立，随标签删除而删除）
   ├─ 聊天标签 A（如 "报告写作"）
   │  ├─ 会话记忆
   │  ├─ 实体记忆
   │  └─ 文档记忆
   │
   └─ 聊天标签 B（如 "项目规划"）
      └─ ...
```

**设计原则**：
- **标签隔离**：每个聊天标签（Agent/Chat）拥有**独立**的记忆库，互不干扰
- **树状展开**：支持按层级展开/折叠，便于浏览和选择
- **单节点可引用**：支持对单个记忆项或记忆节点进行引用

## 二、标签级记忆库（核心）

### 2.1 独立记忆库原则

**每个聊天标签应有独立的记忆库**。

| 维度 | 要求 |
|------|------|
| **存储隔离** | 标签 A 的记忆与标签 B 的记忆完全隔离，不共享 |
| **检索范围** | 用户在该标签内对话时，仅检索该标签的记忆库 |
| **跨标签引用** | 仅当用户**主动 @聊天名称** 时，才读取该标签的对话上下文；默认不读取 |
| **生命周期** | 标签删除时，其记忆库**随标签一并删除**；删除行为及标签大致记忆会写入长期记忆库（见 2.4） |

### 2.2 记忆库内容来源

**自动生成**，来源包括但不限于：

| 来源 | 说明 | 触发时机 |
|------|------|----------|
| 对话摘要 | AI 对多轮对话的摘要提炼 | 对话轮次达到阈值或会话结束时 |
| 实体提取 | 从对话中提取人物、事件、概念、关系等 | 对话结束后或按需 |
| 文档关联 | 用户引用的文档、选中的文本的摘要 | 用户引用或编辑时 |
| 用户偏好 | 用户表达的风格、术语偏好等 | 对话中自然识别 |

### 2.3 自动生成与不可修改

**记忆库需要自动生成，用户不可修改**。

| 规则 | 说明 |
|------|------|
| **自动生成** | 系统根据对话内容、引用文档等自动提取并写入记忆 |
| **用户不可编辑** | 用户不能直接修改记忆项内容；避免与 AI 生成逻辑冲突 |
| **删除方式** | 记忆随标签删除而删除；用户不单独删除记忆项 |
| **用户可引用** | 用户通过拖拽或 @ 引用记忆，不改变记忆本身 |

**设计理由**：记忆库为 AI 提供上下文，若用户可随意修改，可能导致不一致；自动生成保证与对话内容一致。参考 Cursor 的「Remember...」与事实自动捕获机制。

### 2.4 删除与长期记忆的联动

**记忆库的删除跟随聊天标签的变动而变动**，且**不影响长期记忆**。

| 规则 | 说明 |
|------|------|
| **标签删除 → 标签记忆删除** | 用户删除聊天标签时，该标签的记忆库**一并删除** |
| **长期记忆不受影响** | 标签级记忆的删除**不修改**长期记忆库中已写入的内容 |
| **删除行为写入长期记忆** | 用户删除对话标签时，**删除行为**及**此对话标签的大致记忆**（摘要）会写入长期记忆库 |
| **长期记忆不可修改** | 记录进长期记忆库的内容**不再修改**，仅追加 |

**写入长期记忆的「标签删除记录」示例**：
- 删除时间
- 被删标签名称
- 该标签对话的大致主题/摘要（供日后跨项目参考）

## 三、长期记忆库（项目锚定）

### 3.1 是否构建独立长期记忆库

**建议**：构建**工作区级（项目锚定）**的长期记忆库，与标签级记忆库并存。

| 类型 | 作用域 | 锚定 | 内容 | 用途 |
|------|--------|------|------|------|
| **标签级记忆库** | 单个聊天标签 | 项目 | 该标签对话的摘要、实体、关联 | 该标签内 AI 上下文 |
| **长期记忆库** | 整个工作区 | **项目** | 跨标签的概念、结构、知识、标签删除记录 | 新标签或跨标签场景 |

**长期记忆库内容**（参考 PMS 项目记忆系统）：
- **概念记忆**：项目关键概念与主题
- **结构记忆**：项目文档结构、章节关系
- **知识记忆**：从文档、数据集提取的知识
- **工作流记忆**：构建过程中发现的模式（可选）
- **标签删除记录**：用户删除标签的行为及该标签的大致记忆摘要

**长期记忆库不可修改**：一旦写入，**不再修改**，仅支持追加新记忆。

### 3.2 项目锚定与切换

**长期记忆库锚定于项目**：

| 场景 | 行为 |
|------|------|
| **用户切换项目** | 长期记忆库、标签级记忆库**均不复存在**（不随用户进入新项目） |
| **用户返回原项目** | 该项目的长期记忆库、标签级记忆库**恢复可用** |
| **用户级记忆** | 切换项目后**继承**，见第四章 |

### 3.3 与标签级记忆库的关系

```
长期记忆库（工作区级，项目锚定）
    └─ 作为「全局上下文」供各标签按需检索
    └─ 标签级记忆库优先；长期记忆库补充
    └─ 接收标签删除时的「删除行为 + 标签摘要」写入

标签级记忆库（每个标签独立）
    └─ 该标签专属，检索时优先使用
    └─ 随标签删除而删除
```

**检索优先级**：当前标签记忆库 > 项目内容记忆 > 长期记忆库 > 用户级记忆库 > 知识库 > 聊天历史

## 四、用户级记忆库（跨项目）

### 4.1 与长期记忆库的区别

**用户级记忆**区别于长期记忆，二者锚定不同、生命周期不同：

| 维度 | 长期记忆库 | 用户级记忆库 |
|------|------------|--------------|
| **锚定** | 项目/工作区 | **用户** |
| **切换项目后** | 不复存在（不随用户进入新项目） | **继承**，随用户进入新项目 |
| **进化方式** | 仅追加，不修改 | 用户操作各项目时**关联进化** |
| **内容** | 项目内概念、结构、知识、标签删除记录 | 用户偏好、跨项目概念、操作模式、通用知识 |

### 4.2 用户级记忆的特性

| 规则 | 说明 |
|------|------|
| **跨项目继承** | 用户切换项目后，用户级记忆**保留并可用** |
| **关联进化** | 用户在不同项目中操作时，用户级记忆随使用**持续关联、进化** |
| **与项目记忆隔离** | 长期记忆、标签记忆为项目私有；用户级记忆为用户全局 |
| **检索时机** | 在当前项目内对话时，可作为补充上下文（优先级低于标签记忆、长期记忆） |

### 4.3 用户级记忆内容示例

- **用户偏好**：写作风格、术语习惯、常用模板
- **跨项目概念**：用户常涉及的主题、领域知识
- **操作模式**：用户常用的工作流、决策模式
- **通用知识**：与具体项目无关但用户常用的知识

### 4.4 三层记忆的关系

```
用户切换项目
    │
    ├─ 长期记忆库、标签级记忆库：不复存在（不进入新项目）
    │
    └─ 用户级记忆库：继承，随用户进入新项目，并在新项目中关联进化

用户在新项目中操作
    │
    ├─ 新项目的标签级记忆、长期记忆：从零开始积累
    └─ 用户级记忆：持续可用，并根据新项目操作关联进化
```

## 五、项目内容记忆（文档衍生、可更新）

### 5.1 定位与必要性

**项目内容记忆**用于存储从项目文档中自动提取的**语义内容**，如人物关系、关键剧情、章节结构、术语结论等。此类记忆**依赖项目结构理解**，且需**随文档变化而更新**，与长期记忆（不可修改）不同。

**典型场景**：
- **小说项目**：人物关系、关键剧情、章节概要、时间线
- **报告/研究项目**：核心术语、结论、数据要点、章节逻辑
- **技术/代码项目**：模块职责、接口约定、依赖关系、关键设计决策

### 5.2 项目结构理解

**项目内容记忆需要理解项目结构**，才能按结构正确提取和组织记忆。

| 能力 | 说明 |
|------|------|
| **项目类型识别** | 根据文档结构、命名、内容特征识别项目类型（小说/报告/技术等） |
| **结构解析** | 解析文档树、章节层级、文件组织方式 |
| **按类型适配** | 不同项目类型采用不同的提取策略和记忆 schema |

**项目类型与记忆 schema 示例**：

| 项目类型 | 典型结构 | 提取的记忆类型 |
|----------|----------|----------------|
| **小说/创作** | 章节、分卷、角色表 | 人物、关系、剧情、地点、时间线、章节概要 |
| **报告/研究** | 章节、附录、图表 | 术语、结论、数据要点、章节逻辑、引用 |
| **技术/代码** | 模块、目录、文档 | 模块职责、接口、依赖、设计决策、关键概念 |
| **通用/混合** | 文档树 | 概念、实体、关系、文档摘要 |

**非小说场景**：并非所有项目都是小说格式，系统需根据实际项目结构动态适配，避免强行套用小说 schema。

### 5.3 与长期记忆的区分

| 维度 | 项目内容记忆 | 长期记忆库 |
|------|--------------|------------|
| **来源** | 项目文档内容 | 对话、标签删除等 |
| **可修改性** | **可更新**，随文档编辑/保存刷新 | **不可修改**，仅追加 |
| **结构依赖** | **依赖**项目结构理解 | 不依赖 |
| **典型内容** | 人物、剧情、术语、模块等 | 概念、标签删除记录等 |

### 5.4 更新与生命周期

| 规则 | 说明 |
|------|------|
| **触发更新** | 文档打开、保存、批量处理时，按需提取或更新 |
| **增量更新** | 大文档可采用「大纲 + 按需深入」策略（见记忆库功能需求文档） |
| **项目锚定** | 切换项目后不复存在；返回原项目时恢复可用 |
| **自动生成** | 系统自动提取，用户不可编辑 |

### 5.5 与项目结构信息的协作

项目内容记忆**依赖**项目结构，但不替代结构信息的实时获取：

| 信息 | 归属 | 说明 |
|------|------|------|
| **文档树、章节关系** | 上下文/工具 | 实时获取，供项目内容记忆提取时使用 |
| **提取的语义内容** | 项目内容记忆 | 人物、剧情、术语等，可更新 |
| **结构记忆（静态快照）** | 长期记忆库 | 若需持久化结构快照，可写入长期记忆（不可修改） |

## 六、项目结构信息的归属

### 6.1 问题

项目结构（文档树、章节关系、文件路径等）在记忆体系中的归属？

### 6.2 建议方案

**项目结构信息**作为**独立的上下文层与工具能力**，供项目内容记忆提取时使用，不直接放入记忆库：

| 信息类型 | 归属 | 说明 |
|----------|------|------|
| **文档树结构** | 上下文提示词（第二层） | 当前打开的文档、工作区路径、编辑器状态 |
| **章节关系** | 上下文或工具 | 通过 `list_documents`、`get_project_summary` 等获取 |
| **结构解析结果** | 项目内容记忆提取的输入 | 供按项目类型适配的提取逻辑使用 |

### 6.3 数据流示意

```
用户发送消息
    │
    ├─ 上下文层：当前文档、选区、工作区路径、编辑器状态（含 block 结构）
    ├─ 引用层：用户引用的文本、文件、记忆库项等
    ├─ 记忆库层：检索 标签记忆 + 项目内容记忆 + 长期记忆 + 用户级记忆
    └─ 工具层：需要时调用 list_documents、get_project_summary 等

项目内容记忆提取（后台）
    │
    ├─ 输入：项目结构（文档树、章节）、文档内容
    ├─ 项目类型识别 → 选择提取 schema
    └─ 输出：写入项目内容记忆，随文档更新而更新
```

## 七、引用与 @ 交互

### 7.1 拖拽引用

**单个记忆库支持用户拖拽到聊天窗口作为引用**。

| 能力 | 要求 |
|------|------|
| **拖拽对象** | 单个记忆项、记忆节点（如某类型下的所有记忆）、或整个记忆库 |
| **拖拽目标** | 聊天输入框区域 |
| **引用展示** | 在输入框中显示引用标签或引用块 |
| **引用传递** | 前端将 references 传递给 ai_chat_stream，格式见 AI 与其他功能交互规范 |

### 7.2 @ 语法

**用户输入 @ 后，记忆库作为可选项出现**。

| 能力 | 要求 |
|------|------|
| **触发** | 用户输入 `@` 后弹出候选列表 |
| **候选内容** | 记忆库项、记忆节点（如「人物」「事件」）、文件、知识库等 |
| **记忆库展示** | 以树状或列表形式展示可引用的记忆项 |
| **选择** | 用户选择后插入引用标签 |

### 7.3 引用格式协议

与 AI 与其他功能交互规范一致：

| 引用类型 | type | 说明 |
|----------|------|------|
| 记忆库引用 | `memory` | 记忆项 content 完整传入 |

**前端 → 后端**：
```typescript
references?: Array<{
  type: 'memory';
  source: string;   // 记忆库名称或记忆项 ID
  content: string;   // 记忆项完整内容
}>;
```

**后端拼接**：作为「记忆库引用」段落，标注「以下内容已完整传入，无需再读取」。

## 八、记忆库检索协议

### 8.1 接口协议

与 AI 功能前置协议与标准、AI 与其他功能交互规范对齐：

| 协议项 | 内容 |
|--------|------|
| **命令** | `search_memories(query: string, options?: { tabId?, workspacePath?, limit?, scope? })` |
| **scope** | `tab`（仅当前标签）\| `content`（项目内容记忆）\| `workspace`（长期记忆库）\| `user`（用户级记忆）\| `all` |
| **返回** | `{ items: [{ id, content, metadata?, source?, tabId? }], total?: number }` |
| **拼接方式** | 作为「记忆库信息」段落，插入上下文；明确标注来源 |

### 8.2 检索时机

| 场景 | 检索范围 | 说明 |
|------|----------|------|
| 用户发送消息 | 标签记忆 + 项目内容记忆 + 长期记忆 + 用户级记忆 | 自动检索相关记忆注入上下文 |
| 用户 @ 引用 | 标签记忆 / 项目内容记忆 / 长期记忆 / 用户级记忆 | 用户主动选择，直接传入 |
| 用户拖拽引用 | 无需检索 | 内容已完整传入 |

### 8.3 降级策略

- 检索失败：静默返回空数组，不阻塞主流程
- 超时：500ms 内未返回则使用空数组
- 无 tabId：不检索标签级记忆；仅检索工作区级（如有）

## 九、UI 与展示

### 9.1 记忆库面板

| 位置 | 说明 |
|------|------|
| **入口** | 标签栏或资源管理区添加记忆库标签（图标：🧠） |
| **切换** | 点击切换到记忆库面板 |
| **展示** | 树状结构展示当前标签的记忆库；支持切换标签查看不同标签的记忆 |

### 9.2 树状展示规则

- **顶层**：可按来源分组（标签记忆、项目内容记忆、长期记忆）
- **标签记忆内**：按类型分组（会话记忆、实体记忆、文档记忆等）
- **项目内容记忆内**：按项目类型适配的 schema 分组（如人物/剧情/章节、术语/结论等）
- **展开/折叠**：支持点击展开或折叠任意层级
- **数量标识**：每个节点显示包含的记忆数量

### 9.3 只读展示

- 记忆项为**只读**，用户不可编辑内容
- 支持查看详情、引用（拖拽或 @）；记忆随标签删除而删除，无单独删除

## 十、与现有实现的关联

### 10.1 现有记忆库

当前存在 `memory_service`、`search_memories`、`MemoryTab` 等，按**文档路径**分组，**工作区级**。

**本需求文档与之关系**：

- 需新增**标签级记忆库**，与标签（tab_id）绑定
- 需新增**长期记忆库**（不可修改）与**用户级记忆库**（跨项目）

### 10.2 与引用系统

- 记忆库引用类型已定义：`reference.ts` 中的 `Memory`
- 需支持「选中某一条拖入」与「@ 记忆库名称」两种方式
- 需确保 references 传递给 ai_chat_stream（当前待实现）

### 10.3 与知识库、模板库

- 记忆库、知识库、模板库均支持 @ 引用与选中拖入
- 三者格式应对称，便于统一 @ 候选列表的展示

## 十一、边界与限制

### 11.1 不支持的场景

- 辅助续写、局部修改：不接入标签级记忆库（可接入工作区级，见 1.2）
- 用户直接编辑记忆内容：不支持
- 跨标签自动读取：默认不读取；仅 @ 聊天名称时读取

### 11.2 性能考虑

- 单标签记忆数量：建议不超过 200 条（可配置）
- 检索超时：500ms
- 大记忆库：可考虑分页或按需加载

### 11.3 数据管理

- **标签删除**：其记忆库**随标签一并删除**；删除行为及标签摘要写入长期记忆库
- **工作区切换**：长期记忆、标签记忆按 workspace_path 隔离，**不复存在**；用户级记忆**继承**
- **长期记忆**：一旦写入**不再修改**，仅追加
