## 一、文档定位

本文是 Binder 模板库模块的**开发计划文档**。

本文回答：

1. 模板库模块在当前代码与协议基础上应如何分阶段落地。
2. 哪些能力可以直接承接现有实现，哪些必须新建。
3. 工作流模板从资产创建到解析、编译、运行时绑定、执行协同、状态观测应按什么顺序推进。
4. 每个阶段的前置依赖、产出物、风险点与完成判定是什么。
5. 描述文档与技术文档中的核心规则，分别由哪个开发阶段和哪个任务项承接。

本文不负责：

1. 数据表最终字段定稿。
2. API 详细字段清单。
3. UI 视觉稿与交互稿。
4. parser / compiler / runtime 的实现代码细节。
5. sprint 排期、工时估算与人员分配。

本文服务的开发目标是：

**在不回退任何上游决策的前提下，使模板库模块形成“工作流模板资产 -> 结构解析 -> 流程编译 -> 运行时 plan -> 统一执行模型 -> 可观测状态”的最小可用实现闭环。**

## 三、当前实现基础扫描

### 3.1 当前可复用基础

结合当前代码与文档，模板库模块并非从零开始，但现有可复用基础主要分布在 AG、WS、AST 的承接链上。

#### 3.1.1 Agent 主链与运行态基础

当前可直接复用：

1. `src-tauri/src/commands/ai_commands.rs`
   - 已存在 L3 对话执行主链、工具调用循环、任务 stage 事件、artifact 持久化骨架。
2. `src/stores/chatStore.ts`
   - 已存在聊天 tab、workspace 绑定、Agent 模式入口。
3. `src/stores/agentStore.ts`
   - 已存在运行态壳、任务恢复、stageState 承接位。

这说明：

1. 模板库后续接入不需要另起一套独立执行系统。
2. `RuntimeWorkflowPlan`、`ExecutionState` 可以挂接现有 Agent 主链，而不是新造并行执行框架。

#### 3.1.2 上下文注入与装配基础

当前可直接复用：

1. `src-tauri/src/services/context_manager.rs`
   - 已存在上下文层级、截断、注入策略与内容装配基础。
2. `A-AST-M-P-01_上下文注入.md`
   - 已冻结“工作流模板进入上下文前必须先完成结构解析与流程编译”的规则。

这说明：

1. 模板库无需重新设计一套独立 prompt 注入链。
2. 但必须在现有装配链中新增“编译后的结构化流程表示”承接位。

#### 3.1.3 Workspace 持久化与资源管理基础

当前可直接复用：

1. `src-tauri/src/workspace/workspace_db.rs`
   - 已存在 workspace 级数据库、迁移机制、agent_tasks / agent_artifacts 表。
2. `src-tauri/src/workspace/workspace_commands.rs`
   - 已存在 workspace 命令组织方式、门禁与结果结构模式。
3. `src/services/documentService.ts`
   - 已存在文档打开、文档编辑、工作区文件操作基础。

这说明：

1. 模板资产的元信息、状态和索引可以承接现有 `workspace.db` 模式。
2. 用户友好文档表达层可以复用现有工作区文档编辑链，而不是单独做一套编辑器。

#### 3.1.4 引用协议与工作台入口基础

当前可直接复用：

1. `src/types/reference.ts`
   - 已存在 `TemplateReference` 占位。
2. `src/utils/referenceProtocolAdapter.ts`
   - 已存在模板引用适配入口。
3. `A-WS-M-D-01_workspace工作台协同主控文档.md`
   - 已冻结“模板库入口只承接工作流模板”的工作台边界。

这说明：

1. 模板选择 / 引用链已有协议插槽。
2. 但现有协议仍未收敛到“工作流模板唯一类型”，需要优先修正。

### 3.2 当前主要缺口

基于当前代码与文档，模板库模块的主要缺口如下：

1. **没有 TMP 专属实现模块**
   - 当前 `src` 与 `src-tauri/src` 中没有模板库专属 service / command / store / runtime 目录。
2. **没有模板资产对象**
   - `WorkflowTemplate`、`WorkflowTemplateDocument` 尚未进入真实代码。
3. **没有解析与编译链**
   - `ParsedWorkflow`、`CompiledWorkflow` 只存在于文档中，当前代码未落地。
4. **没有运行时绑定链**
   - `TemplateBinding`、`RuntimeWorkflowPlan` 尚未进入 Agent 主链。
5. **没有模板执行状态对象**
   - `ExecutionState`、`StepState`、`ExecutionContext` 尚未形成模板执行专属承接位。
6. **没有模板资产存储闭环**
   - 当前既没有模板元信息表，也没有模板文档表达层存储协议。
7. **前端模板协议仍残留旧语义**
   - `src/types/reference.ts` 中 `templateType` 仍允许 `'document' | 'workflow' | 'skill'`，与当前规则冲突。
8. **模板引用适配仍是占位**
   - `src/utils/referenceProtocolAdapter.ts` 当前仍返回 `[模板库功能未实现]`。

### 3.3 当前实现与上游规则的差距判断

当前代码已经具备“接入模板库”的主链基础，但离规则完全对齐还存在四类差距：

1. **对象差距**
   - 文档已冻结的对象模型未进入代码。
2. **主链差距**
   - 文档表达 -> 解析 -> 编译 -> plan 的主链尚未建立。
3. **边界差距**
   - 前端协议仍残留 document / skill 模板类型。
4. **观测差距**
   - 统一执行模型的状态可观测性尚未接入现有工作台与聊天面板。

### 3.4 可直接承接与必须新建

| 类别 | 可直接承接 | 必须新建 |
|---|---|---|
| 资产管理 | `workspace_db` 迁移机制、workspace 归属模型 | 模板资产表、模板表达层存储协议、TMP 命令入口 |
| 编辑链 | `documentService.ts`、工作区文档打开链 | 模板编辑入口、模板校验回写链 |
| 注入链 | `context_manager.rs`、AST 上下文装配链 | 编译后流程注入载荷、模板绑定装配链 |
| 执行链 | `ai_commands.rs`、`agentStore.ts` | `RuntimeWorkflowPlan`、模板步骤状态、受限推进承接 |
| 引用链 | `TemplateReference` 占位、adapter 入口 | workflow-only 协议收口、模板引用到 plan 绑定链 |
| 观测链 | `agentStore.ts`、任务/artifact 状态展示基础 | 模板执行阶段、当前步骤、受限推进状态暴露 |

## 四、开发目标拆解

模板库模块当前开发目标拆为以下 11 个能力方向：

1. 模板资产层落地
2. 模板编辑与存储链落地
3. 结构解析链落地
4. 流程编译链落地
5. 运行时 plan 映射链落地
6. Agent 执行协同链落地
7. 统一执行模型承接
8. 执行状态可观测性承接
9. 中断 / 失败 / 恢复承接
10. workspace / AST / CBT 接口衔接
11. 治理与边界收口

这些目标之间的关系不是并列开发，而是：

1. 先搭基础对象与协议。
2. 再形成模板资产最小闭环。
3. 再打通解析与编译。
4. 再接入 Agent 运行链。
5. 再补统一执行模型、可观测与恢复。
6. 最后做协同收口与稳定性验收。

## 五、阶段划分与执行顺序总览

### 5.1 阶段总览

| 阶段 | 名称 | 目标 |
|---|---|---|
| P0 | 基础对象与协议收口 | 冻结 workflow-only 协议、代码骨架与命名边界 |
| P1 | 模板资产最小闭环 | 完成模板创建、保存、读取、编辑、归属关系 |
| P2 | 结构解析与流程编译主链 | 完成文档表达到编译结果的主链闭环 |
| P3 | 运行时 plan 映射与 Agent 接入 | 完成模板绑定、plan 生成与 Agent 主链消费 |
| P4 | 统一执行模型与状态可观测 | 完成隐性执行状态、观测字段、中断与恢复挂接 |
| P5 | 协同接入与边界收口 | 完成 WS / AST / CBT 的正式接入与边界修正 |
| P6 | 稳定性、治理与验收补强 | 完成版本治理、规则扫描、回归与总体验收 |

### 5.2 阶段依赖关系

1. `P0 -> P1 -> P2 -> P3 -> P4 -> P6` 必须串行。
2. `P5` 可以在 `P3` 完成后开始局部并行，但最终合并验收必须晚于 `P4`。
3. `P1` 内部的“后端资产存储”与“前端列表入口”可以并行。
4. `P2` 内部的“解析器骨架”和“编译器校验骨架”可以并行，但编译验收必须以解析结果对象冻结为前提。

### 5.3 为什么按这个顺序推进

1. 如果 `P0` 不先做，后续会把旧的 `TemplateReference` 多类型协议继续带入实现。
2. 如果 `P1` 不先完成，解析与编译没有真实模板资产可承接。
3. 如果 `P2` 不先打通，`P3` 会退化成“原始文档文本直接进 Agent”。
4. 如果 `P4` 不独立成阶段，统一执行模型很容易被错误实现成显式 mode system。
5. 如果 `P5` 过早推进，模板与结构型资产、Chat Build 会重新混层。

### 5.4 最小可运行闭环（MVP Path）

模板库模块的第一条可运行闭环，应固定为以下路径：

1. 用户创建工作流模板。
2. 保存模板文档表达层。
3. 用户手动触发结构解析。
4. 系统生成 `ParsedWorkflow`。
5. 系统完成流程编译并生成 `CompiledWorkflow`。
6. 用户在对话任务中显式选择模板。
7. 系统生成 `TemplateBinding` 与 `RuntimeWorkflowPlan`。
8. Agent 基于 `RuntimeWorkflowPlan` 执行第一步。

第一条闭环的验收目标不是“完整流程全部跑通”，而是：

**单步骤执行成功。**

这条最小闭环对应的开发阶段是：

1. `P1`：模板资产与模板文档表达层真源。
2. `P2`：解析与编译主链。
3. `P3`：模板选择、任务绑定、运行时 plan 与 Agent 首步执行。

后续的统一执行模型、可观测性、失败恢复与跨模块协同，都建立在这条最小闭环已真实跑通的前提上。

## 六、分阶段详细开发计划

## 6.1 P0：基础对象与协议收口

### 6.1.1 本阶段目标

1. 冻结模板库在代码层的唯一对象类型与命名边界。
2. 把当前仍残留的 document / skill 模板协议清除出主链。
3. 建立 TMP 模块的代码骨架与实现入口。

### 6.1.2 为什么先做这个阶段

因为当前最大的现实风险，不是“没 parser”，而是“旧协议还允许错误对象进入模板库主链”。不先收口协议，后续所有实现都会带着错误语义前进。

### 6.1.3 任务拆解

| 任务名称 | 任务目标 | 触达模块 | 主要改动层 | 依赖对象 / 依赖文档 | 风险点 | 完成判定 |
|---|---|---|---|---|---|---|
| T0-1 冻结 workflow-only 引用协议 | 把 `TemplateReference` 收敛为工作流模板唯一引用类型 | `src/types/reference.ts`, `src/utils/referenceProtocolAdapter.ts` | 协议层 / 前端适配层 | `TMP-M-D-02`, `TMP-M-T-02` | 旧 UI 或旧引用逻辑仍依赖多类型模板 | `templateType` 不再允许 document / skill；adapter 不再输出旧类型占位 |
| T0-2 建立 TMP 模块代码骨架 | 建立 TMP 前后端模块目录与基础命令壳 | `src-tauri/src/services/`, `src-tauri/src/commands/`, `src/stores/`, `src/types/` | 模块骨架层 | `TMP-M-T-02` §4/§6 | 骨架命名与现有文档对象不一致 | 代码中出现 TMP 专属 service / command / store / type 入口 |
| T0-3 冻结核心对象命名 | 在前后端类型层引入 `WorkflowTemplate` 等最小对象壳 | `src/types/`, `src-tauri/src/services/` | 对象层 | `TMP-M-T-02` §6 | 对象命名与文档不一致，后续扩散修复成本高 | `WorkflowTemplate`, `WorkflowTemplateDocument`, `ParsedWorkflow`, `CompiledWorkflow`, `RuntimeWorkflowPlan` 等对象壳可被统一引用 |
| T0-4 冻结非目标边界 | 在实现入口注释、adapter 和 TODO 中明确不做 doc template / skill / prompt layer | TMP 入口层 | 治理层 | `TMP-M-D-02` §14 | 工程人员误把旧概念带回 TMP | 代码入口不再出现 document template / skill template 作为主对象 |

### 6.1.4 前置依赖

1. `TMP-M-D-02`
2. `TMP-M-T-02`
3. `TMP-M-S-01`
4. `TMP-M-S-02`

### 6.1.5 产出物

1. TMP 模块代码骨架
2. workflow-only 模板引用协议
3. 核心对象命名壳

### 6.1.6 本阶段验收口径

1. 工程中不存在把模板库主对象定义为 document / skill / prompt 的新代码入口。
2. `TemplateReference` 已收敛为工作流模板语义。
3. TMP 模块有明确代码承接位，后续开发不再借道 AG / AST 杂糅实现。

## 6.2 P1：模板资产最小闭环

### 6.2.1 本阶段目标

1. 形成模板资产创建、保存、读取、编辑、归属的最小闭环。
2. 把“工作流模板是用户显式创建资产”落到真实持久化链。

### 6.2.2 为什么此阶段紧随 P0

解析、编译、plan 映射都必须以真实模板资产为输入；如果没有资产闭环，后续只会在示意对象上空转。

### 6.2.3 任务拆解

## 6.3 P2：结构解析与流程编译主链

### 6.3.1 本阶段目标

1. 建立从模板文档表达到 `ParsedWorkflow`、`CompiledWorkflow` 的主链。
2. 确保模板不会以原始文本直接进入执行链。

### 6.3.2 为什么此阶段必须早于 Agent 接入

如果不先把解析与编译建起来，`P3` 会不可避免地退化为“把模板文档塞进 prompt 里让模型自己理解”，这会直接破坏当前技术路线。

### 6.3.3 任务拆解

| 任务名称 | 任务目标 | 触达模块 | 主要改动层 | 依赖对象 / 依赖文档 | 风险点 | 完成判定 |
|---|---|---|---|---|---|---|
| T2-1 建立解析器骨架 | 从文档表达中提取 phase / step / input / output / constraint | TMP parse service | 结构解析层 | `TMP-M-S-01` | 解析器只做模糊文本理解，无法稳定输出边界 | 可以生成 `ParsedWorkflow` 最小结构 |
| T2-2 建立最小步骤边界校验 | 冻结 `step name / input / output / constraint` 的存在性检查 | TMP parse / compile service | 校验层 | `TMP-M-D-02` §6.2.2.1 | 解析结果看似成功，实际不可编译 | 缺少最小步骤语义时，模板被判定不可用 |
| T2-3 建立编译器骨架 | 从 `ParsedWorkflow` 生成 `CompiledWorkflow` | TMP compile service | 编译层 | `TMP-M-S-01` | 编译结果和解析结果职责混淆 | 生成独立 `CompiledWorkflow` 对象 |
| T2-4 建立编译失败阻断机制 | 编译失败时阻断执行引用 | TMP service / WS store | 控制层 | `TMP-M-S-01` | 编译失败后仍可被选中执行 | 编译失败模板不能进入任务绑定链 |
| T2-5 建立编译缓存与失效策略 | 支持编辑后重新解析 / 编译与缓存失效 | TMP service | 中间产物层 | `TMP-M-S-01` | 缓存命中旧结果导致运行时语义失真 | 模板更新后旧解析 / 旧编译缓存失效 |

### 6.3.4 前置依赖

1. P1 完成

### 6.3.5 解析 / 编译失败分层策略

P2 不允许把所有失败都等价处理为“完全不可执行”。

最小失败分层如下：

1. **结构错误（Fatal）**
   - 典型情况：无 step、无 input / output、阶段边界完全不可识别。
   - 处理原则：不允许编译成功，不允许进入执行链。
2. **语义不完整（Recoverable）**
   - 典型情况：缺 constraint、step 描述模糊、依赖关系未写全。
   - 处理原则：允许生成带风险标记的 `CompiledWorkflow`，但模板状态必须标记为风险可执行。
3. **运行时不匹配（Runtime）**
   - 典型情况：任务绑定后 input 不存在、上下文不满足、依赖资源缺失。
   - 处理原则：不在 P2 直接判死，而是在 P3/P4 的运行时链路中转入受限推进状态或等待人工介入。

这三类失败分别作用于：

1. `P2` 的结构校验与编译结果生成。
2. `P3` 的任务绑定与 plan 映射。
3. `P4` 的受限推进、中断与恢复。

### 6.3.6 产出物

1. `ParsedWorkflow`
2. `CompiledWorkflow`
3. 编译错误与可用性状态

### 6.3.7 本阶段验收口径

1. 模板执行引用链只消费 `CompiledWorkflow`，不消费原始模板文本。
2. 编译失败模板不能进入运行时绑定链。
3. 最小步骤边界校验已经落地。
4. 模板编辑后能够触发重新解析与重新编译。
5. 结构错误、语义不完整、运行时不匹配三类失败已被区分，不再等价处理。

## 6.4 P3：运行时 plan 映射与 Agent 接入

### 6.4.1 本阶段目标

1. 把 `CompiledWorkflow` 正式接入 Agent 主链。
2. 形成 `TemplateReference -> TemplateBinding -> RuntimeWorkflowPlan` 的运行时映射闭环。

### 6.4.2 为什么在 P2 之后做

因为 `RuntimeWorkflowPlan` 必须建立在编译产物之上，而不是原始文档表达之上；否则主链会直接偏离技术主控文档。

### 6.4.3 本阶段强约束

1. 不允许“引用了模板，但仍直接按用户指令自由执行”的绕行路径。

3. 不允许在模板任务中同时存在“基于 plan 执行”和“绕过 plan 直接执行”的双执行路径系统。

## 八、模块协同与接口承接

### 8.1 与 AST 的协同

开发承接要求：

1. 模板编译产物与结构型资产必须作为两类不同上下文对象进入装配链。
2. AST 负责事实内容和结构参考，不负责模板治理与流程约束。
3. `P3 / T3-4` 完成最小 AST 注入对齐。
4. `P5 / T5-1` 完成完整边界收口。

实现约束：

1. 不允许把结构型资产并入 TMP。
2. 不允许把工作流模板并入 AST。

### 8.2 与 AG 的协同

开发承接要求：

1. `P3 / T3-5 / T3-6 / T3-7` 是 AG 主链接入点。
2. 所有模板任务都必须经过 `RuntimeWorkflowPlan`。
3. `CompiledWorkflow` 与 `RuntimeWorkflowPlan` 必须进入 `context_manager.rs` 与 `ai_commands.rs` 的正式主链。
4. `P4` 负责把模板执行状态接入 `agentStore.ts` 与 Agent 事件链。

实现约束：

1. Agent 始终是执行主控。
2. 模板只提供过程约束。

### 8.3 与 WS 的协同

开发承接要求：

1. `P1 / T1-4` 负责模板入口的最小接入。
2. `P4 / T4-3` 负责执行状态可观测性在工作台中的展示位。
3. `P5 / T5-2` 负责最终入口分层收口。

实现约束：

1. 模板库入口与知识库结构型资产入口分离。
2. 工作台只展示真实运行状态，不自行推断模板执行状态。

### 8.4 与 CBT 的协同

开发承接要求：

1. `P5 / T5-4` 才是 CBT 正式接入点。
2. CBT 只能消费 `CompiledWorkflow` 或等价结构化约束载荷。

实现约束：

1. Chat Build 不直接执行模板。
2. 模板在构建态中是约束输入，不是 Chat Build 引擎本体。

## 九、风险、依赖与非目标

### 9.1 主要风险

1. **结构解析不稳定**
   - 结果：模板看似可保存，但无法稳定编译。
   - 控制：P2 必须优先落地最小步骤边界校验。
2. **编译结果与执行链脱节**
   - 结果：运行时又回到原始文本执行。
   - 控制：P3 只允许 `CompiledWorkflow` 进入 AG / CBT 消费链。
3. **统一执行模型被错误做成显式模式**
   - 结果：实现偏离文档，产生 mode switch 系统。
   - 控制：P4 只建立状态对象，不建立模式切换接口。
4. **模板与知识库重新混层**
   - 结果：结构型资产与流程约束边界再次模糊。
   - 控制：P3 完成最小 AST 注入对齐，P5 做接口分流与入口收口。
5. **运行状态不可观测导致黑箱体验**
   - 结果：用户无法理解系统处于什么步骤、为什么暂停。
   - 控制：P4 必须落地最小可观测状态集合。

### 9.2 关键依赖

1. `TMP-M-D-02`, `TMP-M-T-02`, `TMP-M-S-01`, `TMP-M-S-02` 的对象与规则命名必须先冻结。
2. `AG-M-T-05`、`AST-M-P-01`、`A-CBT-I-T-02_Chat Build执行链与运行控制.md` 的接口承接口径必须保持一致。
3. `context_manager.rs`、`ai_commands.rs`、`agentStore.ts` 是主链接入的先决代码入口。
4. `workspace_db.rs` 的迁移与 workspace 归属模型，是模板资产持久化的先决基础。

### 9.3 当前阶段非目标

当前开发计划明确不做：

1. 文档模板
2. skills / prompt / rules 层
3. graph editor
4. 模板市场
5. 社区共享
6. 模板涌现生成
7. 自动保存为模板
8. 独立执行引擎
9. 面向模板的显式模式切换系统

## 十、分阶段验收口径

### 10.1 阶段验收

#### P0 验收

1. TMP 主链代码中不存在 document / skill 模板主对象。
2. `TemplateReference` 已收敛到 workflow-only 协议。
3. TMP 模块有独立代码骨架。

#### P1 验收

1. 已形成模板资产创建、保存、读取、编辑最小闭环。
2. 模板资产具备 workspace / project 归属。
3. 模板来源只有用户创建。

#### P2 验收

1. 已形成 `WorkflowTemplateDocument -> ParsedWorkflow -> CompiledWorkflow` 主链。
2. 原始模板文本不会直接进入执行链。
3. 编译失败可被阻断和暴露。

#### P3 验收

1. 已形成 `TemplateReference -> TemplateBinding -> RuntimeWorkflowPlan` 主链。
2. AG 消费编译产物，不消费原始模板文本。
3. 模板不替代 Agent 主控。
4. 已跑通最小可运行闭环中的“单步骤执行成功”。
5. 已完成最小 AST 注入对齐。

#### P4 验收

1. 已形成统一执行模型承接对象。
2. 连续推进 / 受限推进 / 人工介入是隐性状态，不是显式模式。
3. 用户可观测当前阶段、当前步骤、已完成步骤、受限推进状态、等待介入状态。
4. 用户始终可中断、调整、恢复。

#### P5 验收

1. AST 注入、WS 展示、AG 消费、CBT 只读接入全部边界清晰。
2. 模板入口与结构型资产入口分离。
3. Chat Build 只把模板当过程约束输入。

#### P6 验收

1. 版本与状态治理达到最小可用。
2. 主链有回归与规则扫描。
3. 没有旧协议回流。

### 10.2 总体验收

模板库模块总体验收必须同时满足：

1. 已形成最小模板资产闭环。
2. 已完成“文档表达 -> 解析 -> 编译 -> plan 映射 -> 统一执行”的主链。
3. 模板不会以原始文本直接执行。
4. 执行模型仍然是统一执行模型。
5. 三种执行状态没有被做成显式模式系统。
6. 用户始终可以介入。
7. 模板来源只有用户创建。
8. 没有模板涌现生成链路。
9. 与知识库结构型资产边界未被破坏。
10. 与 Agent / workspace / build 接口链路可通。

## 十一、结论

当前模板库模块的开发不应被理解为“新增一个模板中心”，而应被理解为：

**在现有 Agent 主链、workspace 持久化、上下文注入与工作台协同基础上，补齐工作流模板资产、解析、编译、运行时绑定、统一执行模型与状态观测的分阶段落地。**

开发顺序必须坚持：

1. 先收口对象和协议。
2. 再做模板资产闭环。
3. 再打通解析与编译主链。
4. 再接入 Agent 运行时 plan。
5. 再落统一执行模型与状态观测。
6. 最后做协同收口与稳定性验收。

只有按这个顺序推进，才能保证模板库模块不会退化为：

1. 文档模板系统。
2. prompt 包装层。
3. graph editor。
4. 独立自动化执行器。
5. 模式切换系统。

模板库模块当前阶段的正确落地方向只有一个：

**工作流模板资产系统。**
