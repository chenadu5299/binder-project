2. 为什么模板库只保留工作流模板。
3. 工作流模板在 Binder 中以什么形式存在。

## 二、模板库模块定义

### 2.1 模板库是什么

Binder 模板库是：

**位于 Binder 工作环境中的、由用户显式创建与治理的工作流模板资产模块。**

它承接的是：

1. 过程约束。
2. 流程复用。
3. 用户定义的工作方式资产。
4. Agent 主链中的可复用流程协议。

它回答的问题是：

**当前任务应如何分阶段推进。**

### 2.2 模板库不是什么

模板库不是：

1. 知识库。
2. 记忆库。
3. 文档模板中心。
4. prompt 管理器。
5. skill 容器。
6. rule 容器。
7. 节点链产品。
8. 低代码流程编排器。
9. 独立自动化执行引擎。

### 2.3 为什么 Binder 仍然需要模板库

Binder 仍然需要模板库，是因为复杂任务在运行时不仅需要内容和证据，也需要稳定的过程约束。

知识和结构参考只能回答：

1. 有什么资料可用。
2. 成品结构可参考什么样子。

但它们不能稳定回答：

1. 当前任务应先做什么、后做什么。
2. 哪些阶段必须出现。
3. 哪些中间结果应被生成。
4. 哪些位置应校验、收口或暴露结果。

模板库存在的原因，就是把这些过程约束沉淀为可复用资产。

### 2.4 为什么模板库现在只保留工作流模板

模板库当前只保留工作流模板，是因为 Binder 已经明确将不同语义的对象拆分回各自正确归属：

1. 模板库负责过程约束。
2. 知识库负责事实知识与结构型资产。
3. Agent 负责运行时解释、推进、验证与收口。
4. workspace / workbench 负责创建、编辑、选择和使用环境。

因此，模板库不再承接：

1. 文档模板。
2. skills。
3. rules。
4. prompt 模板。
5. 风格模板。
6. 方法模板。
7. 行为模板。

Binder 不提供独立的用户行为定义层。

模板库不承接 AI 行为定义，只承接任务流程约束。

## 三、当前对象范围

### 3.1 唯一对象类型

模板库当前唯一对象类型是：

**工作流模板（Workflow Template）**

除工作流模板外，以下对象均不属于模板库：

1. 文档模板。
2. 模板文档。
3. 常用格式文档。
4. 结构样例。
5. 结构参考文档。
6. skills。
7. prompt 层。
8. 行为定义层。
9. 风格层。
10. 规则层。

### 3.2 文档范本类能力的归属

所有“文档范本 / 模板文档 / 常用格式文档 / 结构样例 / 结构参考文档”相关能力，统一归入：

**知识库中的结构型资产**

两者边界必须固定为：

1. 模板库负责：过程约束。
2. 知识库结构型资产负责：结构参考。

不得再出现“文档模板属于模板库”的表述。

## 四、模板库的能力边界

### 4.1 模板库负责什么

模板库负责：

1. 过程约束表达。
2. 流程复用。
3. 用户定义的工作方式资产。
4. Agent 主链中的可复用流程协议。
5. 在项目任务上下文中被选择、引用、解释与执行的工作流模板资产。

### 4.2 模板库不负责什么

模板库不负责：

1. 文档结构范本。
2. 内容事实。
3. 知识证据。
4. 用户长期偏好。
5. AI 行为定义。
6. prompt 管理。
7. 自动化涌现沉淀。
8. 独立执行流水线。

### 4.3 模板库与其他资产的边界

模板库与其他资产的关系应严格区分：

| 资产类型 | 回答的问题 | 主责模块 |
|---|---|---|
| 工作流模板 | 任务如何推进 | TMP |
| 事实知识 / 证据 | 任务依据什么内容推进 | AST |
| 结构型资产 | 任务产出在结构上可参考什么 | AST |
| 记忆 | 系统在长期上记住了什么 | AST / Memory |
| 当前轮中间态 | 当前轮已进行到哪里 | AG / 当前轮运行链 |

## 五、工作流模板的本质与作用

### 5.1 工作流模板不是哪些东西

工作流模板在 Binder 中不是：

1. 最终内容。
2. 文档范本。
3. 输出样式。
4. skill 定义。
5. 人格设定。
6. 提示词封装。
7. 一段自由 prompt。
8. 一段不可解析的 md 流程说明。
9. 节点链。
10. 可直接运行的自动化流水线。

### 5.2 工作流模板是什么

工作流模板是：

**对任务推进过程的可复用约束表达。**

它回答的是：

1. 任务应如何分阶段推进。
2. 每一步大致做什么。
3. 每一步依赖什么输入。
4. 每一步产出什么中间结果。
5. 在什么位置需要校验、收口、暴露结果或允许介入。

### 5.3 为什么它不是 skills

skills 的语义是行为能力或行为定义。

工作流模板不承接行为定义，它只承接：

1. 流程边界。
2. 阶段顺序。
3. 输入输出约束。
4. 校验与收口位置。

Binder 不再提供独立的用户行为定义层，因此不存在通过“方法模板”“执行风格层”“可编辑 prompt 层”等方式让 skills 借壳回流的空间。

### 5.4 为什么它不是自由 prompt

自由 prompt 的语义是：

1. 松散表达意图。
2. 依赖模型自由理解。
3. 缺乏稳定的结构化流程边界。

工作流模板不是“让 AI 按这个意思做”的松散提示词。

它必须能稳定表达：

1. 阶段。
2. 步骤。
3. 输入。
4. 输出。
5. 约束。
6. 校验点。
7. 可介入点。

## 六、工作流模板的表现形式

### 6.1 总体判断

Binder 工作流模板应被理解为：

**以用户友好文档形式表达、但内部具备结构化流程语义的过程协议资产。**

### 6.2 三层理解

工作流模板至少包含以下三层理解。

#### 6.2.1 用户表现层

在用户侧，工作流模板应看起来接近文档。

这样做的目的，是保证它：

1. 便于阅读。
2. 便于编辑。
3. 便于保存。
4. 便于复用。
5. 便于在项目上下文中被理解。

#### 6.2.2 结构语义层

在内部，工作流模板不能只是普通文本。

它必须具备可被解析的流程语义，例如：

1. 阶段。
2. 步骤。
3. 输入。
4. 输出。
5. 约束。
6. 依赖。
7. 校验点。
8. 可暴露点。
9. 可介入点。

#### 6.2.2.1 最小结构语义示意

工作流模板至少应能稳定表达出一个最小步骤单元，其语义锚点至少包括：

1. `step name`
2. `input`
3. `output`
4. `constraint`

这里给出的不是实现格式，也不是 JSON / DSL 约束，而是最小语义锚点。

如果一个模板连这一级别的步骤边界都不能稳定表达，它就不构成 Binder 所要求的工作流模板。

#### 6.2.3 执行映射层

在运行时，Agent 需要把工作流模板解释并映射为：

1. 运行时 plan。
2. 当前执行步骤。
3. 中间态推进顺序。
4. 校验与收口动作。

因此，工作流模板不是普通文档，而是：

**带结构化流程语义、可被 Agent 解释的流程协议表达。**

### 6.3 为什么不是节点链

Binder 工作流模板不是传统节点链，也不是低代码流程编排器。

原因有三：

1. Binder 的用户表达层不以显式图结构为主。
2. Binder 当前主链是对话与任务协同，不是画布编排。
3. 用户需要的是可读、可写、可复用的流程协议，而不是拖拽节点构图。

因此，Binder 不要求用户通过节点连接构建流程，也不以 Node-RED / n8n / LangGraph 这类显式图结构作为用户表达层。

### 6.4 为什么不是普通 md

工作流模板在外观上可以接近文档，但它本质上不是普通 md。

原因是：

1. 普通 md 只保证可阅读。
2. 工作流模板还必须保证可解释。
3. 普通 md 无法稳定表达运行时所需的流程边界。
4. 工作流模板必须具备从用户表达映射到结构化流程表示的能力。

因此，它不是一段松散说明文字，而是带有内部结构语义的流程协议表达。

### 6.5 为什么需要“用户友好表达”与“内部结构语义”分层

如果只保留内部结构语义，用户将被迫面对工程化表达，使用门槛过高。

如果只保留用户友好表达，系统无法稳定解释与执行。

因此，必须分层：

1. 用户层保证可读可写。
2. 结构层保证可解析可约束。
3. 运行层保证可映射可执行。

### 6.6 为什么运行前需要从用户表达映射到结构化流程表示

运行前需要进行映射，是因为 Agent 不能直接把用户看到的文档式模板当成自由文本执行。

运行时必须获得的是：

1. 当前阶段。
2. 下一步。
3. 所需输入。
4. 预期中间结果。
5. 校验点。
6. 风险点。
7. 可介入点。

只有完成从用户表达层到结构化流程表示的映射，Agent 才能稳定执行。

### 6.7 运行前的结构解析与流程编译

在执行前，工作流模板需要从用户表现层表达映射为结构化流程表示。

这一过程属于：

1. 运行前的结构解析。
2. 运行前的流程编译。

Agent 在执行阶段使用的不是原始文档表达，而是：

**编译后的结构化流程表示。**

这样做的目的，是避免系统退化为“复杂 prompt system”，并为以下能力提供稳定前提：

1. 结构校验。
2. 步骤边界检查。
3. 输入输出约束检查。
4. 执行顺序控制。
5. 风险点识别。
6. 运行状态可观测。
7. 中断与恢复。

## 七、工作流模板的执行语义

### 7.1 工作流模板不是独立执行器

工作流模板不提供独立运行入口。

它不是：

1. 可直接单独运行的自动化流水线。
2. 可脱离 Binder 主链单独启动的 workflow。
3. 类似 GitHub Actions / n8n workflow 的直接执行流。

它是在以下链路中被引用、解释、展开和执行的过程约束资产：

1. 对话主链。
2. 构建主链。
3. Agent 执行链。

### 7.2 工作流模板如何被 Agent 理解

Agent 读取工作流模板时，不是简单照抄其文本，而是要基于编译后的结构化流程表示完成以下动作：

1. 识别流程目标。
2. 识别阶段顺序。
3. 识别步骤边界。
4. 识别每一步所需输入与预期输出。
5. 识别校验点、风险点与可介入点。
6. 将这些信息映射为运行时 plan 与执行推进序列。

### 7.3 工作流模板如何被 Agent 执行

工作流模板本身不执行。

执行主控始终是 Agent。

Agent 负责：

1. 解释模板。
2. 展开步骤。
3. 处理中间态。
4. 推进执行。
5. 在必要时暴露局部结果。
6. 在风险点暂停或放缓。
7. 收口最终结果。

### 7.4 执行阶段的硬约束

在执行阶段，至少应满足以下硬约束：

1. 每一步必须有清晰边界。
2. 每一步必须对应明确输入与预期输出。
3. 执行默认按既定顺序推进。
4. 未经中断、调整或明确判定，不应盲目跳过关键步骤。
5. 编译失败、结构不成立或步骤边界不清时，不得把原始模板文本直接当作执行依据。

### 7.5 工作流模板与运行时 plan 的关系

工作流模板不是运行时 plan 本身。

两者关系是：

1. 工作流模板提供可复用过程约束。
2. Agent 将模板解释并映射为某一次任务的运行时 plan。
3. plan 是当前任务语境下的具体运行实例。

因此：

1. 模板是稳定流程协议。
2. plan 是当前轮任务上的具体执行展开。

### 7.6 失败、中断与恢复原则

在执行过程中，当某一步骤无法完成或结果不满足约束时，Agent 应允许中断当前流程，并在当前上下文基础上进行调整或重新推进，而不是盲目继续后续步骤。

这意味着：

1. step 失败不是默认继续的理由。
2. 中断后应保留当前上下文与已完成状态。
3. 恢复执行时，应从当前可确认边界重新推进，而不是回到无状态执行。
4. 是否跳过步骤，必须建立在运行时明确判断或用户介入基础上。

## 八、统一执行模型与隐性执行状态

### 8.1 统一执行模型

工作流执行采用统一执行模型。

这里不存在显式可选模式，也不存在“模式一 / 模式二 / 模式三”供用户切换。

运行时只有一个执行模型：

**Agent 在工作流模板约束下持续推进任务，并根据上下文、风险、不确定性与用户行为动态调整推进方式。**

### 8.2 三种隐性并存执行状态

在统一执行模型下，运行时会表现为三种隐性并存状态。

#### 8.2.1 连续推进状态

这是默认态。

其表现为：

1. Agent 按工作流步骤自动连续推进。
2. 不进行逐步人工确认。
3. 中间结果不强制逐步展示或确认。
4. 以完整执行结果为主输出。

#### 8.2.2 受限推进状态

这不是用户可选模式，而是运行时隐式触发状态。

当出现以下情况时，系统自动进入受限推进状态：

1. 高风险操作。
2. 模型不确定性上升。
3. 用户设置或历史行为触发更谨慎推进。

其表现为：

1. 执行节奏减缓或阶段性停顿。
2. 局部结果更倾向于暴露给用户。
3. 必要时等待反馈或确认。

#### 8.2.3 人工介入状态

人工介入状态始终允许进入。

其表现为：

1. 用户可在任意时刻打断执行。
2. 用户可修改当前中间结果。
3. 用户可在修改后继续后续步骤。

### 8.3 关于人工确认的当前原则

当前原则是：

1. 默认不采用逐步确认式执行。
2. 中间结果不要求逐步人工确认。
3. 只有在高风险、异常、不确定或用户主动介入时，才进入受限推进或人工介入状态。

同时必须保留：

1. 用户随时介入权。
2. 系统在关键风险点暂停或放缓的能力。

### 8.4 执行状态可观测性

统一执行模型虽然不暴露显式模式切换，但执行状态本身必须可观测。

至少应让用户能够感知以下信息：

1. 当前执行阶段。
2. 当前步骤。
3. 已完成步骤。
4. 是否处于受限推进状态。
5. 是否等待人工介入或反馈。

这里定义的是可观测性原则，不等于要求在本文中展开具体 UI 或状态机实现。

其目的只有一个：

**避免执行过程对用户呈现为不可理解的黑箱推进。**

---

## 九、与 Agent 的关系

### 9.1 Agent 是执行主控

Agent 是执行主控。

工作流模板不替代 Agent。

### 9.2 模板为 Agent 提供过程约束

模板为 Agent 提供的是过程约束，而不是最终执行逻辑全集。

Agent 仍然负责：

1. 解释模板。
2. 将模板映射为 plan。
3. 处理中间态。
4. 推进执行。
5. 处理异常与不确定性。
6. 在必要时暴露结果、暂停、等待介入。
7. 收口最终结果。

### 9.3 模板不替代 Agent 的原因

模板不替代 Agent，是因为运行时永远存在：

1. 任务上下文差异。
2. 资料条件差异。
3. 风险与不确定性差异。
4. 用户介入差异。

模板只能稳定定义流程约束，不能替代运行时判断。

---

## 十、与知识库的关系

### 10.1 两者职责不同

知识库提供：

1. 内容。
2. 证据。
3. 资料。
4. 结构型资产。

模板库提供：

1. 过程约束。

### 10.2 文档结构范本由知识库承接

文档结构范本、结构样例、常用格式文档等能力，不再由模板库承接。

这些能力统一归入：

**知识库中的结构型资产**

### 10.3 工作流模板不得并入知识库

工作流模板不得并入知识库。

原因是：

1. 工作流模板的语义是过程约束。
2. 知识库资产的语义是内容、证据或结构参考。
3. 两者运行时可以共同进入上下文，但语义层不同。

### 10.4 运行时协同方式

在运行时，两者可以共同进入上下文：

1. 工作流模板提供推进方式。
2. 知识文档提供事实依据。
3. 结构型资产提供结构参考。

这三者不能互相替代。

---

## 十一、与 workspace / workbench 的关系

workspace / workbench 是模板被创建、编辑、选择、引用、触发和执行的工作环境。

模板库不是脱离项目上下文的孤立中心仓库。

模板的使用天然与项目任务上下文绑定。

因此，模板库在 workspace / workbench 中的意义是：

1. 用户可以在项目环境内创建模板。
2. 用户可以在项目环境内编辑模板。
3. 用户可以在任务上下文中选择模板。
4. 用户可以在任务执行中让 Agent 引用模板。
5. 模板的复用发生在真实项目语境中，而不是抽空脱离现场使用。

---

## 十二、与构建模式的关系

构建模式可引用工作流模板作为过程约束输入。

但构建模式的执行主控仍然是：

1. Agent
2. build 主链

工作流模板在构建模式中的作用是：

1. 提供规划参考。
2. 提供阶段约束。
3. 提供执行约束。

它不是：

1. 自动化流水线。
2. 独立构建脚本。
3. 可单独运行的构建执行器。

---

## 十三、模板来源与治理原则

### 13.1 模板来源

当前已确认：

**工作流模板仅由用户创建。**

模板属于：

**用户显式定义资产。**

### 13.2 已删除的来源路径

当前不存在以下链路：

1. 执行过程自动涌现工作流模板。
2. 系统建议保存为模板。
3. 模板由真实执行自动沉淀。
4. 自动保存为高影响模板。

模板库不再依赖涌现生成。

### 13.3 治理原则

模板的启用、选择、应用必须保持用户主导。

这意味着：

1. 模板由用户创建。
2. 模板由用户保存。
3. 模板由用户选择。
4. 模板由用户决定是否启用。

系统不静默生成高影响模板。

---

## 十四、当前阶段边界与不做事项

### 14.1 当前阶段做什么

当前阶段模板库只做：

1. 工作流模板。
2. 本地 / 项目内可创建、可编辑、可选择、可复用的工作流模板资产。
3. 在对话主链、构建主链与 Agent 执行链中作为过程约束被引用和解释。

### 14.2 当前阶段不做什么

当前阶段明确不做：

1. 文档模板。
2. skills。
3. prompt 层。
4. rules 层。
5. 行为定义层。
6. 风格层。
7. 图编辑器。
8. 模板市场。
9. 模板自动涌现。
10. 独立执行引擎。
11. 自动化流水线产品。

### 14.3 当前阶段边界的意义

当前阶段的边界不是临时折中，而是为了保证模板库语义稳定：

1. 模板库只负责过程约束。
2. 知识库负责内容与结构参考。
3. Agent 负责解释、推进、验证与收口。
4. workspace / workbench 负责使用环境。

---

## 十五、结论

Binder 模板库的当前真实定位是：

**一个由用户显式创建与治理的工作流模板资产模块。**

工作流模板在 Binder 中的当前真实形态是：

**以用户友好文档形式表达、但内部具备结构化流程语义，并在运行时由 Agent 解释为 plan 与执行步骤的过程协议资产。**

它不是：

1. 文档模板。
2. skill。
3. prompt。
4. rule。
5. 节点链。
6. 独立执行器。

它的唯一职责是：

**为任务执行提供可复用的过程约束。**
