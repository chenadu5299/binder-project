## 二、路径定位

讨论构建是构建模式的**扩展路径**，面向多人协同决策场景。

使用前提：

1. 用户需要多方参与讨论后才能明确构建目标。
2. 参与者可能是真人（团队成员）或 AI 角色（模拟专业视角）。
3. 讨论内容需要被结构化记录，并作为构建依据。

---

## 三、讨论组模型

### 3.1 讨论组定义

讨论组（Discussion Room）是讨论构建路径的协作空间。

类比：参考腾讯会议会议室模型，轻群聊模式，无视频功能。

核心功能：

1. AI 角色参与讨论，提供分析与专业建议。
2. 真人参与者参与讨论，提供决策意见。
3. 主控 AI 实时收束讨论结构（共识/分歧/方案列表）。
4. 讨论内容最终作为构建大纲的依据来源。

### 3.2 讨论组类型

**开放讨论组（Open Discussion）**

- AI 与项目完全隔离，仅使用公开信息。
- 无项目上下文，无隐私风险。
- 适用场景：思路探索、方向讨论、创意发散。

**内部讨论组（Internal Discussion）**

- AI 可访问当前 Workspace 项目（带脱敏）。
- 进入前展示风险提示。
- 适用场景：方案优化、项目决策、内容深化。

### 3.3 讨论组生命周期

```
创建讨论组
→ 配置类型（Open / Internal）
→ 配置角色 AI（可多个，可选）
→ 邀请真人参与者（可选）
→ 进入讨论（IN_PROGRESS）
→ 主控 AI 触发收束（WAITING_CONFIRMATION）
→ 发起人触发构建
→ 讨论结束（FINISHED）
```

## 四、角色体系

### 4.1 主控 AI（Master AI）

- **存在形式**：隐式，用户不直接与主控 AI 对话，感知其输出（收束结果、大纲、进度）。
- **职责**：
  1. 监听讨论内容，判断何时触发收束（进入 WAITING_CONFIRMATION）。
  2. 生成结构化收束摘要（共识、分歧、方案列表）。
  3. 在发起人触发构建时，基于讨论结果生成 Build Outline。
  4. 执行构建计划（同直接构建路径的执行阶段）。
- **边界**：主控 AI 不直接回复用户消息，不参与讨论内容本身。

### 4.2 角色 AI（Role AI）

- **存在形式**：显式，用户配置后可见，以角色名称参与讨论。
- **配置归属**：归 BLD 模块，与编辑模式 AI 和 AST 模块隔离。
- **职责**：
  1. 以所属角色视角参与讨论，提供分析、建议、专业意见。
  2. 在内部讨论组中可访问项目资源（经主控 AI 分发，带脱敏）。
  3. 不参与最终生产（不执行构建，不写入文件）。
- **配置项**：

> 待定：角色 AI 的配置项（角色名称、角色描述、专业领域、访问权限）的完整定义

- **数量限制**：

> 待定：单个讨论组最多支持几个角色 AI？

### 4.3 真人参与者

- **加入方式**：发起人生成分享链接，参与者通过链接加入。
- **特征**：无账号体系（轻量），无社交关系，临时参与。
- **权限**：
  1. 可发送消息参与讨论。
  2. 可在 WAITING_CONFIRMATION 阶段点击「确认」。
  3. 不可触发构建（仅发起人可触发）。
  4. 不可修改角色 AI 配置。

### 4.4 发起人

- **定义**：创建讨论组的用户。
- **特有权限**：
  1. 触发构建（任何阶段均可）。
  2. 修改角色 AI 配置。
  3. 关闭讨论组（触发 FINISHED）。
  4. 最终决策权（覆盖所有参与者意见）。

## 五、讨论机制

### 5.1 消息流结构

讨论组内的消息分为：

| 消息类型 | 发送方 | 可见性 |
|---|---|---|
| 用户消息 | 真人参与者 / 发起人 | 所有参与者可见 |
| 角色 AI 消息 | 角色 AI | 所有参与者可见 |
| 收束摘要 | 主控 AI | 所有参与者可见，标注"主控 AI 整理" |
| 系统通知 | 系统 | 所有参与者可见（状态变更通知等）|

### 5.2 主控 AI 收束输出

主控 AI 在讨论过程中持续输出收束结构：

```
当前共识：
- [条目 1]
- [条目 2]

当前分歧：
- [分歧点 1]：A 方观点 vs B 方观点
- [分歧点 2]：...

方案列表：
- 方案 A：[描述]
- 方案 B：[描述]
```

## 六、状态机概述

讨论组经历三个状态：

```
IN_PROGRESS → WAITING_CONFIRMATION → FINISHED
```

| 状态 | 含义 | 可执行操作 |
|---|---|---|
| IN_PROGRESS | 讨论进行中 | 所有参与者发消息，角色 AI 参与，主控 AI 收束 |
| WAITING_CONFIRMATION | 等待确认 | 真人参与者确认，发起人触发构建 |
| FINISHED | 讨论已结束 | 只读，不可追加消息 |

**详细状态转换规则见 `A-BLD-M-T-02_讨论组状态机与协作机制.md`。**

## 七、确认机制

### 7.1 确认对象

仅真人参与者可以确认（角色 AI 不参与确认）。

### 7.2 确认语义

确认 = "我同意当前主控 AI 的收束结果，支持进入构建阶段"。

确认不代表对所有内容的同意，仅表示"不阻塞构建"。

### 7.3 确认规则

1. 非强制：不要求所有参与者确认才能构建。
2. 发起人最终决策：发起人可在任何参与者确认状态下触发构建。
3. 确认状态记录：系统记录每个参与者的确认时间，写入讨论结果快照。

---

## 八、构建触发

### 8.1 触发条件

发起人点击「构建」按钮，可在以下时机触发：

1. IN_PROGRESS 阶段（跳过等待确认直接构建）。
2. WAITING_CONFIRMATION 阶段（无论参与者是否全部确认）。

> 注意：不允许在 FINISHED 阶段触发构建（讨论已结束，需新建讨论组）。

### 8.2 触发后流程

```
发起人触发构建
→ 讨论组状态锁定为 FINISHED
→ 主控 AI 读取讨论内容与收束摘要
→ 生成 Build Outline（构建大纲）
→ 进入执行阶段（同直接构建路径）
→ 输出 Project Object 写入 Workspace
```

### 8.3 构建大纲来源

Build Outline 由主控 AI 综合以下内容生成：

1. 主控 AI 最新的收束摘要（共识 + 最终采用方案）。
2. 发起人的输入描述与需求（若有额外补充）。
3. 引用资源（文档、数据、模板等）。

## 九、隐私模型

### 9.1 基础脱敏（系统层自动）

进入内部讨论组时，以下内容自动脱敏：

- 姓名
- 联系方式
- 地址细节

> 待定：脱敏规则的完整清单与实现方案。

### 9.2 AI 信息访问模式（用户选择）

| 模式 | AI 访问范围 | 风险提示 |
|---|---|---|
| 关联项目（默认） | 可使用项目上下文（带脱敏） | 进入前展示风险提示 |
| 完全脱离 | 仅使用公开信息，不访问 Workspace | 无 |

### 9.3 风险提示触发时机

用户创建内部讨论组时，系统在进入讨论前展示：

> "此讨论组为内部模式，AI 角色可能间接接触项目信息。系统已做基础脱敏，但仍存在信息暴露风险。请确认是否继续。"

## 十、结果沉淀

讨论结束后，以下内容写入 Workspace：

| 内容 | 形式 | 写入时机 |
|---|---|---|
| 讨论原始记录 | 文档（可选，用户决定） | 构建触发时 |
| 结构化总结 | 文档（主控 AI 生成） | 构建触发时 |
| 决策节点快照 | 结构化数据 | 构建触发时 |
| Project Object | 项目文件结构 | 构建完成时 |

写入规则见 `A-BLD-I-P-01_构建模式与Workspace接口.md`。
