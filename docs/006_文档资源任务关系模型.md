# 006_文档资源任务关系模型

## 一、文档定位

本文定义 Binder MVP 中“文档对象-资源对象-任务对象”的统一关系模型。  
目标是把引用链路与执行任务解耦，补齐任务对象覆盖不足的问题。

本文只定义领域关系与约束，不展开具体代码实现步骤。

---

## 二、建模范围（MVP）

1. 文档域：工作区文件、快照、编辑目标。  
2. 资源域：引用来源对象（文件/文件夹/记忆/知识库/模板/聊天/链接等）。  
3. 任务域：AI 执行任务、工具调用任务、批处理任务。  
4. 关系域：引用关系、依赖关系、执行关系、产出关系。  

---

## 三、对象模型

## 3.1 文档对象（Document Domain）

| 对象 | 定义 | 关键字段 |
|---|---|---|
| Document | 可编辑文档实体 | `documentId`、`workspacePath`、`filePath`、`fileType` |
| DocumentSnapshot | 文档执行快照 | `snapshotId`、`documentId`、`canonicalHtml`、`snapshotHash`、`sourceMtime` |
| DocumentTarget | 本轮任务目标文档 | `targetFile`、`targetResolved`、`source`（current/non-current/unopened） |
| DocumentRange | 文档定位区间 | `startBlockId`、`startOffset`、`endBlockId`、`endOffset` |

说明：DocumentSnapshot 是任务执行语义真源，不等于展示层 DOM。

## 3.2 资源对象（Resource Domain）

| 对象 | 定义 | 关键字段 |
|---|---|---|
| ReferenceResource | 引用资源统一抽象 | `referenceId`、`type`、`source`、`content` |
| ReferenceRange | 可编辑引用定位扩展 | `startBlockId`、`startOffset`、`endBlockId`、`endOffset` |
| ResourceBundle | 一次请求携带的资源集合 | `bundleId`、`chatTabId`、`references[]` |

资源类型（MVP）：
- `text`、`file`、`folder`、`image`、`table`、`memory`、`kb`、`template`、`chat`、`link`

## 3.3 任务对象（Task Domain）

| 对象 | 定义 | 关键字段 |
|---|---|---|
| EditingTask | 一次对话编辑任务实例 | `taskId`、`workspacePath`、`chatTabId`、`baselineId`、`revision`、`status` |
| TaskStep | 任务步骤对象 | `stepId`、`taskId`、`stepType`、`status`、`startedAt`、`endedAt` |
| ToolCallTask | 工具调用步骤对象 | `toolCallId`、`taskId`、`toolName`、`arguments`、`status` |
| DiffTaskItem | 任务产出的 diff 条目对象 | `diffId`、`taskId`、`toolCallId`、`status`、`routeSource` |
| BatchTask | 批量执行任务对象 | `batchId`、`filePath`、`diffIds[]`、`status` |

任务状态（统一）：
- `pending`、`executing`、`completed`、`failed`、`aborted`

---

## 四、关系模型

## 4.1 核心关系

1. `EditingTask -> DocumentTarget`：一对一，任务必须绑定一个目标文档。  
2. `EditingTask -> ResourceBundle`：一对一，请求期资源输入容器。  
3. `EditingTask -> TaskStep`：一对多，任务由步骤组成。  
4. `ToolCallTask -> DiffTaskItem`：一对多，一次工具调用可产出多个 diff。  
5. `DiffTaskItem -> DocumentRange`：一对一，diff 必须有可执行区间。  
6. `BatchTask -> DiffTaskItem`：一对多，批处理消费 diff 集合。  

## 4.2 关系约束

1. 无 `DocumentTarget` 的任务不得进入执行态。  
2. 无 `DocumentSnapshot` 的任务不得进行定位。  
3. 无 `ResourceBundle` 不影响执行，但会降低上下文质量。  
4. `DiffTaskItem` 必须回指 `taskId + toolCallId`，不可孤立存在。  
5. `BatchTask` 只能消费同一 `filePath` 下的 diff 项。  

---

## 五、任务执行链（关系视角）

`UserRequest -> EditingTask -> ResourceBundle + DocumentTarget -> ToolCallTask -> DiffTaskItem -> UserDecision -> DocumentCommit`

各阶段语义：
1. UserRequest：用户请求进入任务系统。  
2. EditingTask：建立任务上下文（workspace、chat、baseline、revision）。  
3. ResourceBundle + DocumentTarget：装配资源与目标。  
4. ToolCallTask：执行工具调用并返回结果。  
5. DiffTaskItem：生成待决策修改项。  
6. UserDecision：接受/拒绝/批量处理。  
7. DocumentCommit：仅接受项写入真实文档。  

---

## 六、任务对象生命周期

## 6.1 EditingTask 生命周期

`Created -> Ready -> Running -> WaitingDecision -> Finalized`

说明：
- `Created`：任务创建。  
- `Ready`：目标文档与上下文准备完成。  
- `Running`：工具步骤执行中。  
- `WaitingDecision`：已有 diff 产出，等待用户决策。  
- `Finalized`：任务结束（完成/失败/中止）。  

## 6.2 ToolCallTask 生命周期

`pending -> executing -> completed | failed`

约束：
- 只有 `completed` 才允许产出可消费的 `DiffTaskItem`。  
- `failed` 必须生成观测事件，不自动修改 diff 状态。  

## 6.3 DiffTaskItem 生命周期

`generated -> pending -> accepted | rejected | expired`

约束：
- `accepted` 才参与真实文档提交。  
- `expired` 是业务态，不等于执行异常。  

---

## 七、文档-资源-任务一致性规则

1. 任务一致性：同一 `taskId` 内 `baselineId` 不可漂移。  
2. 目标一致性：一次任务只允许一个 `targetFile` 真源。  
3. 快照一致性：定位、校验、执行必须绑定同一快照语义。  
4. 引用一致性：资源引用删除标记后即降级文本，不再计入 ResourceBundle。  
5. 批处理一致性：全部接受必须先读后写，执行期间不重算候选集。  

---

## 八、失败与暴露边界

1. 任务失败：`EditingTask` 或 `ToolCallTask` 的执行状态。  
2. diff 失效：`DiffTaskItem` 的业务状态流转。  
3. 两者不可互相替代，必须独立记录。  
4. 失败暴露走统一观测通道；失效走 diff 状态通道。

## 十、MVP 验收标准（关系模型维度）

1. 能用统一模型解释“一个任务如何绑定文档与资源”。  
2. 能追踪任一 diff 的来源任务与来源工具调用。  
3. 批量与单条执行在关系模型上无语义冲突。  
4. 外部修改、任务失败、diff 失效三类事件边界清晰。  
5. 模型可直接支持后续构建模式扩展，而无需推翻现有对象关系。
