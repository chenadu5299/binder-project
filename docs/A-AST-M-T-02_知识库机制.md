## 一、文档定位

本文是 Binder 知识库系统的**机制级主控文档**，定义知识库对象模型、双库结构、处理流水线、检索协议与治理边界。

本文回答：

1. 知识库对象如何建模。
2. 文档库与检索库如何一一映射。
3. 文档导入、替换、删除、移动、索引如何成立。
4. 检索如何返回稳定、可引用、可溯源结果。

本文不负责：

1. 知识库在 Agent 中的上位协同语义。
2. 用户显式引用与自动检索谁优先。

这些问题由 `A-AST-M-D-02_Binder Agent知识库协同主控文档.md` 负责。

## 二、MVP 目标

1. 建立知识库 / 分类 / 条目三级对象模型。

## 四、双库结构与同步规则

## 4.1 双库结构

知识库采用双库结构：

1. **文档库**：存原始文档
2. **检索库**：存分块、索引、检索元数据

二者以 `document_id` 为稳定映射键。

## 4.2 同步规则

| 场景 | 文档库 | 检索库 |
|---|---|---|
| 新增文档 | 写入原始文件 | 解析、分块、建索引 |
| 替换文档 | 覆盖或新版本入库 | 旧索引失效，重建索引 |
| 删除条目 | 删除文档 | 删除对应索引 |
| 重命名条目 | 不改文档 | 不改检索数据，仅改展示映射 |
| 移动条目 | 不改文档内容 | 更新归属元数据 |

---

## 五、处理流水线

## 5.1 导入链

`上传 / 存入 -> 创建条目 -> 写入文档库 -> 解析 -> 分块 -> 索引 -> 状态更新`

## 5.2 状态定义

1. `pending`
2. `processing`
3. `ready`
4. `failed`

## 5.3 分块与索引

1. 按格式解析文本与结构。
2. 采用文档结构优先的分块策略。
3. 为每个块保留来源、位置、条目与知识库归属。

---

## 六、检索协议

## 6.1 检索接口

推荐统一接口：

`query_knowledge_base(kbId, query, options?) -> { items }`

## 6.2 返回要求

每个结果至少包含：

1. `id`
2. `entry_id`
3. `content`
4. `score`
5. `source`
6. `metadata`

## 6.3 结果要求

1. 结果必须可溯源到条目与文档。
2. 结果必须能作为引用对象或上下文注入对象消费。
3. 检索失败按空结果降级，不阻塞主链。

---

## 七、治理边界

1. 条目显示名可改，但稳定 ID 不变。
2. 删除条目必须同步删除双库映射。
3. 知识库默认只在层次三承接，不为层次一/二隐式开口。
