## 一、验证目标

### 1.1 核心验证目标

**验证 ODS 驱动路线的技术可行性，确保：**
1. ✅ LibreOffice 可以将 Excel 转换为 ODS，并完整保留公式
2. ✅ ODS 解析可以准确提取公式、格式、多工作表信息
3. ✅ ODS 生成可以正确写入公式、格式、多工作表信息
4. ✅ HTML 表格模型可以完整表示 Excel 数据（包括公式）
5. ✅ 转换过程数据不丢失、不失真

### 1.2 验证优先级

**P0（必须验证，阻塞开发）：**
- LibreOffice Excel → ODS 转换公式保留
- ODS 解析公式提取
- ODS 生成公式写入
- 格式保留准确性

**P1（重要验证，影响功能）：**
- 多工作表解析和生成
- 合并单元格保留
- 数字格式保留
- 跨工作表引用保留

**P2（可选验证，优化体验）：**
- 性能验证
- 大文件处理
- 复杂公式处理

pub fn compare_excel_files(original: &Path, generated: &Path) -> Result<ComparisonResult, String> {
    // 1. 读取原始 Excel 文件（转换为 ODS）

## 九、替代方案（如果验证失败）

### 9.1 方案A：直接操作 XLSX 格式

**技术路线：**
```
XLSX → Rust 库解析 → 数据模型 → HTML 表格 → TipTap 编辑器
                                                      ↕
XLSX ← Rust 库生成 ← 数据模型 ← HTML 表格 ← 用户编辑
```

**技术栈：**
- Rust `calamine` 库（读取 XLSX）
- Rust `rust_xlsxwriter` 库（写入 XLSX）

**优势：**
- ✅ 不依赖 LibreOffice
- ✅ 直接操作 XLSX，数据准确性高
- ✅ 性能可能更好

**劣势：**
- ⚠️ 需要实现完整的 XLSX 解析和生成
- ⚠️ 工作量大
- ⚠️ 格式支持可能不完整

### 9.2 方案B：简化功能范围

**调整内容：**
- 暂不支持公式（仅支持数据编辑）
- 暂不支持多工作表（仅支持单工作表）
- 暂不支持复杂格式（仅支持基础格式）

**优势：**
- ✅ 技术复杂度低
- ✅ 开发周期短
- ✅ 风险低

**劣势：**
- ❌ 不符合专业用户需求
- ❌ 功能不完整

### 9.3 方案C：使用第三方服务

**技术路线：**
- 使用在线 Excel 处理服务（如 OnlyOffice、Google Sheets API）
- 或使用本地 Excel 处理库（如 Python `openpyxl`）

**优势：**
- ✅ 功能完整
- ✅ 技术成熟

**劣势：**
- ❌ 依赖第三方
- ❌ 可能不符合产品定位
