## 一、知识库定位与架构

### 1.1 功能定位

知识库是 Binder 对话编辑（层次三）的**文档级知识增强系统**，与记忆库（语义记忆）、模板库（工作流）并列，为 Agent/Chat 模式下的 AI 对话提供**可检索的文档知识**。

**核心价值**：
- 用户导入或生成文档，经处理后供 AI 语义检索
- 支持项目文档一键存入知识库，打通项目与知识库
- 与记忆库、引用系统协同，形成完整的上下文增强体系

**与 PMS 知识记忆的边界**（整合版 v1.1）：知识库 = PMS 中的**知识记忆**。记忆库承担概念记忆 + 结构记忆；工作流库承担工作流记忆。三者并列，无重叠。

**与记忆库的区别**：

| 维度 | 知识库 | 记忆库 |
|------|--------|--------|
| **PMS 对应** | 知识记忆 | 概念记忆 + 结构记忆 |
| **内容来源** | 用户导入、项目文档存入 | AI 自动从对话/文档提取 |
| **可编辑性** | 用户可管理（增删改、重命名） | 用户不可编辑，自动生成 |
| **存储形态** | 文档库 + 检索库（双库映射） | 语义记忆项 |
| **AI 消费** | 语义检索后注入上下文 | 语义检索后注入上下文 |

### 1.2 与 AI 三层架构的关系

| 层次 | 知识库接入 | 说明 |
|------|------------|------|
| 层次一：辅助续写 | 不接入 | 单次续写，无工具、无检索 |
| 层次二：局部修改 | 不接入 | 弹窗内无工具、无检索 |
| 层次三：对话编辑 | **必须接入** | 支持 @ 引用、拖拽引用、AI 自动检索 |

**上下文优先级**（CCE 简化，见 AI 与其他功能交互规范）：
```
用户引用 > 当前文档/选区 > 工作区相关 > 记忆库 > 知识库 > 聊天历史
```

## 二、树状结构与命名

### 2.1 树状结构设计

**知识库采用树状结构**，支持层级分类、展开/折叠、拖拽调整归属。

**树状层级**：

```
知识库根
├─ 知识库 A（用户可重命名）
│  ├─ 分类/文件夹 1（可选层级）
│  │  ├─ 条目 1（与文档一一对应）
│  │  └─ 条目 2
│  └─ 条目 3
│
└─ 知识库 B
   └─ 条目 4
```

**设计原则**：
- **知识库为顶层节点**：每个知识库可包含多个条目，可含子分类（文件夹）
- **条目为叶子节点**：每个条目对应一份文档，条目名称与文档一一映射
- **支持用户重命名**：知识库名称、分类名称、条目名称均可由用户修改

### 2.2 知识库名称修改

| 操作 | 行为 | 同步范围 |
|------|------|----------|
| **修改知识库名称** | 用户右键或双击编辑知识库节点名称 | 仅更新显示名称，不影响文档库与检索库 |
| **修改分类名称** | 用户修改文件夹/分类名称 | 仅更新显示名称，不影响文档归属 |
| **修改条目名称** | 用户修改条目显示名称 | 同步更新条目与文档的映射关系；**文档文件名不变** |

**条目名称与文档的映射规则**：
- 条目名称：用户可见、可编辑，用于树状展示与引用
- 文档文件名：存储层不变，用户修改条目名称不触发文档重命名
- 映射表：`条目 ID ↔ 文档路径/ID` 保持稳定；`条目显示名称` 可独立更新

### 2.3 条目删除与同步

| 操作 | 行为 | 同步范围 |
|------|------|----------|
| **删除条目** | 用户在树状结构中删除某条目 | **同步删除**：文档库中对应文档、检索库中对应向量/索引 |
| **删除分类** | 用户删除某分类（文件夹） | 其下所有条目同步删除（含文档、检索数据） |
| **删除知识库** | 用户删除整个知识库 | 其下所有条目、文档、检索数据一并删除 |

**删除确认**：危险操作需用户确认（如「删除后将无法恢复，是否继续？」）。

### 2.4 条目初始命名规则（命名系统）

**问题**：用户文档库中极可能存在**同名文档**（如多个 `README.md`、`报告.docx`、`第一章.md`），若仅以文件名作为条目名，将产生冲突与混淆。**必须有一套命名系统**保证条目名称唯一且可读。

**命名原则**：
- **唯一性**：同一知识库（或同一分类）下，条目名称不得重复
- **可读性**：优先保留用户可理解的上下文（路径、来源等）
- **可修改**：初始名称自动生成后，用户可随时重命名

**命名规则**（按优先级）：

| 来源 | 规则 | 示例 |
|------|------|------|
| **项目文档存入** | 使用**相对路径**（相对于项目根）作为条目名，路径分隔符可转为 `/` 或 ` > ` | `docs/第一章/报告`、`项目A > 附录 > 数据表` |
| **拖拽上传（有路径上下文）** | 同上，使用来源路径的相对部分 | `子文件夹/报告.docx` → `子文件夹/报告` |
| **拖拽/点击上传（无路径）** | 使用**文件名**（不含扩展名）；若与已有条目重名，则追加**去重后缀** | `报告`；重名时 `报告 (2)` 或 `报告_2` |
| **去重后缀格式** | 括号序号 `(2)` 或下划线序号 `_2`；同一基名下递增 | `报告`、`报告 (2)`、`报告 (3)` |

**去重逻辑**：
- 在**同一分类**（或同一知识库根下无分类时）内检查重名
- 重名时：在基名后追加 ` (N)` 或 `_N`，N 从 2 起递增，直至不重复
- 基名：文件名（不含扩展名），或路径最后一段（不含扩展名）

**可选增强**（P1）：
- **时间戳后缀**：无路径且重名时，可用 `报告_20260317` 等，便于区分批次
- **短哈希后缀**：极端重名时用短 ID，如 `报告_a3f2`，保证唯一

**用户重命名**：初始名称生成后，用户可随时修改；修改仅更新显示名称，不影响文档存储路径。

## 三、双库架构：文档库与检索库

### 3.1 双库设计

知识库采用**双库架构**，与成熟 RAG 方案（如 Dify、RAGFlow）一致：

| 库类型 | 用途 | 内容 | 用户可见 |
|--------|------|------|----------|
| **文档库（Document Store）** | 存储用户原始文档 | 用户导入、项目文档存入的原始文件 | ✅ 是，可打开管理 |
| **检索库（Retrieval Store）** | 供 AI 语义检索 | 经解析、分块、向量化后的可检索数据 | ❌ 否，后台使用 |

**映射关系**：
- 文档库与检索库**一一对应**，同一文档在两边各有一份映射记录
- 文档库：`文档 ID → 文件路径/存储位置`
- 检索库：`文档 ID → 向量索引、分块元数据`
- 结构相同：树状层级（知识库 → 分类 → 条目）在两边保持一致

### 3.2 文档库与检索库的同步

| 场景 | 文档库 | 检索库 | 说明 |
|------|--------|--------|------|
| **新增文档** | 存入原始文件 | 解析 → 分块 → 向量化 → 建立索引 | 异步处理，支持进度提示 |
| **删除文档** | 删除文件 | 删除对应向量与索引 | 同步执行 |
| **替换文档** | 覆盖原文件 | 重新解析、分块、向量化 | 异步处理 |
| **修改条目名称** | 不涉及 | 不涉及 | 仅更新映射表中的显示名称 |
| **移动条目归属** | 更新文档归属元数据 | 更新检索库归属元数据 | 同步执行 |

### 3.3 文档库管理入口

**用户可在知识库标签下一键打开文档库**：

| 能力 | 要求 |
|------|------|
| **入口** | 知识库面板提供「打开文档库」或类似入口，切换至文档库视图 |
| **视图** | 文档库与树状结构同构，按知识库 → 分类 → 条目展示 |
| **操作** | 支持选择文档进行**删除**、**编辑**（替换文件） |
| **同步** | 文档库的删除、替换操作**同步更新**检索库；树状条目同步更新 |
| **条目名称** | 条目名称**按命名系统自动生成**（见 2.4），与文档一一对应，用户可修改 |

**条目名称生成**：按 2.4 命名规则生成初始名称，保证唯一；用户可后续修改。

## 四、支持的文档格式

### 4.1 格式支持分级

参考 Dify、RAGFlow、Progress Agentic RAG 等成熟方案，采用**分级支持**策略：

#### 4.1.1 P0（必须支持）

| 格式 | 扩展名 | 说明 | 参考依据 |
|------|--------|------|----------|
| **纯文本** | .txt | 通用支持 | 所有 RAG 方案均支持 |
| **Markdown** | .md | 通用支持 | 所有 RAG 方案均支持 |
| **PDF** | .pdf | 文本型 PDF | 主流方案均支持；影印/扫描 PDF 需 OCR（可选） |
| **Word** | .docx | Office 文档 | 主流方案均支持，含段落、表格、嵌入图片 |

#### 4.1.2 P1（重要支持）

| 格式 | 扩展名 | 说明 | 参考依据 |
|------|--------|------|----------|
| **HTML** | .html, .htm | 网页内容 | 多数方案支持 |
| **CSV** | .csv | 表格数据 | RAGFlow、Dify 等支持；适合结构化数据检索 |
| **Excel** | .xlsx, .xls | 电子表格 | Dify、RAGFlow 支持；可提取表格文本 |

#### 4.1.3 P2（可选支持）

| 格式 | 扩展名 | 说明 | 参考依据与限制 |
|------|--------|------|----------------|
| **PowerPoint** | .pptx | 演示文稿 | Dify 支持；多为文本提取，版式可能丢失 |
| **JSON/JSONL** | .json, .jsonl | 结构化数据 | 部分方案支持批量导入 |
| **EPUB** | .epub | 电子书 | Dify 支持 |

#### 4.1.4 表格与多媒体（客观判断）

**表格**：
- **建议支持**：xlsx、csv 为常见知识载体，Dify、RAGFlow 等均支持
- **处理方式**：提取表格文本或结构化数据，建立索引；RAGFlow 有专门「Table」分块模板
- **结论**：纳入 P1，与文档类格式同等重要

**多媒体**：
- **图片**：独立图片文件（jpg、png 等）在 Dify 中**不作为知识库内容**直接支持；DOCX 内嵌图片可提取；PDF 内嵌图片需文档提取插件
- **音频/视频**：需语音转文字、视频摘要等，实现成本高，主流方案支持有限
- **结论**：
  - **图片**：Phase 1 不支持独立图片上传；支持 DOCX/PDF 内嵌图片的提取（随文档解析）
  - **音频/视频**：Phase 2+ 可选，需单独评估（语音转写、摘要生成等）

### 4.2 格式支持汇总

| 优先级 | 格式 | 说明 |
|--------|------|------|
| **P0** | txt, md, pdf, docx | MVP 必须支持 |
| **P1** | html, csv, xlsx, xls | 重要支持，表格纳入 |
| **P2** | pptx, json, jsonl, epub | 可选支持 |
| **不推荐** | 独立图片、音频、视频 | 实现复杂，可后续评估 |

### 4.3 文件限制建议

| 限制项 | 建议值 | 说明 |
|--------|--------|------|
| 单文件大小 | ≤ 15 MB | 与 Dify 一致，避免解析超时 |
| 单次上传数量 | 5–10 个 | 可配置 |
| 不支持 | 加密文档、损坏文件、非标准格式 | 与主流方案一致 |

## 五、文档导入与来源

### 5.1 导入方式

| 方式 | 说明 | 归属默认 |
|------|------|----------|
| **拖拽上传** | 用户将文件拖入知识库面板 | 放入根目录或当前选中节点下 |
| **点击上传** | 知识库面板提供上传按钮，选择文件 | 同上 |
| **项目文档一键存入** | 项目文档支持一键存入知识库 | 见 5.2 |

### 5.2 项目文档一键存入

| 能力 | 要求 |
|------|------|
| **入口** | 项目文档（文件树、编辑器）提供「存入知识库」操作 |
| **行为** | 用户选择目标知识库（及可选分类）后，将当前文档复制/引用存入 |
| **格式** | 支持 Binder 可编辑的文档格式（如 docx、md 等）；其他格式按 4.1 支持情况处理 |
| **归属** | 用户选择的知识库及分类；未选择时放入根目录 |

### 5.3 拖拽创建条目

| 能力 | 要求 |
|------|------|
| **拖入文件** | 用户将文件拖入知识库面板（或某知识库/分类节点） | 自动创建条目，归属到拖放目标；若拖到根则归属根目录 |
| **拖入项目文档** | 用户从项目文件树拖拽文档到知识库 | 同上，复制或建立引用后创建条目 |

### 5.4 拖拽调整归属

| 能力 | 要求 |
|------|------|
| **拖动条目** | 用户将条目从一个分类拖到另一个分类或知识库 | 同步更新文档归属元数据、检索库归属 |
| **拖动分类** | 用户将分类拖到其他知识库下 | 其下所有条目同步移动 |

## 六、知识库与 AI 的交互

### 6.1 检索协议（与 AI 功能前置协议与标准对齐）

| 协议项 | 内容 |
|--------|------|
| **接口** | `query_knowledge_base(kbId: string, query: string, options?: { limit?, scope? }) → KnowledgeSearchResult` |
| **返回** | `{ items: [{ id, content, score?, metadata?, source? }] }` |
| **拼接方式** | 作为「知识库信息」段落，插入上下文；明确标注来源为「知识库」 |

### 6.2 引用方式

| 方式 | 说明 |
|------|------|
| **@ 语法** | 用户输入 `@` 后，知识库、知识库条目作为候选；选择后插入引用 |
| **拖拽引用** | 用户将知识库条目拖入聊天输入框，作为引用传入 |
| **AI 自动检索** | 用户消息构建提示词时，按 query 自动检索知识库，注入上下文 |

### 6.3 与引用的关系

- 用户 @知识库 或 拖拽条目：走**引用流程**，内容完整传入
- AI 自动检索：走**上下文注入**，检索结果拼接进提示词

## 七、其他优质知识库逻辑

### 7.1 分块（Chunking）策略

| 策略 | 说明 | 参考 |
|------|------|------|
| **按段落/标题分块** | 按文档结构（标题、段落）切分 | 主流方案通用 |
| **重叠窗口** | 块间保留重叠，避免语义断裂 | RAGFlow、LangChain 等 |
| **块大小可配置** | 支持配置块大小、重叠比例 | 便于调优 |
| **表格专用分块** | 表格类文档采用专用分块模板 | RAGFlow TSI 等 |

### 7.2 处理流水线

| 阶段 | 说明 |
|------|------|
| **解析** | 按格式解析文档，提取文本、表格、内嵌图片描述等 |
| **清洗** | 去除无关字符、规范化空白等 |
| **分块** | 按策略切分为检索单元 |
| **向量化** | 将块转为向量，写入检索库 |
| **索引** | 建立向量索引，支持相似度检索 |

### 7.3 处理状态与进度

| 状态 | 说明 | 用户可见 |
|------|------|----------|
| **待处理** | 文档已入库，尚未解析 | 是 |
| **处理中** | 解析、分块、向量化进行中 | 是，显示进度 |
| **已完成** | 可被检索 | 是 |
| **失败** | 解析或处理失败 | 是，提示原因 |

### 7.4 增量更新

| 场景 | 行为 |
|------|------|
| **文档替换** | 删除旧索引，重新解析、分块、向量化 |
| **文档删除** | 删除对应索引与文档文件 |
| **条目移动** | 仅更新归属元数据，不重新向量化 |

### 7.5 检索增强

| 能力 | 说明 |
|------|------|
| **混合检索** | 向量检索 + 关键词检索（BM25 等）可选 |
| **重排序** | 对初筛结果重排序，提升相关性 |
| **引用溯源** | 检索结果标注来源文档、块位置，便于用户核对 |

## 八、UI 与展示

### 8.1 知识库面板

| 位置 | 说明 |
|------|------|
| **入口** | 标签栏或资源管理区添加知识库标签（图标：📚） |
| **切换** | 点击切换到知识库面板 |
| **展示** | 树状结构展示知识库、分类、条目 |

### 8.2 树状展示规则

- **顶层**：知识库列表，支持重命名
- **第二层**：分类（可选）、条目
- **展开/折叠**：支持点击展开或折叠任意层级
- **右键菜单**：新建知识库、新建分类、上传文档、重命名、删除等

### 8.3 文档库视图

- **切换**：从知识库面板可切换到「文档库」视图
- **结构**：与树状结构同构，按知识库 → 分类 → 条目展示
- **操作**：选择文档可删除、替换（编辑）
- **条目名称**：与文档一一对应，默认取文件名，用户可修改

## 九、与记忆库、引用系统的关系

### 9.1 与记忆库

| 维度 | 知识库 | 记忆库 |
|------|--------|--------|
| **内容** | 用户导入的文档 | AI 自动提取的语义记忆 |
| **管理** | 用户可增删改、重命名 | 用户不可编辑 |
| **AI 消费** | 语义检索 | 语义检索 |
| **引用** | @ 知识库、拖拽条目 | @ 记忆库、拖拽记忆项 |

### 9.2 与引用系统

- 知识库引用类型：`reference.ts` 中的 `KnowledgeBase`
- 支持「选中某一条拖入」与「@ 知识库名称」两种方式
- 引用格式与记忆库对称，便于统一 @ 候选列表

## 十、边界与限制

### 10.1 不支持的场景

- 辅助续写、局部修改：不接入知识库
- 独立图片、音频、视频文件：Phase 1 不支持
- 加密或受密码保护的文档：不支持

### 10.2 性能考虑

- 单知识库文档数量：建议可配置上限（如 500 个）
- 检索超时：500ms，与记忆库一致
- 大文档：分块处理，避免单块过大

### 10.3 数据管理

- **项目锚定**：知识库可锚定于项目/工作区，切换项目后按设计决定是否保留
- **导出**：可选支持知识库导出（文档列表、元数据等）

## 十一、需求优先级

### 11.1 P0（必须实现）

- 树状结构：知识库、分类、条目，支持重命名
- 双库架构：文档库与检索库，一一映射，结构相同
- 文档库管理：一键打开文档库，删除、编辑文档，同步检索库
- 条目与文档一一对应，条目名称按命名系统自动生成（见 2.4），用户可修改
- 树状操作：删除条目同步删除文档；修改条目名称同步映射，文档名不变
- 支持格式：txt、md、pdf、docx（P0）
- 拖拽上传、拖拽调整归属
- 项目文档一键存入知识库
- query_knowledge_base 检索协议
- @ 引用、拖拽引用

### 11.2 P1（重要功能）

- 支持格式：html、csv、xlsx、xls（P1）
- 分块策略可配置
- 处理状态与进度展示
- 文档库与知识库面板切换

### 11.3 P2（增强功能）

- 支持格式：pptx、json、epub（P2）
- 混合检索、重排序
- 引用溯源
- 表格专用分块
