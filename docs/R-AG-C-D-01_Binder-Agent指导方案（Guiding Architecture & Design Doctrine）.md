## 0. 文档使用边界（必须阅读）

### 0.1 文档性质

本文是 Binder Agent 的**上位指导文档 / 主控指导文档**。  
其作用是：

1. 统一设计方向
2. 约束演进边界
3. 提供架构评审依据
4. 提供产品决策依据
5. 为后续执行文档提供上位约束

### 0.2 非执行文档声明

本文**不是**：

1. 功能实现文档
2. 开发任务拆解文档
3. 当前系统能力说明
4. 运行时行为说明
5. 流程实现规范
6. sprint 级开发计划

本文中的原则、模型、分层、状态与路径，用于定义**目标系统的指导形态**，不直接构成工程执行依据。

### 0.3 当前系统状态说明

本文所描述的多数能力，例如：

1. 完整状态系统
2. 分层验证体系
3. runtime gate 体系
4. 中间态驱动协作结构
5. 阶段闭合语义

均属于 Binder Agent 的**目标系统形态**或**目标设计模型**。  
Binder 当前并未完整实现这些能力。

因此：

1. 不应将本文作为当前系统能力依据
2. 不应将本文中的逻辑模型直接理解为当前已存在的运行逻辑
3. 不应将本文中的状态、验证、gate 语义直接视为当前工程现实

### 0.4 正确使用方式

本文应被正确用作：

1. 设计评审依据
2. 架构对齐依据
3. 后续执行文档的上位约束

具体实现、优先级、任务拆解、交互细节、运行时行为与工程承接，必须通过独立执行文档定义。  
后续执行文档可由 `511` 系列或同级文档承担，但无论采用何种编号体系，均应服从本文的上位约束。

## 第一部分：Binder Agent 的根定义

### 1.1 Binder Agent 的本质定义

**Binder Agent 不是一个追求最大自治的任务代理。**  
它的本质是：

**一个运行在 Binder 协作平台之上的人-AI协作执行系统，用于围绕内容、文档、项目与工作区资产，完成“澄清 -> 结构化 -> 执行 -> 验证 -> 确认 -> 沉淀”的协作闭环。**

它的核心不是替代用户做完一切，而是：

1. 帮用户推进任务
2. 帮用户暴露风险
3. 帮用户降低判断成本
4. 帮系统托底流程与状态
5. 把过程沉淀为可复用资产

### 1.2 Binder Agent 的目标

Binder Agent 的目标不是“自动完成任务最多”，而是：

1. **让模糊任务变清晰**
2. **让复杂任务变可协作**
3. **让执行过程变可见**
4. **让验证成本变可控**
5. **让阶段结果可确认、可回退、可沉淀**

### 1.3 Binder Agent 的边界

Binder Agent 明确**不以以下目标为主**：

1. 不以“完全自动完成任务”为目标
2. 不以“最大化模型自治能力”为目标
3. 不以“尽量减少用户介入”为目标
4. 不以“黑箱流程跑完即算完成”为目标
5. 不以“系统替用户做最终质量判断”为目标

### 1.4 Binder Agent 与其他系统的本质区别

#### 与 Coding Agent 的区别

Coding Agent 的核心对象是代码与可执行环境，常有较强的客观终态验证。  
Binder Agent 的核心对象是内容、文档、结构、协作状态与项目资产，很多任务不存在单一客观终态。

所以 Binder 不能照搬：

1. `跑通 = 完成`
2. `测试通过 = 质量成立`
3. `默认自动执行越多越好`

#### 与通用任务 Agent 的区别

通用任务 Agent 更偏向“替用户异步完成任务”。  
Binder Agent 更偏向“与用户共同推进任务，并保持中间态可见、可接管、可确认”。

所以 Binder 不追求：

1. 强黑箱自治
2. 无干预长时间执行
3. 尽量少打扰用户的通用代理心智

#### 与 Chat 型 AI 的区别

Chat 型 AI 主要提供理解、回答、建议。  
Binder Agent 不止回答，还要：

1. 管理状态
2. 管理范围
3. 管理中间态
4. 管理验证与确认
5. 管理资产沉淀

#### 与办公辅助 AI 的区别

办公辅助 AI 多停留在单点功能层：改写、总结、生成草稿。  
Binder Agent 要更进一步，把这些能力放进一个协作运行框架里，形成：

1. 统一状态语义
2. 统一验证语义
3. 统一确认语义
4. 统一资产语义

## 第二部分：三元协作模型（Human / AI / System）

### 2.1 总体原则

Binder Agent 必须建立在 **Human / AI / System** 三元协作模型上。

三者缺一不可，也不允许塌缩为“AI 全权执行”。

### 2.2 Human 的职责

Human 负责：

1. 目标定义
2. 主观判断
3. 风格与审美判断
4. 策略选择
5. 高影响内容确认
6. 在候选之间做决策

Human 不应被系统替代的部分包括：

1. 最终主观质量判断
2. 对外表达与风险承担
3. 对组织、项目、创作方向的最终负责

### 2.3 AI 的职责

AI 负责：

1. 生成候选
2. 推进执行
3. 承担改写、组织、整合、补写、比较
4. 提供计划、摘要、风险提示
5. 提供启发式检查与下一步建议

AI 不应被赋予的职责包括：

1. 最终主观质量裁决
2. 隐式越权修改
3. 黑箱结束任务并宣称已完成

### 2.4 System 的职责

System 负责：

1. 状态管理
2. 范围约束
3. 结构验证
4. 约束验证
5. 中间态可视化
6. 审批与中断
7. 可回退与局部接受
8. 资产沉淀

System 的核心价值不是“替 AI 更聪明”，而是：

1. 托底
2. 约束
3. 降低用户验证成本
4. 维护协作秩序

### 2.5 三者关系的不可替代性

Binder 的正确形态不是：

`Human -> AI`

而是：

`Human 定义与判断 -> AI 生成与推进 -> System 约束与托底`

任何设计如果让三者之一消失，都会导致系统退化：

1. 没有 Human：退化为黑箱代理
2. 没有 AI：退化为普通编辑器
3. 没有 System：退化为高负担 AI 草稿器

## 第三部分：Binder 的“完成观”

> 边界说明：
> 本部分定义的是 Binder Agent 的目标状态语义与目标完成观，
> 用于指导系统设计、架构演进与后续实现承接。
> 本部分不构成当前系统行为说明，也不表示 Binder 当前已经实现完整状态机或完整状态语义。

### 3.1 完成不是终态，而是状态跃迁

Binder 对“完成”的定义，不采用二元 done / not-done 逻辑。  
Binder 采用：

**状态跃迁模型**

原因：

1. 内容任务往往没有单一终版
2. 很多任务只有相对更优，没有绝对终态
3. 当前轮结束不等于项目终结

因此，Binder 的任务目标不是“把一切做完”，而是：

**把任务推进到更高确定性、更低认知成本、更可确认的状态。**

### 3.2 Binder 的阶段状态模型

本节中的状态定义属于**目标语义模型**，用于统一后续设计和实现语言。  
这些状态不应被直接理解为 Binder 当前已完整存在的运行时状态集合。

建议统一采用以下核心状态语义：

1. **draft**  
   任务仍处于模糊、初始、未收敛状态。

2. **structured**  
   目标、范围、约束、对象已经结构化。

3. **candidate-ready**  
   AI 已生成候选内容、候选方案、diff、提纲或草稿。

4. **constraint-satisfied**  
   系统已完成结构层与约束层检查，基础条件成立。

5. **review-ready**  
   结果已被整理为低成本可判断形态。

6. **user-confirmed**  
   用户已对当前关键结果做出确认或选择。

7. **refined**  
   结果经过一轮或多轮 refinement，已提升确定性。

8. **stage-complete**  
   当前轮任务达到稳定可结束、可保存、可继续状态。

9. **invalidated**  
   因范围变化、上下文变化、文档变化，当前中间态失效。

### 3.3 阶段闭合的标准

“阶段闭合”不是“最终最好”，而是满足以下条件：

1. 当前轮目标范围已明确
2. 执行动作已完成
3. 结构层与约束层问题已暴露或已通过
4. 需要用户判断的部分已显式暴露
5. 当前结果已处于可结束、可继续、可回溯状态

只有满足这些条件，Binder 才能把任务标记为 `stage-complete`。

### 3.4 如何避免“伪完成”

Binder 必须明确禁止以下表达：

1. 在主观内容任务上宣称“最终完成”
2. 在用户尚未确认时宣称“质量已满足”
3. 在只完成执行动作时宣称“任务已结束”

系统允许的表达应是：

1. 已生成候选
2. 已满足结构约束
3. 已进入可评估状态
4. 已进入待确认状态
5. 已阶段闭合

## 第四部分：Binder 的验证体系

Binder 采用三层验证体系。

### 4.1 第一层：结构验证（System）

#### 验证什么

1. 作用范围是否正确
2. 是否只修改允许修改的区域
3. 结构是否完整
4. 必要字段、章节、块是否覆盖
5. diff 是否可应用
6. 是否存在未处理执行错误

#### 不验证什么

1. 内容是否最优
2. 文风是否最恰当
3. 是否足够有说服力

#### 谁负责

System 负责。

### 4.2 第二层：约束验证（System + AI）

#### 验证什么

1. 是否符合显式约束
2. 是否存在风格偏离
3. 是否遗漏关键要求
4. 是否有高风险表达
5. 是否有不确定断言
6. 是否需要补充证据或引用

#### 不验证什么

1. 用户偏好本身
2. 创作价值本身
3. 最终主观质量

#### 谁负责

System 负责结构化规则，AI 负责启发式提示与风险揭示。

### 4.3 第三层：质量判断（Human）

#### 验证什么

1. 是否符合真实意图
2. 是否符合审美与策略
3. 是否适合对外使用
4. 是否进入下一轮 refinement
5. 是否接受当前候选

#### 不验证什么

1. 基础结构正确性
2. 工具执行是否成功
3. 显式规则是否满足

#### 谁负责

Human 负责。

### 4.4 三层如何协作

Binder 的协作顺序应为：

1. 先做结构验证
2. 再做约束验证
3. 再把结果转化为低成本确认对象
4. 最后交给用户做质量判断

这意味着：

1. 用户不应被迫做基础结构检查
2. AI 不应替用户宣布最终质量成立
3. 系统不应跳过前两层直接让用户全文复查

> 边界说明：
> 本部分定义的是验证体系的职责分层、责任边界与目标协作关系。
> 本部分不定义具体验证触发机制、执行顺序实现、交互形态、事件编排或工程落点。
> 验证的执行逻辑、触发时机、异常处理与交互承接，需在后续执行文档中单独定义。

## 第五部分：Binder Agent 的运行模型

> 边界说明：
> 本部分描述的是 Binder Agent 的逻辑运行模型（Logical Execution Model）。
> 它用于表达目标协作结构、核心循环与上位运行原则。
> 它不等同于 Binder 当前系统的实际执行路径，也不直接构成流程实现规范。

### 5.1 核心循环

Binder Agent 的目标逻辑运行模型为：

`plan -> execute -> verify -> confirm -> transition`

其中：

1. **plan**  
   澄清目标、锚定范围、结构化约束、暴露风险。

2. **execute**  
   生成候选、执行改写、组织内容、形成中间产物。

3. **verify**  
   执行结构层与约束层验证，生成风险与不确定性提示。

4. **confirm**  
   用户低成本判断、局部接受、拒绝、选择或继续 refinement。

5. **transition**  
   任务进入下一状态，或阶段闭合，或失效重算。

### 5.2 Runtime Gate

Binder 的目标系统应在 runtime 层设置 gate，而不是只在提示词里写要求。

runtime gate 至少包含：

1. 权限 gate
2. 范围 gate
3. 状态 gate
4. 变更可视化 gate
5. 验证 gate
6. 确认 gate
7. 中断与回退 gate

### 5.3 中间态驱动

Binder 的目标运行形态不应允许黑盒执行直接跳到最终回答。  
Binder 的核心对象应当是可见中间态，例如：

1. plan
2. diff
3. 候选稿
4. 风险摘要
5. 来源 / 引用
6. 结构覆盖结果
7. 待确认项

### 5.4 非黑盒执行原则

对高影响任务，Binder 必须满足：

1. 用户知道 AI 改了什么
2. 用户知道 AI 影响了哪里
3. 用户知道哪些部分系统已验证
4. 用户知道哪些部分仍需自己判断

如果做不到这四点，该执行流程不应被视为合格设计。

### 5.5 适用于内容协作的改写

Binder 对 coding agent 架构的吸收必须经过改写：

1. `verifier` 不再是最终裁判，而是结构与约束验证器
2. `complete` 不再代表终态，而是阶段闭合
3. `execution success` 不等于任务完成
4. `approval` 不是形式化点击，而是低成本质量确认

### 6.1 左侧：协作前置

包括：

1. 需求澄清

4. 计划生成

中到高。

需要，且通常较强。

#### 是否可验证

1. 把模糊需求变结构化
2. 把风险显式化
3. 在动手前把作用范围锚定清楚

### 6.2 中段：执行

包括：

1. 修改
2. 生成
3. 重组
4. 多候选
5. 多点修改
6. 内容补写
7. 跨文档联动
8. 中间产物生成

#### agent 参与程度

高。

#### 是否需要 human-in-loop

需要，但不必每一步都重确认。

#### 是否可验证

2. 高影响改动必须可局部接受与拒绝
3. 不允许执行结果直接黑箱落地

### 6.3 右侧：验证 / 收口

包括：

1. 检查
2. 用户确认

4. 阶段闭合
5. 资产化
6. 经验沉淀
7. 项目级记忆沉淀

#### agent 参与程度

中到高，但最终判断不应由 AI 独占。

#### 是否需要 human-in-loop

需要，尤其在质量判断与阶段确认上。

#### 是否可验证

结构与约束可验证，最终质量与偏好需用户判断。

#### Binder 的设计重点

1. 降低用户确认成本
2. 不伪造完成
3. 把阶段结果变成可复用资产

## 第七部分：核心设计原则

以下原则可直接用于否决设计。

### 原则 1：所有高影响变更必须以可见中间态呈现

若一个设计让高影响修改以黑盒方式直接落地，该设计应被否决。

### 原则 2：系统不得宣称主观内容任务“最终完成”

系统只能声明阶段状态，不能伪造最终确定性。

### 原则 3：执行成功不等于任务完成

任何把“动作已执行”直接当成“任务已完成”的设计都不成立。

### 原则 4：用户必须能低成本验证关键结果

若验证关键结果需要用户全文重审，该设计不合格。

### 原则 5：AI 执行必须可中断、可回退、可局部接受

若用户无法打断、回退或局部接受，设计应被否决。

### 原则 6：系统必须先做结构层与约束层验证，再请求用户判断

不能把基础检查责任直接丢给用户。

### 原则 7：所有作用范围必须可见、可追踪、可解释

若用户不知道 AI 改了哪里、影响到哪里，设计不合格。

### 原则 8：中间态必须可转化为资产，而不是一次性日志

若过程结果无法保存、复用、沉淀，设计价值不完整。

### 原则 9：runtime 约束优先于 prompt 约束

凡是需要稳定执行的关键限制，都应落到系统结构中。

### 原则 10：多 agent 不是默认答案

若单 agent + 强 runtime 已能满足目标，不应引入多 agent 复杂度。

### 原则 11：用户确认是协作设计，不是负担转嫁

确认机制必须降低认知成本，而不是把最终审稿工作全量甩给用户。

### 原则 12：任何新功能都必须回答其状态语义

若一个设计无法说清楚它会把任务推到哪个状态，该设计不应通过。

## 第八部分：Binder 的架构吸收与拒绝清单

### 8.1 可以吸收

1. runtime gate
2. hook / validator
3. plan -> execute -> verify 结构
4. context governance
5. event / state-machine 驱动
6. 有限 delegation
7. artifact persistence

### 8.2 必须改写

1. `verifier` 机制  
   从最终裁判改写为结构 / 约束 / 风险验证器。

2. `completion` 定义  
   从终态完成改写为阶段状态跃迁与阶段闭合。

3. `approval` 机制  
   从形式审批改写为低成本质量确认。

4. `intermediate state`  
   从调试日志改写为主产品对象和项目资产。

### 8.3 必须拒绝

1. 黑盒自动完成
2. 无中间态执行
3. 伪确定性完成
4. 默认强自治 background 代理
5. 把 coding agent 的“跑通逻辑”直接套到内容任务
6. 把用户确认理解为“用户自己负责审完一切”

---

## 第九部分：落地路径

> 边界说明：
> 本部分描述的是 Binder Agent 的能力演进方向划分，
> 用于帮助区分短期 / 中期 / 长期的目标层级。
> 本部分不构成具体开发优先级、不构成任务拆解、不构成 sprint 级计划。
> 具体优先级、排期、依赖关系与工程承接，应由独立执行文档确定。

### 9.1 短期（可立即实现）

目标：建立最小闭环。

重点包括：

1. 统一阶段状态语义
2. 可见中间态呈现
3. 结构层验证
4. 约束层验证
5. 低成本确认界面
6. 局部接受 / 拒绝
7. 中断 / 回退能力

短期标准：

1. 不再伪造完成
2. 高影响改动可见
3. 用户确认成本明显下降

### 9.2 中期（增强阶段）

目标：完善协作状态系统。

重点包括：

1. 阶段闭合规则
2. 风险加权确认机制
3. 更完善的 context governance
4. 资产化中间态
5. 项目级记忆沉淀
6. 结构化 plan / verify / confirm 面板

中期标准：

1. 任务可以稳定在状态机中流转
2. 中间态可保存、可恢复、可复用
3. 验证与确认逻辑更清晰

### 9.3 长期（前瞻）

目标：形成 Binder 独有的协作型 agent 能力。

重点包括：

1. 更复杂的专项验证器
2. 构建模式下的阶段性协作
3. 有限多 agent / delegation
4. 面向项目的流程资产化
5. 模板、风格、流程、项目知识沉淀

长期标准：

1. Binder 形成自己的协作型 agent 基因
2. 多 agent 只在真正必要时引入
3. 资产沉淀与项目协作能力成为平台优势

---

## 结论

Binder Agent 的指导总纲可以收敛为一句话：

**Binder Agent 不是追求最大自治的任务代理，而是一个以阶段状态跃迁为完成观、以分层验证为托底、以低成本确认为关键体验、以中间态资产化为平台能力的人-AI协作运行系统。**

后续所有设计、评审与架构演进，都应以这一定义为准绳。

**文档结束。**
