## 二、目标与范围

本方案定义“对话编辑”完整业务链：

1. 文档选中 / 光标定位 / 引用
2. 发送对话消息
3. 注入定位上下文与文档快照
4. 后端 Resolver 定位并生成 canonical diff
5. 前端创建 Diff 卡片与文档侧删除标记
6. 单卡接受 / 拒绝 / 全部接受 / 全部拒绝
7. 文档生效、过期判定、下一轮编辑基准

不在本方案范围内：

- Excel / 演示文稿编辑
- 图片 Diff
- 超长文档分段读取策略
- 移动端完整交互定稿

## 三、核心原则

### 3.1 两变量原则

对话编辑本质上同时解决两个变量：

- 内容：改什么
- 位置：改哪里

任何场景都必须有明确的内容来源与位置来源。无法明确时，按降级阶梯处理，不允许隐式猜测。

### 3.2 职责边界

模型负责：

- 理解用户意图
- 从系统给出的块列表中选择目标块
- 提供 `edit_mode`、`block_index`、`target`、`content`

系统负责：

- 维护 `blockId`
- 维护块编号到 `blockId` 的映射
- 提供定位上下文与文档快照
- 在块内完成文本定位与偏移计算
- 在必要时自动降级到整块替换
- 输出 canonical diff 并驱动 Diff 卡 UI

约束：

- 模型永远不直接接触 `blockId`

### 3.3 双状态模型

文档同时存在两种状态：

- 显示状态：带删除标记的视觉层
- 逻辑状态：仅包含已明确接受的修改

规则：

- 用户沉默不等于接受
- pending diff 不改变逻辑状态
- AI 下一轮编辑必须以逻辑状态为基准

## 四、完整业务链

### 4.1 入口信号

系统接收三种位置输入：

1. 用户显式选区
2. 无选区，仅有光标位置
3. 用户通过“精确引用标签”（由选区复制/粘贴生成）提供区间坐标

三者都属于“定位线索”，但精度等级不同。

统一入口规则：

- 用户选中后如果只是直接发起编辑请求，没有粘贴引用，系统走当前文档选区识别路径
- 用户选中后通过复制粘贴形成引用，系统走引用识别路径
- 两条路径共用同一套精确坐标语义
- 引用标签、标题展示、来源信息等显示层逻辑可以引用这些坐标变量
- 编辑定位只保留一条真源，不允许选区路径和引用路径分别生成两套不一致的定位结果

### 4.2 发送前构建上下文

发送消息前，前端必须构造统一的请求上下文：

```ts
RequestContext {
  targetFile: string
  L: string
  revision: number
  baselineId: string
  editorTabId: string
}
```

其中：

- `targetFile`：本轮真正用于编辑的文件
- `L`：本轮用于定位与校验的 HTML 序列化快照
- `revision`：当前文档版本号
- `baselineId`：本轮唯一基线标识，绑定本轮生成/校验/执行主链
- `editorTabId`：对应编辑器标签页

同时根据场景补充：

- `selection_start_block_id`
- `selection_start_offset`
- `selection_end_block_id`
- `selection_end_offset`
- `cursor_block_id`
- `cursor_offset`
- `selected_text`

### 4.3 后端定位与生成 Diff

后端 `edit_current_editor_document` 只接受结构化编辑协议，并走 Resolver 主路径：

1. 解析 HTML 中全部带 `data-block-id` 的块
2. 构建 `block_index -> block_id -> text_content`
3. 构建只读 `BlockTreeIndex`（与本轮 `baselineId` 绑定）
4. 根据输入进入不同定位分支
5. 统一输出 canonical diff

### 4.4 前端展示

前端收到 canonical diff 后，做两类展示：

- 聊天侧：Diff 卡片
- 文档侧：删除标记

新增内容只在聊天卡片中完整展示，不直接插入文档。

### 4.5 用户决策

用户可执行：

- 单卡接受
- 单卡拒绝
- 全部接受
- 全部拒绝

只有“接受”才真正改写文档内容。

### 4.6 生效与下一轮编辑

接受后：

- 编辑器 DOM 写入新内容
- 文档版本递增
- 已接受 diff 进入逻辑状态
- 下一轮请求重新构建新的 `L + revision + baselineId`

## 五、场景覆盖

### 场景1：用户有选区

- 内容来源：`selected_text`
- 位置来源：选区对应的 `blockId + offset`
- 处理策略：零搜索，直接生成精确 diff
- 精度：最高

### 场景2：无选区，文件已打开

- 内容来源：编辑器当前 HTML
- 位置来源：块列表 + `block_index`
- 处理策略：块内搜索 `target`

### 场景3：全文扫描修改

- 内容来源：完整块列表
- 位置来源：整篇块映射
- 处理策略：模型逐块发起多次编辑，不允许偷懒改成一次全文替换
- 降级规则：仅当命中严格定义的结构化降级条件时，允许从精确定位降级到块级替换；不得直接降级为全文重写

### 场景4：文件未打开，但曾打开过

- 内容来源：缓存中的 canonical HTML
- 位置来源：缓存中的 `blockId`
- 处理策略：静默加载缓存后走场景2

### 场景5：文件从未打开（html/htm/docx）

- 内容来源：磁盘原始内容
- 位置来源：后端管道生成 canonical HTML
- 处理策略：先生成带 `blockId` 的缓存，再进入场景4

### 场景6：文件从未打开（md/txt）

- 内容来源：磁盘原始文本
- 位置来源：后端为纯文本分段注入 `blockId`
- 处理策略：注入 `<p data-block-id="block-ws-{uuid}">` 后写入缓存，再进入场景4

### 场景统一规则：跟需求走，不跟状态走

当用户意图或 AI 任务规划要求同步修改某个目标文档时：

- 如果目标是非当前已打开文档，系统先切换目标注入上下文，再执行编辑
- 如果目标文档未打开但存在缓存，系统先静默加载缓存，再执行编辑
- 如果目标文档未打开且无缓存，系统先静默构建 canonical HTML，再执行编辑

文档是否当前打开不是是否允许编辑的决定因素。用户任务意图才是决定因素。

非当前文档执行门禁（强制）：

1. `targetFileResolved`
2. `canonicalLoaded`
3. `blockMapReady`
4. `contextInjected`

任一状态未就绪，本次调用不得进入工具执行链。

## 六、定位与降级策略

### 6.1 定位精度梯度

1. 零搜索
2. 块内搜索
3. 整块替换降级
4. 报错重试

### 6.2 入口门禁统一（分流只在 Resolver）

规则：

- 存在精确区间坐标（选区或精确引用）时，必须强制零搜索
- 不存在精确区间坐标时，才允许进入块内搜索
- 前端不得做猜测式分流，分流判定权统一在 Resolver

路由标记：

- 每条 diff 必须记录 `route_source`：`selection | reference | block_search`

### 6.3 零搜索

适用：

- 用户选区
- 带精确坐标的引用

行为：

- 直接使用 `selection_start_* / selection_end_*`
- 不再做额外文本搜索
- 输出 `diff_type = precise`

### 6.4 块内搜索

适用：

- 无精确坐标
- 但模型能给出 `block_index + target`

行为：

- 在指定块内搜索 `target`
- 多次出现时通过 `occurrence_index` 决定
- 唯一命中则输出精确偏移

### 6.5 整块替换降级

适用：

- 指定块内未找到目标文本

行为：

- 自动整块替换
- 无需用户额外确认
- diff 卡标记 `block_level`

严格降级约束：

- 先尝试精确定位，再进入降级
- 仅允许降级到对应块的 `block_level`
- 多块局部编辑禁止降级为 `document_level`

严格降级白名单：

- 表格
- 代码块中的复杂结构化片段
- 特殊字符序列
- 富文本嵌套格式
- 数学公式样式文本

### 6.6 全文重写

适用：

- 用户明确要求重写整篇
- 系统已明确判定为全文重写任务

行为：

- 输出 `document_level`
- 仅作为明确模式使用，不得作为多块局部编辑的默认偷懒路径
- 全文扫描任务必须逐块调用，不得用一次 `rewrite_document` 代替多块编辑

### 6.7 内容无变化规则

当一次编辑的结果与原内容完全一致时：

- 视为无修改
- 不生成 diff
- 不生成删除标记
- 不生成 Diff 卡片
- 不推进 `documentRevision`

## 七、工具协议

### 7.1 edit_current_editor_document 输入

```json
{
  "block_index": 1,
  "edit_mode": "replace | delete | insert | rewrite_block | rewrite_document",
  "target": "原始文本",
  "content": "新文本",
  "occurrence_index": 0
}
```

补充字段由系统注入：

- `current_file`
- `current_content`
- `document_revision`
- `baseline_id`
- `_sel_start_block_id`
- `_sel_start_offset`
- `_sel_end_block_id`
- `_sel_end_offset`
- `_sel_text`

引用协议字段（精确引用标签）：

- `TextReference.startBlockId`
- `TextReference.startOffset`
- `TextReference.endBlockId`
- `TextReference.endOffset`

规则：

- 无显式选区但存在精确引用时，后端必须按零搜索执行
- 选区未形成引用时，仍走当前文档选区识别路径

### 7.2 废弃字段

以下字段不再属于新方案：

- `edit_target.anchor`
- `element_identifier`
- `instruction`
- `target_content`
- `scope=block`

## 八、Resolver 统一输出

后端统一输出 canonical diff：

```json
{
  "diffId": "uuid-v4",
  "startBlockId": "block-xxx",
  "startOffset": 0,
  "endBlockId": "block-xxx",
  "endOffset": 15,
  "originalText": "old",
  "newText": "new",
  "type": "replace | delete | insert",
  "diff_type": "precise | block_level | document_level"
}
```

输出要求：

- 同一批返回中的 diff 禁止“部分重叠”；仅允许“无交集”或“完全包含”
- `originalText` 必须与生成时逻辑状态中的实际内容一致
- 支持跨 Block diff
- 跨 Block diff 必须同时满足 `startBlockId/startOffset/endBlockId/endOffset` 的闭区间语义一致
- `offset` 计数口径、换行归一口径、首/中/末块边界口径必须全链一致

### 8.1 originalText 统一规则

`originalText` 抽取、归一化、校验必须使用同一语义，不允许在生成链和执行链使用不同文本口径。

统一项：

- 文本抽取层级一致
- 空白归一规则一致
- 换行归一规则一致
- 特殊字符计数口径一致
- 单块/跨块拼接规则一致

跨 Block 当前版本要求：

- 后端必须输出完整起止块与偏移
- 前端必须支持跨块区间解析、校验与应用

## 九、Diff 效果全案

### 9.1 聊天侧 Diff 卡片

每个连续修改区间对应一张卡片。

卡片至少包含：

- 标题
- 文件信息
- 原文区
- 新文区
- 状态
- 接受 / 拒绝按钮

标题规则：

- 定位信息统一显示块号
- 不显示行号
- `document_level` 仅显示标题，不显示块号

卡片状态包括：

- pending
- accepted
- rejected
- expired

历史态展示规则：

- accepted / rejected 默认折叠，可展开查看完整内容
- expired 不折叠，整体灰色，且不提供操作入口

### 9.2 文档侧删除标记

文档侧只承担删除标记，不承载新增内容。

样式基线：

- 红底
- 删除线
- 标准删除色

技术基线：

- ProseMirror Decoration 或等价机制
- 携带 `diffId`
- 位置跟随 Mapping 漂移

### 9.3 diff_type 视觉语义

- `precise`：正常显示
- `block_level`：显示“块级替换”
- `document_level`：显示“全文重写”

### 9.4 已过期展示

过期卡片与过期标记统一转为不可继续操作状态，不弹出提示，不发 toast，不做额外打断。

失效原因类型：

- 内容已被修改
- 原文校验失败
- 位置解析失败
- 非法部分重叠

卡片展示规则：

- 待执行 diff：显示路径、标题+位置（全文仅标题）、完整内容、操作键
- 失效 diff：显示路径、标题+位置（全文仅标题）、完整内容，不折叠，整体灰色
- 已执行 diff：显示路径、标题+位置（全文仅标题）、内容默认折叠，可展开查看

## 十、接受、拒绝与生效机制

### 10.1 单卡接受

单卡接受流程：

1. baselineId 校验
2. 块位置解析
3. `originalText` 校验
4. 应用替换
5. 标记 accepted
6. 更新文档内容与版本

### 10.2 全部接受

全部接受流程分两段：

读阶段：

- 解析全部 pending diff 的区间
- 校验 `startBlockId/endBlockId` 在 `BlockTreeIndex` 中存在
- 校验跨块顺序合法
- 做基线校验
- 做区间交叉检测（禁止非法部分重叠）
- 校验失败的标记 expired

写阶段：

- 按稳定排序执行
- 逐条应用修改

统一执行口径：

- 先读后写，写阶段不重新改写候选集合
- 稳定排序键为：`from` 降序 -> `to` 降序 -> `createdAt` 升序 -> `diffId` 升序
- 整批提交后统一刷新一次：DOM、`L`、`revision`、decoration、分页

### 10.3 拒绝

- 单卡拒绝：移除该卡对应的 pending 状态
- 全部拒绝：清空当前作用域全部 pending diff

拒绝不会改变逻辑状态。

### 10.4 生效后的状态推进

接受后：

- 文档内容更新
- `documentRevision` 前进
- 已接受 diff 进入逻辑状态历史
- 同文件其余 pending diff 继续保留，仅当命中“内容已被修改 / 区间不可解析 / 非法部分重叠”时转为 expired

### 10.5 失效处理

失效处理规则：

- 任何失效都不弹提示
- 文档侧红色删除标记立即消失
- 失效卡片转为历史态
- 历史态不再提供接受 / 拒绝入口
- 用户通过文档当前状态直接理解该建议已不再生效
- 失效处理仅负责业务状态流转，不承担错误观测职责

### 10.6 同一 diff 池规则

整个工作区内的对话编辑 diff 使用同一套失效语义。

规则：

- 单次对话单个 diff 不单独定义特殊规则
- 单次对话多个 diff：
  - 若接受其中一个后，其他 diff 与其内容重复或交错，则失效
  - 若其他 diff 与其内容无交集，则继续有效
- 单次对话“全部接受”：
  - 按 `from` 降序执行
  - 命中“内容已被修改”的 diff 失效
  - 未命中的 diff 正常执行
- 多次对话产生的多批 diff：
  - 进入同一 diff 池
  - 适用与单次对话多 diff 相同的内容失效逻辑
- 多对话标签产生的多批 diff：
  - 同样进入同一 diff 池
  - 适用同一套内容失效逻辑

失效判断基准：

- 以“目标内容是否已经被修改”作为第一判定标准
- 不以“来自哪一轮对话”或“来自哪个聊天标签”单独决定生效与否

### 10.7 撤销隔离规则

Diff 系统不参与撤销逻辑。

规则：

- 用户执行撤销时，只对已经真实改写进文档的内容生效
- Diff 系统本身不提供回撤入口
- 如果撤销区域与新的 pending diff 重叠或部分重叠：
  - 视为内容被手动修改
  - 相关新的 diff 直接失效

### 10.8 跳过继续与统一暴露机制

执行原则：

1. 局部失败不等于整轮失败
2. 能跳过则跳过，继续执行剩余可执行项
3. 跳过必须输出统一暴露记录

跳过规则：

- 单条失败：该条标记 `expired`，继续执行下一条
- 批量失败：只跳过失败条，不回滚已成功条
- 仅在事务前置条件失效时中断整批（例如目标文档未就绪）

统一暴露结构：

```ts
ExecutionExposure {
  exposureId: string
  level: "info" | "warn" | "error"
  phase: "route" | "resolve" | "validate" | "apply" | "refresh"
  code: string
  message: string
  targetFile: string
  diffId?: string
  baselineId?: string
  routeSource?: "selection" | "reference" | "block_search"
  timestamp: number
}
```

暴露渠道：

- 结构化日志（必开）
- 临时执行面板（可开关，字段与日志完全一致）
- 用户展示层保持静默失效策略（不弹 toast / 不阻断）
- 调试观测层必须完整记录执行暴露

标准错误码：

- `E_ROUTE_MISMATCH`
- `E_TARGET_NOT_READY`
- `E_RANGE_UNRESOLVABLE`
- `E_ORIGINALTEXT_MISMATCH`
- `E_PARTIAL_OVERLAP`
- `E_BASELINE_MISMATCH`
- `E_APPLY_FAILED`
- `E_REFRESH_FAILED`
- `E_BLOCKTREE_NODE_MISSING`
- `E_BLOCKTREE_STALE`
- `E_BLOCKTREE_BUILD_FAILED`

完成判定：

1. 执行层完成：可执行项完成，失败项已跳过并暴露
2. 业务层完成：用户目标是否达成由结果内容判定

### 10.9 失败暴露与失效处理隔离

强制隔离：

- 失效处理是 `diffItem` 业务状态流转（`pending -> expired`）
- 失败暴露是 `ExecutionExposure` 观测事件上报
- 二者允许同时发生，但不得互相替代

判定矩阵：

- 仅失效无错误：允许
- 仅错误不失效：允许
- 失效且有错误：允许
- 无失效无错误：正常执行

## 十一、逻辑状态与 baseline

### 11.1 逻辑状态定义

逻辑状态 = 基线文档 + 已接受修改的有序累积结果

### 11.2 baseline 规则

- 每轮发送消息前，使用当前 `positioningCtx.L` 作为 baseline
- 每轮必须生成唯一 `baselineId`
- baseline 与 `revision + baselineId` 一起构成该轮定位真源
- 接受/拒绝判定以“当前文档真实内容 + 区间与原文校验”为准，不按“轮次”批量失效
- 同一轮主链禁止临时改用“重新抓取的编辑器 HTML”替代该轮 baseline 真源

### 11.3 注入来源

现阶段整合口径：

- 发送给后端的 `current_content` 以 `positioningCtx.L` 为准
- `getLogicalContent()` 保留为逻辑状态重建能力
- 不允许在多个来源之间随意切换，必须保证一轮请求的定位、校验、Diff 生成基于同一份 `L`
- 同一 `baselineId` 下禁止混用不同来源的内容快照

### 11.4 格式保真规则

所有 diff 必须保留原内容的格式关系。

规则：

- 不能因为 diff 应用而抹除原有换行关系
- 不能因为 diff 应用而抹除原有颜色、背景、字体、段落样式等格式信息
- 定位允许基于纯文本
- 应用必须在保留原块结构与样式关系的前提下执行
- 当某种精确替换无法在保留结构与样式关系的前提下稳定执行时，必须走严格降级流程

### 11.5 BlockTreeIndex 绑定规则

在 `positioningCtx.L` 基础上构建只读 `BlockTreeIndex`，并与 `baselineId` 强绑定。

规则：

- 每轮发送前构建一次 `BlockTreeIndex`
- 同一 `baselineId` 下，生成/校验/单卡接受/全部接受共用同一份索引
- 禁止在接受阶段临时重建另一份树覆盖当前判定
- 树索引不可用时允许线性回退解析；回退失败的条目按跳过规则转 `expired` 并暴露错误码

索引最小字段：

- `blockId`
- `blockIndex`
- `path`
- `textStart`
- `textEnd`
- `textHash`

## 十二、两条完整业务链推演

### 12.1 链路A：当前打开文档 + 选区 / 精确引用 + 零搜索精确编辑

目标：

- 保证当前文档内局部编辑从输入到生效走单一精确路径

上游输入：

1. 用户在编辑器内选中内容，或通过复制粘贴形成精确引用标签
2. 前端保留完整坐标：
   - `selection_start_block_id`
   - `selection_start_offset`
   - `selection_end_block_id`
   - `selection_end_offset`
   - `selected_text`
3. 发送前构建统一 `RequestContext`
4. 同时注入：
   - `current_file`
   - `current_content = positioningCtx.L`
   - `document_revision`
   - `baseline_id`

入口分流：

- 用户未粘贴引用，只是选区发起编辑：走当前文档选区识别路径
- 用户粘贴引用：走引用识别路径
- 两条路径最终收敛到同一套零搜索定位输入

协议执行：

1. 前端调用 `ai_chat_stream`
2. 模型根据上下文调用 `edit_current_editor_document`
3. 工具参数使用结构化协议：
   - `edit_mode`
   - `content`
   - `replace/delete/insert` 必带 `target`
4. 系统注入 `_sel_*` 与 `_sel_text`
5. 后端 `Resolver` 检测到 `_sel_*` 存在，直接进入零搜索路径
6. 后端记录 `route_source = selection | reference`
7. 后端直接生成 `diff_type = precise` 的 canonical diff

中游处理：

1. 工具结果经流式事件返回前端
2. 前端只走单一路径把 canonical diff 写入 `diffStore`
3. 文档侧生成删除标记
4. 聊天侧生成 Diff 卡片

下游交互：

1. 用户点击接受
2. 系统执行：
   - baselineId 校验
   - `BlockTreeIndex` 区间存在性校验
   - 区间解析
   - `originalText` 校验
3. 校验通过后应用替换
4. 编辑器 DOM 更新
5. `documentRevision` 前进
6. `positioningCtx.L` 刷新
7. 下一轮编辑基于新的 `L + revision + baselineId`

终态：

- 文档内容已真实改写
- 本条 diff 进入 accepted 历史态
- 同文件其余 pending diff 按“内容是否已被修改”继续判定；无交集 diff 保持有效

### 12.2 链路B：无选区 / 多块扫描 / 非当前文档 + 块列表驱动编辑

目标：

- 保证无精确坐标场景仍然通过块列表和严格降级机制稳定落地

上游输入：

1. 用户没有显式选区，或目标文件不是当前打开文档
2. 前端根据目标文件建立 `RequestContext`
3. 当前文件未打开时：
   - 若已有缓存，直接使用缓存 canonical HTML
   - 若无缓存，先静默生成带 `blockId` 的 canonical HTML
4. 发送时统一注入：
   - `current_file`
   - `current_content = positioningCtx.L`
   - `document_revision`
   - `baseline_id`

协议执行：

1. 系统在 prompt 中注入结构化块列表
2. 模型按块列表返回：
   - `block_index`
   - `edit_mode`
   - `target`
   - `content`
   - `occurrence_index`
3. 后端 Resolver 进入块内搜索路径
4. 后端记录 `route_source = block_search`
5. 若命中唯一位置，输出 `precise`
6. 若命中多处，按 `occurrence_index` 定位
7. 若命中严格降级条件，输出 `block_level`
8. 若是全文扫描任务，模型逐块调用工具，形成多张独立 diff 卡

内容无变化规则：

- 若任一候选修改的结果与原内容一致，该条视为无修改，不入池

中游处理：

1. 每次工具返回只生成一个 canonical diff
2. 前端按 toolCall 分组写入 `diffStore`
3. 聊天侧显示多张卡片
4. 文档侧为每一条 pending diff 挂删除标记

下游交互：

1. 用户执行“全部接受”
2. 系统按文件聚合全部 pending diff
3. 读阶段完成：
   - 目标块存在性校验（BlockTreeIndex）
   - 全部区间解析
   - 全部基线校验
   - 全部交叉检测
4. 写阶段按稳定排序执行（`from -> to -> createdAt -> diffId`）
5. 每一条成功应用后更新 accepted 状态
6. 整个文件完成后统一刷新：
   - `editor.getHTML()`
   - `positioningCtx.L`
   - `baselineId`
   - `documentRevision`

终态：

- 多块修改逐块落地
- 未命中基线或原文校验的条目直接静默失效
- 历史卡片统一折叠

## 十三、高风险链路与固定控制

本节保留高风险点清单，但全部改为固定控制要求，不再使用“确认事项”。

### 13.1 引用坐标保真链

风险：

- 精确引用坐标跨层传递字段较多，任何一层丢字段都会退化为普通文本参考

固定控制：

- 引用坐标只走一条链：复制/粘贴引用标签 -> 消息结构化字段 -> 后端注入 -> Resolver 消费
- 禁止并行旁路再次推导坐标

### 13.2 零搜索与块内搜索双路并存

风险：

- 同一输入若在前后端使用不同分流条件，会产生定位漂移

固定控制：

- 存在精确区间坐标时强制零搜索
- 无精确区间坐标时才允许块内搜索
- 分流判定权只在 Resolver

### 13.3 全文扫描降级边界

风险：

- 降级过宽会导致块级替换泛滥，降级过窄会导致复杂结构频繁失败

固定控制：

- 先尝试精确定位
- 仅在严格降级白名单命中时降级到 `block_level`
- 禁止多块局部编辑降级为 `document_level`

### 13.4 跨 Block 执行一致性

风险：

- 跨块抽取、校验、替换语义不一致会导致前后端同 diff 不同结果

固定控制：

- `startOffset/endOffset` 一律采用各自块内纯文本字符偏移
- `originalText` 拼接顺序固定：起始块尾段 -> 中间完整块 -> 结束块前段
- 替换应用必须使用同一块顺序与边界语义

### 13.5 失效与删除标记时序

风险：

- 失效状态、Decoration 清理、卡片状态不同步会造成视觉残留

固定控制：

- diff 进入 `expired` 的同一事务内完成：文档侧标记清理 + 聊天侧状态切换 + 操作入口禁用

### 13.6 revision 推进与多轮并存

风险：

- 把 revision 当作唯一失效条件会误杀无交集 diff

固定控制：

- 仅按内容冲突与区间可执行性判定失效
- 无交集 diff 保持有效
- 该规则统一适用于单轮、多轮、多聊天标签

### 13.7 历史卡片展示一致性

风险：

- 三类状态展示不一致会造成理解成本和操作误判

固定控制：

- 待执行：路径、标题+块号（全文仅标题）、完整内容、操作键
- 失效：路径、标题+块号（全文仅标题）、完整内容，不折叠，灰色
- 已执行：路径、标题+块号（全文仅标题）、默认折叠，可展开
