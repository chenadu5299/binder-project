## 一、流程总览

Chat Build 交互流程由六个阶段组成：

1. `自由讨论`
2. `构建触发 / 轻量确认`
3. `大纲生成`
4. `大纲确认`
5. `正式构建`
6. `结束态`

## 二、阶段定义

### 2.1 自由讨论

这是构建开始前的自然对话阶段。

用户可以：

1. 描述目标。
2. 补充约束。
3. 继续澄清需求。
4. 引入文件、知识库、记忆、模板等引用。

系统在这一阶段：

1. 不进入正式构建。
2. 只做理解、追问、收束和准备。

### 2.2 构建触发 / 轻量确认

构建意图可以通过两类方式成立：

1. 用户显式点击“开始构建”类入口。
2. 系统从自然对话中识别到明确的项目级生成意图，并请求进入大纲阶段。

构建触发不等于直接开跑。

构建触发后，先进入轻量确认层。

轻量确认层只决定：

1. 当前是否进入大纲生成。

构建触发 / 轻量确认后，只能进入：

1. 大纲生成
2. 返回继续讨论

### 2.3 大纲生成

系统根据当前对话与引用输入生成 `Build Outline`。

大纲的职责是：

1. 冻结本轮构建的目标。
2. 说明预计产物范围。
3. 给出阶段性执行骨架。

### 2.4 大纲确认

这是 Chat Build 的核心用户闸口。

用户在这一阶段只能做三类事：

1. 确认并开始构建。
2. 取消，回到自由讨论。
3. 提出补充意见，重新生成或修正大纲。

只有大纲确认后，才能进入正式构建。

### 2.5 正式构建

正式构建开始后，交互语义发生切换：

1. 不再接受自然语义改向。
2. 不再把新聊天输入当成需求变更。
3. 主界面只展示进度、状态、阶段和过程信息。

用户在这一阶段允许的动作只有：

1. 查看当前状态。
2. 手动中断。

### 2.6 结束态

结束态包括：

1. `Completed`
2. `Interrupted`
3. `Failed`

结束后，用户可以：

1. 查看生成结果。
2. 在编辑模式继续处理生成产物。
3. 重新发起新的 Chat Build，而不是在旧构建运行中继续改需求。

## 三、阶段动作矩阵

| 阶段 | 用户允许动作 | 系统允许动作 | 禁止事项 |
|---|---|---|---|
| 自由讨论 | 继续聊天、补充约束、加引用 | 收敛需求、追问、识别构建意图 | 直接进入正式构建 |
| 构建触发 / 轻量确认 | 确认进入大纲、返回聊天 | 进入大纲生成 | 直接开始正式构建 |
| 大纲生成 | 等待、补输入 | 生成 Build Outline | 当作最终执行 |
| 大纲确认 | 确认、取消、修改大纲 | 进入正式构建或返回讨论 | 跳过确认闸口 |
| 正式构建 | 查看状态、手动中断 | 执行构建、更新进度 | 边构建边改需求 |
| 结束态 | 查看结果、重新发起新构建 | 展示结果 | 把旧构建继续当成对话继续改向 |

## 四、当前版本的关键冻结规则

以下规则以 [`A-CBT-I-P-01_Chat Build最小协议与状态.md`](./A-CBT-I-P-01_Chat%20Build%E6%9C%80%E5%B0%8F%E5%8D%8F%E8%AE%AE%E4%B8%8E%E7%8A%B6%E6%80%81.md) 为统一规则源头：

1. `BR-CBT-RUN-001`：轻量确认层只负责是否进入大纲阶段。
2. `BR-CBT-VERIFY-001`：大纲确认是正式构建前的硬边界。
3. `BR-CBT-STATE-001`：构建开始后不可被自然语义改向。
4. `BR-CBT-STATE-002`：中断控制与中断结果语义。
5. `BR-CBT-MODEL-001`：chat 在构建前后角色切换。

1. 大纲确认前，系统仍处于“可讨论、可变更”的聊天阶段。
2. 大纲确认后，系统进入“冻结式构建”。
3. 构建中的聊天区域不是需求编辑通道，而是状态展示通道。
4. 用户若要改目标，必须先中断或结束当前构建，再回到新一轮讨论。

## 五、当前版本明确不包含的流程

以下流程不属于当前版本：

1. 多角色 AI 发言
2. 真人参与确认
3. 房间内群聊式讨论
4. 分享链接邀请加入
5. 构建过程中边执行边审批多轮方案
