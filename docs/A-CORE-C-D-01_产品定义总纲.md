## 三、产品定义

### 3.1 产品一句话定义

Binder 是一个面向创作者和知识工作者的本地优先 AI 文档工作台，以“文档编辑 + AI 对话编辑 + 工作区资源管理”为核心，提供可控、可确认、可追踪的内容生产体验。

英文副标题：Binder — AI-native local document workspace.  
对外极简介绍：A local-first AI writing workspace with controllable editing flows.

### 3.2 MVP 版本定义（开源首版）

MVP 聚焦“文档工作流”单核心场景，首版交付目标：
1. 文档编辑可用且稳定（不以分页能力作为开源首版前提）
2. 三层 AI 能力可闭环运行（辅助续写/局部修改/对话编辑）
3. 对话编辑 Diff 链路可闭环（生成、展示、接受、失效、继续）
4. 工作区内文件与资源可管理（本地优先）
5. 可观测、可排障、可快速复现关键问题

## 四、产品要解决的问题

1. 写作与修改在多个工具间割裂，流程中断严重。  
2. AI 生成内容不可控，定位不准、结果不可验证。  
3. 本地文档管理和 AI 操作脱节，跨文档上下文不连贯。  
4. 用户需要“看得见、可确认、能回到过程”的编辑机制，而不是黑盒改写。  

## 五、目标用户

1. 专业写作者与内容创作者
2. 产品/研发文档作者
3. 需要高频改写、润色、结构重组的知识工作者

MVP 不做细分行业纵深方案，只覆盖通用文档生产主场景。

## 六、核心价值主张

1. 可控：AI 修改先呈现 Diff，再由用户决策。  
2. 连续：编辑与对话在同一工作区连续完成。  
3. 可信：状态、版本、失效和执行结果可解释可追踪。  
4. 高效：三层 AI 分工明确，避免一个入口承载所有任务。  

## 七、MVP 能力边界

### 7.1 必做能力

1. 文档编辑：md/txt/html/docx 主链路可编辑  
2. 对话编辑：统一整合方案定义的完整链路可运行  
3. Diff 机制：文档侧删除标记 + 聊天侧卡片三态  
4. AI 三层：层次一、层次二、层次三职责清晰且可独立使用  
5. Workspace：文件树、引用、资源管理基础能力  
6. 稳定性：最小可观测体系（错误码、关键日志、复现路径）

### 7.2 非 MVP 范围

1. 旧 Build Mode 扩展方案（Discussion Build / Multi-Actor Build / 主控 AI + 角色 AI 双层协作）
2. 移动端完整方案
3. Excel/演示文稿完整编辑链路
4. 图片 Diff 与复杂富媒体语义编辑

### 7.3 开源首版开放边界

1. 开源核心：本地文件管理、文档编辑主链、AI 三层能力、对话编辑主链（含 Diff 机制）  
2. 非开源范围（首版）：预览子系统、T-DOCX 逻辑、分页子系统  
3. 非开源范围不等于停止研发：内部可继续并行推进，不阻断后续版本合并计划  

### 7.4 模型支持策略（开源首版）

1. 官方适配名单（MVP）：DeepSeek、OpenAI、Anthropic  
2. 通用扩展能力：提供 OpenAI 兼容接口适配层（支持自定义 Base URL、API Key、Model Name）  
3. 推进顺序：先跑通 DeepSeek 全链路回归，再完成 OpenAI、Anthropic 验证  
4. 默认测试模型：DeepSeek（成本低，适合作为持续回归基线）  

## 八、产品级统一原则

1. 单一真源：同一业务语义只能有一个规则主源文档。  
2. 先确认再生效：涉及文档修改默认走“建议 -> 决策 -> 生效”。  
3. 显示态与逻辑态分离：视觉展示不等于真实内容状态。  
4. 失败可暴露：允许局部失败，但必须可追踪且不阻断主流程。  
5. 禁止隐式猜测：定位和执行必须基于显式上下文与协议。  

## 九、MVP 发布定义（产品口径）

满足以下条件即达到开源首版发布线：
1. 对话编辑统一整合方案可稳定跑通核心场景  
2. 文档编辑主链路无阻断性故障  
3. AI 三层入口可用，边界不混淆  
4. 文档体系具备最小一致性（主规范、实现规范、排障规范）  
5. 已知限制清单公开且可复现  

### 9.1 首版必须稳定的 5 个场景

1. 资源库目录管理完整逻辑（含与编辑器交互）  
2. 编辑器对不同格式文档稳定解析与可编辑（含 docx，不要求分页与 T-DOCX）  
3. AI 编辑完整逻辑稳定（三层能力 + 工具边界清晰）  
4. AI 聊天窗口功能完整（基础记忆 + 必要工具调用 + Diff 卡链路正确）  
5. 本地交互整体稳定流畅（文件、编辑、AI 调用全过程不卡主流程）  

### 9.2 发布性能红线（建议值）

1. 启动性能（macOS）：冷启动 ≤ 6 秒，热启动 ≤ 2 秒  
2. 工作区加载：中等项目（≤ 3000 文件）首次树渲染 ≤ 3 秒  
3. 文档打开：md/txt/html（≤ 2MB）≤ 1.5 秒；docx（≤ 20MB）≤ 10 秒  
4. AI 响应（DeepSeek 基线）：首 token P95 ≤ 4 秒；短任务完成 P95 ≤ 20 秒  
5. Diff 执行：单卡接受 ≤ 200ms；20 张卡批量接受 P95 ≤ 3 秒  

## 十、术语最小集合（首版）

1. Workspace：用户当前工作的本地项目容器  
2. 对话编辑：通过聊天触发的结构化编辑能力（非自由改写）  
3. Diff：建议修改单元，需用户决策  
4. baseline：本轮编辑定位与校验基线  
5. revision：真实内容版本推进序号  
6. 逻辑状态：仅包含已生效内容的真实状态  
7. 显示状态：包含待确认装饰的可视状态  

## 十一、已确认发布口径（v0.1.0）

1. 目标平台：仅 macOS 单平台发布  
2. 开源范围：仅开源核心链路（文件管理、文档编辑、AI 三层、对话编辑主链）  
3. 首版模型策略：官方适配 DeepSeek/OpenAI/Anthropic，并提供 OpenAI 兼容接口扩展能力  
4. 稳定性优先级：优先保证你确认的 5 个核心场景  
5. 协作策略：v0.1.0 阶段只收 Issue，不合并外部 PR（允许 clone/fork/试用）
