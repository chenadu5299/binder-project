## 一、引用内容的格式与逻辑

### 1.1 现状

- **前端**：`reference.ts` 已定义 Text、File、Folder、Image、Table、Memory、Link、Chat、KnowledgeBase 等类型
- **后端**：`context_manager.build_reference_prompt()` 已实现，`ai_commands` 中从消息提取引用并构建 ReferenceInfo
- **问题**：引用用途未区分（可编辑 vs 只读 vs 信息获取），AI 消费格式未统一约定

### 1.2 三种引用用途（与引用功能完整设计文档对齐）

| 用途 | 场景 | AI 行为 | 数据要求 |
|------|------|---------|----------|
| **可编辑** | 精确引用或选区定位到目标文档 | 调用 edit_current_editor_document；有精确区间坐标时直接进入零搜索；无精确坐标时使用 `block_index + edit_mode + target + content + occurrence_index` | `startBlockId`、`startOffset`、`endBlockId`、`endOffset` 或目标文件块列表 |
| **只读** | 内容已传入；File path 可信 | 直接使用，无需 read_file；AI 根据指令判断是否 update_file | type、source、content；path 已解析，prompt 统一声明 |
| **信息获取** | 图片、链接等 | 需获取内容再传入 | 占位或实时拉取 |

**原则**：引用层只传内容，不做意图预判；已删除 isOperationTarget。

### 1.3 引用格式协议（引用功能完整设计文档 6.1）

**已定义**：

1. **references 结构**：`{ type, source, content, preciseRange?, templateType? }`；无 isOperationTarget
2. **可编辑引用**：精确引用含完整区间坐标；系统将其作为零搜索定位输入
3. **build_reference_prompt**：含 File 时统一追加「路径已解析，可直接使用」
4. **消息展示**：以标签形式，不展开；完整 content 仅传给 AI

## 二、记忆库交互

### 2.1 现状

- 已有 `memory_service`、`MemoryTab`、`search_memories` 等
- 各层次设计文档均提及「记忆库集成」，但检索接口、返回格式、与提示词的拼接方式分散

### 2.2 建议：先定义「记忆库读取协议」，后做架构

**不必先做完整记忆库架构**，只需约定：

| 协议项 | 内容 |
|--------|------|
| **调用方** | AI 相关模块（useAutoComplete、useInlineAssist、ai_chat_stream） |
| **接口** | `search_memories(query: string, options?: { limit, documentPath? }) -> MemorySearchResult` |
| **返回格式** | `{ items: [{ id, content, metadata?, source? }], total?: number }` |
| **拼接方式** | 作为「记忆库信息」段落，插入上下文或引用层；明确标注来源为「记忆库」 |

**记忆库开发时**：只需实现上述接口并输出约定格式，AI 侧按协议消费即可。架构演进（概念记忆、结构记忆等）可后续扩展，不影响当前协议。

### 4.3 工作流模板格式协议（与 Binder 工作流模板 PRD 对齐）

**开发前完成**：

| 协议项 | 内容 |
|--------|------|
| **结构** | Plan JSON：`{ id, name, description, source, weighted_score, steps: [{ id, type, label, params, depends_on }] }` |

**当前阶段**：工作流由 Agent 按 steps 自主执行，无独立执行引擎。Plan JSON 为数据格式，执行内嵌在对话流中。详见 Binder模板库需求文档 二、工作流模板详细设计。

## 五、AI 文档格式输出协议

### 5.1 问题

当前 Diff 格式中 `newText` 为**纯文本**，`tr.replaceWith(from, to, schema.text(newText))` 插入无格式内容。若不做约定，AI 输出的文档大概率是**不分页的纯文本**，无字体、颜色、字号等格式。

### 5.2 现状

| 能力 | 编辑器支持 | Diff 格式 | 差距 |
|------|------------|-----------|------|
| 粗体/斜体/下划线 | ✅ TextStyle、Bold、Italic、Underline | newText 纯文本 | AI 无法输出格式 |
| 字体颜色 | ✅ Color 扩展 | 无 | 无 |
| 背景高亮 | ✅ Highlight 扩展 | 无 | 无 |
| 字号 | ✅ FontSize（px） | 无 | 无 |
| 字体族 | ✅ FontFamily | 无 | 无 |
| 标题层级 | ✅ Heading | 块级，非 newText | 需单独约定 |

### 5.3 需先定义的协议

| 协议项 | 内容 |
|--------|------|
| **newText 格式** | 纯文本 vs 简单 HTML 子集（如 `<strong>`, `<em>`, `<span style="...">`）vs 结构化 JSON |
| **字体白名单** | AI 可使用的字体列表；**必须指定**，避免 AI 输出系统不存在的字体（如「微软雅黑」在 macOS 可能不可用） |
| **字号范围** | 当前工具栏：10/12/14/16/18/20/24/28/32/36 px；需与 DOCX pt 对应关系（见 Word样式需求清单） |
| **颜色格式** | 十六进制 `#RRGGBB`；是否限制为预设色板 |

### 5.4 字体白名单建议（基于当前 EditorToolbar）

**西文字体**（跨平台可用性较高）：Arial, Times New Roman, Courier New, Verdana, Georgia, Palatino, Garamond, Trebuchet MS

**中文字体**（需按目标平台验证）：宋体、黑体、微软雅黑、仿宋、楷体（macOS/Windows 可用性不同，建议先列「已验证可用」清单）

**原则**：只允许白名单内字体；AI 提示词中注入该列表；未列出的字体请求时降级为默认或最近似白名单字体。

### 5.5 方案选项

| 方案 | 说明 | 复杂度 | 建议 |
|------|------|--------|------|
| **A. 纯文本 MVP** | Phase 1 保持 newText 纯文本，文档化限制；格式由用户手动调整 | 低 | 可先采用，但需在提示词中说明「输出纯文本，不包含格式标记」 |
| **B. 简单 HTML 子集** | newText 允许 `<strong>`, `<em>`, `<u>`, `<span style="color:#hex;font-size:Npx">`；前端解析为 ProseMirror 节点 | 中 | 推荐；需定义允许的标签与 style 属性白名单 |
| **C. Markdown 转换** | newText 为 Markdown（**bold**, *italic*），前端解析后应用 | 中 | 与 B 类似，AI 更熟悉 Markdown，但需处理嵌套与边界 |

### 5.6 建议

1. **开发前**：确定 newText 格式（A/B/C）及字体白名单
2. **提示词注入**：在系统提示词或工具定义中明确「可用字体：…」「可用字号：…」「颜色格式：#RRGGBB」
3. **辅助续写 / 局部修改**：若返回内容也插入文档，需与对话编辑 Diff 共用同一格式协议

- 已约定：`{"tool":"tool_name","arguments":{"key":"value"}}`，JSON 严格格式
- 需保持：新增工具时，参数 schema 与前端、AI 的约定一致

### 6.3 错误与降级策略

| 场景 | 策略 |
|------|------|
| 记忆库检索失败 | 静默降级，不阻塞主流程 |
| 引用内容过大 | 仅传摘要，提示 AI 需要时可 read_file |
| 无精确定位输入时 AI 要编辑 | 要求用户必须通过选区、精确引用或目标文件块列表提供位置（见《对话编辑-主控设计文档》第二节、第三节） |

### 6.5 块 ID 稳定性

- 已确认：DOCX 不保存 blockId，100% 不跨会话继承；单会话内有效

## 八、结论

**需要先定义的标准/协议**：

1. **引用格式协议**：三种用途（可编辑、只读、信息获取）的 AI 消费格式
2. **AI 文档格式输出协议**：newText 格式（纯文本/HTML 子集）、字体白名单、字号、颜色；避免 AI 输出无格式纯文本
3. **记忆库读取协议**：search_memories 接口与返回格式
4. **知识库读取协议**：与记忆库对称
5. **工作流模板格式**：goal、inputs、steps 等结构（构建模式前）

**可延后**：引用样式、记忆库/知识库完整架构、工作流存储实现。

**原则**：契约先于实现；AI 消费的格式一旦确定，尽量不轻易变更，避免前后端、多模块联调时反复修改。
