## 一、命名与模式总览

### 1.1 命名统一

| 层次 | 名称 | 触发方式 |
|------|------|----------|
| 层次一 | **辅助续写** | Cmd+J / Ctrl+J |
| 层次二 | **局部修改** | Cmd+K |
| 层次三 | **对话编辑** | 聊天窗口工具调用 |

### 1.2 双模式

| 模式 | 定位 | 典型场景 |
|------|------|----------|
| **编辑模式** | 交互式 AI 辅助编辑与微构建 | 写作、修订、增量创建 |
| **构建模式** | 自动化多智能体项目构建 | 生成完整报告、书籍、知识库 |

### 1.3 Chat 模式与构建模式的演进关系

当前聊天窗口存在 **Agent 模式**（可调用工具）与 **Chat 模式**（仅对话）两种子模式。**Chat 模式是构建模式的前身**，其演进路径如下：

```
当前 Chat 模式（纯对话，无工具调用）
        │
        ▼
  增加多 Agent 讨论功能（多个 AI 角色协作讨论）
        │
        ▼
  可能引入真人多角色讨论（或需开发社交系统）
        │
        ▼
  多角色讨论成熟后，升级为构建模式（多智能体；工作流内嵌对话流，无独立引擎）
```

- **当前实现**：Chat 模式已存在，与 Agent 模式并列，用于纯对话场景（咨询、建议、快捷应用到文档等）
- **中期规划**：Chat 模式增加多 Agent 讨论，多个 AI 角色围绕议题协作讨论
- **远期可能**：引入真人参与的多角色讨论，或需配套社交系统
- **终态**：多角色讨论能力成熟后，在此基础上升级为完整的构建模式

## 三、提示词构建流程

### 3.1 辅助续写：单次构建

```
上下文收集 ──→ 拼接提示词 ──→ 单次 AI 调用 ──→ 返回 3 条续写建议（悬浮卡展示）
```

无意图预识别，无资源预搜集，直接一次性构建并调用。

### 3.1.1 局部修改：弹窗内多轮构建

```
弹窗打开 ──→ 用户输入指令 ──→ 构建提示词 ──→ AI 返回 ──→ 用户可继续追问/调整 ──→ 循环
                                    │
                                    └──→ 用户确认 ──→ 应用修改，关闭弹窗
```

弹窗内支持多轮对话，用户可反复说明或调整编辑需求，直至满意后确认应用。每轮对话构建一次提示词，无工具调用。

### 3.2 对话编辑：多层提示词（当前实现）

```
┌─────────────────────────────────────────────────────────────────┐
│ 第一层：基础系统提示词                                             │
│ 角色定义、行为规范、意图识别原则、工具调用规范                         │
└─────────────────────────────────────────────────────────────────┘
                              +
┌─────────────────────────────────────────────────────────────────┐
│ 第二层：上下文提示词                                                │
│ 当前文档、选中文本、工作区路径、编辑器状态                            │
│ 注：用户用自然语言描述位置（如「修改第十段」）时，文档格式需含 block 类型与 blockId，由 AI 理解意图，歧义时反问确认；见《基础环境与协议差距分析》5.1、7.7 │
└─────────────────────────────────────────────────────────────────┘
                              +
┌─────────────────────────────────────────────────────────────────┐
│ 第三层：引用提示词                                                  │
│ 用户引用的文本、文件、文件夹、图片等                                 │
└─────────────────────────────────────────────────────────────────┘
                              +
┌─────────────────────────────────────────────────────────────────┐
│ 第四层：工具定义（仅 Agent 模式）                                    │
│ 工具名称、参数、调用格式                                            │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    合并为完整 system + messages
                              │
                              ▼
                        AI 流式调用
```

**每轮用户消息**：构建一次完整提示词；**每次工具调用后**：将工具结果加入 messages，再次调用 AI，相当于「重新构建」下一轮提示词。

### 3.3 两阶段模式（可选增强）

部分 Agent 可采用「先识别意图，再搜集资源，最后执行」的两阶段模式：

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段一：意图识别                                                   │
│ 轻量提示词：用户消息 + 最小上下文                                    │
│ → AI 返回：意图类型、所需资源（记忆、知识库、文件等）、执行计划概要      │
└────────┬──────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│ 执行资源搜集      │  根据意图调用：search_memories、read_file、query_dataset 等
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────────┐
│ 阶段二：完整执行                                                   │
│ 完整提示词：系统 + 意图结果 + 搜集到的资源 + 工具定义                  │
│ → AI 执行实际操作（编辑、创建、分析等）                               │
└─────────────────────────────────────────────────────────────────┘
```

**适用场景**：用户请求复杂（如「结合记忆库和这份数据，重写市场分析」），需要先明确意图再按需拉取资源，避免一次性塞入过多上下文。

**当前实现**：对话编辑主要依赖「意图识别原则」写在系统提示词中，由 AI 在单次调用内自行决定是否调用工具；尚未实现显式的两阶段拆分。

## 四、Diff 效果方案（引用自 Diff效果优化方案.md）

### 4.1 辅助续写：悬浮卡 + 多选建议

- **幽灵文字已放弃**：移除 GhostTextExtension，悬浮卡为唯一方式
- **触发**：Cmd+J，光标位置弹出悬浮卡；**须光标激活**（编辑器聚焦），否则快捷键无效
- **光标未激活**：按 Cmd+J 无效，窗口底部状态栏「保存状态处」滚动显示提示「指令无效」等
- **展示**：3 条续写建议（AI 一次返回），点击或 Tab+Enter 应用
- **布局**：首选光标右下方；右侧空间不足时紧贴右边缘向左；下方空间不足时改为右上方
- **零重排**：`position: fixed/absolute`，不参与文档流
- **关闭**：手动输入、点击空白、保存、缩放等 → 关闭；已显示时再次 Cmd+J → 重构

### 4.2 局部修改：弹窗 + 对话交互

- **触发**：Cmd+K 调出弹窗；**支持无选区调出**（无选区时以光标所在块为上下文），**须光标激活**（编辑器聚焦），否则快捷键无效
- **光标未激活**：按 Cmd+K 无效，窗口底部状态栏「保存状态处」滚动显示提示「指令无效」等
- **交互**：弹窗内可多轮对话，**无对话历史**（每次覆盖上一次回复）；只回复可直接插入的内容
- **Diff**：弹窗内**不做 Diff 效果**；原文档只显示选区
- **关闭**：关闭按钮、再次 Cmd+K、点击应用 → 关闭；弹窗跟随窗口缩放

### 4.3 对话编辑：聊天侧 Diff 卡片 + 文档侧删除标记

- **文档侧**：只承载红色删除标记（#FCEBEB 背景 + 删除线），不插入新增内容，版式零影响
- **聊天侧**：每张 Diff 卡片含删除区、新增区、接受/拒绝按钮；底部全局操作栏「全部接受」「全部拒绝」
- **双状态模型**：文档显示状态（含删除标记）与文档逻辑状态（仅已接受修改）分离，AI 以逻辑状态为基准
- **时序**：全部接受时倒序执行（最新卡片优先），originalText 校验防「李石头」问题
- **后端输出**：diffId、startBlockId/endBlockId、startOffset/endOffset、originalText、newText、type

## 五、核心设计原则（ consolidated）

### 5.1 项目优先

```
项目 > 文档 > 块
```

### 5.2 双模式均具备构建能力

- 编辑模式：交互式构建（章节、大纲、表格、文档）
- 构建模式：自动化构建

### 5.3 AI 提议，编辑器执行

```
AI → 提议操作
编辑器 → 校验并执行操作
```

（当前实现可为「全文 + Diff」或「操作协议」，见 Diff效果优化方案.md）

### 5.4 工作流从生产中涌现

工作流来源于**真实任务执行中涌现**的可复用计划，不由系统预设。

**涌现机制**：
- 用户执行任务 → AI 记录工具调用链 → 任务完成 → AI 整理 Plan JSON
- 加权分 = Σ(节点敏感度权重)；加权分 ≥ 阈值（建议 8）→ AI 询问是否保存
- 出现 `memory_save` 节点时，无论加权分多少，直接触发涌现询问
- 用户也可手动创建工作流

**使用方式**：对话中 `@` 引用或 AI 自动调起；无独立执行入口；执行内嵌在对话流中。详见 Binder模板库需求文档、Binder 工作流模板 PRD。

## 七、编辑模式核心子系统（ consolidated）

### 7.1 编辑协议 (BEP)

Binder 的骨架，定位清晰，已有实现基础（replace_range、insert_text、originalText 校验）。

- **操作类型**：replace_range、insert_text、create_block、create_section、move_section、generate_outline、split_document 等（含结构级操作，原 SEE 能力合并为 BEP 高级操作类型）
- **文档寻址**：doc://block/{id}、doc://block/{id}/range/{start}-{end}、doc://section/{id}、doc://cursor、doc://selection
- **操作校验**、操作日志、撤销/重做
- **持续验证**：BEP 执行器内轻量检查，每次操作后校验文档完整性

**说明**：跨文档推理由 CCE 的项目摘要 + 文档摘要覆盖。

### 7.2 AI 工具（按需实现，无独立协议框架）

以下工具在对话编辑工作流中真实调用（read_file、edit_current_editor_document 等为当前实现）。

- **项目工具**：list_documents、get_project_summary
- **文档工具**：read_block、read_section、read_document_summary
- **数据工具**：load_dataset、query_dataset
- **编辑工具**：propose_edit_operations、generate_outline、create_document

**实现方式**：不做通用的工具注册/调度框架，按需实现函数调用。ContextManager.build_multi_layer_prompt 直接注入工具定义，为正确做法。

### 7.3 上下文压缩引擎 (CCE)

优先级应更高，为核心能力。四层提示词构建（3.2 节）即 CCE 的体现，已有实现。

- **层次**：项目摘要、文档摘要、章节摘要、块内容
- **选择优先级**：选中文本 > 当前章节 > 相邻块 > 文档摘要 > 项目摘要
- **动态摘要**：超出 token 限制时截断 + 摘要，简单策略即可
- **语义检索**：在知识库中复用 embedding，CCE 自身不需独立向量索引

### 7.4 项目记忆系统 (PMS)

**范围收缩**：仅负责概念记忆 + 结构记忆；知识记忆 = 知识库（独立 PRD）；工作流记忆 = 工作流库。

- **概念记忆**：术语、约定、用户偏好
- **结构记忆**：文档结构偏好

**实现建议**：一个 JSON 文件，存项目级术语表和结构偏好，注入 CCE 系统提示词第一层。数据结构 + 读写接口，非复杂系统。

### 7.5 AI 编辑稳定性（仅保留版本检查点）

- **版本检查点**：AI 批量编辑前自动创建快照，出错可回滚，实现成本低
- **操作沙箱**：本地应用无隔离需求，不保留
- **结构护栏**：BEP 校验机制已覆盖，不单独保留
- **持续验证**：已并入 BEP 执行器内的轻量检查

## 八、构建模式核心子系统

**说明**：构建模式由当前 Chat 模式演进而来，**当前未实现**。演进路径：Chat → 多 Agent 讨论 → 真人多角色（或需社交系统）→ 构建模式。MAR 落地至少在两个演进阶段之后。

**架构前置**（当前版本需打基础，见《Workspace改造可落地实施方案》Phase 0.5）：
- **TruncationStrategy**：上下文截断策略可扩展（KeepRecent / SummarizeMiddle / KeepTaskGoal），避免构建模式长任务时早期任务目标被截掉
- **ToolErrorKind**：工具调用失败类型标记（Retryable / Skippable / Fatal），任务调度层据此决定重试/跳过/中止，避免反向解析错误字符串

### 8.1 多智能体角色（概念保留，运行时降级）

**角色定义**（planner、researcher、writer、editor、reviewer、analyst）保留为概念文档。

**实现策略**：不做 MAR 运行时框架（智能体生命周期、通信总线）。当前及近期采用「带角色 system prompt 的顺序 AI 调用」替代；等 Chat 多 Agent 讨论功能稳定后再评估是否需要调度层。

### 8.2 工作流（无独立执行引擎）

工作流执行**内嵌在对话流中**，AI 读取 Plan JSON 后顺序调用工具，非独立引擎。

- **Plan JSON 数据格式**：goal、inputs、steps、source、weighted_score 等（见 Binder 工作流模板 PRD）
- **执行**：在对话编辑上下文里，由 Agent 按 steps 自主执行
- **工作流执行**：内嵌在对话编辑中，AI 读取 Plan JSON 后顺序调用工具，非独立执行引擎。

### 8.3 冲突处理（已由 Diff 方案覆盖）

对话编辑中唯一真实存在的冲突：多张 Diff 卡片接受时，其他卡片的 originalText 可能失效。

**已有方案**（4.3 节）：倒序执行 + originalText 校验。无需独立的 CRS 模块。其他冲突类型（结构冲突、版本冲突）在单用户本地场景下不会发生。

## 九、完整 AI 运行时架构

```
用户界面
     │
编辑模式 / 构建模式
     │
├─ 编辑模式：辅助续写 | 局部修改 | 对话编辑
├─ 构建模式：未实现；由 Chat 演进，多智能体角色为概念保留
     │
AI 运行时
     │
├─ 工具（按需实现，无独立协议框架）
├─ 上下文引擎 (CCE)
├─ 工作流：内嵌对话流，无独立引擎；Plan JSON 为数据格式
     │
编辑协议 (BEP)（含结构级操作、版本检查点）
     │
文档引擎
     │
项目存储（含 PMS 概念/结构记忆、工作流库、操作历史）
```

MAR 降级为概念保留；冲突由倒序+originalText 覆盖。
