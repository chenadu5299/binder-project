## 一、文档定位

本文是构建模式（Build Mode）的产品级定义文档。

本文回答三个问题：

1. 构建模式是什么，解决什么问题。
2. 构建模式在 Binder 产品体系中处于什么位置。
3. 构建模式与编辑模式的边界在哪里。

本文不负责：路径内部设计细节、技术架构、AI 协议。这些由 `R-BLD-M-D-01` 及后续文档承接。

## 二、构建模式定义

### 2.1 一句话定义

构建模式是 Binder 的项目构建引擎，以"需求 → 决策 → 执行 → 项目产出"为完整闭环，支持单人直接构建与多人讨论后构建两种路径。

### 2.2 解决的问题

编辑模式（对话编辑）解决的是"已有内容的修改与优化"问题。

构建模式解决的是"从零生成一个完整项目"问题：

1. 用户有明确的目标，但项目结构、内容、资源需要从无到有生成。
2. 用户需要多方协同讨论后再执行，决策过程需要记录和收束。
3. 生成任务规模大、持续时间长，需要后台执行而不阻断用户。

### 2.3 与编辑模式的关系

| 维度 | 编辑模式（对话编辑） | 构建模式 |
|---|---|---|
| 核心目标 | 修改、优化已有文档 | 从零生成完整项目 |
| 触发方式 | 用户打开文档后随时触发 | 用户主动开启构建流程 |
| AI 架构 | 三层 AI（续写/局部修改/对话编辑） | 独立主控 AI + 角色 AI 体系 |
| 执行时长 | 短时（秒级到分钟级） | 长时（分钟到几十分钟） |
| 中断性 | 用户可随时停止 | 执行阶段不可打断，可后台运行 |
| 产出形式 | Diff（需用户决策） | Project Object（直接写入 Workspace） |
| 与编辑器关系 | 强耦合（实时操作编辑器） | 解耦（构建完成后写入 Workspace） |

**核心结论：构建模式与编辑模式完全独立。代码实现上不共享 Agent 执行链，不共享编辑器状态，不共享 Diff 机制。**

## 三、构建模式在产品体系中的位置

### 3.1 Binder 能力层级

```
Binder
├── 编辑能力（核心能力）
│   ├── 文档直编（M1）
│   ├── AI 辅助续写（M2，Cmd+J）
│   ├── AI 局部修改（M3，Cmd+K）
│   └── AI 对话编辑（M4，聊天窗口）
│
└── 构建能力（补充能力）
    └── 构建模式（Build Mode）
        ├── 直接构建（Direct Build）
        └── 讨论构建（Discussion Build）
```

### 3.2 定位说明

1. 编辑能力是 Binder 的核心能力，MVP 首版的发布主链。
2. 构建模式是 Binder 的补充能力，在编辑能力稳定后推进。
3. 构建模式不替代编辑模式，构建完成后生成的文档可在编辑模式下继续编辑。

### 3.3 MVP 发布状态

构建模式**不属于 MVP（v0.1.0）发布范围**。

- `CORE-C-D-01` 第七节明确：构建模式属于"非 MVP 范围"。
- `PROD-C-D-04` 第七节明确：构建模式交互编排"暂不纳入"。

构建模式在文档层先完成设计，工程实现在 MVP 发布后排期推进。

## 四、两条路径总纲

### 4.1 路径一：直接构建（Direct Build）

**定位：** 主路径，单人高效模式。

**特征：**
1. 用户提供描述、需求、引用后直接触发构建。
2. 主控 AI 生成执行计划并自动执行，无需用户干预。
3. 执行期间不可中断，可后台运行。
4. 输出 Project Object，写入 Workspace。

### 4.2 路径二：讨论构建（Discussion Build）

**定位：** 扩展路径，多人决策模式。

**特征：**
1. 用户创建讨论组（Discussion Room）。
2. AI 角色与真人参与者共同讨论，主控 AI 持续收束结论。
3. 讨论经历三个阶段：进行中 → 待确认 → 已结束。
4. 发起人触发构建，主控 AI 基于讨论结果生成大纲并执行。

## 五、产品级约束

以下约束适用于构建模式全部路径，不可在模块内部被覆盖：

1. **结果导向**：构建模式的所有流程必须导向"生成项目"，不允许构建模式变成普通聊天。
2. **阶段驱动**：讨论必须有阶段边界，讨论组不允许无限期持续。
3. **决策显性化**：谁参与、谁确认、谁决策，系统必须有记录，不允许隐式决策。
4. **不可逆节点**：讨论组已结束状态不可回退，需继续讨论必须新建。
5. **执行不可打断**：构建执行阶段（主控 AI 执行计划期间）不允许用户修改输入，可后台运行。
6. **模块隔离**：构建模式的 AI 角色、主控 AI、执行链与编辑模式完全隔离，不共享实现。
7. **用户控制隐私边界**：系统不自动判断什么内容敏感，隐私访问范围由用户在进入讨论组前主动选择。

## 六、术语定义（构建模式专属）

| 术语 | 定义 |
|---|---|
| Build Mode | 构建模式，Binder 的项目构建引擎 |
| Direct Build | 直接构建路径，单人触发，主控 AI 自动执行 |
| Discussion Build | 讨论构建路径，多方讨论后触发构建 |
| Discussion Room | 讨论组，讨论构建路径的协作空间，类似轻量级会议室 |
| Master AI | 主控 AI，隐式存在，负责任务分发、触发判断、构建执行 |
| Role AI | 角色 AI，用户自定义，只提供分析和建议，不参与最终生产 |
| Project Object | 项目实体，构建模式的输出单元，包含主文档、子文档、资源等 |
| Build Outline | 构建大纲，主控 AI 在执行前生成的执行规划 |
