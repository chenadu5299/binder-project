# 102_场景边界清单

## 一、文档定位

本文定义 Binder MVP 的场景边界清单，用于回答三件事：
1. 哪些场景必须进入当前版本。  
2. 哪些场景明确不进入当前版本。  
3. 新需求进入前如何做边界判定。  

本文与 `101_使用场景总览` 配套：  
`101` 负责“场景地图”，`102` 负责“边界裁剪”。

---

## 二、边界判定基线

统一按以下四条判定：
1. 是否服务于 MVP 主路径（S0-1/S0-2/S0-3/S0-4）。  
2. 是否能在当前平台与资源约束内稳定交付（macOS 单平台）。  
3. 是否会打破已确认的主链规则（先确认再生效、单一真源、可观测）。  
4. 是否会显著增加发布阻塞风险（新依赖、新协议面、新渲染体系）。  

仅当四条都满足，场景才进入 MVP 范围。

---

## 三、MVP 纳入边界（In Scope）

## 3.1 主路径场景（必须）

1. `S0-1` 本地文档生产主链。  
2. `S0-2` AI 辅助编辑主链。  
3. `S0-3` 对话编辑可确认主链。  
4. `S0-4` Workspace 连续工作主链。  

## 3.2 关键子场景（必须）

1. `S2-1` 文件管理子场景（打开、新建、导入、重命名、删除、移动）。  
2. `S2-2` 文档编辑子场景（富文本/md/txt/html/docx 可编辑，保存与回读一致）。  
3. `S2-3` AI 三层子场景（层次一/层次二/层次三主链可用）。  
4. `S2-4` Diff 子场景（生成、展示、单卡决策、批量决策、失效处理）。  
5. `S2-5` 引用子场景（文本/文件/文件夹/链接进入上下文）。  
6. `S2-6` 可观测子场景（错误分级、跳过继续、最小复现链路）。  

## 3.3 能力边界（必须）

1. 开源首版仅要求核心能力可用：文件管理、文档编辑、AI 三层、对话编辑主链。  
2. 模型适配首批官方名单：DeepSeek、OpenAI、Anthropic。  
3. 提供 OpenAI 兼容接入能力（自定义 Base URL/API Key/Model）。  

---

## 四、MVP 排除边界（Out of Scope）

1. 移动端完整交互与适配。  
2. Excel/演示文稿完整编辑链路。  
3. 构建模式全流程。  
4. 分页/T-DOCX 作为开源首版发布前置条件。  
5. 图片 Diff 与复杂富媒体语义编辑。  
6. 非核心子系统的 UI 完整重构。  

这些项可以并行研发，但不作为 v0.1.0 发布阻塞项。

---

## 五、条件性纳入边界（Conditional）

以下场景允许进入 MVP，但必须满足附加条件：

1. docx 可编辑能力：必须保证“可读、可改、可保存、可回读”，不要求分页一致性。  
2. 非当前文档静默加载编辑：必须满足目标文件就绪链（解析完成、上下文注入完成）再执行。  
3. 多轮 Diff 并行池：必须保证“未交集可继续、被修改区失效、不弹阻断提示”。  
4. 扩展模型接入：不新增协议分支，仅走统一 provider 适配层。  

---

## 六、边界冲突处理规则

当新需求与当前边界冲突时，按以下顺序处理：

1. 优先保证 S0 主路径可发布。  
2. 不破坏“先确认再生效”的对话编辑主链。  
3. 不引入第二套状态真源或第二套执行协议。  
4. 可降级则降级，不可降级则延期到下一版本。  

---

## 七、边界准入流程（用于评审）

任何新增场景进入开发前，必须填写：

1. 目标场景 ID（映射到 `101` 的 S0/S1/S2/S3）。  
2. 所属边界类型（In Scope / Out of Scope / Conditional）。  
3. 对主链影响（是否影响 S0-1~S0-4）。  
4. 风险级别（发布阻塞/非阻塞）。  
5. 处理结论（纳入本版/延期/拆分）。  

无以上信息的需求，不进入迭代排期。

---

## 八、发布前边界核对清单（MVP）

发布前逐项核对：

1. In Scope 项全部具备可演示闭环。  
2. Out of Scope 项未被隐式混入主链。  
3. Conditional 项均满足附加条件且有验证记录。  
4. 失败场景具备统一暴露形态（日志与错误码）。  
5. 文档口径与实现口径一致，无“文档已纳入、代码未实现”的误判项。
