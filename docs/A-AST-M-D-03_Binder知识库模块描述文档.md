## 一、文档定位

本文是 Binder 知识库模块的**模块级主控文档**。

本文回答：

1. Binder 知识库模块是什么。
2. Binder 知识库模块承接什么，不承接什么。
3. 知识库与 workspace、记忆库、模板库、当前轮 artifact 如何分层协同。
4. 知识库模块的对象语义、行为闭环和消费规则应该如何理解。
5. 哪些能力属于知识库模块，哪些能力应明确否决。

本文不替代：

1. `A-AST-M-T-02_知识库机制.md` 的具体机制与数据结构。
2. `A-AST-M-D-02_Binder Agent知识库协同主控文档.md` 的 Agent 协同优先级裁定。
3. `A-AST-M-P-01_上下文注入.md` 的注入顺序、预算与裁剪。

## 二、模块定位

Binder 知识库模块的定位是：

**位于 workspace 工作台中的、由用户主导管理的文档级知识资产模块。**

它是：

1. 用户导入、存入、组织和管理知识资料的模块。
2. AI 在层次三场景下进行事实补强、证据召回、资料引用的知识来源模块。
3. 连接“项目文档现场”与“长期知识资产”的受控桥梁。

它不是：

1. 记忆库。
2. 模板库。
3. 当前轮状态仓。
4. 企业统一搜索平台。
5. 黑箱式自动问答引擎。

---

## 三、模块目标

知识库模块的核心目标有且仅有以下几类：

1. 让用户把真正值得长期管理的资料沉淀为知识资产。
2. 让这些资产保持可管理、可命名、可移动、可替换、可删除。
3. 让 AI 在层次三中能以检索、引用、补强的方式消费这些资产。
4. 让知识结果具备稳定来源、可定位、可核验、可注入的能力。
5. 为未来 workspace 扩展、构建模式、讨论构建、外部知识源接入留出接口，但不提前越界承担这些职责。

## 四、核心边界

### 4.1 与记忆库的边界

知识库与记忆库必须严格分层：

| 对象 | 主要来源 | 管理权 | 主要用途 |
|---|---|---|---|
| 知识库 | 用户导入、用户存入、项目文档显式存入 | 用户主导 | 文档级知识、证据、资料召回 |
| 记忆库 | AI 自动提取与升格 | 系统主导 | 语义延续、结构沉淀、长期偏好 |

知识库模块不得承接：

1. 自动记忆写入。
2. 长期偏好抽取。
3. 对话标签连续语义沉淀。

### 4.2 与模板库的边界

知识库提供内容与证据，模板库提供过程约束。

知识库模块不得承接：

1. 工作流模板解释与执行。
2. 工作流模板治理。
3. 模板资产管理。
4. 过程约束定义。

### 4.3 与当前轮 artifact 的边界

当前轮 artifact 是当前轮协作对象，不是知识资产对象。

知识库模块不得承接：

1. plan / candidate / diff / verification / confirmation 的存储。
2. 当前轮状态恢复。
3. 高影响中间态的默认长期保存。

### 4.4 与 workspace / 项目文档的边界

workspace 文件系统是 Binder 的工作现场，知识库是其中的知识资产分区。

关系如下：

1. 项目文档可以成为知识库来源。
2. 但项目文档默认不等于知识库条目。
3. 只有用户显式存入或冻结规则显式触发后，项目文档才可转入知识库链。

### 4.4.1 与结构型资产的边界补充

知识库可以承接结构型资产。

结构型资产回答的是：

1. 产出结构可参考什么。
2. 栏目组织可参考什么。

它不回答：

1. 任务应按什么流程推进。
2. 执行步骤应如何组织。

后者属于模板库中的工作流模板语义。

### 4.5 与企业搜索平台的边界

知识库模块可以为未来外部源接入留接口，但当前不得承接：

1. 组织级统一身份。
2. 企业级跨 SaaS 连接器平台。
3. 全域权限图谱。
4. 搜索到行动的组织级执行平台。

Binder 当前知识库首先服务于**文档工作台**，不是服务于**跨系统办公搜索平台**。

## 五、核心对象与结构语义

知识库模块的最小对象层级为：

1. `KnowledgeBase`
2. `KnowledgeFolder`
3. `KnowledgeEntry`
4. `KnowledgeDocument`
5. `KnowledgeChunk`

### 5.1 KnowledgeBase

`KnowledgeBase` 是知识资产容器。

它负责表达：

1. 一个知识库的稳定身份。
2. 它所属的 workspace / owner scope。
3. 它当前是否可用、可见、可检索。

### 5.2 KnowledgeFolder

`KnowledgeFolder` 是知识资产分类节点。

它负责表达：

1. 用户对知识的组织方式。
2. entry 的归属与管理结构。
3. 非检索主语义的分类上下文。

### 5.3 KnowledgeEntry

`KnowledgeEntry` 是知识库中的用户可见条目。

它负责表达：

1. 条目的显示名称与归属。
2. 条目与文档对象之间的一一对应关系。
3. 条目级状态、来源、验证与版本切换语义。

### 5.4 KnowledgeDocument

`KnowledgeDocument` 是文档库中的原始知识对象。

它负责表达：

1. 原始文件或原始文档内容的稳定身份。
2. 文档版本、校验信息、解析状态与索引状态。
3. 原始证据的回溯锚点。

### 5.5 KnowledgeChunk

`KnowledgeChunk` 是检索库中的检索与引用对象。

它负责表达：

1. chunk 内容。
2. 它对应的 entry/document。
3. heading、页码、块路径、来源定位等引用锚点。

### 5.6 结构语义补充原则

五对象骨架是正确的，但仅有字段名还不够。  
在 Binder 体系中，这些字段必须上升为**对象级语义控制点**，否则知识库模块仍然是“字段级设计”，不能作为可否决设计的主控依据。

模块层要求机制文档必须补出以下语义：

1. `source_type`
2. `source_ref`
3. `ingestion_mode`
4. `sync_mode`
5. `verification_status`
6. `visibility_scope / access_policy`
7. `active_version_id`
8. `provenance`
9. `deletion_status`
10. `retrieval_state`

### 5.7 `source_type / source_ref` 语义约束

1. 每个 `KnowledgeEntry` 与 `KnowledgeDocument` 都必须能明确回答“它从哪里来”。
2. 任何知识条目若无法表达稳定来源，不得进入可引用状态。
3. `source_ref` 必须指向可审计来源，不得使用只对当前会话有效的临时引用作为长期来源键。
4. 若来源是 workspace 文件，必须明确其导入方式属于 `copy / link / snapshot` 之一，不得模糊处理。

### 5.8 `sync_mode` 语义约束

1. `sync_mode` 决定知识对象是否允许后续同步更新，不只是“是否保留一个字段”。
2. 凡是存在外部同步能力的知识对象，必须通过 ingestion pipeline 更新，不得直接改写检索库。
3. `sync_mode=none` 的对象不得被未来连接器或后台任务静默改写。
4. `sync_mode` 发生变化时，必须产生可观测状态变更，不得静默切换。

### 5.9 `verification_status` 语义约束

1. `verified` 内容在检索排序中必须获得稳定加权提升。
2. `unverified` 内容在证据型任务中不得作为默认最高可信引用来源。
3. 若任务显式要求“证据、出处、依据、对照”，未验证内容进入上下文时必须带风险标记。
4. `verification_status` 改变时，必须能影响至少以下三个面向：
   - 检索排序
   - 引用可信度展示
   - 注入时的风险标识

### 5.10 `active_version_id / version` 语义约束

1. `KnowledgeDocument` 必须具备版本语义，不能只保留“最新文件”这一种模糊状态。
2. `active_version_id` 是当前对检索、引用、注入生效的唯一版本真源。
3. 替换流程未完成前，新版本不得抢占 `active_version_id`。
4. 旧版本可以被归档、下线或标记 superseded，但不得在没有显式状态迁移的情况下消失。

### 5.11 `provenance` 语义约束

1. 每个可检索、可引用结果都必须保留 provenance 路径。
2. provenance 至少要能回溯到 `kb -> entry -> document -> chunk` 或等价链。
3. 若结果缺失 provenance，则该结果不得进入默认引用链。
4. provenance 不只是审计信息，也必须服务于用户核验与 UI 打开原文。

### 5.12 `visibility_scope / access_policy` 语义约束

1. 可见性与访问控制不是未来连接器阶段才需要的附属字段，而是知识资产边界的一部分。
2. 任何未来外部知识源若无法映射到 `visibility_scope / access_policy`，则不得直接接入知识库主链。
3. 自动检索范围必须受可见性约束，不得把不可见对象混入默认检索结果。
4. `visibility_scope` 变化必须经过显式治理动作，不得随同步任务静默扩散。

### 5.13 `retrieval_state` 语义约束

1. `retrieval_state` 用于表达对象当前是否可被检索、可被引用、可被自动注入。
2. `index_ready` 不等于 `reference_ready`；若结果尚不可稳定溯源，则不得视为可引用。
3. 文档处于 `reindexing / pending_delete / superseded` 等状态时，必须明确限制其自动检索资格。
4. 任何对象若处于状态不明或状态冲突，必须默认退出自动检索范围。

## 六、核心行为闭环

### 6.1 导入闭环

知识库模块的导入闭环是：

`用户上传 / 存入 -> 创建条目 -> 写入文档库 -> 解析 -> 分块 -> 建索引 -> ready`

该闭环的意义是：

1. 确保资产先被建立，再进入检索链。
2. 把“用户资产创建”与“后台索引构建”区分开。

### 6.2 替换闭环

知识库模块的替换闭环是：

`用户确认替换 -> 新文档版本入库 -> 新索引构建 -> active version 切换 -> 旧版本降级或归档`

该闭环必须满足：

1. 替换失败不破坏旧版本可用性。
2. 替换成功后，引用与检索应指向新 active version。

### 6.3 删除闭环

知识库模块的删除闭环是：

`用户确认删除 -> 条目标记删除 -> 检索数据删除 -> 文档数据删除 / 归档 -> 完成状态回写`

该闭环必须满足：

1. 删除是双库一致性动作。
2. 删除失败必须可观测、可恢复、可重试。

### 6.4 移动与重命名闭环

移动和重命名的原则是：

1. 重命名优先是展示层变更，不改变 document identity。
2. 移动优先是归属层变更，不改变 document content。
3. 只有在结构语义确实影响索引时，才触发轻量 metadata reindex。

### 6.5 检索与消费闭环

知识库模块的检索闭环是：

`任务形成查询 -> 知识库检索 -> chunk 命中 -> entry 聚合 -> document 溯源 -> 引用 / 注入 / 打开原文`

这意味着：

1. 知识库不是“只返回一段文本”的模块。
2. 它必须同时服务于 AI 注入、引用消费和用户核验。

### 6.6 检索稳定性与一致性约束

#### 6.6.1 命中稳定性

1. 在同一知识库、同一查询条件、同一 active version 集合下，检索结果应保持可解释的相对稳定。
2. 若排序变化，必须能归因于版本切换、验证状态变化、可见性变化、索引重建或显式排序策略变化。
3. 多轮对话中的重复查询，不得因无关状态噪声而产生不可解释的大幅漂移。

#### 6.6.2 引用稳定性

1. `chunk -> entry -> document` 的映射必须唯一且可追踪。
2. 一条引用一旦生成，必须能定位其当时所指向的 active version 或历史版本。
3. 文档替换后，旧引用不得被静默重写为新内容；必须保留“旧引用已指向历史版本或已失效”的可判定语义。

#### 6.6.3 失效策略

1. 文档删除、版本切换、chunk 重建、entry 下线等行为，必须定义对应的引用失效语义。
2. 已失效结果不得继续作为默认自动注入来源。
3. 若旧引用仍需展示，必须明确标记为 `stale / superseded / deleted / unavailable` 中的某一种，而不是静默消失。

#### 6.6.4 冲突裁决

1. 多个 chunk 命中但语义冲突时，系统不得直接拼接为同等可信事实。
2. 多个 entry 内容矛盾时，应优先按以下控制点裁决：
   - 用户显式引用
   - 当前文档事实
   - verification_status
   - active_version
   - provenance 完整度
3. 若冲突不可裁决，必须降级为“并列候选证据”，不得伪装成单一确定事实。

## 七、消费与协同规则

### 7.1 与 workspace 的协同

知识库模块在 workspace 中承担“知识资产入口”。

它与 workspace 的关系是：

1. 以 `workspace_path` 为默认边界。
2. 接受来自项目文档的显式存入。
3. 不反向吞并文件系统主链。

### 7.2 与 Agent / 对话编辑的协同

知识库默认只在层次三承接。

它在层次三中的消费方式包括：

1. `@知识库` 或 `@知识条目`
2. 拖拽知识条目进入输入区
3. AI 自动检索知识库补强当前任务

### 7.3 显式引用与自动检索的规则

冻结规则如下：

1. 用户显式引用高于自动检索。
2. 自动检索只做补强，不得伪装成用户指定。
3. 自动检索不得覆盖当前文档事实与当前轮状态。

### 7.4 与记忆库的协同

知识库与记忆库只通过上下文协议协同，不通过资产对象混用协同。

协同原则：

1. 记忆库更偏语义延续。
2. 知识库更偏文档证据。
3. 记忆库自动检索优先于知识库自动检索。
4. 二者都不得压过当前文档与显式引用。

### 7.5 与模板库的协同

知识库提供内容，模板库提供约束。

一个任务同时使用二者时：

1. 知识库提供证据、背景、材料。
2. 模板库提供过程步骤与依赖约束。

### 7.6 与构建模式 / 讨论构建的协同

知识库模块可以为未来模式留接口，但当前规则是：

1. 可作为背景知识输入。
2. 不作为构建流程的执行引擎。
3. 不因未来模式预期而提前承担工作流职责。

### 7.7 外部扩展接口约束

为未来 workspace 扩展、外部知识源与连接器预留接口时，必须遵守以下约束：

1. 任何外部知识源都必须映射为 `KnowledgeBase / KnowledgeFolder / KnowledgeEntry / KnowledgeDocument / KnowledgeChunk` 对象链中的合法对象，不允许绕过对象模型直接接入。
2. 任何外部同步都必须通过 ingestion pipeline，不允许直接写入检索库。
3. 外部数据不得默认进入自动检索范围，必须具备用户可控的启用边界、可见性边界与 workspace 边界。
4. 外部知识对象必须补齐 `source_type / source_ref / sync_mode / access_policy / provenance`，否则不得进入 Binder 知识库主链。

## 八、治理规则

### 8.1 用户主导规则

知识库模块必须坚持用户主导：

1. 用户决定哪些资料进入知识库。
2. 用户决定条目如何命名与归类。
3. 用户决定是否替换、删除、移动。

### 8.2 自动写入否决规则

默认禁止：

1. AI 自动把生成结果写入知识库。
2. 系统自动把所有项目文档收入知识库。
3. AI 自动把任意引用材料归档成条目。

### 8.3 可溯源规则

知识库结果必须具备：

1. 稳定 ID。
2. 来源标签。
3. 文档回溯能力。
4. 定位锚点。

### 8.4 一致性规则

知识库模块必须坚持双库一致性：

1. 删除不是只删文档或只删索引。
2. 替换不是只换文件名或只换 chunk。
3. 失败必须可记录、可恢复、可重试。

### 8.5 可治理规则

知识库模块不是“越多越好”，而是“越可治理越好”。

至少应支持或预留：

1. 验证状态。
2. 版本状态。
3. 删除状态。
4. 来源状态。
5. 可见性状态。

### 8.6 违反治理规则的默认处理

1. 若某设计无法表达来源、版本、可见性、验证或 provenance 中的关键控制点，则该设计不得进入实现排期。
2. 若某实现绕开 ingestion pipeline、双库一致性或状态绑定规则，则该实现必须回退为实验性方案，不得进入主链。
3. 若某扩展方案会削弱用户主导、显式引用优先或当前文档优先，则必须直接否决，而不是通过工程兜底“后续再补”。

## 九、设计否决条件

### 9.1 直接否决条件

以下任一情况成立，必须直接否决设计：

1. 任何设计允许 AI 在未经用户确认的情况下写入知识库。
2. 任何设计将当前轮 artifact 直接落入知识库主链。
3. 任何设计让知识库自动检索结果覆盖当前文档事实。
4. 任何设计把知识库作为当前轮状态恢复层或任务状态仓。
5. 任何设计把 workspace 文件默认等价为知识库条目。
6. 任何设计允许外部知识源绕过对象模型或 ingestion pipeline 直接写入检索库。
7. 任何设计把知识库主链改写为企业统一搜索平台、组织级动作平台或黑箱式问答平台。
8. 任何设计在层次一、层次二默认承接层次三知识库主链。

### 9.2 违反判定规则

出现以下任一情形，即可判定为违反：

1. 设计文档或实现方案中出现“自动保存到知识库”“后台自动归档当前结果”“默认收录项目文档”等表述，但没有用户显式确认门。
2. 设计把 plan、candidate、diff、verification、confirmation 或等价当前轮对象映射为 `KnowledgeEntry` 或等价长期知识对象。
3. 检索或注入流程中存在“知识库结果高于当前文档”“知识库结果覆盖当前现场事实”的规则表述。
4. 恢复、续跑、回滚、任务状态机等语义被绑定到知识库对象而不是 artifact / task / state 对象。
5. 外部同步链直接生成 chunk 或索引而不经过 entry/document 建模与 ingestion 状态链。

### 9.3 违反处理规则

1. 被判定违反的方案，不得进入机制文档冻结阶段。
2. 已进入实现阶段但被发现违反的，必须停止继续扩散，回退到符合主控语义的上一层设计。
3. 若违反涉及数据结构或持久化主链，必须在继续开发前先完成机制层修正，不得以“先做后补”为由保留在主分支语义中。

## 十、MVP 范围与否决项

### 10.1 MVP 范围

知识库模块的 MVP 应聚焦以下内容：

1. 知识库 / 分类 / 条目 / 文档 / chunk 五对象闭环。
2. 上传、存入、删除、替换、移动、重命名基础链路。
3. 双库一致性。
4. chunk 检索 + entry/document 溯源。
5. `@引用`、拖拽引用、自动检索补强。
6. 层次三消费。

### 10.2 明确否决项

MVP 明确不做：

1. 企业统一搜索平台。
2. 组织级连接器中台。
3. 默认图增强检索。
4. 默认大规模向量平台。
5. 层次一、层次二默认承接知识库。
6. AI 自动落库。
7. 当前轮状态托管。

### 11.2 行为闭环与机制状态绑定要求

本文中的行为闭环，必须在 `A-AST-M-T-02_知识库机制.md` 中找到对应状态位与状态迁移；否则视为机制分层未打通。

最小绑定要求如下：

| 模块行为 | 机制文档必须绑定的状态位 |
|---|---|
| 导入闭环 | `entry.status`, `document.parse_status`, `document.index_status`, `chunk.retrieval_state` |
| 替换闭环 | `document.version`, `active_version_id`, `index_status`, `superseded/superseding` 或等价状态 |
| 删除闭环 | `deletion_status`, `pending_delete`, `deleted`, `index_removed` 或等价状态 |
| 移动 / 重命名闭环 | `metadata_version`、归属状态、是否需要 `metadata_reindex` |
| 引用与检索闭环 | `reference_ready`, `retrieval_state`, `provenance_complete` 或等价状态 |

若 `A-AST-M-T-02_知识库机制.md` 无法承接这些状态位，则：

1. 本文中的行为闭环不视为已真正闭合。
2. 机制文档必须先补状态语义，再进入实现文档冻结。

## 十二、验收口径

当以下问题都能被稳定回答时，可认为知识库模块描述层合格：

1. 能明确说明知识库是什么，不是什么。
2. 能明确区分知识库、记忆库、模板库、当前轮 artifact、workspace 文件系统。
3. 能明确说明为什么知识库默认只在层次三承接。
4. 能明确说明为什么显式引用高于自动检索。
5. 能明确说明为什么自动检索知识库只能补强，不能覆盖当前文档事实。
6. 能明确说明知识库模块承接哪些行为闭环。
7. 能明确列出知识库模块的否决项。
8. 能为机制文档、实现文档和开发计划继续拆分提供稳定上位语义。
9. 能把模块行为闭环稳定映射到机制文档的状态位与状态迁移。
10. 能明确判断未来扩展接口是否越权污染知识库主链。
