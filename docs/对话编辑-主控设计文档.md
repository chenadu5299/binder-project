## 一、核心设计原则

### 1.1 两个变量原则

对话编辑的本质是同时解决两个变量：**内容**（改什么）和**位置**（改哪里）。

每个场景都必须有确定的内容和位置来源。当任一变量无法确定时，
按降级阶梯处理，而不是强行猜测。

### 1.2 职责边界

**模型的职责**：
- 理解用户编辑意图
- 从系统提供的块列表中找到目标块（通过块编号）
- 提供 target 文本和 content 替换内容

**系统的职责**：
- 维护文档块结构（blockId）和位置映射，对模型透明
- 将块编号转换为 blockId
- 在块内执行精确文本定位（字符偏移）
- 块内定位失败时自动降级为整块替换（无需用户确认）
- 生成 canonical diff，展示 diff 卡供用户确认

**模型永远不接触 blockId。**

### 1.3 定位精度降级阶梯
```
第一级（零搜索）：前端直接提供精确位置
  来源：用户选区坐标或精确引用坐标（startBlockId/startOffset/endBlockId/endOffset）
  → 后端直接生成 canonical diff，跳过所有文本搜索

第二级（块内搜索）：模型指定块编号 + target 文本
  → 在指定块内搜索 target
  → 找到唯一匹配或指定 occurrence_index → 精确字符偏移
  → 块内未找到 → 自动降级第三级

第三级（整块替换）：块内搜索失败
  → 整块替换直接应用，不需用户确认
  → diff 卡标注"块级替换"

第四级（报错）：块编号超出范围，无法确定目标位置
  → ToolError(Retryable)，模型重新规划
```

## 二、用户意图信号的三种状态

### 状态A：精确引用标签（由选区复制/粘贴生成）

执行口径：

- `TextReference` 必须携带 `startBlockId/startOffset/endBlockId/endOffset`
- 当无显式选区且存在精确引用时，后端强制走零搜索
- 引用展示可标注块号，模型不接触 `blockId`

### 状态B：编辑器显式选区

执行口径：

- 前端注入完整选区坐标：`selection_start_block_id/start_offset/end_block_id/end_offset`
- Resolver 检测到精确坐标后，直接走零搜索
- 禁止退回块内搜索

### 状态C：无选区，仅光标位置

执行口径：

- 捕获 `cursor_block_id/cursor_offset` 作为上下文提示，不作为强制编辑目标
- 块列表可对光标所在块做提示标记
- 该状态默认进入块内搜索路径

## 三、场景分析与处理路径

### 场景1：用户有选区（精度最高）

内容：`selected_text` ✅  位置：`editTarget`（blockId+offset）✅

后端 Resolver 走零搜索路径，直接生成精确 canonical diff。
**当前已有数据流，后端改造量极小。**

### 场景2：无选区，文件在编辑器中打开

内容：`editor.getHTML()` 的纯文本块列表 ✅  位置：blockId 映射 ✅

系统注入结构化块列表，模型指定块编号和 target，
Resolver 块内搜索，失败时自动整块替换。

### 场景3：全文扫描修改（如"改所有英文为中文"）

内容：完整纯文本块列表 ✅  位置：完整 blockId 映射 ✅

系统识别全文扫描意图，注入完整块列表。
模型遍历所有块，逐块调用工具，每次一个 diff。
禁止将多块局部编辑退化为一次 `rewrite_document`。

### 场景4：文件未打开，但曾打开过（所有文件类型）

内容：`file_cache` 中的 HTML ✅  位置：blockId ✅

后台静默调用 `open_file_with_cache` 读取缓存，走场景2路径。

### 场景5：文件从未打开（html/htm/docx）

内容：磁盘原始内容 ✅  位置：P4 管道可生成 ✅

后台静默调用 `open_file_with_cache` / `open_docx_with_cache`，
生成带 blockId 的 canonical HTML 写入缓存，走场景4路径。

### 场景6：文件从未打开（md/txt）

内容：磁盘原始文本 ✅  位置：需要生成 ✅（可行）

**处理路径（静默加载）**：
```
后端实现 inject_blockids_for_plain_text：
  按空行分割段落
  每段生成 <p data-block-id="block-ws-{uuid-v4}">内容</p>
  写入 file_cache（复用 upsert_file_cache）

BlockIdExtension 已有保留逻辑：
  节点已有 data-block-id 且唯一 → 保留，不覆盖
  因此后端生成的 blockId 在 TipTap 打开时保持有效

用户打开文件后：
  sync_workspace_file_cache_after_save 在保存时
  用 TipTap 导出的 HTML 覆盖缓存（含 TipTap 的 blockId）
  此后走正常路径
```

**blockId 格式**：`block-ws-{uuid-v4}`（与 TipTap 的 `block-{uuid-v4}` 通过前缀区分）

### 场景统一门禁：跟需求走，不跟状态走

非当前文档执行必须满足就绪状态链：

1. `targetFileResolved`
2. `canonicalLoaded`
3. `blockMapReady`
4. `contextInjected`

任一状态未就绪，本次调用不得进入工具执行链。

## 四、内容注入：结构化块列表

### 4.1 块列表格式
```
[文档块列表]
Block 0 [标题]: 第一章 概述
Block 1 [正文]: 这是第一段，含有 some English text 和中文混排。
Block 2 [正文]: 第二段内容，这里有重复词重复词出现两次。  ← [光标位置]
Block 3 [列表]: 列表项内容
Block 4 [正文]: 最后一段。
[/文档块列表]

总块数：5。编辑时通过块编号（0起）指定目标块。
```

- 内容为 `strip_html_tags` 后的纯文本
- 块类型从 HTML 标签推断（`p`→正文，`h1/h2/h3`→标题，`li`→列表等）
- 光标所在块用 `← [光标位置]` 标注（状态C时）
- blockId 不暴露给模型

### 4.2 注入策略

| 任务类型 | 识别条件 | 注入内容 | 状态 |
|---------|---------|---------|------|
| 有选区或精确引用 | 存在精确区间坐标 | 坐标所在块 ± 前后各2块 | 强制生效 |
| 文件系统操作 | 含文件操作关键词 | 只注入文件名 | 保持不变 |
| 全文扫描修改 | 含全文扫描意图关键词 | 完整块列表 | 强制生效 |
| 短文档（< 800字） | 纯文本总长 < 800字 | 完整块列表 | 建议生效 |
| 其他局部编辑 | 默认 | 摘要块列表（前10块 + 所有标题块） | 建议生效 |

### 4.3 全文扫描意图识别

`contains_full_scan_intent` 函数识别以下模式：

中文："所有" + "改/替换/修改/统一/翻译"，"全文" + 动词，"把所有X改成Y"
英文："all occurrences"，"replace all"，"throughout the document"，"every instance of"

## 五、工具接口规范

### 5.1 edit_current_editor_document 参数
```json
{
  "block_index": 1,
  "edit_mode": "replace | delete | insert | rewrite_block | rewrite_document",
  "target": "some English text",
  "content": "一些英文文本",
  "occurrence_index": 0
}
```

| 字段 | 必填条件 | 说明 |
|------|---------|------|
| block_index | edit_mode ≠ rewrite_document 时必填 | 块编号（0起），来自块列表 |
| edit_mode | 必填 | 见下表 |
| target | edit_mode 为 replace/delete/insert 时必填 | 块内精确原始纯文本 |
| content | edit_mode=delete 时可省略 | 替换/插入内容 |
| occurrence_index | 可选，默认0 | 块内有多处相同 target 时，指定第几处（0起） |

| edit_mode | 说明 |
|-----------|------|
| replace | 替换块内 target 为 content |
| delete | 删除块内 target |
| insert | 在块内 target 后插入 content |
| rewrite_block | 整块替换为 content，不需要 target |
| rewrite_document | 全文重写，不需要 block_index 和 target |

**废弃字段**（后端忽略，不得在新代码中生成）：
`edit_target.anchor`、`element_identifier`、`instruction`、`target_content`、`scope=block`

**系统注入字段**：

- `current_file`
- `current_content`
- `document_revision`
- `baseline_id`
- `_sel_start_block_id`
- `_sel_start_offset`
- `_sel_end_block_id`
- `_sel_end_offset`
- `_sel_text`

### 5.2 系统提示编辑规则
```
## Editing Rules

Use edit_current_editor_document for ALL edits to the open file.

Steps:
1. Find the target block in [文档块列表]
2. Call edit_current_editor_document with block_index and target text

Rules:
- block_index: from [文档块列表], 0-based
- target: exact plain text as shown in the block, no HTML tags
- Multiple same text in one block: use occurrence_index (0-based)
- Full rewrite: edit_mode=rewrite_document, no block_index needed
- Block rewrite: edit_mode=rewrite_block with block_index, no target needed
- For multi-block tasks: call the tool ONCE PER BLOCK
- Never use rewrite_document for multi-block local edits
- Never fabricate block_index values not in [文档块列表]
```

## 六、后端 Resolver 规范

### 6.1 输入
```rust
pub struct ResolverInput {
    // 模型提供
    pub block_index: Option<usize>,
    pub edit_mode: String,
    pub target: Option<String>,
    pub content: Option<String>,
    pub occurrence_index: usize,           // 默认0
    // 系统提供（链路绑定）
    pub baseline_id: String,
    pub document_revision: Option<u64>,
    // 系统提供（零搜索路径）
    pub selection_start_block_id: Option<String>,
    pub selection_start_offset: Option<usize>,
    pub selection_end_block_id: Option<String>,
    pub selection_end_offset: Option<usize>,
    pub selected_text: Option<String>,
    // 文档来源
    pub current_editor_content: String,    // 完整 HTML，含 data-block-id
}
```

### 6.2 执行流程
```
Step 1: 解析块结构
  遍历 current_editor_content 中所有 data-block-id 节点，按文档顺序：
  block_map[index] = {
    block_id: String,
    block_type: String,       // p | h1 | h2 | h3 | li | ...
    text_content: String,     // strip_html_tags 结果
    char_count: usize,
  }
  构建 BlockTreeIndex（只读）：
  tree_node = { block_id, block_index, path, text_start, text_end, text_hash }
  与 baseline_id 绑定

Step 2: 路由分发
  selection_start_block_id 或精确引用存在 → Step 2a（零搜索）
  edit_mode == "rewrite_document"        → Step 2e
  edit_mode == "rewrite_block"           → Step 2d
  其他                                   → Step 2b

Step 2a: 零搜索路径（选区/引用直接定位）
  使用 selection_start_block_id/offset + selection_end_block_id/offset
  originalText = selected_text（或从 HTML 提取）
  newText = content
  diff_type = "precise"
  → Step 3

Step 2b: 块内搜索
  block_index 缺失            → ToolError(Fatal)
  block_index >= len          → ToolError(Retryable): "block_index {N} out of range, document has {M} blocks"
  在 block_map[block_index].text_content 中搜索 target：
    匹配数 == 0               → Step 2c（自动降级整块替换）
    匹配数 == 1，idx=0        → 精确定位 → Step 3
    匹配数 > 1，idx 在范围内  → 精确定位 → Step 3
    匹配数 > 1，idx 超出范围  → ToolError(Retryable): "Found {M} in block {N}, range 0-{M-1}"

Step 2c: 整块替换降级（自动，不需用户确认）
  new_text = 将 text_content 中 target 替换为 content（找不到则直接用 content）
  startOffset = 0，endOffset = char_count
  originalText = text_content（整块原文）
  diff_type = "block_level"
  → Step 3

Step 2d: rewrite_block（模型主动请求整块替换）
  block_index 缺失            → ToolError(Fatal)
  startOffset = 0，endOffset = char_count
  originalText = text_content，newText = content
  diff_type = "block_level"
  → Step 3

Step 2e: rewrite_document（整篇替换）
  startBlockId = block_map[0].block_id，startOffset = 0
  endBlockId = block_map[last].block_id，endOffset = block_map[last].char_count
  originalText = 全文纯文本（各块拼接），newText = content
  diff_type = "document_level"
  → Step 3

Step 3: 生成 canonical diff
  {
    diffId: uuid-v4,
    startBlockId, startOffset,
    endBlockId, endOffset,
    originalText, newText,
    type: "replace" | "delete" | "insert",
    diff_type: "precise" | "block_level" | "document_level",
    route_source: "selection" | "reference" | "block_search"
  }

Step 4: 推入 pending_diffs，触发前端 diff 卡渲染
```

### 6.3 错误处理

| 场景 | Kind | error（给模型） | display_error（给用户） |
|------|------|----------------|----------------------|
| block_index 超出范围 | Retryable | "block_index {N} out of range. Has {M} blocks (0 to {M-1})." | "块编号超出范围，AI 正在修正" |
| 块内多处匹配，idx 超范围 | Retryable | "Found {M} in block {N}. occurrence_index: 0 to {M-1}." | "块内多处相同文本，AI 正在确认位置" |
| block_index 缺失 | Fatal | "block_index required for edit_mode={X}." | "编辑参数不完整" |
| target 缺失 | Fatal | "target required for edit_mode={X}." | "缺少目标文本" |
| 块内未找到 target | 不报错，自动降级 | — | diff 卡标注"块级替换" |

统一暴露码（执行观测层）：

- `E_ROUTE_MISMATCH`
- `E_TARGET_NOT_READY`
- `E_RANGE_UNRESOLVABLE`
- `E_ORIGINALTEXT_MISMATCH`
- `E_PARTIAL_OVERLAP`
- `E_BASELINE_MISMATCH`
- `E_APPLY_FAILED`
- `E_REFRESH_FAILED`
- `E_BLOCKTREE_NODE_MISSING`

**光标场景**：`from === to` 时，调用 `findBlockAtPos(doc, from)` 提取光标块信息，

### 7.3 @ 引用场景

`TextReference.formatForAI` 补充块编号标注（块编号在系统侧计算，不暴露 blockId）：

引用对象已有的 `startBlockId/startOffset/endBlockId/endOffset` 作为备用定位输入传入后端

### 7.4 diff 卡 diff_type 展示

| diff_type | diff 卡展示 |

## 十三、附录 A：前端 diffStore 与可选扩展字段

本节承接原《对话编辑 Diff 数据格式规范》中前后端契约与演进字段，避免实现时仅依赖 JSON 示例。

### A.1 单条 DiffEntry（前端，与 canonical 对齐）

除第九节 JSON 外，前端 `diffStore` 单条通常还包含：

| 字段 | 说明 |
|------|------|
| status | `pending` \| `accepted` \| `rejected` \| `expired` |
| acceptedAt | 接受时间戳；逻辑状态正序重放时使用 |
| mappedFrom / mappedTo | 用户编辑文档后 Mapping 映射后的 PM 位置 |

约束：同一次工具返回中各 diff 区间不重叠；`originalText` 须与生成时文档逻辑状态中该区间一致。

### A.2 可选扩展字段（v1.1+ 建议，非强制）

与定位真源 `L`、revision 门禁、失效可解释性对齐，可在工具返回或 `diffStore` 元数据择一层承载：

| 字段（建议名） | 说明 |
|----------------|------|
| baselineId / documentRevision | 绑定该轮定位真源 |
| routeSource | `selection` \| `reference` \| `block_search` |
| executionExposure | 结构化观测对象（与错误码枚举对齐） |
| expireReason | 过期枚举，供 UI：`content_modified`、`original_text_mismatch`、`range_unresolvable`、`partial_overlap` |

### A.3 表格与代码块

- 表格：table 节点需有 blockId；整表 Diff 时 startBlockId = endBlockId = 表 blockId；块内 offset 约定由实现定义。
- 代码块：按块内 textContent 定位。
- 图片：MVP 不做图片 Diff。
