## 一、定义

模板库仅包含一种资产：

**工作流模板（Workflow Template）**

工作流模板的定义是：

1. 对用户复杂操作流程的结构化记录。
2. 供 AI 在对话中按需调用或由用户主动引用的过程资产。
3. 以 Plan JSON 为底层结构、以自然语言为表达层的执行计划。

模板库只承接：

1. 步骤顺序。
2. 任务依赖。
3. 执行意图。
4. 过程约束。

模板库不承接：

1. 输出结构定义。
2. 风格定义。
3. 行为定义。
4. 事实内容存储。

---

## 二、核心定位

工作流模板是 Binder 对用户真实执行过程的可复用沉淀。

工作流模板不是：

1. 独立运行的自动化程序。
2. 黑箱执行引擎。
3. 知识内容容器。

工作流模板是：

1. 从真实任务执行中涌现出来的可复用计划。
2. 对话中的上下文资产，由用户主动引用或 AI 自动建议。
3. 只约束过程，不替代事实、证据和输出结果。

## 三、工作流模板详细设计

### 3.1 生命周期

```text
用户在聊天中执行任务
        │
        ▼
AI 执行并记录工具调用链
        │
        ▼
任务完成 → AI 整理 Plan JSON
        │
        ▼
加权分 ≥ 阈值？
   │              │
  是              否
   │              │
AI 询问：        不触发
是否保存为工作流？
   │
用户确认 ──→ 存入工作流模板库
             （可编辑 · 可删除）
```

### 3.2 涌现触发机制

涌现触发基于节点加权分数。

触发规则：

1. 加权分达到阈值时触发保存询问。
2. 出现 `memory_save` 节点时直接触发保存询问。
3. 未经用户确认不得入库。

### 3.3 节点架构

Binder 的工作流节点代表信息在任务推进过程中的一次变换。

常见节点分为四层：

1. 获取：`knowledge_fetch`、`memory_fetch`、`doc_read`、`web_fetch`
2. 加工：`extract`、`summarize`、`compare`、`classify`、`translate`
3. 生成：`draft_write`、`section_expand`、`table_gen`
4. 输出：`format_export`、`doc_insert`、`memory_save`

### 3.4 敏感度权重

| 权重 | 含义 | 判断标准 |
|------|------|----------|
| 3 | 强复用信号 | 调用了用户特定资产，重建成本高 |
| 2 | 中等复用信号 | 操作有特定意图，但通用性较强 |
| 1 | 弱复用信号 | 通用操作，任何时候都可重做 |

## 四、使用方式

工作流模板没有独立执行入口，只在对话中被调用。

两种路径：

1. 用户主动引用：用户在聊天中通过 `@` 引用工作流模板。
2. AI 自动建议：用户任务意图与已保存工作流高度匹配时，AI 提出建议，用户确认后加载。

执行过程完全内嵌在对话流中，用户看到的是语义化进度，不是技术节点。

## 五、AI 调用机制

### 5.1 调用目标

目标是让工作流模板生成后可稳定调用和执行。

### 5.2 调用协议

#### 5.2.1 模板检索接口

| 协议项 | 内容 |
|--------|------|
| **命令** | `list_templates(options?: { scene?, tags?, limit? })` |
| **命令** | `search_templates(query: string, options?: { limit? })` |
| **返回** | `{ items: [{ id, name, goal, steps, scene }], total? }` |

#### 5.2.2 模板引用格式

```typescript
references?: Array<{
  type: 'template';
  source: string;
  content: string;
  templateType?: 'workflow';
}>;
```

#### 5.2.3 工具调用

| 工具名 | 说明 | 参数 |
|--------|------|------|
| `list_templates` | 列出可用模板 | scene?, limit? |
| `search_templates` | 语义检索模板 | query, limit? |
| `get_template` | 获取单个模板详情 | templateId |

### 5.3 提示词注入

模板作为上下文注入时，只注入：

1. `name`
2. `goal`
3. `steps`
4. `depends_on`

模板注入语义固定为过程约束。

## 六、核心数据模型与协议

### 6.1 模板结构

```typescript
interface WorkflowTemplate {
  id: string;
  name: string;
  description?: string;
  scene?: string;
  tags?: string[];
  source?: 'system' | 'user' | 'emerged';
  createdAt?: string;
  updatedAt?: string;
  goal: string;
  weightedScore?: number;
  steps: Array<{
    id: string;
    type: string;
    label: string;
    params?: Record<string, unknown>;
    depends_on?: string[] | null;
  }>;
}
```

### 6.2 Plan JSON

工作流模板的底层存储格式为 Plan JSON。

```json
{
  "id": "wf_20260318_001",
  "name": "竞品对比分析报告",
  "description": "从知识库获取竞品资料并完成对比分析",
  "source": "emerged",
  "weighted_score": 11,
  "goal": "完成一次竞品对比分析流程",
  "steps": [
    {
      "id": "step_1",
      "type": "knowledge_fetch",
      "label": "检索竞品知识库",
      "params": {
        "knowledge_base_id": "kb_competitor_2026",
        "query_hint": "竞品核心功能与定价"
      },
      "depends_on": null
    },
    {
      "id": "step_2",
      "type": "doc_read",
      "label": "读取当前产品文档",
      "params": {
        "doc_scope": "current_editor"
      },
      "depends_on": null
    },
    {
      "id": "step_3",
      "type": "compare",
      "label": "横向对比分析",
      "params": {
        "dimensions": ["功能", "定价", "目标用户"]
      },
      "depends_on": ["step_1", "step_2"]
    }
  ]
}
```

## 七、UI 与交互

### 7.1 模板库面板

| 位置 | 说明 |
|------|------|
| **入口** | 标签栏或资源管理区添加模板库标签 |
| **展示** | 工作流模板列表，可按场景和标签筛选 |
| **与记忆库、知识库** | 三者并列，支持 `@` 统一候选列表 |

### 7.2 工作流模板侧边栏

工作流模板在侧边栏的作用是管理，不是触发执行。

**详情面板内容**：

1. 工作流名称
2. 来源标注（涌现 / 手动创建）
3. 步骤摘要
4. 操作：编辑步骤、删除

### 7.3 引用方式

1. `@` 引用
2. 拖拽引用
3. 选择后发送

## 九、术语表

| 术语 | 说明 |
|------|------|
| **工作流模板** | 用户复杂操作流程的结构化记录，从真实执行中涌现；以 Plan JSON 为底层结构，供 AI 在对话中按需调用 |
