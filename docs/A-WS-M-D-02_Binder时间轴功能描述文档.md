## 2. 时间轴定义

### 2.1 唯一口径

Binder 的时间轴，只记录：

**项目逻辑状态的已成立变更。**

也就是说，只有当某项操作已经真正改变了项目当前成立的逻辑状态，才允许进入时间轴。

时间轴在 Binder 中的正式产品语义是：

**项目逻辑状态历史的可浏览、可定位、可还原时间视图。**

### 2.2 项目逻辑状态的定义方式

在 Binder 当前实现中，项目逻辑状态应理解为：

1. 已真实写入工作区文件系统的内容状态
2. 已真实生效的工作区资源结构状态
3. 已经正式进入项目范围的资源对象状态

它不是：

1. 编辑器内尚未保存的内容
2. 候选 diff 的待确认内容
3. UI 展示层看到但尚未写盘的中间内容

### 2.3 当前阶段项目逻辑状态范围

当前阶段，Binder 时间轴覆盖的“项目逻辑状态”范围只包括：

1. 工作区文件内容状态
2. 工作区资源结构状态

其中：

1. 工作区文件内容状态，指工作区内文档、文本、HTML、DOCX 等文件在真实写盘后的内容状态
2. 工作区资源结构状态，指工作区内文件、文件夹及其命名、位置、存在性关系在真实生效后的结构状态

当前阶段明确不扩展到以下更高层或旁路状态：

1. 项目编排状态
2. 协同状态
3. 知识状态
4. 任务状态
5. 记忆状态
6. 搜索与索引状态
7. 任何不直接属于工作区文件内容或资源结构的系统状态

这意味着：

当前阶段的时间轴不是“Binder 全部状态历史”，而是：

**工作区文件内容状态与资源结构状态的时间轴。**

### 2.4 何时算“时间轴节点成立”

时间轴节点成立必须满足两个条件：

1. 操作已经通过 Binder 当前主链完成确认
2. 操作已经真实改变项目成立状态

在当前实现中，这对应的成立点主要包括：

1. 已打开文件链：`saveFile` 成功写盘
2. 未打开文件链：`accept_file_diffs` 成功写盘
3. 资源结构链：`create_file`、`create_folder`、`rename_file`、`delete_file`、`duplicate_file`、`move_file` 成功完成

### 2.5 时间轴节点粒度定义

当前阶段，每个时间轴节点对应：

**一次已成立的项目逻辑状态变更，并形成一个可被引用和还原的状态断点。**

这一定义同时冻结了三层含义：

1. 一个时间轴节点不是一段聊天
2. 一个时间轴节点不是一条 diff
3. 一个时间轴节点不是一个 task 阶段

它对应的是：

1. 一次已经成立的状态变化事实
2. 一次把项目从 `S(n)` 推进到 `S(n+1)` 的状态推进
3. 一个后续可被定位、展示和还原的状态断点

因此，当前阶段时间轴节点的正式粒度不是“过程粒度”，而是：

**成立后的状态断点粒度。**

### 2.6 关键说明

必须明确：

1. `acceptDiff` 不等于时间轴节点成立
2. `pending diff` 不等于时间轴
3. `file_cache` 不等于时间轴版本
4. 外部修改后的被动同步不等于时间轴节点成立
5. 一次成立入口执行成功，不必然新增时间轴节点
6. 时间轴节点记录的是状态断点，不是操作过程回放

## 3. 时间轴边界

### 3.1 进入时间轴的内容

以下内容属于 Binder 时间轴候选：

1. 已确认并真实写入项目文件的内容修改
2. 已确认并写入项目的新生成内容
3. 已确认并正式进入项目的新建文档
4. 已确认并正式进入项目的新建文件夹与资源结构变化
5. 已确认并真实生效的重命名、移动、复制、删除
6. 已确认并真实写入项目的格式编辑、结构编辑、块编辑
7. 任何已使项目从状态 `S(n)` 进入 `S(n+1)` 的成立操作

### 3.2 不进入时间轴的内容

以下对象明确不进入时间轴：

#### 3.2.1 AI / diff / 审阅过程

1. 聊天历史
2. AI 思考过程
3. agent task 执行过程
4. artifact 过程
5. 工具调用过程
6. `pending diff`
7. `reject / expired diff`
8. `DiffCard / FileDiffCard`
9. diff 审阅过程本身

原因：

这些对象描述的是“候选、分析、审阅、执行过程”，不是项目逻辑状态已经成立后的事实。

#### 3.2.2 外部修改与被动同步

1. 文档被外部修改后的状态更新
2. 外部文件变动检测
3. `mtime` 变化
4. `content hash` 变化
5. `reload / refresh / cache sync`
6. 外部覆盖后的被动对齐

原因：

这些对象描述的是 Binder 对外部现实状态的被动同步，不属于 Binder 内部确认后的项目状态成立。

#### 3.2.3 缓存 / 校验 / 临时态

1. `file_cache`
2. `baseline`
3. `revision`
4. `positioning snapshot`
5. 编辑器内未保存状态
6. 仅用于校验或显示的中间结构

原因：

这些结构只服务当前一致性、定位和显示，不自动等价于时间轴版本。

#### 3.2.4 其他旁路系统

1. `memory`
2. `knowledge version`
3. `search index`
4. 调试日志
5. 错误暴露记录
6. 本地 `localStorage` 的旧历史占位结构

原因：

这些对象各自有独立职责，不应污染项目逻辑状态时间轴。

#### 3.2.5 失败操作与非成立动作

以下对象明确不进入时间轴：

1. 失败操作
2. 未成立动作
3. 未改变项目逻辑状态的空操作
4. 外部同步导致的被动状态刷新

原因：

1. 时间轴只记录已经成立的项目逻辑状态变更
2. 失败操作没有把项目推进到新的成立状态
3. 未成立动作不构成可还原的状态断点
4. 如需保留失败留痕，应进入独立机制，而不是时间轴

### 3.3 为什么必须隔离

时间轴必须与上述对象隔离，否则会立刻出现以下问题：

1. 候选态被误记为已成立事实
2. UI 展示被误当为状态事实
3. 外部同步被误记为用户时间轴
4. `cache / snapshot / revision` 被误当成版本链
5. 用户无法区分“项目已变更”与“系统正在处理”

## 4. 时间轴与现有系统的关系

### 4.1 与当前前端“时间轴”UI 标签的关系

当前前端已经存在历史入口面板，后续正式标签应统一为“时间轴”，位置在：

1. [HistorySection.tsx](/Users/chenzhenqiang/Desktop/test/binder-project/src/components/FileTree/HistorySection.tsx)
2. [ResourceToolbar.tsx](/Users/chenzhenqiang/Desktop/test/binder-project/src/components/FileTree/ResourceToolbar.tsx)

当前事实是：

1. 该入口位置继续沿用，不另起新入口
2. 该标签的正式语义应修正为：**项目逻辑状态时间轴**
3. 当前代码中的 `localStorage` 占位实现不是正式时间轴系统
4. 后续该标签展示的内容必须以项目逻辑状态时间轴为准，而不是以旧 `binder-history` 数据为准

### 4.2 与 diff 审阅链的关系

diff 审阅链的职责是：

1. 生成候选修改
2. 帮助用户确认或拒绝
3. 在确认前阻止错误修改直接进入项目

时间轴的职责是：

1. 只记录已经成立的项目状态变化
2. 展示时间轴节点
3. 允许基于时间轴节点执行时间轴还原

结论：

1. diff 可以是时间轴节点成立前的候选载体
2. diff 不是时间轴本体
3. `pending diff`、`reject diff`、`expired diff` 都不进入时间轴主链

### 4.3 与 AI 过程链的关系

AI 做了什么，不等于项目状态发生了什么。

因此：

1. AI 过程链不直接构成时间轴
2. 只有 AI 相关操作最终真实改变了项目逻辑状态，才以“项目状态变更”身份进入时间轴
3. 时间轴节点展示的是项目状态变化，不展示 AI 过程细节

### 4.4 与 `file_cache / revision / snapshot` 的关系

这些结构在当前系统中负责：

1. 当前态缓存
2. 一致性校验
3. 定位与失效控制

它们与时间轴的关系是：

1. 可作为实现时的辅助校验来源
2. 不构成时间轴版本本体
3. 不应直接作为时间轴展示对象

### 4.5 与外部修改同步的关系

外部修改同步只负责把 Binder 的当前界面与磁盘现实状态重新对齐。

因此：

1. 外部修改同步不计入时间轴
2. 被动 reload / refresh / cache sync 不计入时间轴
3. 时间轴只认 Binder 内部确认后的成立变更

## 5. 时间轴展示语义

### 5.1 UI 标签沿用原则

Binder 继续沿用当前前端 UI 的入口位置，但正式标签统一为“时间轴”。

该标签后续承载的是：

**Binder 项目逻辑状态时间轴列表。**

不再承载：

1. localStorage 占位历史
2. 聊天记录
3. diff 审阅过程
4. task 过程摘要

### 5.2 时间轴节点展示对象是什么

每个时间轴节点展示的是：

**一个已经成立的项目逻辑状态变更。**

不是：

1. 一段聊天
2. 一条 diff
3. 一个 task 阶段
4. 一个工具调用结果

### 5.3 单个时间轴节点的展示信息建议

单个时间轴节点至少应展示以下语义维度：

1. 操作类型
2. 作用对象
3. 发生时间
4. 简要摘要
5. 是否可执行时间轴还原
6. 时间轴还原后的影响范围

说明：

1. 这里定义的是展示语义，不是数据库字段设计
2. 展示摘要应服务用户理解“项目发生了什么变化”
3. 不要求把候选 diff 或 AI 推理细节搬进时间轴节点

### 5.4 当前占位实现与正式时间轴系统的关系

当前 `HistorySection` 的事实状态是：

1. 使用 `localStorage` 的 `binder-history`
2. 固定上限 50 条
3. 当前只有资源栏创建文件/文件夹成功会写入
4. 类型枚举中混入了 `ai_edit_file`、`save_file` 等旧占位项

因此必须明确：

1. 当前实现只是 UI 占位
2. 当前实现不能被解释为正式时间轴系统
3. 后续正式时间轴系统应复用“时间轴”标签，但不能复用当前占位数据语义

## 6. 时间轴还原流程与用户操作路径

### 6.1 “时间轴还原”的定义

时间轴中的“还原”，指的是：

**针对某个时间轴节点对应的项目逻辑状态断点，执行项目状态层面的恢复动作。**

它作用于：

1. 项目文件内容
2. 项目资源结构
3. 项目当前成立状态

它不作用于：

1. 聊天现场
2. AI 过程现场
3. diff 候选界面
4. task 运行现场

### 6.2 时间轴还原对项目意味着什么

时间轴还原意味着：

1. 选定一个时间轴节点对应的已成立状态断点
2. 将该状态重新作用到当前项目
3. 让项目进入一个新的当前成立状态

因此，时间轴还原不是“回看过去”，而是：

**用时间轴节点驱动一次新的项目状态恢复。**

### 6.3 时间轴还原的断点语义

当前阶段，Binder 的“时间轴还原”正式指向：

**某个时间轴节点所对应的项目逻辑状态断点。**

这意味着：

1. 还原恢复的是该时间轴节点对应的状态断点
2. 还原不是简单逆转该节点本身的单次操作
3. 还原不是把某条 diff 重新反向应用
4. 还原不是补丁级回滚

必须明确：

当前阶段的还原语义是“恢复到状态断点”，而不是“逆放一次操作”。

这样定义的原因是：

1. Binder 当前冻结的时间轴节点粒度是“已成立状态断点”
2. 若把还原定义成单次逆操作，后续很容易把时间轴系统错误实现成操作日志回滚系统
3. 若把还原定义成 diff 重放，后续又会退回 diff 审阅链语义

因此，Binder 时间轴中的“还原”应始终理解为：

**以时间轴节点为锚点，对项目逻辑状态执行断点恢复。**

### 6.4 用户操作路径

当前阶段的时间轴还原操作路径应保持简单、稳定、可执行：

1. 用户在“时间轴”列表中选中一个时间轴节点
2. 用户点击“还原”，系统弹出确认信息并说明覆盖当前状态与影响范围
3. 用户确认后，系统执行状态恢复并刷新当前项目状态

这是当前阶段唯一的正式用户路径。

### 6.5 确认弹窗与影响提示

时间轴还原前必须出现确认弹窗。

确认弹窗至少要明确提示：

1. 将覆盖当前项目逻辑状态
2. 将影响哪些文件内容或资源结构
3. 该操作不会恢复聊天、AI 过程、pending diff 或 task 现场

这样做的原因是：

1. 时间轴还原作用于项目当前成立状态，而不是局部 UI
2. 用户需要在执行前明确知道影响范围
3. 时间轴还原不应被误解成轻量级的编辑器撤销

### 6.6 还原成功、生成节点与失败处理

时间轴还原成功后，系统应把恢复后的当前状态视为新的当前成立状态。

当前阶段的正式口径是：

1. 时间轴还原成功且当前项目状态确实发生变化时，生成新的时间轴节点
2. 如果当前状态与目标断点状态实际一致，则不新增节点
3. 时间轴还原失败时，不写入时间轴
4. 时间轴还原失败时，只给出错误提示，不写入失败节点

之所以要求成功还原后生成新节点，是因为：

1. 时间轴记录的是状态演进结果，而不是单向回看清单
2. 还原本身会把项目带入一个新的当前成立状态
3. 生成新节点可以保持时间轴主链收敛，不把“回到过去”误做成“删除现在”

### 6.7 时间轴还原边界

时间轴还原只恢复：

1. 文件内容
2. 资源结构

时间轴还原明确不恢复：

1. 聊天现场
2. AI 过程现场
3. pending diff
4. task / artifact 现场
5. 外部修改同步现场

### 6.8 与 undo / redo / restore 的关系

当前阶段应明确区分：

1. `undo / redo`
   - 指短时、交互级、编辑器内的撤销重做语义
   - 面向当前会话和当前编辑动作
   - 不等于项目级时间轴

2. `restore`
   - 指基于时间轴节点对项目逻辑状态执行恢复
   - 面向已成立的项目状态
   - 是时间轴语义下的正式用户动作

结论：

1. Binder 时间轴功能当前阶段的正式用户动作应描述为“还原 / restore”
2. 不应把时间轴描述成编辑器 `undo / redo` 栈
3. 不应把还原描述成 diff 重放

### 6.9 时间轴还原时必须坚持的边界原则

1. 还原作用于项目逻辑状态层
2. 还原不回放 AI 过程
3. 还原不重新生成 pending diff
4. 还原不恢复聊天上下文
5. 还原不把外部修改同步当成可还原对象

## 7. 时间轴数量上限确认

### 7.1 最终确认值

**当前阶段时间轴上限 = 50 个时间轴节点。**

这是当前阶段的正式确认值，不是临时占位值。

### 7.2 时间轴节点新增条件

时间轴节点的新增必须同时满足以下条件：

1. 已经过了 Binder 当前主链中的成立点
2. 相对上一已成立状态，项目逻辑状态发生了实际变化
3. 该变化属于当前阶段时间轴覆盖范围

因此必须明确：

1. `saveFile` 成功，不等于必然新增时间轴节点
2. 每次走到成立入口，不等于必然追加一个时间轴节点
3. 只有当项目逻辑状态相对上一已成立状态发生实际变化时，才新增时间轴节点

当前阶段，Binder 时间轴的正式生成策略冻结为：

**基于成立点收口、并以“实际状态变化过滤”为准的时间轴生成策略。**

这条策略的含义是：

1. 时间轴节点的候选来源来自当前主链中的各类成立点
2. 进入成立点只代表“具备生成资格”，不代表“必须生成节点”
3. 只有当成立点之后的项目逻辑状态相对上一已成立状态产生稳定且可恢复的实际变化时，才写入时间轴

必须进一步明确：

1. 时间轴不是“每次成立入口调用一次就记一条”
2. 时间轴也不是“只在保存时才生成”
3. 时间轴记录的是经过成立点过滤后的稳定状态断点，而不是成立点调用日志

冻结规则：

**时间轴节点只在项目逻辑状态发生稳定且可恢复变化时生成，而不是在每次成立入口触发。**

这条规则同样适用于：

1. 手动保存
2. 自动保存
3. `accept_file_diffs`
4. 资源结构变更入口
5. 其它真实成立入口

以下情况不应新增时间轴节点：

1. 保存成功，但内容与上一已成立状态一致
2. 成立入口执行成功，但没有形成实际状态变化
3. 失败操作
4. 外部同步
5. 仅缓存刷新或校验状态变化

### 7.3 为什么确认 50 个时间轴节点

当前阶段保持 50 个时间轴节点的理由如下：

1. Binder 当前是桌面端本地工作区产品，时间轴首先服务当前项目近期回看与还原，不服务长期审计归档
2. 本次定义下的时间轴节点是“项目逻辑状态成立后的变更”，粒度明显粗于编辑器按键级撤销栈，因此不需要过高数量
3. 时间轴标签需要具备浏览性和可操作性，50 个节点足以覆盖当前阶段用户最常回看的近期项目变更
4. 当前实现基础尚未形成正式时间轴系统，首版应优先保证语义清晰、列表可读、还原边界稳定，而不是无限累计
5. 现有 UI 占位实现本身已经采用 50 条上限，保持该数字有利于过渡期语义收口，但正式理由不再是“沿用占位”，而是“当前阶段产品控制值”

### 7.4 时间轴上限的适用对象

当前阶段的 50 条上限，只适用于：

**正式时间轴节点。**

这里的“正式时间轴节点”指的是：

1. 已成立的项目逻辑状态变更记录
2. 已被正式纳入时间轴主链的状态断点记录

这 50 条明确不包含：

1. `pending diff`
2. `reject / expired diff`
3. 失败操作
4. 外部同步
5. 缓存刷新
6. `file_cache` 更新
7. 旧 `localStorage` 占位项

因此，不能把当前旧占位实现中的 50 条逻辑，直接等同为正式时间轴系统的统计口径。

正式时间轴系统中的“50 条”，统计对象只能是：

**正式时间轴节点本身。**

### 7.5 当前阶段固定值说明

当前阶段，50 条应视为：

1. 首版固定上限
2. 进入正式实现时默认采用的明确产品值

### 7.6 后续调整触发条件

后续只有在以下条件成立时，才应调整该上限：

1. 时间轴已形成正式持久化与筛选能力
2. 时间轴节点来源显著扩展，50 条不足以覆盖主要使用场景
3. 时间轴列表已具备更强的检索、过滤或分段浏览能力

在这些条件未成立前，时间轴上限固定为 50 条。

## 8. 当前实现基础与功能落点说明

### 8.1 当前实现中会产生时间轴节点候选的成立点

基于当前代码实现，以下成立点会产出时间轴节点候选：

1. 打开文件链的 `saveFile`
2. 未打开文件链的 `accept_file_diffs`
3. 新建文件 `create_file`
4. 新建文件夹 `create_folder`
5. 重命名 `rename_file`
6. 删除 `delete_file`
7. 复制 `duplicate_file`
8. 移动 `move_file`
9. 直接写盘的 `update_file(use_diff=false)` 与 `create_file` 工具分支

### 8.2 当前 UI 标签如何承接这些时间轴节点

当前前端“时间轴”标签可以继续作为承接面，但必须完成以下语义修正：

1. 标签名统一为“时间轴”
2. 标签内容改为展示项目逻辑状态时间轴节点
3. 每个时间轴节点可提供“还原”
4. 列表来源必须来自正式时间轴事实层，而不是当前 localStorage 占位

### 8.3 当前实现距离正式时间轴系统还缺什么

当前实现还缺少以下基础：

1. 时间轴事实层
2. 时间轴节点持久化主链
3. 各成立点的统一收口
4. 时间轴节点与时间轴还原动作的正式衔接

当前已经具备的，是若干真实的状态成立入口；当前尚不具备的，是把这些入口收口成正式时间轴系统的事实层与展示层。

## 9. 结论

Binder 的时间轴，当前阶段必须被正式定义为：

**项目逻辑状态的已成立变更记录。**

它不是 AI 历史，不是 diff 历史，不是聊天历史，也不是 task 历史。

Binder 继续沿用当前前端 UI 的入口位置，但其正式语义必须收口为：

**项目逻辑状态时间轴列表。**

时间轴展示的是已经成立的项目状态变化；用户可以针对任意一个时间轴节点执行“还原”；“还原”作用于项目逻辑状态层，而不是聊天、diff 或 AI 过程层。

当前阶段，时间轴数量上限正式确认为：

**50 条。**

这个值在当前阶段固定生效，用于保证 Binder 时间轴功能的首版语义清晰、展示可控和还原边界稳定。
