# 007_系统设计原则总纲

## 一、文档定位

本文定义 Binder MVP 的全局系统设计原则。  
用途：统一产品、架构、协议、实现、交互文档中的优先级和裁决口径，避免模块各自为政。

本文是原则文档，不替代具体功能方案。

---

## 二、适用范围

适用于以下全链路：
1. Workspace 与资源管理  
2. 文档编辑主链  
3. AI 三层能力（辅助续写、局部修改、对话编辑）  
4. 引用、记忆、知识库、模板库交互  
5. Diff 生成、决策、生效、失效、观测  

---

## 三、原则优先级（从高到低）

## P0 级（不可违背）

### 原则 1：单一真源

同一业务语义只能有一个规则主源和一个运行时真源。  
示例：
- 目标文件只有一个 `targetFile` 真源  
- 一轮定位只有一个 `baselineId + L + revision` 语义源

### 原则 2：先确认再生效

AI 输出修改建议必须经用户决策后才写入真实内容。  
`pending` 不等于已应用。

### 原则 3：显示态与逻辑态分离

视觉展示不得反向定义真实内容状态。  
删除标记、卡片状态属于显示层；真实内容由逻辑状态定义。

### 原则 4：禁止隐式猜测

定位与执行必须基于显式上下文和结构化输入，不允许“猜位置、猜目标文件、猜执行意图”。

---

## P1 级（默认必须满足）

### 原则 5：接口契约先行

先定义输入/输出协议，再推进实现。  
包括：
- tools 参数 JSON Schema  
- references 协议  
- diff 输出协议

### 原则 6：跨模块边界清晰

上游负责输入，下游负责执行，中间层负责转换，不做职责越权。  
示例：引用层传内容，不做最终意图裁决；执行层不改写业务语义。

### 原则 7：项目隔离优先

所有状态、资源、检索、缓存默认按 `workspace_path` 隔离，禁止跨工作区污染。

### 原则 8：失败可暴露，主流程可继续

允许局部失败，但必须可观测、可追踪，且不阻断可继续完成的任务路径。

---

## P2 级（优化增强）

### 原则 9：降级一致且可解释

降级策略要有固定触发条件、统一语义和可追踪记录，不做临时兜底拼凑。

### 原则 10：扩展向后兼容

新增模型接入、资源类型、任务类型时，优先沿既有协议扩展，不破坏主链。

### 原则 11：性能约束前置

首版即纳入关键性能红线（启动、打开、AI首 token、批量接受），避免后期返工。

---

## 四、全局裁决规则（冲突时怎么判）

## 4.1 裁决顺序

当两条规则冲突时，按以下顺序裁决：
1. P0 级原则优先于 P1/P2  
2. 单一真源优先于局部体验优化  
3. 业务正确性优先于展示完整性  
4. 可追踪性优先于静默吞错  

## 4.2 典型冲突裁决

1. “为了快，跳过结构化定位” vs “禁止隐式猜测”  
裁决：不得跳过，必须走结构化定位或显式失败。

2. “为了连贯，把 pending 当已应用继续推理” vs “先确认再生效”  
裁决：禁止；pending 只能作为待决策状态。

3. “为了少提示，不暴露执行错误” vs “失败可暴露”  
裁决：业务层可静默，但观测层必须记录。

4. “为了兼容旧实现，继续用旧字段” vs “接口契约先行”  
裁决：旧字段仅可读兼容，不可作为新链路输入。

## 五、原则到实现的落地约束

## 5.1 架构层约束

1. 模块职责按“输入层/转换层/执行层/展示层”分离。  
2. 分流判定权集中到单点（如 Resolver），避免前后端重复分流。  
3. 状态机对象必须有明确生命周期，不允许状态隐式跃迁。  

## 5.2 协议层约束

1. 关键对象必须有版本化字段或等价演进策略。  
2. 关键执行对象（task/toolCall/diff）必须可追溯来源。  
3. 任何“临时字段”不得进入主协议。  

## 5.3 交互层约束

1. 用户决策入口必须稳定可达（单卡/批量）。  
2. 失效与失败语义分离，UI 不得混淆。  
3. 不以行号作为主定位口径，使用结构化块语义。  

## 5.4 运维与排障约束

1. 关键事件必须有统一错误码或观测事件。  
2. 复现链路必须覆盖：输入 -> 路由 -> 执行 -> 状态变化。  
3. 允许降级执行，但禁止无记录降级。  

---

## 六、MVP 最小合规检查清单

1. 是否存在同一语义多真源（文档、协议、运行时）？  
2. 是否存在“未确认先写入”的路径？  
3. 是否把显示状态当作真实状态参与执行判断？  
4. 是否存在无显式输入的猜测式分流？  
5. 是否存在无法追踪来源的 diff/task/toolCall？  
6. 是否存在错误被吞并且无法观测的链路？  
7. 是否存在跨 workspace 的状态串用？
