## 一、冻结原则

### 1.1 文档职责

**需求文档的职责**：
- 定义业务目标、功能范围、用户价值
- 不定义技术选型和接口细节

**主控文档的职责**：
- 冻结技术选型
- 冻结功能逻辑链
- 冻结接口协议
- 冻结能力边界和非目标

### 1.2 接入层次冻结

记忆库**仅接入层次三：对话编辑**。

以下层次不使用记忆库，也不具备记忆库引用逻辑：
- 层次一：辅助续写
- 层次二：局部修改

### 1.3 设计原则

- 记忆库是**上下文增强系统**，不是通用笔记系统
- 记忆默认**系统生成、只读、可引用**
- 显式引用优先于自动检索
- 作用域隔离优先于召回规模
- 先结构化存储，再做语义增强

## 二、能力边界

### 2.1 支持的能力

- 标签级记忆生成、检索、引用
- 项目内容记忆生成、更新、检索、引用
- 工作区长期记忆追加写入、检索、引用
- 用户级记忆跨项目继承与检索
- 记忆树状展示、搜索、详情查看
- `@` 记忆引用与拖拽引用
- 对话发送前自动检索相关记忆并注入 AI 上下文

### 2.2 不支持的能力

- 辅助续写使用记忆库
- 局部修改使用记忆库
- 用户手动编辑记忆内容
- 用户单独删除单条标签级记忆
- 跨标签自动读取对话上下文
- 将记忆库当作知识库全文检索替代品

### 2.3 约束规则

- 记忆项必须可追溯来源
- 自动生成失败不得阻塞主对话流程
- 检索超时必须降级为空结果
- 任何写入都必须带作用域信息

## 三、技术选型

### 3.1 存储选型

记忆库采用**本地 SQLite** 作为唯一事实源，数据库位置固定在工作区 `.binder/` 下。

选择理由：
- 与当前 Tauri 本地架构一致
- 便于作用域隔离、事务写入、生命周期管理
- 便于与工作区数据、文件路径、标签关系协同

### 3.2 检索选型

P0/P1 使用**结构化过滤 + SQLite 全文检索**。

冻结结论：
- **不引入外部向量数据库**
- **不把纯语义向量检索作为首版前提**
- 后续如需引入 embedding，只能作为增强层，不能替代结构化主存

### 3.3 数据组织选型

记忆库采用**分层 + 分作用域 + 可追溯来源**模型，而不是单表平铺。

冻结的数据实体：
- `memory_items`：记忆项主表
- `memory_sources`：来源表
- `memory_edges`：关系表
- `memory_jobs`：后台生成/更新任务表
- `memory_usage_logs`：检索与注入日志表

### 3.4 UI 选型

前端采用**树状只读展示**：
- 顶层按来源层级分组
- 下层按类型分组
- 叶子为具体记忆项

不采用：
- 可编辑表单式管理
- 手工维护型知识卡片模式

## 四、记忆模型

### 4.1 四层记忆

#### 4.1.1 标签级记忆

- 作用域：单个聊天标签
- 来源：多轮对话、引用、当前编辑任务
- 生命周期：随标签删除而删除
- 用途：当前标签对话的首要上下文

#### 4.1.2 项目内容记忆

- 作用域：当前工作区
- 来源：项目文档自动提取
- 生命周期：随文档内容变化而更新
- 用途：项目实体、术语、结构、关系的稳定语义层

#### 4.1.3 工作区长期记忆

- 作用域：当前工作区
- 来源：跨标签沉淀、标签删除摘要、项目概念归纳
- 生命周期：仅追加，不修改
- 用途：跨标签复用的项目长期上下文

#### 4.1.4 用户级记忆

- 作用域：用户
- 来源：跨项目偏好与通用模式
- 生命周期：跨项目继承，持续演化
- 用途：风格偏好、术语习惯、通用工作模式

### 4.2 记忆项最小结构

每条记忆至少包含：
- `id`
- `layer`
- `scope_type`
- `scope_id`
- `entity_type`
- `entity_name`
- `content`
- `summary`
- `source_kind`
- `source_ref`
- `confidence`
- `freshness_status`
- `readonly`
- `created_at`
- `updated_at`

### 4.3 来源结构

来源类型冻结为：
- `conversation_summary`
- `conversation_entity`
- `document_extract`
- `document_outline`
- `document_detail_enrichment`
- `tab_deletion_summary`
- `user_preference`

## 五、功能逻辑链

### 5.1 对话记忆生成链

```
用户消息 / AI 回复
  -> 对话轮次累积
  -> 达到阈值或会话空闲
  -> 生成会话摘要 / 实体 / 偏好
  -> 写入标签级记忆
  -> 必要时归纳为长期记忆
```

冻结规则：
- 生成在后台执行
- 主对话不等待记忆写入完成
- 写入失败仅记录日志，不影响回复

### 5.2 文档记忆生成链

```
文档打开 / 保存 / 批量处理
  -> 识别项目类型
  -> 解析项目结构
  -> 小文档直接抽取完整记忆
  -> 大文档先生成大纲记忆
  -> 用户实际使用时按需深入
  -> 写入或更新项目内容记忆
```

冻结规则：
- 项目内容记忆允许更新
- 长期记忆不允许回写修改
- 大文档必须采用“大纲优先、按需深入”策略

### 5.3 标签删除链

```
用户删除聊天标签
  -> 删除该标签全部标签级记忆
  -> 生成标签摘要
  -> 将删除行为 + 标签摘要写入长期记忆
```

### 5.4 对话使用链

```
用户发送消息
  -> 构建检索 query
  -> 先检查显式引用
  -> 自动检索标签级记忆
  -> 自动检索项目内容记忆
  -> 自动检索长期记忆
  -> 自动检索用户级记忆
  -> 组装 [记忆库信息] 提示词段
  -> 调用 AI
```

## 六、检索与排序协议

### 6.1 检索顺序

固定顺序如下：

`显式引用 > 标签级记忆 > 项目内容记忆 > 长期记忆 > 用户级记忆`

### 6.2 检索原则

- 先按作用域过滤，再按相关性排序
- 同层优先最近更新、置信度更高、来源更明确的记忆
- 显式引用不参与自动排序淘汰

### 6.3 检索范围

`scope` 冻结为：
- `tab`
- `content`
- `workspace_long_term`
- `user`
- `all`

### 6.4 降级策略

- 检索失败：返回空数组
- 检索超时：500ms 超时返回空数组
- 无 `tabId`：不检索标签级记忆
- 无 `workspacePath`：不检索项目内容记忆与长期记忆

## 七、接口协议

### 7.1 公开读取接口

#### `search_memories`

```ts
search_memories(
  query: string,
  options: {
    tabId?: string;
    workspacePath?: string;
    limit?: number;
    scope?: 'tab' | 'content' | 'workspace_long_term' | 'user' | 'all';
    entityTypes?: string[];
    includeOutline?: boolean;
  }
) -> {
  items: MemorySearchItem[];
  total: number;
}
```

#### `get_memory_tree`

```ts
get_memory_tree(
  options: {
    tabId?: string;
    workspacePath?: string;
    includeLayers?: Array<'tab' | 'content' | 'workspace_long_term' | 'user'>;
  }
) -> MemoryTreeNode[]
```

#### `get_memory_item`

```ts
get_memory_item(id: string) -> MemoryItemDetail | null
```

### 7.2 内部写入接口

以下为系统内部能力，不对普通 UI 暴露“手工编辑”入口：

- `upsert_tab_memories`
- `upsert_project_content_memories`
- `append_long_term_memory`
- `upsert_user_memory`
- `record_memory_usage`
- `schedule_memory_job`

### 7.3 返回结构冻结

`MemorySearchItem` 至少包含：
- `id`
- `layer`
- `scopeType`
- `scopeId`
- `entityType`
- `entityName`
- `content`
- `summary`
- `sourceKind`
- `sourceRef`
- `confidence`
- `freshnessStatus`

### 7.4 协议约束

- 搜索接口必须支持作用域过滤
- 搜索接口必须支持 `limit`
- 返回结果必须带来源信息
- 不允许再沿用仅 `query + workspacePath -> Vec<Memory>` 的旧接口作为最终协议

## 八、AI 交互协议

### 8.1 显式引用

用户通过拖拽或 `@` 选择记忆时：
- 直接把记忆内容作为 `references` 传入
- 标记来源为 `memory`
- 该内容视为已完整提供，无需再次检索

### 8.2 自动检索

用户发送消息时：
- 后端统一执行记忆检索
- 前端不负责自动检索拼装
- 后端根据当前 `tabId/workspacePath` 构建上下文

### 8.3 提示词拼接

记忆库内容统一拼接为：

```text
[记忆库信息]
- [实体名]（[类型] / [层级]）：[内容摘要]
...
```

拼接规则：
- 显式引用优先完整传入
- 自动检索仅传摘要，不传超长正文
- 必须带层级或来源标记，便于模型区分

### 8.4 使用记录

每次 AI 调用后，系统需记录：
- 命中的记忆项
- 注入方式：显式 / 自动
- 使用时间
- 所属标签与工作区

## 九、UI 协议

### 9.1 面板结构

记忆库面板冻结为只读树：
- 顶层：标签级记忆 / 项目内容记忆 / 长期记忆 / 用户级记忆
- 第二层：类型分组
- 第三层：记忆项

### 9.2 交互动作

支持：
- 查看详情
- 搜索
- 展开/折叠
- 拖拽引用
- `@` 选择引用

不支持：
- 手动编辑内容
- 手动删除单项
- 手动合并记忆

### 9.3 大纲记忆状态

项目内容记忆允许两种状态：
- `outline`
- `full`

仅项目内容记忆允许从 `outline` 升级为 `full`。

## 十、非功能要求

### 10.1 性能

- 检索超时：500ms
- 单标签默认记忆上限：200 条
- 详情加载可延迟
- 树结构支持按需加载

### 10.2 稳定性

- 任何记忆生成失败不得阻塞聊天
- 任何记忆检索失败必须静默降级
- 数据写入必须保证作用域隔离

### 10.3 可观测性

必须可追踪：
- 记忆生成来源
- 记忆最后更新时间
- 记忆被哪些对话使用过
- 哪些记忆处于过期或待更新状态
