## 一、文档定位与控制权威

### 1.1 文档定位

本文是 Binder Agent 的**记忆协同控制文档**。  
本文回答以下问题：

1. Binder 的“对话记忆”“记忆库”“当前轮 Agent 中间态”分别是什么。
2. 这三类对象如何分工，而不是互相吞并。
3. 什么内容应留在当前轮状态，什么内容应进入标签记忆，什么内容才能升格为项目 / 工作区 / 用户记忆。
4. Binder Agent指导方案中与资产沉淀、项目记忆、context governance 相关的规则，哪些落在记忆系统，哪些不落在记忆系统。
5. ``A-AST-M-T-01_记忆模型.md``、``A-AST-M-P-01_上下文注入.md`` 与 Agent 主控文档组之间如何协同。

### 1.2 控制权威

本文是记忆协同层的主控文档。  
本文不替代：

1. `A-AST-M-T-01_记忆模型.md` 的存储模型与生命周期定义。
2. `A-AST-M-P-01_上下文注入.md` 的注入顺序、预算与降级定义。
3. ``A-AG-M-D-01_Binder Agent能力描述文档.md`/`A-AG-M-T-04_Binder Agent技术主控文档.md`` 的 Agent 上位语义和技术对象主控。

但凡涉及以下问题，必须以本文为准：

1. 对话记忆与记忆库的分工。
2. Agent artifact 与 memory 的边界。
3. 哪些结果允许写入记忆，哪些结果只允许保留为当前轮 artifact。
4. 记忆升格、降级、失效与引用协同。

### 1.3 本文不负责什么

本文不负责：

1. SQLite 表结构细节。
2. 检索算法实现细节。
3. 注入 token 预算的具体数值。
4. 某个具体 UI 面板的交互稿。

这些内容分别由 ``A-AST-M-T-01_记忆模型.md``、``A-AST-M-P-01_上下文注入.md`` 和实现文档承接。

## 二、为什么需要这份文档

``A-AST-M-T-01_记忆模型.md`` 当前主要回答“记忆存什么”。  
``A-AST-M-P-01_上下文注入.md`` 当前主要回答“上下文怎么注入”。

但 Binder Agent 在收敛到 `R-AG-C-D-01_Binder-Agent指导方案（Guiding Architecture & Design Doctrine）.md` 之后，多出了旧记忆体系中没有被单独主控的几个问题：

1. **当前轮 Agent 状态与 artifact** 不是记忆库，但又会参与后续轮次协作。
2. **对话记忆** 既不是原始聊天历史，也不是整个项目长期记忆。
3. **中间态资产化** 不能直接等于“全部写入记忆库”。
4. **阶段闭合后的沉淀** 需要有升格规则，否则会把记忆库变成杂乱日志库。

因此，Binder 需要一份单独文档，冻结：

1. 记忆协同对象边界。
2. 写入与升格规则。
3. 读取与注入协同规则。
4. Binder Agent指导方案相关规则在记忆系统中的承接关系。

## 四、记忆协同总模型

## 4.1 总体立场

Binder 的记忆协同不是“把一切都写进记忆库”。  
正确关系应为：

`当前轮状态 / artifact -> 对话记忆 -> 项目 / 工作区 / 用户记忆`

这三层分别解决不同问题：

1. **当前轮状态 / artifact**：解决当前轮协作与确认。
2. **对话记忆**：解决当前标签连续多轮对话的稳定语义延续。
3. **项目 / 工作区 / 用户记忆**：解决跨轮、跨标签、跨项目的可复用沉淀。

### 4.2 四类对象的边界

| 对象 | 作用 | 默认生命周期 | 是否属于记忆库 |
|---|---|---|---|
| 当前轮 Agent 状态 | 表达当前任务处于什么阶段 | 当前轮 / 当前任务 | 否 |
| 当前轮 artifact | plan、candidate、diff、verification、confirmation 等 | 当前轮，可跨轮继续消费 | 否 |
| 对话记忆（标签级） | 稳定保存当前标签的摘要、实体、文档关联、偏好线索 | 随标签存在，标签删除则删除 | 是 |
| 项目 / 工作区 / 用户记忆 | 稳定沉淀项目概念、结构、长期习惯、跨项目偏好 | 按作用域长期存在 | 是 |

### 4.3 不得误读的点

1. 当前轮 artifact 可以被后续轮继续消费，但这不等于它已经成为记忆库内容。
2. 对话记忆不是原始聊天历史，也不是简单消息摘要集合。
3. 记忆库不是日志回收站，也不是任意对象的自动落库区。
4. 项目级沉淀不等于所有标签记忆都自动升格为长期记忆。

## 五、Binder 的记忆分工

## 5.1 当前轮状态 / artifact 的职责

当前轮状态和 artifact 负责：

1. 支撑 `plan -> execute -> verify -> confirm -> transition` 当前轮闭环。
2. 支撑可见中间态、局部接受、回退和 refinement。
3. 支撑当前轮上下文构建。

它们默认**不负责**：

1. 跨标签长期保留。
2. 作为稳定“事实记忆”被自动复用。
3. 直接成为用户级或项目级偏好。

### 5.1.1 允许进入记忆协同链的当前轮对象

以下对象可作为 `memory seed` 候选：

1. 已收敛的 `plan`
2. 已确认的 `scope`
3. 已通过结构 / 约束验证的 `verification summary`
4. 已确认的 `confirmation result`
5. 具有复用价值的 `template seed`
6. 具有复用价值的 `memory seed`

### 5.1.2 默认不得直接进入记忆库的对象

1. 未确认的候选稿
2. 已失效的 artifact
3. 仅用于当前轮调试的中间日志
4. 无来源、无作用域、无状态信息的临时片段

## 5.2 对话记忆（标签级）的职责

对话记忆的核心职责是：

1. 让当前聊天标签在多轮协作中保持语义连续。
2. 为当前标签内的计划、约束、实体、文档关联提供稳定的摘要层。
3. 作为标签删除前可被升格或沉淀的中间层。

对话记忆不等于：

1. 原始聊天历史
2. 当前轮所有 artifact
3. 工作区长期记忆

## 5.3 记忆库分层职责

| 记忆层 | 主要职责 | 典型来源 | 说明 |
|---|---|---|---|
| 标签级记忆 | 当前标签的持续语义 | 多轮对话、引用、确认后的稳定线索 | 当前连续性第一优先 |
| 项目内容记忆 | 从项目文档提取的可更新语义 | 文档内容与结构解析 | 偏项目事实与结构 |
| 工作区长期记忆 | 跨标签的项目概念与结构沉淀 | 标签摘要、概念归纳、结构快照 | 仅追加，不回写 |
| 用户级记忆 | 跨项目偏好与通用模式 | 用户长期行为、稳定偏好 | 用户维度，不混入项目私有事实 |

### 5.4 层次影响边界

延续既有三层独立要求：

1. 层次三：当前必须完整承接对话记忆与记忆库协同。
2. 层次一：如需吸收记忆能力，只允许单独立项吸收有限工作区级补强，不默认接入标签级对话记忆。
3. 层次二：当前不接入记忆库主链。

## 六、写入、升格与删除协同

## 6.1 写入原则

1. 记忆写入默认后台异步。
2. 写入失败不阻塞主协作链。
3. 每次写入必须带来源、作用域、时间和关联任务。

## 6.2 升格原则

一个对象从当前轮 artifact 或标签级记忆升格为更高层记忆，至少应满足：

1. 已通过本轮必要验证。
2. 已完成必要确认，或符合显式豁免规则。
3. 具有跨轮复用价值。
4. 作用域明确，知道它属于标签 / 项目 / 工作区 / 用户哪一层。
5. 不是仅对当前一次执行有效的临时噪声。

### 6.2.1 推荐升格链

`artifact -> memory seed -> 标签级记忆 -> 工作区长期记忆 / 用户级记忆`

说明：

1. 不是所有对象都要经过完整升格链。
2. 大多数对象只应停留在当前轮 artifact 或标签级记忆。
3. 只有稳定、复用价值高的对象才进入长期记忆或用户级记忆。

## 6.3 禁止性写入规则

以下对象不得直接写入长期记忆或用户级记忆：

1. 未通过结构 / 约束验证的内容
2. 用户尚未确认但主观性极强的候选内容
3. 已被 `invalidated` 标记的 artifact
4. 仅来自一次失败尝试的噪声结果

## 6.4 删除与失效协同

1. 标签删除时，标签级记忆删除，但允许将”删除行为 + 标签摘要”追加写入工作区长期记忆。
2. 当前轮 artifact 失效时，应标记失效，不自动清洗已合法升格的记忆。
3. 项目内容记忆允许更新；工作区长期记忆默认只追加；用户级记忆允许演化，但不得混入项目私有事实。

### 6.4.1 升格链工程细节

升格链路径：`artifact -> memory seed -> 标签级记忆 -> 工作区长期记忆`

各节点的工程触发条件与落地位置：

| 升格节点 | 触发条件 | 工程位置 | 实施阶段 |
|---------|---------|---------|---------|
| artifact → memory seed | 已通过本轮验证 + 具有复用价值 | `chatStore` 中维护 memory seed 候选列表 | P2 |
| memory seed → 标签级记忆 | 轮次阈值（5 轮）或标签空闲 | `ai_commands.rs` 触发 `memory_generation_task_tab` | P0 |
| 项目内容记忆生成 | 文件保存成功 | `sync_workspace_file_cache_after_save` 钩子 | P0 |
| 标签级记忆 → 工作区长期记忆 | 标签被删除 | `on_tab_deleted` 后台任务 | P1 |
| 用户级记忆演化 | 跨项目行为模式稳定 | `user_memory.db` 写入任务 | P2 |

P0 阶段实现：标签级记忆生成（触发 + 写入）+ 项目内容记忆生成。
P1 阶段实现：标签→长期记忆升格。
P2 阶段实现：用户级记忆 + AI 工具主动写入。

### 6.4.2 失效链工程细节

| 失效场景 | 影响范围 | freshness_status 变化 | 工程位置 |
|---------|---------|----------------------|---------|
| 标签删除 | 该 tab_id 所有 tab 层记忆 | `fresh/stale` → `expired` | `on_tab_deleted()` |
| 文件重新保存 | 该 file_path 来源的 content 层记忆 | `fresh` → `superseded` | `upsert_project_content_memories()` 写入前 |
| 工作区切换 | 整个 workspace.db 不再使用 | 随 db 生命周期 | workspace 切换逻辑 |
| 7 天未更新 | 所有 `fresh` 超时记忆 | `fresh` → `stale` | P1 后台定期任务 |

失效记录的处理原则：
- `expired` 记忆：不参与自动检索，只在 MemoryTab UI 中可见（P1 实现展示）
- `superseded` 记忆：不参与自动检索，UI 不展示（内部保留供审计）
- `stale` 记忆：参与检索但排序降权（系数 0.5x），注入时标注”（信息可能已过时）”

## 七、读取、检索与注入协同

## 7.1 优先级规则

在 Agent 场景下，推荐统一为：

`用户显式引用 > 当前文档 / 选区 > 当前轮状态 / artifact > 标签级记忆 > 项目内容记忆 > 工作区长期记忆 > 用户级记忆 > 知识库 > 聊天历史摘要`

解释：

1. 当前轮状态 / artifact 优先于记忆库，因为它们代表当前最真实的任务现场。
2. 标签级记忆优先于其他记忆层，因为它最接近当前标签的连续语义。
3. 记忆库只做补强，不得压过当前轮现场状态。

## 7.2 检索与注入边界

1. 显式引用的记忆项视为用户主动指定，不参与自动裁剪逻辑的同层竞争。
2. 自动检索到的记忆不得覆盖当前文档事实或用户显式引用。
3. 记忆项注入时必须带来源与作用域标签。
4. 对话记忆与聊天历史摘要不是同一对象，不得重复注入。

## 7.3 `A-AST-M-P-01_上下文注入.md` 的承接边界

`A-AST-M-P-01_上下文注入.md` 负责：

1. 注入顺序
2. 预算分配
3. 裁剪与降级

本文负责：

1. 为什么当前轮 artifact 要先于记忆库
2. 为什么标签级记忆先于长期记忆
3. 哪些对象属于“可注入记忆”，哪些只是当前轮临时状态

对 Binder 而言：

## 十、设计评审否决条件

以下任一情况成立，都应直接否决：

1. 把当前轮所有 artifact 默认写入记忆库。
2. 把聊天历史直接等同于对话记忆。
3. 让自动检索记忆覆盖用户显式引用或当前文档事实。
4. 未经确认就把高主观性结果升格为长期记忆或用户级记忆。
5. 未明确作用于层次一或层次二，却把层次三记忆协同逻辑强行下压。
