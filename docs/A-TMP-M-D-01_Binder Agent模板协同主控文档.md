## 一、文档定位与控制权威

### 1.1 文档定位

本文是 Binder Agent 的模板协同控制文档。

本文回答：

1. 模板库在 Agent 体系中到底是什么。
2. 模板库唯一有效对象是什么。
3. 工作流模板如何与 Agent 协同。
4. 工作流模板如何被用户创建、选择与使用。
5. AI 可以如何建议与消费工作流模板，但不应越权到什么程度。

### 1.2 控制权威

本文是模板协同层的主控文档。

本文不替代：

1. `A-TMP-M-T-01_模板机制.md` 的对象模型与机制规则。
2. `A-AG-M-T-05_文档生成流程.md` 的文档生成主链。
3. `A-AG-M-D-01_Binder Agent能力描述文档.md` 与 `A-AG-M-T-04_Binder Agent技术主控文档.md` 的 Agent 上位语义与技术对象主控。

但凡涉及以下问题，必须以本文为准：

1. 模板是否属于约束资产。
2. 模板是否必须由用户主动选择。
3. 工作流模板是否只允许用户创建。
4. Agent 与工作流模板的协同边界是什么。

### 1.3 本文不负责什么

本文不负责：

1. Plan JSON 具体字段定义。
2. 模板管理界面与编辑器 UI 细节。
3. 生成流程中每一步的代码实现。

这些内容由 `A-TMP-M-T-01_模板机制.md` 和实现文档承接。文档生成流程由 `A-AG-M-T-05_文档生成流程.md` 负责。

## 二、模板库在 Binder Agent 中的本质

模板库不是知识库。
模板库也不是记忆库。

模板库的本质是：

**由用户主动选择、主动保存、主动治理的工作流模板资产系统。**

工作流模板在 Binder 中只承担一种职责：

**过程约束。**

它回答的问题只有一个：

**当前任务应该按什么执行过程推进。**

它不承担：

1. 事实来源。
2. 内容结论。
3. 输出结构定义。
4. 风格偏好定义。
5. 行为方法定义。

模板库与知识库的区别是：

1. 知识库提供内容与证据。
2. 模板库提供过程约束。

## 四、模板库边界

### 4.1 与知识库、记忆库、当前轮 artifact 的区别

| 对象 | 主要来源 | 主要管理者 | 主要作用 |
|---|---|---|---|
| 记忆库 | 系统提取 | 系统主导 | 语义延续与沉淀 |
| 知识库 | 用户导入 / 存入 | 用户主导 | 文档知识检索 |
| 模板库 | 用户创建 / 选择 / 保存 | 用户主导 | 过程约束 |
| 当前轮 artifact | 当前轮协作生成 | 系统 + AI + 用户确认 | 当前轮推进与收口 |

### 4.2 工作流模板的唯一职责

工作流模板是模板库唯一有效类型。

工作流模板只回答：

1. 任务过程怎么推进。
2. 步骤顺序怎么组织。
3. 哪些阶段需要先后依赖。

工作流模板不回答：

1. 产物应该长什么样。
2. 语言风格应该如何表达。
3. 某类任务应该采用什么个人方法。
4. 内容事实是否成立。
5. 最终质量是否成立。

### 4.3 不得误读的点

1. 模板不是知识内容本身。
2. 模板不是记忆沉淀的自动仓。
3. 模板不是 AI 可随时自行启用的隐藏规则层。
4. 工作流模板不是独立执行引擎。

## 五、工作流模板的创建、保存与治理原则

### 5.1 用户确认入库原则

默认情况下：

1. 用户可以主动创建、编辑、选择工作流模板。
2. 模板成为正式资产前，必须经过用户显式保存。
3. 模板库中的工作流模板属于用户显式定义资产。

### 5.2 非静默生成原则

默认禁止：

1. AI 自动从真实执行中生成工作流模板并入库。
2. AI 静默提出高影响模板并直接启用。
3. AI 用模板生成替代当前轮验证与确认。

### 5.3 模板作为用户资产的语义

模板偏用户侧的关键在于：

1. 用户对模板拥有选择权。
2. 用户对模板拥有编辑权。
3. 用户对模板是否进入长期资产有决定权。

这点与记忆库、知识库不同。

## 六、工作流模板的选择、引用与注入协同

### 6.1 用户显式选用优先

模板调用的默认优先级应为：

1. 用户显式选择 / `@` 引用 / 拖拽模板。
2. AI 提出可追溯的模板建议。
3. 用户接受 AI 建议后再注入模板。

#### 6.1.1 AI 自动建议的边界

AI 可以建议适合的工作流模板。

但默认不应：

1. 静默启用高影响模板。
2. 静默替换用户已显式选择的模板。

### 6.2 模板作为过程约束输入

工作流模板进入上下文时，应被理解为：

1. 步骤约束。
2. 顺序约束。
3. 依赖约束。

而不是：

1. 当前任务事实。
2. 证据来源。
3. 输出结构定义。
4. 风格偏好定义。
5. 质量已成立的证明。

#### 6.2.1 与知识内容的关系

如果一个任务既使用知识库也使用模板库：

1. 知识库提供内容 / 证据。
2. 模板库提供过程约束。

两者不能混为一层。

## 七、与结构文档的协同关系

### 7.1 与 `A-TMP-M-T-01_模板机制.md` / `A-AG-M-T-05_文档生成流程.md` 的关系

`A-TMP-M-T-01_模板机制.md` 负责：

1. 工作流模板对象模型。
2. 调用机制。
3. 结构解析、流程编译、执行约束与失败降级。

`A-AG-M-T-05_文档生成流程.md` 负责：

1. 工作流模板如何进入文档生成主链。
2. 工作流模板与上下文、工具、写入、导出的流程关系。

本文负责：

1. 模板为何是用户主导约束资产。
2. 显式选用与 AI 建议之间的边界。
3. 工作流模板与 Agent 协同的主控语义。

### 7.2 与层次边界的关系

当前默认边界：

1. 层次三：完整承接工作流模板协同。
2. 层次一、层次二：不默认下压工作流模板主链。
3. Chat Build（CBT 模块）：执行引擎独立，不接入本文的生成流程框架；但可在输入阶段引用模板库资源作为约束输入。被 Chat Build 引用的工作流模板必须具备高可执行性与高兼容性，以保障主控链路能稳定解析并映射为 Build Outline 步骤。

### 8.3 关键判断

模板系统真正负责的是：

1. 让用户把“我希望按这个过程推进”变成稳定资产。
2. 让 AI 在被授权时稳定按该过程推进。

它不负责：

1. 替用户做事实判断。
2. 替系统做状态判断。
3. 替知识库提供内容。

## 九、设计评审否决条件

以下任一情况成立，都应直接否决：

1. AI 自动从真实执行中生成并保存高影响工作流模板。
2. AI 静默启用用户未选择的高影响模板。
3. 把模板内容当作事实来源或证据来源。
4. 未单独决策就把工作流模板主链下压到层次一或层次二。

## 十、MVP 验收口径

1. 模板库仅包含工作流模板，且语义稳定为过程约束。
2. 用户显式选用模板与 AI 自动建议模板的边界明确。
3. 工作流模板仅由用户创建与保存，不存在执行过程自动入库链路。
4. `R-AG-C-D-01_Binder-Agent指导方案（Guiding Architecture & Design Doctrine）.md` 中落在模板系统的规则已有 ``A-TMP-M-D-01_Binder Agent模板协同主控文档.md`` / ``A-TMP-M-T-01_模板机制.md`` 的明确承接位；文档生成流程规则由 `A-AG-M-T-05_文档生成流程.md` 承接。
