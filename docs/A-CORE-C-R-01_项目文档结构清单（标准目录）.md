## 1. 文档编码规则

### 1.1 文件名格式

统一格式：

`[状态]-[模块]-[大类]-[类型]-[序号]`

示例：

1. `A-CORE-C-D-01`
2. `A-WS-M-T-01`
3. `R-AG-M-R-13`
4. `X-AG-M-R-20`

说明：

1. 文件名由**状态字段 + 结构编码**两部分组成。
2. 结构编码用于表达文档在体系中的位置。
3. 状态字段用于表达文档当前是否进入主链阅读。
4. 当前主结构文档、参考文档和归档文档已按该格式进入真实文件名。
5. 后续新增文档时，应先确定状态字段，再分配结构编码。
6. 旧数字命名仅代表历史阶段，不再作为新增文档命名方式。
7. Chat Build 当前主线文档进入 `docs/` 根目录的正式 Active 序列，文件名采用 `A-CBT-*`。
8. `docs/README.md`、`docs/active/README.md`、`docs/reference/README.md` 属于目录级状态索引文档，不参与结构编码，但属于当前必须维护的入口文档。

### 1.2 状态字段

分类逻辑说明：

`1.2` 回答的是：

**这份文档当前在项目中的生效状态是什么。**

也就是说，这里分的不是模块、层级和职责，而是“当前要不要进入主阅读链”。

判断方法：

1. 当前生效、属于主结构链路的文档，使用 `A`。
2. 仍有阅读价值、但不属于主结构主链的文档，使用 `R`。
3. 已被替换、只保留历史回看价值的文档，使用 `X`。

| 编码 | 状态 | 含义 |
|---|---|---|
| `A` | Active | 当前生效，进入主链阅读 |
| `R` | Reference | 参考保留，不属于主链，但仍可能需要阅读 |
| `X` | Replaced | 已被替换，默认不进入当前开发阅读链 |

### 1.3 四大类编码

分类逻辑说明：

`1.2` 回答的是：

**这份文档在整个文档治理体系里，处于哪一个层级。**

也就是说，这里分的不是“主题模块”，而是“治理层级”。

例如：

1. `C` 关注总纲、原则、基础模型，属于上位约束层。
2. `M` 关注某个模块本身，属于模块主控层。
3. `I` 关注跨模块契约，属于接口层。
4. `X` 关注落地、验证、运维，属于执行层。

判断方法：

1. 如果一份文档主要定义系统级原则和总边界，应归 `C`。
2. 如果一份文档主要定义某个模块自身，应归 `M`。
3. 如果一份文档主要定义模块之间如何对接，应归 `I`。
4. 如果一份文档主要定义如何开发、验证、运维，应归 `X`。

| 编码 | 类别 | 含义 |
|---|---|---|
| `C` | Constitution | 宪法级 / 总纲级文档 |
| `M` | Module | 模块主控与模块技术文档 |
| `I` | Interface | 跨模块接口 / 协议 / 数据契约 |
| `X` | Execution | 执行、落地、验证、运维、审计 |

### 1.4 模块编码

分类逻辑说明：

`1.4` 回答的是：

**这份文档主要属于哪个业务/系统模块。**

也就是说，这里分的不是层级，也不是文档职责，而是“主题归属”。

例如：

1. `WS` 说明它主要属于 Workspace 模块。
2. `AG` 说明它主要属于 Agent 模块。
3. `ED` 说明它主要属于 Editor 模块。

判断方法：

1. 如果一份文档的主要内容围绕某一个模块展开，就放到该模块。
2. 如果一份文档同时服务多个模块，不应强行塞入某一个模块，优先考虑进入 `1.2` 的 `I` 类。
3. 模块编码的合理性，决定了模块化开发的边界是否稳定。

注意：

1. `1.4` 只应该放真正的模块。
2. 研究属性、旧体系属性、历史属性，本质上不是模块，只是文档属性。
3. 因此像 `RSH`、`LEG` 这类编码，后续更适合独立为“文档属性标签”，而不是继续放在模块位。

| 编码 | 模块 | 含义 |
|---|---|---|
| `CORE` | Core | 产品定义、术语、原则、基础模型 |
| `PROD` | Product | 场景、任务、交互、路线图 |
| `SYS` | System | 系统级架构与全局流转 |
| `WS` | Workspace | Workspace、工作台、文件、资源、跨文档 |
| `DE` | Dialog Edit | 对话编辑系统 |
| `ED` | Editor | 编辑器系统 |
| `AG` | Agent | AI / Agent / Prompt / 工具 / 任务闭环 |
| `AST` | Asset | 记忆、知识库、上下文资产 |
| `TMP` | Template | 模板库系统（工作流模板资产） |
| `CBT` | Chat Build | 当前唯一生效的项目级构建主线 |
| `BLD` | Build Mode Legacy | 旧 BLD / Discussion Build / Multi-Actor Build 参考体系 |
| `ENG` | Engineering | 工程实现层 |
| `VAL` | Validation | 验证、测试、运维、审计 |

### 1.5 文档类型编码

分类逻辑说明：

`1.5` 回答的是：

**这份文档在所属层级和所属模块中，承担什么职责。**

也就是说，这里分的不是治理层级，也不是模块主题，而是“文档功能角色”。

例如：

1. `D` 用来定义“做什么、边界是什么”。
2. `T` 用来定义“技术上怎么承接”。
3. `P` 用来定义“接口和协议怎么对接”。
4. `L` 用来定义“如何落地”。

判断方法：

1. 如果文档主要讲功能边界、模块定义、规则边界，应归 `D`。
2. 如果主要讲技术模型、架构、状态机、对象关系，应归 `T`。
3. 如果主要讲接口、协议、Schema、数据契约，应归 `P`。
4. 如果主要讲计划、排期、迁移、开发承接，应归 `L`。
5. 如果主要讲测试、验收、排障、限制，应归 `V`。
6. 如果主要讲索引、参考、研究、分析、清单，应归 `R`。

三层关系可以这样理解：

1. `1.2` 决定它当前是否进入主链阅读。
2. `1.4` 决定它主要讲哪个模块。
3. `1.3` 决定它在该模块中处于哪一层。
4. `1.5` 决定它在该模块和该层里扮演什么角色。

| 编码 | 类型 | 含义 |
|---|---|---|
| `D` | Description | 描述主控 / 功能主控 / 规则主控 |
| `T` | Technical | 技术主控 / 架构 / 模型 / 机制 |
| `P` | Protocol | 协议 / 接口 / Schema / 契约 |
| `R` | Reference | 索引 / 参考 / 研究 / 清单 |
| `L` | Launch | 落地计划 / 开发计划 / 迁移计划 |
| `V` | Validation | 测试 / 验证 / 运维 / 限制 |

### 1.6 文档属性标签（非编码）

以下标签不是结构编码的一部分，只用于补充说明文档当前属性：

1. `主结构`
2. `参考`
3. `旧体系参考`
4. `指导参考`
5. `归档参考`

说明：

1. 结构编码负责表达层级、模块、职责。
2. 属性标签负责表达文档当前在项目中的使用身份。
3. 研究属性、旧体系属性、历史属性不再占用模块位。

### 1.7 使用约束

1. 一份文档只能有一个主结构编码。
2. 一份文档只能有一个状态字段。
3. 同一模块内，原则上最多只有一份 `D` 主控文档和一份总 `T` 主控文档。
4. `P/R/L/V` 只能承接，不应重写 `D/T` 的主源规则。
5. 历史研究文档、旧体系文档可以保留，但不得与主结构位混写。
6. `X` 类文档只保留历史回看价值，不进入当前开发主阅读链。

### 1.8 当前展示规则

过渡期统一写法：

`[状态字段][结构编码][属性标签] 现文件名`

例如：

1. `[A][CORE-C-D-01][主结构] A-CORE-C-D-01_产品定义总纲.md`
2. `[A][WS-M-T-01][主结构] A-WS-M-T-01_workspace架构.md`
3. `[R][AG-M-R-13][旧体系参考] R-AG-M-R-13_Binder AI 功能方案整合版.md`
4. `[X][AG-M-R-20][归档参考] X-AG-M-R-20_AI功能文档清单.md`

## 6. 兼容型统一规则体系（试行）

### 6.1 目的

本节用于定义一套**面向后续收敛的通用规则体系**。  
目标不是立刻替换现有 `BA / DE / AX / PR / TC / TP`，而是：

1. 给后续新增模块和新增规则提供统一编码入口。
2. 避免继续扩张新的命名空间前缀。
3. 在不破坏当前文档映射的前提下，为未来统一迁移预留标准。

### 6.2 兼容原则

1. 当前已落地的 `BA-* / DE-* / AX-* / PR-* / TC-* / TP-*` 全部继续有效。
2. 现有文档中的规则 ID 不要求立即替换。
3. 后续如果继续新增规则，优先按本节的统一规则体系设计。
4. 若某条旧规则后续需要进入统一体系，应采用“旧规则保留 + 新规则映射”的方式渐进迁移。
5. 在未完成专项迁移前，旧规则仍是对应文档中的有效引用标识。

### 6.3 统一规则格式

统一格式：

`BR-[模块]-[规则域]-[序号]`

示例：

1. `BR-AG-CORE-001`
2. `BR-AG-STATE-001`
3. `BR-DE-DIFF-001`
4. `BR-WS-RESOURCE-001`

说明：

1. `BR` 表示 Binder Rule，为统一规则前缀。
2. 第二段表示规则所属模块。
3. 第三段表示规则域。
4. 第四段表示该规则在该模块和规则域中的顺序号。

### 6.4 统一规则模块编码

| 编码 | 模块 | 含义 |
|---|---|---|
| `CORE` | Core | 总原则、总边界、总治理 |
| `PROD` | Product | 产品、场景、交互、路线图 |
| `SYS` | System | 系统级架构、状态、数据流 |
| `WS` | Workspace | 工作台、文件、资源、跨文档 |
| `DE` | Dialog Edit | 对话编辑系统 |
| `ED` | Editor | 编辑器与分页系统 |
| `AG` | Agent | Agent / Prompt / Tool / Task 主链 |
| `AST` | Asset | 记忆、知识库、上下文资产 |
| `TMP` | Template | 模板库系统（工作流模板资产） |
| `CBT` | Chat Build | 当前唯一生效的项目级构建主线 |
| `BLD` | Legacy Build System | 历史构建体系（仅作为参考层存在，不属于当前生效主线） |
| `ENG` | Engineering | 工程实现与基础设施 |
| `VAL` | Validation | 验证、测试、运维、审计 |

### 6.5 统一规则域建议

| 规则域 | 含义 |
|---|---|
| `CORE` | 根定义、目标、边界 |
| `MODEL` | 模型、对象、职责分层 |
| `STATE` | 状态语义、阶段闭合、生命周期 |
| `VERIFY` | 验证、确认、验收 |
| `RUN` | 运行模型、gate、编排、调度 |
| `PROMPT` | 提示词主链与上下文构建 |
| `TOOL` | 工具契约、调用、执行、结果回流 |
| `TASK` | 任务规划、任务收口、阶段推进 |
| `SCENE` | 能力场景与适用边界 |
| `ASSET` | 中间态、记忆、知识、模板、资产沉淀 |
| `DATA` | 数据结构、字段、Schema、协议 |
| `GOV` | 文档协同、门禁、变更治理 |
| `OBS` | 可观测性、错误、恢复、审计 |

### 6.6 与当前规则体系的兼容映射

当前规则体系与统一规则体系的兼容关系如下：

| 当前命名空间 | 当前作用 | 统一规则建议落点 |
|---|---|---|
| `BA-*` | Binder Agent 上位规则 | `BR-AG-*` |
| `DE-*` | 对话编辑专项规则 | `BR-DE-*` |
| `AX-*` | `A-AG-M-T-01_ai执行架构.md` 执行架构本地规则 | `BR-AG-RUN-*` 或 `BR-SYS-RUN-*` |
| `PR-*` | `A-AG-M-T-02_prompt架构.md` 提示词架构本地规则 | `BR-AG-PROMPT-*` |
| `TC-*` | `A-AG-M-P-01_工具调用体系.md` 工具调用本地规则 | `BR-AG-TOOL-*` |
| `TP-*` | `A-AG-M-T-03_任务规划执行.md` 任务规划本地规则 | `BR-AG-TASK-*` |
| `CBT` 主线规则 | `A-CBT-*` 主线中的 Chat Build 规则源头 | `BR-CBT-*` |

说明：

1. `AX / PR / TC / TP` 当前仍保留为文档局部命名空间。
2. 未来若迁移到统一体系，应优先落到 `AG` 模块下对应规则域。
3. 若某条规则本质上属于更高层的系统规则，也允许落到 `BR-SYS-*` 或 `BR-CORE-*`。

### 6.7 迁移策略

为保证最大兼容度，统一规则体系采用以下迁移策略：

1. **现状保持**：当前规则继续用原 ID，不做全量替换。
2. **新增优先**：后续新增通用规则，优先使用 `BR-*`。
3. **映射过渡**：旧规则若需要统一，只新增映射，不强制回改全仓。
4. **分模块迁移**：仅在某模块发生规则体系重构时，才对该模块执行专项迁移。
5. **主链优先**：若未来迁移，应先迁 `A-*` 主结构文档，再迁 `R-* / X-*` 文档。

### 6.8 当前结论

当前项目规则体系采用“双轨制”：

1. **现行规则轨**：`BA / DE / AX / PR / TC / TP`
2. **统一收敛轨**：`BR-[模块]-[规则域]-[序号]`

在未启动专项迁移前：

1. 现行规则继续作为当前文档主引用。
2. 统一规则体系作为后续扩展与收敛标准。
3. 禁止继续新增新的散乱命名空间前缀。
