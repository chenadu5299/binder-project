## 一、文档定位

本文定义构建模式路径一：直接构建（Direct Build）的完整功能设计。

本文覆盖：入口设计、输入结构、执行流程、各阶段边界、产出规范。

本文不覆盖：主控 AI 内部编排机制（见 `R-BLD-M-T-04`）、Project Object 数据结构（见 `R-BLD-M-T-03`）。

---

## 二、路径定位

直接构建是构建模式的**主路径**，面向单人高效场景。

使用前提：

1. 用户目标明确，无需多方协同决策。
2. 用户希望快速进入执行，不需要讨论环节。
3. 项目结构、内容范围由主控 AI 自动规划。

## 三、入口设计

### 3.1 入口位置

> 待定：构建模式在 UI 层的具体入口位置（独立 Tab？侧边栏入口？命令面板？）

### 3.2 触发方式

### 3.3 入口前置条件

1. 用户已打开一个 Workspace。
2. 无其他进行中的构建任务（同一 Workspace 不允许并发构建）。

## 四、输入结构

直接构建的输入分为三类：

### 4.1 描述（必填）

用户对目标项目的自然语言描述。

- 用途：主控 AI 解析需求，理解项目目标。
- 形式：自由文本输入框。
- 约束：不限字数，但建议引导用户描述"做什么"而非"怎么做"。

### 4.2 需求（选填）

用户对构建过程的约束和细节要求。

- 用途：约束主控 AI 的执行边界（格式要求、内容范围、禁止事项等）。
- 形式：自由文本输入框。

> 待定：需求是否支持结构化标签（如：语言/格式/字数/范围）？

### 4.3 引用（选填）

用户提供的上下文资源，用于辅助构建。

支持引用类型：

| 引用类型 | 来源 | 用途 |
|---|---|---|
| 文档 | 当前 Workspace 文件 | 提供内容参考或约束 |
| 数据 | 当前 Workspace 文件 | 提供结构化输入 |
| 资源 | 当前 Workspace 文件 | 图片/附件等 |
| 模板 | 模板库（TMP） | 提供工作流模板过程约束 |
| 工作流 | 模板库（TMP） | 提供 SOP 约束 |

> 待定：引用的具体交互形式（@mention？文件选择器？）

## 五、执行流程

```
[输入阶段]
用户填写描述 + 需求 + 引用
→ 点击「开始构建」

[规划阶段]
主控 AI 解析输入
→ 生成 Build Outline（构建大纲）
→ 展示大纲给用户确认

[确认节点]（可选）
用户查看构建大纲
→ 选择「确认执行」或「修改后执行」
→ 修改仅限于此阶段，进入执行后不可修改

[执行阶段]
主控 AI 按 Build Outline 分步执行
→ 支持后台运行（用户可切走）
→ 执行期间不接受新输入
→ 实时展示执行进度

[产出阶段]
主控 AI 完成所有步骤
→ 生成 Project Object
→ 写入 Workspace
→ 通知用户构建完成
```

### 5.1 规划阶段说明

主控 AI 在规划阶段需要完成：

1. 解析用户描述与需求。
2. 确定项目结构（主文档/子文档/资源组织方式）。
3. 确定生成顺序与依赖关系。
4. 生成 Build Outline，以可读形式呈现给用户。

Build Outline 格式定义见 `R-BLD-M-T-04_构建执行引擎.md`。

### 5.2 执行阶段说明

1. 主控 AI 按 Build Outline 顺序执行每个步骤。
2. 每个步骤完成后，更新进度状态。
3. 执行阶段不接受用户修改输入，不接受新引用。
4. 执行失败处理机制见 `R-BLD-M-T-04_构建执行引擎.md`。

### 5.3 后台执行

1. 执行阶段支持后台运行：用户可切换到编辑模式继续工作。
2. 后台运行时，状态栏/通知区域展示构建进度。
3. 构建完成后，推送通知，用户可切回查看结果。

> 待定：后台执行的具体 UI 通知形式。

## 六、确认节点设计

### 6.1 是否强制确认

> 待定：规划阶段的大纲确认是否强制？（方案 A：强制查看后才能执行；方案 B：可跳过直接执行）

### 6.2 确认后的修改范围

如果用户在确认节点选择修改：

1. 用户只能修改 Build Outline 的内容，不能修改底层执行机制。
2. 修改后重新进入规划阶段（主控 AI 重新生成大纲）。
3. 不允许在执行阶段中途修改。

## 七、产出规范

### 7.1 产出单元

直接构建的产出是一个 **Project Object**。

Project Object 的数据结构定义见 `R-BLD-M-T-03_项目实体模型.md`。

### 7.2 写入位置

Project Object 写入当前 Workspace，写入规则见 `R-BLD-I-P-01_构建模式与Workspace接口.md`。

### 7.3 构建完成后

1. Workspace 文件树中出现新生成的项目结构。
2. 用户可在编辑模式下继续编辑任意生成文档。
3. 构建结果不产生 Diff，直接为最终文件，无需用户决策。

---

## 八、边界约束

1. **不产生 Diff**：直接构建的产出直接写入文件，不经过 Diff 决策链路。
2. **不操作编辑器**：执行阶段不向 TipTap 编辑器写入内容，只操作 Workspace 文件系统。
3. **不共享 AG 执行链**：主控 AI 的实现独立于编辑模式的 Agent，不调用 `ai_chat_stream` 等编辑模式命令。
4. **单任务约束**：同一 Workspace 同一时刻只能有一个构建任务在执行。
