## 二、整体结构

### 2.1 模式层级

```
Build Mode（构建模式）
│
├── Direct Build（直接构建）              ← 主路径
│   ├── 输入阶段（描述 + 需求 + 引用）
│   ├── 规划阶段（主控 AI 生成 Build Outline）
│   ├── 执行阶段（主控 AI 调度工具执行，不可打断）
│   └── 产出阶段（Project Object 写入 Workspace）
│
└── Discussion Build（讨论构建）          ← 扩展路径
    ├── Open Discussion（开放讨论组）
    └── Internal Discussion（内部讨论组）
        ├── 讨论阶段（角色 AI + 真人 + 主控 AI 收束）
        ├── 确认阶段（结构化输出 + 发起人决策）
        └── 构建阶段（同 Direct Build 执行链）
```

### 2.2 AI 层级

```
Build Mode AI 体系
│
├── Master AI（主控 AI）                  ← 隐式，用户不直接感知
│   ├── 任务分发（给角色 AI）
│   ├── 触发判断（何时进入构建阶段）
│   ├── 构建大纲生成
│   └── 构建执行编排
│
└── Role AI（角色 AI）                    ← 用户可配置，可多个
    ├── 专属领域分析与建议
    ├── 资源分享与引用
    └── 不参与最终生产（只提供输入，不执行）
```

## 三、路径分工

### 3.1 直接构建（Direct Build）

**适用场景：** 用户目标明确，需求清晰，单人操作。

**核心特征：**

1. 用户提供输入后，主控 AI 直接生成执行计划并运行。
2. 执行阶段不需要用户介入，也不可中断（可后台运行）。
3. 无讨论环节，无角色 AI 参与，只有主控 AI 与用户输入。
4. 模板库资源可在输入阶段引用，作为构建约束。

**详细设计：** 见 `A-BLD-M-D-02_直接构建功能设计.md`

### 3.2 讨论构建（Discussion Build）

**适用场景：** 需要多方协同决策，目标需要讨论收束后才能执行。

**核心特征：**

1. 用户创建讨论组，邀请真人参与者和 AI 角色加入。
2. 主控 AI 在后台持续收束讨论结构（共识/分歧/方案）。
3. 讨论经历三态状态机，已结束状态不可逆。
4. 发起人在任何阶段均可触发构建，不依赖所有参与者确认。
5. 讨论记录与结构化结论写入 Workspace，作为项目资产。

**详细设计：** 见 `A-BLD-M-D-03_讨论构建功能设计.md`

## 四、与其他模块的分工

### 4.1 与编辑模式（DE/AG 模块）的关系

| 边界维度 | 编辑模式 | 构建模式 |
|---|---|---|
| AI 执行链 | AG 模块三层 AI 独立实现 | BLD 模块独立实现，不复用 AG |
| 编辑器状态 | 强耦合 TipTap 编辑器 | 构建期不操作编辑器 |
| Diff 机制 | 核心机制，用户决策 | 不使用 Diff，直接写入文件 |
| 会话历史 | 聊天 Tab 独立管理 | 讨论组独立管理，不进入聊天 Tab |

**禁止：** 构建模式的实现不允许直接调用编辑模式的 AI 服务、编辑器 API 或 Diff 机制。

### 4.2 与 Workspace（WS 模块）的关系

构建模式的产出（Project Object）最终写入 Workspace 文件系统。
接口契约见 `A-BLD-I-P-01_构建模式与Workspace接口.md`。

### 4.3 与模板库（TMP 模块）的关系

构建模式可在输入阶段引用模板库资源（工作流模板）。
模板库只提供过程约束，不参与构建执行。
被引用的工作流模板在进入构建主链前，应先完成结构解析与流程编译。
构建主链消费的是模板编译后的结构化流程表示，而不是原始模板文本。
接口契约见 `A-BLD-I-P-02_构建模式与模板库接口.md`。

### 4.4 与记忆/知识库（AST 模块）的关系

讨论组作为聊天标签，走标签记忆流程规范（`A-AST-M-T-01_记忆模型.md`）。
知识库（用户主动生产）不与构建模式直接交互。

## 五、核心设计原则落地

以下原则来自 `BLD-C-D-01` 第五节，本节说明其在模块内如何具体落地。

### 5.1 结果导向

- 构建模式入口不提供"随意聊天"功能。
- 直接构建：输入表单结构化，强制用户提供目标描述。
- 讨论构建：每个讨论组必须绑定一个"最终构建"的预期目标。

### 5.2 阶段驱动

- 直接构建分为：输入 → 规划 → 执行 → 产出，四个明确阶段。
- 讨论构建三态状态机（IN_PROGRESS / WAITING_CONFIRMATION / FINISHED）强制阶段流转。

### 5.3 决策显性化

- 讨论构建中，主控 AI 在每次收束时输出结构化摘要（共识/分歧/方案）。
- 确认行为由真人参与者主动操作，系统记录每个确认人的时间戳。
- 发起人触发构建时，系统记录当时的确认状态快照。

### 5.4 不可逆节点

- 讨论组进入 FINISHED 状态后，对话锁定，不允许追加消息或修改内容。
- 如需继续讨论，系统引导用户"新建讨论组"，不允许解锁原讨论组。

### 5.5 执行不可打断

- 主控 AI 开始执行构建计划后，用户输入被屏蔽。
- 允许"后台执行"：用户可切换到编辑模式继续工作，构建在后台进行。
- 执行完成后通知用户，Project Object 出现在 Workspace 中。
