## 一、文档定位与控制权威

### 1.1 文档定位

本文是 Binder 的 **Workspace 工作台协同主控文档**。

本文回答以下问题：

1. Binder 的 `workspace 工作台` 到底是什么，不是什么。
2. 顶部工具栏、主内容区、底部状态栏以及资源分区如何形成统一工作台语义。
3. 工作台中的工作区、记忆库、知识库、模板库、时间轴如何分工，而不是互相吞并。
4. AI 与工作台如何协同，哪些入口属于显式引用，哪些只是资源浏览，哪些不应直接触发执行。
5. 旧体系中散落在多个 AI 文档里的 workspace / 工作台规则，在新结构文档中分别由谁承接。

### 1.2 控制权威

本文是 workspace 工作台层的主控文档。

本文不替代：

1. `A-WS-M-T-03_文件系统.md` 对文件主链、缓存、一致性的控制。
2. `A-WS-M-T-04_资源管理.md` 对资源对象、资源操作与资源状态的控制。
3. `A-WS-M-T-05_跨文档操作.md` 对跨文档引用、目标解析、任务回流的控制。
4. ``A-AST-M-P-01_上下文注入.md`/`A-AST-M-D-01_Binder Agent记忆协同主控文档.md`/`A-AST-M-D-02_Binder Agent知识库协同主控文档.md`/`A-TMP-M-D-01_Binder Agent模板协同主控文档.md`` 对上下文、记忆、知识库、模板库语义的控制。

但凡涉及以下问题，必须以本文为准：

1. Workspace 工作台的边界与分区语义。
2. 工作台中的资源入口如何与 AI 协作链对接。
3. 工作台级搜索、显式引用、状态暴露和资源分区的协同规则。
4. 旧体系中“资源管理区 + AI 交互入口 + workspace 上下文锚点”在新结构中的统一承接关系。

### 1.3 本文不负责什么

本文不负责：

1. 文件监听实现细节。
2. 索引库、向量库、检索算法实现细节。
3. 某个按钮、面板、卡片的最终 UI 稿。
4. 某个具体工具调用或后端命令实现。

这些内容由 ``A-WS-M-T-03_文件系统.md`/`A-WS-M-T-04_资源管理.md`/`A-WS-M-T-05_跨文档操作.md`/`A-AST-M-P-01_上下文注入.md`/`A-AST-M-D-01_Binder Agent记忆协同主控文档.md`/`A-AST-M-D-02_Binder Agent知识库协同主控文档.md`/`A-AST-M-T-02_知识库机制.md`/`A-TMP-M-T-01_模板机制.md`/`A-TMP-M-D-01_Binder Agent模板协同主控文档.md`/`A-AG-M-T-05_文档生成流程.md`` 和工程文档承接。

## 三、Workspace 工作台的根定义

### 3.1 根定义

Binder 的 workspace 工作台不是单纯的文件树，也不是 AI 的执行面板。

它的本质是：

**围绕当前 workspace 组织文件、上下文资产、用户资源入口和执行状态暴露的统一协作工作台。**

进一步说，workspace 工作台在 Binder 中应被视为：

**Binder 的工作基础框架。**

其他功能进入 Binder 后，通常不是各自再造一套独立工作面，而是：

1. 基于工作台框架获取资源入口。
2. 基于工作台框架完成任务分发。
3. 基于工作台框架完成中间态与结果呈现。
4. 基于工作台框架回到受控保存、受控确认、受控沉淀链。

### 3.2 工作台回答的问题

Workspace 工作台主要回答：

1. 当前用户在什么 workspace 中工作。
2. 当前有哪些可浏览、可选择、可引用、可管理的资源入口。
3. 哪些资源进入当前任务，哪些只是工作台中的可见资产。
4. AI 需要读取、引用、写回工作区内容时，如何从工作台入口进入受控主链。

### 3.3 工作台不回答的问题

Workspace 工作台不直接回答：

1. 某个文件如何保存与回写。
2. 某个知识条目如何分块和索引。
3. 某个模板如何解析为生成流程。
4. 某次跨文档任务如何具体拆步执行。

这些都应继续由专项文档负责。

### 3.4 工作台作为基础框架的边界判断

把 workspace 工作台视为基础框架，不等于把一切逻辑都塞进工作台。

正确关系应是：

1. 工作台定义工作入口、作用域、资源可见性和协作承载面。
2. 专项系统定义各自的执行语义、验证语义和对象规则。
3. 专项系统的结果应回到工作台框架中呈现、确认和回流。

因此，不符合该定位的典型误区包括：

1. 把工作台简化成单纯文件树侧栏。
2. 让某个 AI 功能绕开工作台，直接自带一套平行资源入口和状态面板。
3. 把工作台误写成资源展示层，而不承认其任务分发与结果承载职责。

## 五、工作台区域模型

## 5.1 三段式区域模型

Workspace 工作台至少由三段式区域构成：

1. **顶部工具栏**：提供创建、搜索、导入、更多操作等入口。
2. **主内容区**：承载各类 workspace 资源分区与列表。
3. **底部状态栏**：暴露当前 workspace 路径、全局状态、关键提示。

这三段共同构成工作台对象，不能被理解为彼此无关的 UI 碎片。

### 5.1.1 顶部工具栏的作用域规则

顶部工具栏不是“全局万能操作栏”，而是工作台入口栏。

其最小语义应冻结为：

1. **新建文件夹 / 新建文件**
   - 默认作用于工作区分区。
   - 在无当前 workspace 时不得静默成功，必须暴露缺失工作区状态。
2. **搜索**
   - 搜索的是当前激活分区或当前显式指定分区。
   - 不能把“搜索哪个分区”偷换成“AI 自动把哪个分区全量注入上下文”。
3. **云存储 / 更多操作**
   - 属于工作台扩展入口。
   - 在未实现阶段只保留占位语义，不得伪装为已成立的主链能力。

### 5.1.2 主内容区的分区展开规则

主内容区默认应满足：

1. 工作区分区是默认主展开分区。
2. 记忆库、知识库、模板库、时间轴可折叠，不要求一次只展开一个。
3. 分区是否展开，解决的是“用户当前看什么”，而不是“系统当前默认注入什么”。

### 5.1.3 底部状态栏的语义

底部状态栏在工作台层至少承担：

1. 暴露当前 workspace 路径或当前 workspace 身份。
2. 暴露关键系统状态与提示。
3. 作为工作台级全局状态的只读反馈层。

它不应承担：

1. 文件主链状态机定义。
2. AI 阶段状态定义。
3. 资源分区语义定义。

## 5.2 五类资源入口分区

在新结构下，workspace 工作台应统一承接以下五类入口：

1. **工作区**：文件树、文件夹、文档资源。
2. **记忆库**：标签 / 项目 / 工作区 / 用户记忆入口。
3. **知识库**：用户导入、存入、管理的知识资产入口。
4. **模板库**：工作流模板入口。
5. **时间轴**：工作台级项目逻辑状态演进入口。

说明：

1. 旧体系中的“工作区 / 记忆库 / 历史记录”三分区仍成立，但不再足以覆盖新结构。
2. 知识库与模板库既然已进入主结构，就应在工作台层获得明确入口语义，而不是继续悬空。
3. 时间轴节点的生成应基于成立点收口后的实际状态变化，而不是以成立入口触发次数作为展示依据。
4. 模板库入口只承接工作流模板；结构型资产入口属于知识库，不得在工作台层混层展示。

## 5.3 分区语义边界

工作台分区解决的是“资源从哪里进入、用户如何选择与管理”。

它不等于：

1. 上下文注入优先级。
2. Agent 状态机。
3. 验证体系分层。

因此：

1. **工作台里可见**，不等于 **当前轮必注入**。
2. **用户点击进入某分区**，不等于 **AI 自动把该分区内容当上下文**。
3. **资源在工作台可管理**，不等于 **AI 有权自动修改该资源**。

### 5.3.1 当前激活分区的作用

当前激活分区只应直接影响：

1. 当前工作台搜索范围。
2. 当前资源操作面板和快捷入口。
3. 当前用户视觉焦点。

当前激活分区默认**不应直接决定**：

1. AI 任务目标文档。
2. 自动注入的上下文全集。
3. 高影响写入操作的默认目标。

## 六、工作台与 AI 的协同规则

## 6.1 显式引用与自动检索必须区分

从工作台进入 AI 的资源，至少分为两类：

1. **显式引用**
   - 用户通过 `@`、拖拽、点击插入、明确选择条目进入当前任务。
   - 这类资源应进入“用户显式引用层”。
2. **自动检索 / 自动补强**
   - AI 基于当前任务，在工作区、记忆、知识库等范围内自动补强。
   - 这类资源进入自动注入层，不得伪装成用户显式指定。

## 6.2 工作台不做意图判断

工作台提供的是：

1. 资源入口
2. 作用域锚点
3. 搜索与选择
4. 状态暴露

工作台不负责：

1. 判断用户究竟想改哪一份文档
2. 判断某份资源是不是最终操作目标
3. 决定 AI 是否应该执行高影响操作

这些判断应继续由 ``A-WS-M-T-05_跨文档操作.md``、``A-AG-M-T-03_任务规划执行.md``、``A-DE-M-D-01_对话编辑统一方案.md`~`A-DE-X-L-01_对话编辑落地开发计划.md``、``A-AG-M-D-01_Binder Agent能力描述文档.md`/`A-AG-M-T-04_Binder Agent技术主控文档.md`` 承接。

## 6.3 workspace 边界与隔离

工作台层必须继续承接旧体系的以下约束：

1. 所有资源默认按 `workspace_path` 隔离。
2. 当前工作台中的资源搜索、引用、检索与写回，都应以当前 workspace 为边界。
3. 跨 workspace 的资源切换必须显式发生，不允许静默串用旧状态。

## 6.4 “应用到工作区文档”类入口的正确语义

旧体系中存在“将 AI 回复内容应用到工作区文档”的工作台式入口。

在新结构里，它必须被改写为：

1. 进入目标文档解析链。
2. 进入作用范围与写入方式确认链。
3. 进入文件系统 / 对话编辑 / diff / 保存主链。

它**不能**被实现成：

1. 工作台直接改文件内容。
2. 绕过确认与验证直接写盘。
3. UI 快捷入口直接替代受控编辑链。

### 7.1.1 工作分发与结果呈现的归口

在 Binder 中，很多能力最终都表现为：

1. 从工作台进入某类资源或任务。
2. 在专项系统中执行具体逻辑。
3. 再把中间态、结果、状态和后续入口回到工作台框架内呈现。

因此：

1. 工作台不是所有能力的执行内核，但它是多数能力的统一承载面。
2. “其他功能基于工作台框架做工作分发和结果呈现”应作为结构口径长期成立。
3. 若后续出现新能力，需要先判断它是工作台入口扩展、专项执行系统，还是两者协同，而不是直接并列再开一套工作面。

| `A-AST-M-D-02_Binder Agent知识库协同主控文档.md` | 定义知识库入口的用户主导与注入边界 |

工作台层只负责：

1. 让这些资产在 workspace 内有清晰入口。
2. 让用户明确知道自己是在“浏览 / 选择 / 引用 / 管理”什么。
3. 保证入口语义不与执行链、注入链、存储链混淆。

### 8.2 当前缺口的收口结论

本轮审查后的结论是：

1. 旧体系的 workspace 工作台规则在新结构中**不是没有承接**，而是**承接链过于分散**。
2. 新体系此前缺少一个专门说明“工作台级协同语义”的文档。
3. ``A-WS-M-D-01_workspace工作台协同主控文档.md`` 的职责就是把这些散落在 ``A-WS-M-T-03_文件系统.md`/`A-WS-M-T-04_资源管理.md`/`A-WS-M-T-05_跨文档操作.md`/`A-AST-M-P-01_上下文注入.md`/`A-AST-M-D-01_Binder Agent记忆协同主控文档.md`/`A-AST-M-D-02_Binder Agent知识库协同主控文档.md`/`A-TMP-M-D-01_Binder Agent模板协同主控文档.md`` 的规则统一收口。
4. 当前结构文档应明确承认：workspace 工作台已经不再只是辅助区域，而是 Binder 的工作基础框架；其他能力更多是在其上完成工作分发和结果呈现。

## 九、MVP 验收口径

1. 结构文档层面已经能明确回答“workspace 工作台是什么、由谁主控、与谁协同”。
2. 旧体系中与 workspace 工作台相关的关键规则，在新结构中都有明确承接位。
3. 工作台中的资源入口分区不再停留在旧三分区口径，而是与记忆库、知识库、模板库的新体系一致。
4. 工作台入口不会被误解为 AI 执行引擎或直接写盘入口。
