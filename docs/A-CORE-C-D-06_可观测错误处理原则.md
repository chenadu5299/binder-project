## 一、文档定位

本文定义 Binder MVP 的全局可观测错误处理原则。  
目标：保证错误可识别、可归因、可追踪、可复现，同时不破坏主流程可继续性。

本文是原则文档，不替代具体故障修复方案。

---

## 二、适用范围

适用于以下链路：
1. AI 流式响应（SSE/chunk/tool_call）  
2. 工具调用执行链  
3. 文档编辑与 Diff 执行链  
4. Workspace 与资源加载链  
5. 前端状态更新与事件分发链  

---

## 三、核心原则

## P0 级（不可违背）

### 原则 1：错误必须可观测

任何进入执行链的错误，必须产生至少一种统一观测记录：
- 结构化错误码  
- 结构化日志事件  
- 调试面板事件

禁止仅在控制台打印自由文本后结束。

### 原则 2：错误与状态分离

错误观测事件不等于业务状态。  
示例：
- `ExecutionExposure` 是观测对象  
- `pending/accepted/rejected/expired` 是业务对象

禁止用“是否报错”直接改写 diff 业务状态语义。

### 原则 3：局部失败可跳过，链路不可失控

可恢复错误允许“跳过继续”，但必须满足：
1. 当前条目失败被明确记录  
2. 其余条目继续执行  
3. 整批执行结果可解释

## P1 级（默认必须满足）

### 原则 4：错误分级先行

所有执行错误必须归入统一级别：
- `Retryable`：可重试  
- `Skippable`：可跳过  
- `Fatal`：必须中断

未分级错误视为治理缺口。

### 原则 5：通道统一

所有错误暴露走统一输出通道，不允许“各模块各发一套”：
1. 运行日志通道（可检索）  
2. 调试观测通道（可聚合）  
3. 用户提示通道（仅必要）

### 原则 6：用户提示最小打断

对业务可继续场景：
- 允许静默失效  
- 禁止高频 toast 打断  

对业务不可继续场景：
- 必须明确提示中断原因与下一步动作

## 四、错误分类模型

## 4.1 按阶段分类

1. 输入阶段错误：参数缺失、schema 不匹配、引用结构不合法。  
2. 路由阶段错误：目标文件未解析、门禁未就绪、路径冲突。  
3. 执行阶段错误：tool_call 失败、I/O 失败、定位失败。  
4. 状态阶段错误：状态机非法跃迁、状态源冲突。  
5. 展示阶段错误：渲染失败、重复追加、事件错序。  

## 4.2 按可恢复性分类

1. 可重试：网络波动、短时解析失败、资源暂不可用。  
2. 可跳过：单条 diff 失效、单条 tool_call 数据不完整。  
3. 必中断：目标文件不存在、基线语义失效、协议破坏。  

---

## 五、观测数据最小字段集

每条错误事件至少包含：
1. `error_code`  
2. `error_kind`（Retryable/Skippable/Fatal）  
3. `stage`（input/routing/execution/state/render）  
4. `trace_id` 或 `taskId`  
5. `workspace_path`  
6. `target_file`（如有）  
7. `tool_call_id` / `diff_id`（如有）  
8. `timestamp`

可选扩展：
- `baseline_id`、`revision`、`route_source`、`retry_count`

## 六、执行链行为约束

## 6.1 流式响应链

1. 增量内容必须做去重与错序保护。  
2. 工具调用 arguments 支持跨 chunk 累积。  
3. JSON 未完成时不得提前进入执行。  
4. 解析失败必须记录事件，不得静默吞并。

## 6.2 工具调用链

1. `tool_call` 进入执行前必须参数校验通过。  
2. 执行失败必须带错误分级返回。  
3. `is_complete=false` 的工具调用不得进入最终提交阶段。  

## 6.3 Diff 执行链

1. 单条失败可跳过，不得污染整批排序与执行。  
2. 失效处理与执行失败暴露强制分离。  
3. 批量执行结束后统一汇总结果（applied/skipped/failed）。  

---

## 七、冲突裁决规则

当“继续执行”与“保持正确性”冲突时，按顺序裁决：
1. 协议正确性优先于体验连续性  
2. 状态一致性优先于局部成功率  
3. 可观测性优先于静默吞错  
4. 中断范围最小化优先于全量回滚  

---

## 八、最小排障闭环（MVP）

1. 能定位错误发生在哪个阶段。  
2. 能定位错误影响到哪个对象（task/toolCall/diff/file）。  
3. 能判断本次错误是否可重试/可跳过/必须中断。  
4. 能看到本次错误后的系统行为（继续/中断/降级）。  
5. 能复现同类错误（具备最小上下文字段）。  

---

## 九、反模式（禁止）

1. 把流式重复内容问题只当 UI 展示问题，不记录执行链事件。  
2. 把“用户未看到报错”当作“系统无错误”。  
3. 把业务失效状态当作错误观测结果。  
4. 不做错误分级，统一当 fatal 或统一静默。  
5. 失败后无 trace_id/taskId，导致不可追踪。
