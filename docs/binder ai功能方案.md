# Binder AI 系统架构规范 v1

> **说明**：本文档为原始架构规范。若需查看 **AI 工作流程图、提示词构建流程、去重整合后的方案**，请参阅 [Binder AI 功能方案整合版.md](./Binder%20AI%20功能方案整合版.md)。

## 1. 引言

Binder 被设计为一个 **AI 原生内容生产系统**，其核心对象不是单个文档，而是 **项目**。一个项目可包含多份文档、数据集、图片、研究材料及其他资产，共同构成完整生产成果。

Binder 的典型产出包括：

* 研究报告
* 小说
* 项目文档
* 内容系列
* 结构化知识项目

因此，Binder 中的项目更适合被描述为 **文档集合或项目包**，而非单个文本文件。

Binder 将 AI 深度集成到生产流程中，提供两种主要操作模式：

* **编辑模式**
* **构建模式**

这两种模式服务于不同目的，但基于同一底层项目系统运行。

---

# 2. 核心设计原则

Binder 遵循若干基础设计原则。

### 原则 1 — 项目优先

Binder 以 **项目** 为操作单位，而非单个文档。

```
项目 > 文档 > 块
```

文档是项目的组成部分，块是文档内最小的可编辑单元。

---

### 原则 2 — 双操作模式

Binder 提供两种操作模式：

**编辑模式**

交互式 AI 辅助编辑与微构建。

**构建模式**

自动化多智能体项目构建与大规模生成。

---

### 原则 3 — 两种模式均具备构建能力

Binder 的核心原则是：

**构建能力必须同时存在于两种模式中。**

* 编辑模式提供 **交互式构建**
* 构建模式提供 **自动化构建**

编辑模式必须始终允许用户在写作过程中创建和重组内容。

编辑模式中的构建能力包括：

* 创建章节
* 生成大纲
* 插入表格
* 重组内容
* 生成新段落
* 创建文档

因此，编辑模式充当 **半自动化构建环境**。

---

### 原则 4 — AI 提议，编辑器执行

AI 从不直接修改文档。

而是：

```
AI -> 提议操作
编辑器 -> 执行操作
```

所有 AI 编辑均被转换为 **结构化编辑操作**。

---

### 原则 5 — 工作流从生产中涌现

生产工作流不是固定模板，而是从实际使用中演化而来。

工作流可能来源于：

* 用户定义流程
* AI 对过往构建的总结
* 可复用的生产模式

# 5. 编辑模式

编辑模式是用户进行写作、修订和构建内容的 **主要交互环境**。

它必须同时支持 **编辑** 和 **构建**。

编辑模式是 Binder 中最关键的 AI 交互层。

---

## 5.1 编辑模式的角色

编辑模式支持：

* 实时写作辅助
* 结构编辑
* 增量内容创建
* 局部文档重组

其设计目标是 **深度专注写作与迭代精炼**。

---

## 5.2 编辑模式中的构建

编辑模式必须始终支持 **构建操作**。

包括：

### 内容构建

* 生成段落
* 扩展章节
* 撰写子章节
* 创建列表

### 结构构建

* 生成大纲
* 创建新章节
* 插入标题结构

### 文档构建

* 创建新文档
* 生成草稿文档
* 将文档拆分为章节

### 数据构建

* 创建表格
* 生成数据摘要

因此，编辑模式充当 **半自动化构建系统**。

用户可在编辑模式内完整启动项目。

---

# 12. Binder 编辑协议 (BEP)

## 12.1 目的

Binder 编辑协议定义 AI 在编辑模式中如何与文档交互。

该协议确保：

* AI 编辑是 **确定性的**
* 文档结构是 **受保护的**
* 操作是 **可追溯的**
* 所有变更均可 **撤销**

AI 不直接生成修改后的文档，而是产出 **结构化编辑操作**。

编辑器负责验证和执行这些操作。

核心规则：

```text
AI 从不直接修改文档。
AI 提议操作。
编辑器执行操作。
```

---

# 12.2 编辑操作模型

每个 AI 动作被表示为 **操作对象**。

示例结构：

```
Operation
id
type
target
payload
metadata
timestamp
```

示例：

```
operation_id: op_98342
type: replace_range
target: doc://block/21/range/10-45
payload: "Updated text content"
timestamp: 2026-03-13
```

操作存储在 **操作日志** 中。

---

# 12.3 操作分类

编辑操作分为四类。

## 文本操作

细粒度文本编辑。

示例：

```
replace_range
insert_text
delete_range
```

---

## 块操作

影响块的结构编辑。

示例：

```
create_block
delete_block
move_block
merge_blocks
split_block
```

---

## 结构操作

文档结构构建。

示例：

```
create_section
create_outline
create_table
create_list
create_heading
```

这些是 **构建操作**，在编辑模式中不可或缺。

编辑模式必须支持结构生成。

---

## 文档操作

文档级修改。

示例：

```
create_document
duplicate_document
split_document
merge_document
```

这些允许编辑模式构建新的项目内容。

---

# 12.4 操作验证

执行前，每个操作必须由编辑器验证。

验证规则包括：

```
目标存在性
块有效性
范围边界
文档完整性
```

若验证失败：

```
操作被拒绝
```

AI 可尝试重新生成。

# 13. 项目语义图 (PSG)

## 13.1 目的

项目语义图表示 **Binder 项目的语义结构**。

它使 AI 能够理解以下关系：

* 文档
* 数据集
* 资产
* 章节
* 概念

没有 PSG，AI 会将项目视为无关文件的集合。

PSG 支持 **跨文档推理**。

---

# 13.2 图结构

项目语义图是有向图。

节点表示实体。

边表示关系。

示例结构：

```
项目
 ├ 文档
 │   ├ 章节
 │   │   ├ 块
 │   │   └ 概念
 │   └ 数据集
 └ 资产
```

# 17. AI 工具协议 (ATP)

## 17.1 目的

AI 模型无法直接与 Binder 内部交互。

而是通过 **结构化工具接口** 交互。

AI 工具协议定义模型如何：

* 读取项目信息
* 查询文档
* 检索数据集
* 提议编辑
* 创建结构
* 触发工作流

该协议确保：

```text
可预测的 AI 行为
安全的文档访问
受控的项目修改
一致的工具使用
```

---

# 17.2 工具调用模型

AI 生成结构化工具调用。

示例结构：

```
ToolCall
tool_name
parameters
reason
context_reference
```

示例：

```
tool_name: read_section
parameters:
  document: report.md
  section: market_size
reason:
  update analysis using dataset
```

Binder 运行时验证并执行工具。

insert_table

这些返回 **操作**，而非修改后的文本。

# 18. 上下文压缩引擎 (CCE)

## 18.1 问题

示例项目规模：

```

10000 个块

```

上下文压缩引擎解决这一问题。

```

```

优先级顺序：

```

相邻块

相关数据集
```

目的

关键数据集
```

这些摘要使 AI 能对大型文档进行推理。

---

# 18.5 语义检索

示例：

用户请求：

```
使用新数据集更新市场分析。
```

系统检索：

```

数据集摘要
相关图表
```

仅相关上下文发送给 AI。

# 25. 项目记忆系统 (PMS)

## 25.1 目的

大型项目可能迅速超出 AI 模型的上下文限制。

示例：

```
项目规模
20 份文档
50,000 个块
300,000+ 词
```

没有长期记忆系统，AI 无法在项目范围内保持连贯理解。

**项目记忆系统** 为 AI 提供持久语义记忆。

---

## 25.2 记忆层次

Binder 将项目记忆组织为四层。

```
项目记忆
 ├ 概念记忆
 ├ 结构记忆
 ├ 知识记忆
 └ 工作流记忆
```

---

## 25.3 概念记忆

概念记忆存储项目的 **关键概念与主题**。

示例：

```
Concept
id
title
summary
related_documents
related_sections
```

示例概念条目：

```
Concept
AI 市场增长
summary: 全球 AI 市场扩张趋势
related_sections:
  report.md#market_analysis
  report.md#future_trends
```

这使 AI 能追踪 **跨文档的主题连续性**。

---

## 25.4 结构记忆

结构记忆追踪项目的演化结构。

示例结构树：

```
项目
 ├ 报告
 │   ├ 引言
 │   ├ 市场分析
 │   ├ 技术趋势
 │   └ 未来展望
```

结构记忆使 AI 能对以下内容进行推理：

```
文档层次
章节关系
结构缺口
内容重复
```

---

## 25.5 知识记忆

知识记忆存储从数据集、文档和研究提取的知识。

示例：

```
Knowledge Item
source
summary
related concepts
confidence score
```

示例：

```
Knowledge
AI 市场复合年增长率：32%
source: dataset_market_growth.xlsx
related concept: AI Market Growth
```

---

## 25.6 工作流记忆

工作流记忆存储构建过程中发现的 **生产模式**。

示例工作流记忆：

2 analyze market

4 review content
```

这些工作流可能日后成为可复用模板。

## 25.7 记忆更新流程

```

构建流程执行
```

AI 定期重新生成记忆摘要。

# 27. AI 编辑稳定性框架 (AESF)

## 27.1 问题

AI 编辑系统在长时间编辑会话中往往失效。

示例场景：

```
AI 执行 200 次编辑
文档结构损坏
内容变得不一致
```

Binder 必须保证编辑稳定性。

---

## 27.2 稳定性原则

Binder 执行若干稳定性原则。

```
操作必须可逆
结构必须保持有效
编辑必须沙箱化
状态必须可验证
```

---

## 27.3 操作沙箱

所有 AI 操作首先在 **模拟环境** 中执行。

流程：

```
AI 提议操作
操作在沙箱中执行
结果验证
若有效 → 提交
若无效 → 拒绝
```

这防止文档损坏。

---

## 27.4 结构护栏

结构护栏防止危险编辑。

示例：

```
不能意外删除整个文档
不能破坏章节层次
不能创建无效块结构
```

护栏保护文档模型。

---

## 27.5 版本检查点

Binder 自动创建检查点。

示例：

```
checkpoint
重大编辑后
构建流程后
结构变更后
```

这支持安全回滚。

---

## 27.6 持续验证

每次编辑后，系统验证：

```
文档结构
块完整性
语义一致性
引用链接
```

若验证失败，编辑被撤销。
