# 202_workspace架构

## 一、文档定位

本文定义 Binder MVP 的 Workspace 架构，统一对象模型、核心主链、状态边界和对外契约。  
本文是 Workspace 子系统架构文档，不替代资源管理细节实现文档。

---

## 二、架构目标

1. 提供稳定的本地工作区容器（文件、资源、上下文的一致管理）。  
2. 支撑“文件管理 -> 文档编辑 -> AI 调用 -> 结果生效”的连续主链。  
3. 保证跨文档与未打开文档场景下的可执行与可恢复。  

---

## 三、Workspace 对象模型

## 3.1 核心对象

1. Workspace  
本地项目容器，包含路径、索引、缓存、会话上下文。  

2. WorkspaceFile  
工作区内文件对象，包含 filePath、fileType、mtime、缓存状态。  

3. FileCache  
文件缓存对象，存储 canonical HTML（或等价可编辑快照）与版本信息。  

4. PendingDiffSet  
文件级待确认修改集合，支持按文件与会话追踪。  

5. DependencyLink  
文件间引用/依赖关系（用于上下文注入与影响分析）。  

## 3.2 对象关系

1. 一个 Workspace 包含多个 WorkspaceFile。  
2. 每个 WorkspaceFile 对应一个 FileCache（按版本更新）。  
3. PendingDiffSet 绑定 WorkspaceFile。  
4. DependencyLink 在 Workspace 作用域内建立文件关系图。  

---

## 四、核心主链

## 4.1 打开文件主链

1. 解析目标文件路径。  
2. 优先读取 FileCache（命中则直接返回快照）。  
3. 未命中则读盘并规范化，生成缓存后返回。  
4. 编辑器加载快照并建立 tab 绑定。  

## 4.2 保存文件主链

1. 编辑器提交内容。  
2. 写盘成功后同步更新 FileCache。  
3. 更新 mtime 与版本状态。  
4. 通知依赖链和相关视图刷新。  

## 4.3 未打开文件编辑主链

1. 通过 Workspace 服务加载目标文件快照。  
2. 生成 pending diff（不直接写盘）。  
3. 用户确认后再执行落盘。  
4. 更新缓存与状态。  

## 4.4 外部修改处理主链

1. 监听文件 mtime 变化。  
2. 检测冲突后触发外部修改处理流程。  
3. 选择加载外部内容时，关联 pending diff 进入失效流。  
4. 保持主链可继续，不阻断其他文件操作。  

---

## 五、分层结构（Workspace 子系统内）

1. 接口层（Commands/API）  
对前端暴露 open/save/cache/diff 相关命令。  

2. 领域层（Workspace Domain）  
管理 WorkspaceFile、FileCache、PendingDiffSet、DependencyLink 规则。  

3. 存储层（workspace.db + file system）  
持久化缓存、pending diff 元数据、依赖关系与状态索引。  

---

## 六、状态与一致性约束

1. 同一文件只允许一个当前有效缓存快照。  
2. 写盘成功后必须同步更新缓存，禁止“磁盘新、缓存旧”。  
3. pending diff 不等于已生效内容。  
4. 外部修改与手动修改触发的失效语义保持一致。  
5. 未打开文件场景必须先完成就绪链再进入编辑执行。  

---

## 七、对外契约（与其他子系统）

1. 对编辑器子系统  
提供文件打开、缓存加载、保存回写、外部修改通知。  

2. 对 AI 执行子系统  
提供目标文件快照读取、未打开文件上下文准备、pending diff 容器。  

3. 对对话编辑子系统  
提供按文件维度的 diff 池与确认后落盘能力。  

4. 对可观测子系统  
输出关键事件：缓存命中、缓存回写、冲突处理、批量确认结果。  

---

## 八、MVP 验收口径

1. Workspace 打开/切换稳定，不出现路径错位。  
2. 文件打开与保存后缓存一致。  
3. 未打开文件可进入 pending diff 主链并确认落盘。  
4. 外部修改冲突可处理且状态一致。  
5. 与编辑器/AI/对话编辑三方交互无协议断裂。
